# Процесс: регулярные интервью с пользователями

> **Контекст РП:** [WP-284](../../../../../DS-my-strategy/inbox/WP-284-interviews-pipeline.md) — зонтичный РП «Регулярные интервью → обратная связь → РП».

<details open>
<summary><b>1. Service Clause</b></summary>

**Имя:** «Поток валидированных задач из интервью пользователей».

**Триггер:** проведено интервью с пользователем IWE / клиентом МИМ / наставником / пилотом первой когорты.

**Входы:**
- транскрипт интервью (текст, ≥10 минут материала);
- цель интервью из 5 классов (см. § 4);
- идентификатор сегмента (пилот когорты / подписчик / наставник / внешний эксперт).

**Действие** (что делает процесс):
1. Транскрипт нормализуется и кладётся в `transcripts/` под именем `YYYY-MM-DD-<class>-<initials>.md`.
2. Делается разбор по согласованному списку направлений (см. § 5) → саммари в `analysis/`.
3. Из разбора извлекаются **задачи** (атомарные предложения «что сделать»), каждая попадает строкой в `tasks-registry.md`.
4. Каждая задача маршрутизируется: **новая фаза** существующего РП / **новый РП** / **отказ** (с обоснованием) — по правилам Routing (§ 7).
5. Маршрутизированные задачи попадают в WP-REGISTRY/WeekPlan; реестр обновляется с обратной ссылкой на конкретный РП и фазу.

**Выходы:**
- транскрипт в `transcripts/`;
- сводка интервью в `analysis/<date>-<initials>.md`;
- ≥1 строка в `tasks-registry.md`;
- ≥0 новых строк/фаз в `DS-my-strategy/docs/WP-REGISTRY.md`, `DS-my-strategy/current/WeekPlan W{N}.md`.

**Время отклика:** транскрипт + сводка + первая маршрутизация ≤7 календарных дней от даты интервью. (Метрика lag в § 9.)

**Инвариант:** **ни одна задача не теряется** — для каждого пункта в реестре есть либо ссылка на РП, либо явная пометка «отклонено: <причина>». Реестр — единственный source-of-truth по задачам из интервью.

**Режим отказа** (когда Service Clause не выполняется):
- транскрипт >7 дней без сводки → задача попадает в Day Open следующего рабочего дня;
- задача в реестре без маршрутизации >14 дней → автоматически в `pending-review` для следующего Strategy Session.

<summary><b>2. Сценарии использования</b></summary>

### Сценарий 1. Пилот первой когорты IWE рассказывает о болях

**Кто:** Тсерен (R14, Заказчик IWE) + Менеджер продукта IWE — как со-интервьюеры.
**Когда:** еженедельно, 1–2 интервью с пилотами WP-188.
**Вход:** запрос пилота на встречу или приглашение от Заказчика.
**Что делает с результатом:** боли пилота → новая фаза WP-188, либо новый РП (например, по UX бота); ключевые цитаты → драфты постов (DS-my-strategy/drafts).
**Пример:** боль «ведущий залип на одном участнике 10 мин» → задача «чек-лист наставника по тайм-менеджменту групповой сессии» → маршрут: новая фаза в РП по преподаванию (если есть) или новый РП «Методика ведения резидентуры».

### Сценарий 2. Аудит подписки и продуктов МИМ

**Кто:** Тсерен + Менеджер продукта МИМ + Маркетолог — как со-интервьюеры.
**Когда:** ежемесячно, выборка из ~10 БР (базовых резидентов) + наставников.
**Вход:** длительность подписки, уровень квалификации, история курсов.
**Что делает с результатом:** проблемы продукта → задачи в А2 (Systems-Builder) или WP-188 (когортная программа); красные флаги (попсовость, агрессивные продажи) → JTBD-карты сегментов в `1.1.2. Marketing/` (передача Маркетологу).
**Пример:** «не зависеть от одного LLM-вендора, готовность к VPN-/гос-блокировкам» → задача «universal MCP-agnostic архитектура» → маршрут: новая фаза WP-150 (мультиагентность) или новый РП.

### Сценарий 3. Внешний эксперт-практик тестирует IWE

**Кто:** Тсерен + Менеджер продукта IWE — как со-интервьюеры; эксперт = носитель отдельной школы (TOC, Lean, Agile, Data Governance).
**Когда:** по запросу или по приглашению, 1–2 в месяц.
**Вход:** профиль эксперта + предметный Pack-репо, который он мог бы создать.
**Что делает с результатом:** идеи Pack-репо (предметные паки) → задачи в Pack-программу; интеграционные сценарии (корп. AI-инфраструктура) → задачи в R2 (новая архитектура).
**Пример:** «Pack-репо предметной школы как живая система» → задача «методология построения предметного Pack-репо» → маршрут: новая фаза в WP-202 (template maturity) или Pack-методы.

### Сценарий 4. Наставник делится практикой преподавания

**Кто:** Тсерен + Алексей (методист программ) + Менеджер продукта МИМ — как со-интервьюеры.
**Когда:** после каждого потока резидентур.
**Вход:** журнал занятий + обратная связь участников.
**Что делает с результатом:** улучшения методики → задачи в `2.2.1. Development-Programs/`; шаблоны → методические материалы наставников; повторяющиеся боли участников → передача Маркетологу как сигнал JTBD.

### Сценарий 5. Sales Discovery до подписки

**Кто:** Тсерен / Ильшат (R6 трек передачи) + Маркетолог — как со-интервьюеры; интервью с лидами до конверсии.
**Когда:** по запросам с лендинга, ≥2 в месяц.
**Вход:** предыдущий опыт лида с курсами / AI / системным мышлением.
**Что делает с результатом:** возражения и контр-нарративы → `1.1.3. Sales/`; JTBD → `1.1.2. Marketing/` (Маркетолог обновляет JTBD-карты сегментов).

### Сценарий 6. Пилот программы личного развития (ЛР) — мировоззренческая трансформация

> Отдельный сценарий, **не сводится к Сценарию 1**: фокус — внутренняя трансформация (мировоззренческая дуга, baseline слотов мастерства, характеристики выпускника), а не UX продукта IWE.

**Кто:** Тсерен (R14) + Навигатор R27 (Claude Sonnet sub-agent под фасадом роли) — как со-интервьюеры.
**Когда:** ежемесячно с каждым активным пилотом ЛР (репо `personal-guide` существует и обновляется).
**Вход:** профиль из Память.Derived (rcs_profile, baseline 7 слотов W/M1-M4/IT/A, текущая ступень/фаза дуги PD.FORM.080), репо `personal-guide`, история Day Open/Week Close, журнал ритма слотов.
**Что делает с результатом:**
- мировоззренческие сдвиги (D15) → коррекция дуги в PD.FORM.080 / методбуки в `PACK-personal/methods/`;
- bottleneck по слотам (D16) → задачи в WP-151 (характеристики деятеля), WP-245 (содержание руководства);
- gap по характеристикам выпускника (D17) → задачи в `DS-principles-curriculum/` или новые методбуки;
- слабые места ритуалов и качества `personal-guide` (D18) → задачи в WP-245 / extension'ы протоколов Day Open/Close, Week Close.

**Пример:** пилот застрял между ст. 2 → ст. 3 («Я — система» удерживается, «Окружение влияет на меня» не открывается) → задача «методбук перехода 2→3 с эмфазой на среду сопроизводителей» → маршрут: новая фаза WP-245 или новый РП в PACK-personal.

<summary><b>6. Организация процесса (рабочая процедура)</b></summary>

> Полный чеклист — в `templates/interview-checklist.md` (Ф6 WP-284). Здесь — шаги верхнего уровня.

```
Шаг 1. Запрос на интервью
  - Источник: пилот WP-188 / запрос лида / решение Strategy Session / план R3-R6
  - Триггер: запись в tasks-registry.md (колонка «Интервью запланировано»)
  - Действие: согласовать дату + класс (C1-C7) + интервьюера / ведущего

Шаг 2. Подготовка вопросов
  - Открыть templates/c{N}-*.md для класса
  - Адаптировать под конкретного собеседника (контекст, история, специфика)
  - Записать в transcripts/<date>-<class>-<initials>-questions.md (опционально)

Шаг 3. Проведение
  - Запись (Zoom/Телемост/локально) с авто-расшифровкой
  - Краткие заметки сразу после (≤30 мин) — фиксируют горячее впечатление

Шаг 4. Транскрипция
  - Если автоматическая запись доступна — выгрузить в transcripts/
  - Если только аудио — запустить skill /transcribe <file> (MLX Whisper)
  - Имя файла: YYYY-MM-DD-<class>-<initials>.md

Шаг 5. Разбор (Аналитик)
  - Прочитать транскрипт + заметки
```

<details>
<summary><b>7. Routing (правила маршрутизации задач)</b></summary>

Решение «новая фаза / новый РП / отказ» принимает R14 (Заказчик) по правилам:

| Признак задачи | Маршрут |
|----------------|---------|
| Задача ≤4 часов и явно дополняет существующий РП | **Новая фаза** существующего РП (строка в его context file) |
| Задача ≥4 часов или с собственным lifecycle / собственным артефактом | **Новый РП** (через Ритуал согласования, CLAUDE.md §2) |
| Задача дублирует уже существующую (в реестре или WP-REGISTRY) | **Слияние** — ссылка на исходную строку |
| Задача из «нет, не сейчас» (за пределами текущей стратегии) | **Отказ** с обоснованием (P-приоритет / horizon / сферa) |
| Задача — «вопрос», не «действие» | переоформить как вопрос для следующего Strategy Session |

Каждое решение фиксируется в колонке «Маршрут» реестра. Reject — не удаление: видимо в реестре навсегда, чтобы не возвращаться к одному и тому же.

<details>
<summary><b>8. Транскрипты — конвенции</b></summary>

- **Имя файла:** `YYYY-MM-DD-<class>-<initials>.md` (например `2026-05-02-c2-vv.md`).
- **Frontmatter:**
  ```yaml
  ---
  date: 2026-05-02
  class: C2
  participant: "Виктор Васильевич (initials VV)"
  duration_min: 37
  interviewer: "Tseren"
  source: "Telemost / Zoom / IRL"
  recording: "<path-or-link>"
  status: "transcribed"  # transcribed | analyzed | routed
  related_wp: ["WP-188", "WP-150"]
  ---
  ```
- **Тело:** чистый текст без таймстампов (или с таймстампами в отдельном поле, если нужно для цитат).
- **Аудио** не хранится в Git (большой объём) — ссылка на Drive/локальный путь в `recording`.
- **PII:** транскрипт может содержать имена и контакты участника. Перед публикацией наружу — анонимизация (см. § 11).

</details>

<summary><b>9. Метрики процесса</b></summary>

| Метрика | Цель | Источник |
|---------|------|----------|
| Интервью / месяц | ≥4 (по 1 в неделю) | подсчёт файлов в `transcripts/` |
| Lag интервью → транскрипт | ≤2 дня | `created` файла - `date` |
| Lag транскрипт → сводка | ≤5 дней | `analyzed` - `transcribed` (из frontmatter) |
| Lag сводка → маршрутизация | ≤7 дней | `routed` - `analyzed` |
| Доля задач маршрутизированных | ≥80% за 14 дней | `tasks-registry.md` колонка «Маршрут» |
| Доля implemented (из routed) | ≥40% за 60 дней | сверка с WP-REGISTRY (статус ✅) |
| Активных направлений | 8-12 из 18 | `analysis/` |

Метрики снимаются на Week Close (раздел «User Research» в WeekPlan; реализуется в Ф11).

<summary><b>11. Этика и приватность (БЛОКИРУЮЩЕЕ)</b></summary>

**Базовое правило:** в репо `DS-ecosystem-development` (включая `1.1.4. User-Research/`) **запрещено хранить**:
1. Фамилии участников интервью (имена-отчества, ФИО). Используются только **инициалы** (например, «ВВ», «ИИ», «ГП»).
2. Названия конкретных корпоративных клиентов / работодателей участников. Используются **описательные категории**: «крупная FMCG», «маркетплейс», «завод-производитель», «консалтинг», «корпорация», «IT-вендор».
3. Email-адреса, телефоны, Telegram-хэндлы.

**Исключение:** имена внутренних сотрудников / партнёров команды Аистанта (Тсерен, Андрей, Алексей, Ильшат, Дима, Олег) допустимы — они являются носителями ролей, не PII третьих лиц.

**Процедура** при миграции / создании транскриптов:
- **Перед коммитом** — обязательный grep по whitelist (regex): `(Mondelez|Mondelēz|Wildberries|Ozon|Сбер|Яндекс|Газпром|X5|Магнит|М.Видео)` + `[А-Я][а-я]+ [А-Я][а-я]+(ович|евич|вна|овна)` (отчества) + `\\bООО\\b|\\bАО\\b|\\bПАО\\b`.
- Если grep даёт совпадения → ручной обзор + замена → повторный grep → коммит.

**Список участников** (маппинг `s1..s8 → инициалы + класс + сегмент`) хранится **локально**, **не коммитится в Git**, в файле:
- `~/IWE/DS-my-strategy/inbox/WP-284-participants.md` (private, в `.gitignore` репо `DS-my-strategy`)

**Согласие участника** (этика):
- Устное согласие на запись + согласие на использование цитат (фиксируется в frontmatter транскрипта: `consent: oral|written; quote_use: anonymous|with_initials|with_name`).
- При цитировании наружу (публикации) — повторно спрашивать форму (анонимно / инициалы / имя).

**Корпоративные NDA:** при фиксации компании участника — описательная категория, без названия и без доменной специфики, по которой компанию легко вычислить.

**Аудио / видео** (`.mp4`, `.m4a`, `.wav`) — **не коммитится** в публичные репо. Хранится локально либо в Drive (с приватными правами); путь к источнику — в frontmatter `recording: <path>`.
