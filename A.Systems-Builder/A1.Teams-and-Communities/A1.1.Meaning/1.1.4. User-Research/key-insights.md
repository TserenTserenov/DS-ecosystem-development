# 10 ключевых инсайтов из первого раунда интервью

> Синтез 9 интервью (C1 × 3, C2 × 5, C3 × 1) + групповой сеанс резидентуры. Каждый инсайт — паттерн, замеченный у 2+ респондентов.

---

## И1. IWE не встраивается — он конкурирует

**Что увидели.** Пользователи с рабочей системой (PKM в Obsidian, Notion, AI-workflow, еженедельный AI-обзор) не видят, *что IWE добавляет поверх*, а видят угрозу: «ещё одна система, с которой надо синхронизироваться». Феномен у ЮЮ (три раздельных пространства), ВП (PKM + Code + Whisper), АТ (Notion + MD как рабочий стек).

**Почему важно.** Это не барьер онбординга — это ошибка позиционирования. IWE предлагается как замена, а нужно предлагать как расширение. Пока этого нет, самый продвинутый сегмент аудитории (те, кому IWE нужнее всего) остаётся снаружи.

**Как реализовать.**
- WP-289 (Интеграция IWE с Notion/Obsidian/PKM): шаблоны интеграции, онбординг «у тебя уже есть система — вот как IWE встраивается»
- WP-245 Ф-new: онбординг для продвинутых практиков с устоявшейся системой

---

## И2. Корпоративная среда — отдельный мир

**Что увидели.** Три независимых кейса: ДМ (личный Mac + корпоративный ноутбук с ограниченным GitHub-доступом), Уч-С (корпоративный ноутбук вообще без интернета, 64GB RAM), ИК (PM без IDE, «привык к продуктовым инструментам»), VS (завод в Израиле, корпоративное сопротивление культуры). IWE спроектирован под разработчика с личным Mac и свободным интернетом.

**Почему важно.** Корпоративный рынок — крупнейший нереализованный сегмент. «Разработчик на личном Mac» = потолок текущей аудитории. Без корпоративного режима IWE остаётся нишевым инструментом.

**Как реализовать.**
- WP-286 (IWE в корп. среде): личный Mac + GitHub-мост, офлайн-режим с локальной LLM, B2B-оффер с дашбордом прогресса команды, кейс «IWE-архитектура для маркетплейса»
- WP-287 (Руководство по использованию IWE): путь для PM/менеджеров без IDE, браузерный онбординг через Claude.ai

## И3. Барьер — не цена и не VPN, а язык терминологии

**Что увидели.** ИО: «3 месяца, чтобы привыкнуть к терминологии». ВВ: «я себя называл не дисциплиной, а традицией» — сидел рядом, но говорил на другом языке 5 лет. Это не техническая проблема доступа, а семиотический барьер: специфический язык ШСМ отталкивает раньше, чем пользователь добирается до ценности.

**Почему важно.** Терминологический барьер стоит дороже технического: его не видно в воронке (пользователь уходит молча), и он бьёт по самым ценным потенциальным участникам — экспертам с собственным языком.

**Как реализовать.**
- WP-245 Ф-new: онбординг для «сомневающихся» — ускоренная адаптация языка (3 мес → 3 нед)
- WP-242 (Граф понятий): мультиязычность через граф — перевод терминов, а не подстрочник, при генерации на языке пользователя

## И4. Долгосрочный подписчик ≠ пользователь IWE

**Что увидели.** ВП: 2 года подписчик MIM, активный AI builder, организовал группу совместного изучения СМ — и ни разу не открыл IWE. ЮЮ: «последние обновления не знаю». Статус «подписчик» не коррелирует с использованием ключевого продукта.

**Почему важно.** Это значит, что LTV-модель сломана: люди платят, не получая ценность. При этом — они лояльны («не обременительно»), не уходят, но и не вовлекаются. Конверсия подписчик → пользователь IWE = ключевая метрика роста, которая сейчас не измеряется и не оптимизируется.

**Как реализовать.**
- WP-251 Ф-new: ЦП для сегмента «давний участник, не обновлялся» — что изменилось за N лет в 2-3 фразах
- WP-251 Ф-new: исследование и закрытие разрыва «подписчик MIM → non-adopter IWE»

## И5. Контент для 80%-экспертов

**Что увидели.** АТ (ГД с 15+ годами, строит AI-продукты): «рациональная работа — 80% знаю, скучно». ВВ (консультант-методолог): параллельно ведёт 5 сессий, интегрирует TOC и ТРИЗ как сторонние PAK. ЮЮ: глубоко в теме, хочет «фронтир», не азы. Для этого сегмента линейная программа ЛР воспринимается как движение назад.

**Почему важно.** 80%-эксперты — потенциальные R2 (рекомендатели) и кейсы. Но текущий контент не даёт им ценность. Они приходят «посмотреть» и уходят разочарованными — а уносят с собой мнение.

**Как реализовать.**
- WP-251 Ф-new: ЦП для сегмента «ГД/основатель 15+ лет» — масштабирование через онтологию, не обучение с нуля
- WP-251 Ф-new: формат «обучение по дороге делания» — разбор своих артефактов через системную линзу
- WP-55 Ф-new: материал «границы разумного использования ИИ» как контент для экспертного уровня

## И6. Качество резидентуры зависит от конкретного наставника

**Что увидели.** К (ученица резидентуры): первый наставник Пион — эталон (Сократика, равное вовлечение, включение записи). Второй — полный провал по тем же критериям. g1 (наставник): методика не задокументирована, передаётся устно через демонстрацию. Нет стандарта — нет масштабирования.

**Почему важно.** Резидентура — премиальный продукт. Непредсказуемое качество разрушает репутацию быстрее, чем любой баг. Стандартизация наставника = предусловие масштабирования формата.

**Как реализовать.**
- WP-288 (Рекомендации наставникам резидентур): чеклист наставника, формализация Сократики, стандарт тайминга, систематическая ОС участников, методика «нет отчёта = нет работы»

## И7. Создатели знаний — незакрытый сегмент

**Что увидели.** ВВ: строит PAK-TOS (Theory of Constraints) и PAK-ATB как отдельные базы знаний, пишет книгу, ведёт параллельные сессии в разных модальностях. ИО: строит чекап-продукт для ITSM/ERP-клиентов поверх своего PAK. Оба — не «ученики», а производители контента и методов. Для них IWE = инструмент производства, но путь «PAK → DS → книга/курс» нигде не описан.

**Почему важно.** Создатели знаний производят контент, который привлекает новых пользователей. Поддержать их = получить внешних контент-мультипликаторов бесплатно.

**Как реализовать.**
- WP-149 Ф-new: онбординг для создателей знаний — путь PAK → DS → книга/курс; шаблон PAK для профессионального консультанта
- WP-258 Ф-new: интеграция стороннего PAK (TOC, ТРИЗ) как подключаемого агента через Plugin API L2
- WP-167 Ф-new: кейс «консультант-методолог × IWE × книга» как публикация

## И8. Пользователи не знают, что устаревают

**Что увидели.** ЮЮ: «в фоне беспокойство, что безнадёжно устарел» — но не мониторит активно. АТ: «нельзя всё доверить ИИ по онтологии», при этом не знает где граница. ВП: «через год-два скиллы сгорят» — острое ощущение, но нет системы мониторинга. Пользователи не получают сигнал об устаревании конкретных своих знаний.

**Почему важно.** Это боль, которую платформа может закрыть лучше любого другого инструмента — у неё есть контекст пользователя (его PAK, его практики). «Персонализированный сигнал фронтира» = killer feature, которого нет нигде.

**Как реализовать.**
- WP-117 Ф-new: push-дайджест «что обновилось в практиках» — персонализированный сигнал об устаревании конкретного раздела
- WP-117 Ф-new: формат «сигнал фронтира» — 1-2 абзаца: что устарело, что появилось, как перейти
- WP-55 Ф-new: материал «границы разумного использования ИИ» — учебный контент

## И9. Безопасность данных — реальный блокер для B2B

**Что увидели.** ИО: риск нарушить NDA клиента при загрузке его данных в IWE — нет паттерна анонимизации. ИК: корпоративный ноутбук с ограниченным GitHub-доступом — вопрос безопасности данных. Уч-С: вообще без интернета — вопрос утечки. При работе с корпоративными клиентами это юридический, а не технический вопрос.

**Почему важно.** B2B-продажа блокируется на этапе «а можно ли грузить клиентские данные?». Без ответа на этот вопрос корпоративный рынок закрыт.

**Как реализовать.**
- WP-212 Ф-new (Программа безопасности продуктов): паттерн работы с NDA-данными клиентов в IWE — анонимизация, локальный режим, документированные гарантии
- WP-286 Ф-new: офлайн/локальный режим IWE с локальной LLM для полной изоляции данных

## И10. R2-потенциал есть, конверсии нет

**Что увидели.** ВП уже организовал группу совместного изучения СМ — сильнейший R2-сигнал, не конвертированный. ИК делает внутренний доклад «IWE-архитектура для маркетплейса» в своей компании — уже R2, но без поддержки материалами. Уч-М написал 2 статьи за месяц после 3 месяцев застоя — публичный кейс, не оформленный. Рекомендатели есть, но система не собирает и не усиливает их истории.

**Почему важно.** Реферальный канал — самый дешёвый и доверенный. При этом текущие R2 работают «несмотря на систему», а не благодаря ей. Оформить кейсы, дать материалы, поднять в публичность — бесплатный рост.

**Как реализовать.**
- WP-251 Ф-new: кейс ВП публичным («AI builder × СМ × группа»), кейс Уч-М («автор × IWE»)
- WP-286 Ф-new: кейс ИК «IWE-архитектура для маркетплейса» как B2B-материал
- WP-199 Ф-new: Telegram-марафон как официальный онбординг-инструмент — масштабирование механики Уч-М
