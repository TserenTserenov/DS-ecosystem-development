# Описание хаба активностей

Единая событийная шина экосистемы — центральный журнал всех действий участников. Собирает, агрегирует и распределяет события от всех подсистем: обучение, проекты, публикации, продажи, peer reviews, коммуникации. Обеспечивает полную трассировку активности от действия до результата, реализуя принципы FPF: эволюционность через непрерывный мониторинг, сквозную целостность от события до доказательств, дидактическую прозрачность каждой активности.

## Назначение

Activity Hub — это **нервная система экосистемы**, которая фиксирует, обрабатывает и распределяет информацию обо всех происходящих активностях.

### Ключевые функции

1. **Централизованный сбор событий**
   - Приём событий от всех подсистем экосистемы
   - Нормализация и валидация event payloads
   - Присвоение уникальных идентификаторов (event IDs)
   - Обогащение событий метаданными (timestamps, context)

2. **Event Sourcing (источник истины)**
   - Полная хронология всех действий участников
   - Immutable event log (неизменяемый журнал)
   - Возможность replay (воспроизведение истории)
   - Audit trail для compliance и отладки

3. **Агрегация и аналитика**
   - Real-time агрегация метрик (activity scores, streaks)
   - Построение derived views (проекции событий)
   - OLAP-запросы для аналитики
   - Временные ряды для трендов

4. **Распределение событий (Event Distribution)**
   - Pub/Sub для подписчиков (downstream systems)
   - Webhooks для внешних интеграций
   - Real-time stream processing
   - Batch processing для тяжёлых аналитических задач

5. **Интеграция с Proof-of-Impact**
   - Трассировка Work → Evidence → Metrics
   - Агрегация доказательств для токеномики
   - Связывание активностей с результатами
   - Валидация эффекта во времени

### Роль в архитектуре

Activity Hub — центральная точка данных:
- Все системы **публикуют** события в Hub
- Все системы **подписываются** на релевантные события
- Hub обеспечивает **temporal coupling** (временную связь)
- Hub является **single source of truth** для истории

## Архитектура

### Компоненты

#### 1. Event Ingestion (Приём событий)

**Назначение:** Приём, валидация и первичная обработка входящих событий.

**Функции:**
- **Event Reception** — приём событий через API (REST, gRPC, WebSocket)
- **Schema Validation** — валидация payload по JSON Schema
- **Deduplication** — удаление дубликатов (idempotency)
- **Enrichment** — обогащение метаданными (timestamp, source, user context)
- **Routing** — маршрутизация по типам событий

**Типы событий:**
- **Learning Events** — обучение (course started, lesson completed, test passed)
- **Work Events** — выполнение работ (task started, work completed, artifact created)
- **Social Events** — социальные (post created, comment added, peer review submitted)
- **Economic Events** — экономика (payment received, token earned, token spent)
- **System Events** — системные (user logged in, error occurred, alert triggered)

**Event Schema (базовый):**
```json
{
  "event_id": "uuid",
  "event_type": "string",
  "event_version": "semver",
  "timestamp": "ISO8601",
  "source_system": "string",
  "user_id": "uuid",
  "session_id": "uuid",
  "payload": {
    "// специфичные для типа данные"
  },
  "metadata": {
    "ip_address": "string",
    "user_agent": "string",
    "locale": "string"
  }
}
```

**Интеграции:**
- Все подсистемы экосистемы — публикация событий
- ORY (4.14) — обогащение identity контекстом

#### 2. Event Store (Хранилище событий)

**Назначение:** Долговременное хранение всех событий в immutable log.

**Функции:**
- **Append-only log** — только добавление, никаких изменений
- **Event persistence** — надёжное хранение с репликацией
- **Indexing** — индексация по user_id, timestamp, event_type
- **Partitioning** — шардирование по времени и user
- **Compression** — сжатие старых событий

**Storage layers:**
- **Hot storage** — последние 3 месяца (fast queries)
- **Warm storage** — 3-12 месяцев (slower queries)
- **Cold storage** — >1 год (archival, S3)

**Event ordering:**
- Per-user ordering (строгий порядок для одного участника)
- Global ordering (best effort для всей системы)
- Causal ordering (с учётом причинно-следственных связей)

**Интеграции:**
- Kafka для real-time streaming
- PostgreSQL для structured events
- TimescaleDB для time-series queries
- S3 для архивного хранилища

#### 3. Stream Processor (Обработчик потоков)

**Назначение:** Real-time обработка и агрегация событийных потоков.

**Функции:**
- **Filtering** — фильтрация событий по критериям
- **Transformation** — преобразование форматов
- **Aggregation** — агрегация метрик (counts, sums, averages)
- **Windowing** — временные окна (hourly, daily, weekly)
- **Joins** — соединение событий из разных источников

**Агрегаты:**
- **Activity Score** — общий score активности (за день, неделю, месяц)
- **Streaks** — серии последовательных действий (daily streak, weekly streak)
- **Completion Rate** — доля завершённых активностей
- **Engagement Level** — уровень вовлечённости (active, moderate, low, churned)
- **Velocity** — скорость прогресса (tasks per week, lessons per day)

**Обработка:**
```python
def process_event_stream(event):
    # 1. Фильтрация
    if not is_relevant(event):
        return

    # 2. Агрегация
    update_activity_score(event.user_id, event.type)
    update_streaks(event.user_id, event.timestamp)

    # 3. Генерация derived events
    if milestone_reached(event):
        publish_event(MilestoneReachedEvent(...))

    # 4. Алерты
    if anomaly_detected(event):
        trigger_alert(event.user_id, anomaly_type)
```

**Интеграции:**
- Kafka Streams для обработки
- Redis для in-memory агрегатов
- Digital Twin (4.11) — обновление профиля

#### 4. Event Dispatcher (Диспетчер событий)

**Назначение:** Распределение событий подписчикам.

**Функции:**
- **Subscription Management** — управление подписками
- **Topic-based routing** — маршрутизация по топикам
- **Fan-out** — рассылка события множеству подписчиков
- **Delivery Guarantees** — гарантии доставки (at-least-once, exactly-once)
- **Retry Logic** — повторные попытки при ошибках

**Топики событий:**
- `activity.learning.*` — события обучения
- `activity.work.*` — события работ
- `activity.social.*` — социальные события
- `activity.economic.*` — экономические события
- `activity.system.*` — системные события

**Subscription patterns:**
- **Direct** — прямая подписка на топик
- **Wildcard** — подписка на несколько топиков (`activity.learning.*`)
- **Filter** — подписка с фильтром по payload
- **Dead Letter Queue** — очередь для failed deliveries

**Интеграции:**
- Memory Bank (4.12) — агрегация Work
- Proof-of-Impact (4.17) — трассировка доказательств
- Digital Twin (4.11) — обновление состояния
- Tokenomics (4.10) — начисление токенов
- Guide (4.5) — мониторинг прогресса

#### 5. Query Interface (Интерфейс запросов)

**Назначение:** Предоставление доступа к историческим данным и аналитике.

**Функции:**
- **Event Lookup** — поиск событий по критериям
- **Aggregated Queries** — запросы агрегированных метрик
- **Timeline Queries** — временные ряды (trends, charts)
- **User Activity Summary** — сводка активности участника
- **System-wide Analytics** — аналитика по всей экосистеме

**Query types:**
- **Point queries** — конкретное событие по ID
- **Range queries** — события за период времени
- **Aggregated queries** — сводные метрики
- **Pattern queries** — поиск паттернов (sequences)

**Оптимизация:**
- Materialized views для частых запросов
- Кеширование агрегатов в Redis
- Pre-computed summaries (daily, weekly)
- Indexing стратегии (B-tree, Hash, BRIN)

**Интеграции:**
- Dashboard/UI для визуализации
- Reporting tools для отчётов
- Analytics platform для OLAP

#### 6. Replay Engine (Движок воспроизведения)

**Назначение:** Воспроизведение истории событий для восстановления или анализа.

**Функции:**
- **Full Replay** — полное воспроизведение всей истории
- **Partial Replay** — воспроизведение за период времени
- **Filtered Replay** — воспроизведение определённых типов событий
- **Speed Control** — контроль скорости воспроизведения (fast-forward, slow-motion)

**Use cases:**
- Восстановление состояния системы после сбоя
- Миграция данных в новый формат
- Тестирование новых агрегаций
- Анализ поведения пользователей (debugging)
- Аудит и compliance

**Интеграции:**
- Testing framework для regression tests
- Migration tools для обновлений схем
- Analytics tools для исследований

## Процессы

### П1. Приём и публикация события

**Назначение:** Стандартный flow от генерации события до его сохранения.

**Этапы:**
1. **Event Generation** — подсистема генерирует событие
2. **API Call** — POST /activity/events
3. **Validation** — проверка schema
4. **Enrichment** — добавление метаданных
5. **Persistence** — сохранение в Event Store
6. **Acknowledgment** — подтверждение источнику
7. **Fan-out** — рассылка подписчикам

**Пример кода:**
```python
def publish_event(event_type, user_id, payload):
    event = {
        "event_id": generate_uuid(),
        "event_type": event_type,
        "event_version": "1.0.0",
        "timestamp": now_iso8601(),
        "source_system": calling_system(),
        "user_id": user_id,
        "payload": payload,
        "metadata": enrich_metadata()
    }

    # Валидация
    validate_schema(event, event_type_schema)

    # Сохранение
    event_store.append(event)

    # Публикация в топик
    kafka.publish(f"activity.{event_type}", event)

    return event["event_id"]
```

**Интеграции:**
- Все подсистемы как источники событий
- Kafka как шина

### П2. Real-time агрегация метрик

**Назначение:** Непрерывное обновление агрегированных метрик.

**Этапы:**
1. **Event Arrival** — новое событие в потоке
2. **Window Assignment** — определение временного окна
3. **Aggregation** — обновление счётчиков и сумм
4. **State Update** — обновление состояния агрегата
5. **Derived Event** — генерация derived события (если нужно)
6. **Cache Update** — обновление кеша для быстрого доступа

**Пример агрегации:**
```python
def aggregate_activity_score(event):
    window = hourly_window(event.timestamp)
    key = (event.user_id, window)

    # Обновление счётчика
    activity_counts[key] += 1

    # Обновление score (с весами по типу активности)
    weight = activity_weights[event.event_type]
    activity_scores[key] += weight

    # Проверка достижений
    if activity_scores[key] > threshold:
        publish_event("milestone.reached", event.user_id, {...})
```

**Интеграции:**
- Kafka Streams для обработки
- Redis для кеширования
- Digital Twin (4.11) для обновления

### П3. Построение Work Trace (трассы работы)

**Назначение:** Агрегация связанных событий в единую трассу работы (Work).

**Этапы:**
1. **Event Collection** — сбор событий, связанных с одной работой
2. **Correlation** — связывание по work_id или session_id
3. **Ordering** — упорядочивание событий по времени
4. **Artifact Extraction** — извлечение артефактов (code, documents, etc.)
5. **Evidence Binding** — привязка доказательств (evidence)
6. **Work Submission** — публикация Work в Memory Bank

**Логика:**
```python
def build_work_trace(work_id):
    # 1. Собрать все события по work_id
    events = event_store.query(
        filter={"payload.work_id": work_id},
        order_by="timestamp"
    )

    # 2. Извлечь артефакты
    artifacts = extract_artifacts(events)

    # 3. Извлечь доказательства
    evidence = extract_evidence(events)

    # 4. Сформировать Work
    work = {
        "work_id": work_id,
        "user_id": events[0].user_id,
        "start_time": events[0].timestamp,
        "end_time": events[-1].timestamp,
        "events": events,
        "artifacts": artifacts,
        "evidence": evidence,
        "status": "completed"
    }

    # 5. Отправить в Memory Bank
    memory_bank.store_work(work)

    return work
```

**Интеграции:**
- Memory Bank (4.12) — хранение Work
- Epistemic Graph (4.16) — статусы артефактов
- Proof-of-Impact (4.17) — валидация эффекта

### П4. Мониторинг аномалий

**Назначение:** Обнаружение необычных паттернов активности.

**Аномалии:**
- **Suspicious Patterns** — подозрительные паттерны (спам, gaming)
- **Inactivity** — длительное отсутствие активности (churn risk)
- **Overactivity** — слишком много активности (burnout risk, bot?)
- **Quality Drops** — снижение качества работы
- **Failed Actions** — много неудачных попыток

**Действия:**
- **Alert** — алерт админу/ментору
- **Auto-intervention** — автоматическое вмешательство (throttling, nudge)
- **Investigation** — ручное расследование

**Алгоритмы:**
- Statistical anomaly detection (Z-score, IQR)
- ML-based (Isolation Forest, Autoencoders)
- Rule-based (threshold alerts)

**Интеграции:**
- Digital Twin (4.11) — мониторинг состояния
- Guide (4.5) — adaptive intervention
- CRM (4.7) — escalation

### П5. Архивирование и cleanup

**Назначение:** Перемещение старых событий в холодное хранилище.

**Этапы:**
1. **Identify Old Events** — выбор событий старше retention period
2. **Compress** — сжатие событий
3. **Move to Cold Storage** — перемещение в S3
4. **Update Indexes** — обновление индексов
5. **Delete from Hot Storage** — удаление из горячего хранилища
6. **Verify** — проверка целостности архива

**Retention policy:**
- **Hot (3 месяца)** — полные события, быстрый доступ
- **Warm (12 месяцев)** — полные события, медленный доступ
- **Cold (5 лет)** — сжатые события, archival access
- **Aggregates** — бессрочно

## Хранилище данных

### Схема данных

**Таблицы (PostgreSQL):**

```sql
-- События (горячее хранилище)
CREATE TABLE Event (
  event_id UUID PRIMARY KEY,
  event_type VARCHAR NOT NULL,
  event_version VARCHAR NOT NULL,
  timestamp TIMESTAMP NOT NULL,
  source_system VARCHAR NOT NULL,
  user_id UUID,
  session_id UUID,
  payload JSONB NOT NULL,
  metadata JSONB,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Индексы для быстрого поиска
CREATE INDEX idx_event_user_time ON Event(user_id, timestamp DESC);
CREATE INDEX idx_event_type_time ON Event(event_type, timestamp DESC);
CREATE INDEX idx_event_timestamp ON Event(timestamp DESC);

-- Агрегаты активности
CREATE TABLE ActivityAggregate (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  period VARCHAR,  -- hourly, daily, weekly, monthly
  period_start TIMESTAMP NOT NULL,
  period_end TIMESTAMP NOT NULL,
  event_counts JSONB,  -- counts по типам событий
  activity_score DECIMAL,
  streak_count INT,
  metadata JSONB,
  computed_at TIMESTAMP
);

-- Стрики (серии действий)
CREATE TABLE Streak (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  streak_type VARCHAR,  -- daily, weekly
  start_date DATE NOT NULL,
  end_date DATE,
  current_count INT DEFAULT 1,
  max_count INT DEFAULT 1,
  active BOOLEAN DEFAULT true,
  last_activity TIMESTAMP
);

-- Подписки на события
CREATE TABLE Subscription (
  id UUID PRIMARY KEY,
  subscriber_id VARCHAR NOT NULL,  -- service name or user ID
  topic_pattern VARCHAR NOT NULL,
  filter JSONB,
  delivery_url VARCHAR,
  active BOOLEAN DEFAULT true,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

### Retention policy

- **Hot events:** 3 месяца (PostgreSQL)
- **Warm events:** 12 месяцев (PostgreSQL + compression)
- **Cold events:** 5 лет (S3 + Parquet format)
- **Aggregates:** Бессрочно
- **Derived events:** Same as source events

### Алерты

**Критические:**
- Event ingestion pipeline stopped
- Kafka cluster unavailable
- Database connection lost
- Event loss detected (data loss)
- Storage capacity critical (>90%)

**Предупреждения:**
- High Kafka lag (>10k messages)
- Slow ingestion (>1 sec P95)
- High error rate (>1%)
- Storage capacity warning (>80%)
- Subscription delivery failures (>1%)
- Anomaly detection spike
