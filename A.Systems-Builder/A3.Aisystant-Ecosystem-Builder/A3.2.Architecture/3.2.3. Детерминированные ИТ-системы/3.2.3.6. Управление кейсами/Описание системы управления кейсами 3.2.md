# Описание системы управления кейсами

Подсистема для управления проектами, кейсами, задачами и треккинга прогресса — координационный центр для работы над прикладными инициативами участников. Обеспечивает структурированное ведение проектов от идеи до завершения, с прозрачной трассировкой вклада, распределением ролей и интеграцией с системой вознаграждений. Реализует принципы FPF: эволюционность через итеративное развитие кейсов, сквозную целостность от цели до результата, дидактическую прозрачность каждого этапа работы.

## Назначение

Система управления кейсами — это **операционный хаб экосистемы**, который превращает идеи участников в структурированные проекты с измеримыми результатами.

### Ключевые функции

1. **Управление проектами и кейсами**
   - Создание кейсов (проекты, исследования, разработка агентов)
   - Определение целей, критериев приёмки, вех (milestones)
   - Декомпозиция на задачи и подзадачи
   - Управление зависимостями между задачами
   - Треккинг прогресса по кейсу

2. **Координация команд**
   - Формирование команд для кейсов
   - Распределение ролей (owner, contributor, reviewer, mentor)
   - Управление доступом и разрешениями
   - Коммуникация внутри команды
   - Координация с внешними участниками (клиенты, партнёры)

3. **Треккинг задач и активности**
   - Создание задач с приоритетами и дедлайнами
   - Назначение задач участникам
   - Статусы задач (todo, in_progress, review, done, blocked)
   - Логирование времени и усилий (Pomodoro tracker)
   - Трассировка зависимостей

4. **Интеграция с воронками развития**
   - Кейсы как часть персональных маршрутов (Guide)
   - Связь задач с обучением (LMS)
   - Подтверждение компетенций через выполнение задач
   - Экспорт результатов в Digital Twin
   - Вознаграждения за вклад через Proof-of-Impact

5. **Аналитика и отчётность**
   - Дашборды по кейсам и командам
   - Метрики производительности (velocity, cycle time)
   - Анализ рисков и блокеров
   - Ретроспективы и улучшения процессов
   - Публикация отчётов для заинтересованных лиц

### Роль в архитектуре

Система управления кейсами координирует работу участников:
- Получает задачи от **Personal Path** (индивидуальные маршруты)
- Интегрирует с **Activity Hub** (логирование работы)
- Связана с **CRM** (b2b проекты для клиентов)
- Взаимодействует с **ОС ИИ** (ИИ-ассистенты для управления кейсами)
- Передаёт результаты в **Proof-of-Impact** (вознаграждения за вклад)

## Архитектура

### Компоненты

#### 1. Case Registry (Реестр кейсов)

**Назначение:** Управление жизненным циклом кейсов.

**Функции:**
- **Case CRUD** — создание, чтение, обновление, архивирование кейсов
- **Case Templates** — шаблоны для типовых кейсов
- **Goal Definition** — определение целей и критериев успеха
- **Milestone Management** — управление вехами
- **Acceptance Criteria** — критерии приёмки (Definition of Done)

**Типы кейсов:**
- **Learning Projects** — учебные проекты для отработки навыков
- **Research** — исследовательские инициативы
- **Product Development** — разработка продуктов/агентов
- **Client Projects** — проекты для внешних клиентов (b2b)
- **Internal Ops** — операционные задачи экосистемы
- **Community Initiatives** — инициативы сообщества (events, workshops)

**Жизненный цикл кейса:**
1. **Ideation** — идея, черновой концепт
2. **Proposal** — проработанное предложение с целями
3. **Approved** — одобрено для реализации
4. **In Progress** — активная работа
5. **Review** — на ревью/приёмке
6. **Completed** — завершено успешно
7. **Archived** — архивировано (успешно или нет)

- Digital Twin (4.11) — экспорт опыта

#### 2. Task Management (Управление задачами)

**Назначение:** Разбиение кейсов на задачи и управление их выполнением.

**Функции:**
- **Task CRUD** — создание, чтение, обновление, закрытие задач
- **Task Assignment** — назначение задач участникам
- **Priority Management** — управление приоритетами
- **Dependency Tracking** — трассировка зависимостей между задачами
- **Time Tracking** — логирование времени выполнения (Pomodoro)

**Статусы задач:**
- **Todo** — запланировано, не начато
- **In Progress** — в работе
- **Review** — на ревью
- **Done** — завершено и принято
- **Blocked** — заблокировано (с указанием причины)
- **Cancelled** — отменено

**Приоритеты:**
- **Critical** — критичная задача, блокирует другие
- **High** — высокий приоритет
- **Medium** — средний приоритет
- **Low** — низкий приоритет

**Типы задач:**
- **Permanent** — постоянные задачи (еженедельные сессии стратегирования)
- **Temporary** — временные задачи (обучение на курсе)
- **One-time** — разовые задачи (изучить материал по ссылке)

**Схема задачи:**
```json
{
  "id": "uuid",
  "case_id": "uuid",
  "title": "string",
  "description": "markdown",
  "status": "todo|in_progress|review|done|blocked|cancelled",
  "priority": "critical|high|medium|low",
  "recurrence": "permanent|temporary|one_time",
  "assignee_id": "uuid",
  "reviewer_id": "uuid",
  "estimated_hours": "number",
  "actual_hours": "number",
  "due_date": "date",
  "dependencies": ["task_id"],
  "tags": ["string"],
  "created_at": "timestamp",
  "updated_at": "timestamp",
  "completed_at": "timestamp"
}
```

**Интеграции:**
- Activity Hub (4.6) — логирование работы
- Digital Twin (4.11) — обновление навыков
- ОС ИИ (4.2) — ИИ-ассистенты для задач

#### 3. Team Coordination (Координация команд)

**Назначение:** Управление командами и ролями в кейсах.

**Функции:**
- **Team Formation** — формирование команд для кейсов
- **Role Assignment** — назначение ролей участникам
- **Access Control** — управление доступом к кейсам
- **Collaboration Tools** — инструменты для совместной работы
- **Communication** — коммуникация внутри команды

**Роли в кейсе:**
- **Owner** — владелец кейса, отвечает за результат
- **Contributor** — участник, выполняет задачи
- **Reviewer** — ревьюер, проверяет работу
- **Mentor** — ментор, консультирует команду
- **Stakeholder** — заинтересованное лицо (клиент, партнёр), наблюдает за прогрессом

**Роли участников экосистемы:**
- **Стажёры** — создают личные проекты, получают задачи от преподавателей
- **Преподаватели** — создают обязательные задачи для курсов
- **Сотрудники МИМ** — получают задачи от руководителей служб
- **Руководители проектов** — управляют проектами, назначают задачи
- **Деканат** — создаёт расписание обучения, образовательные проекты

**Разрешения:**
- **Owner**: полный доступ, управление кейсом
- **Contributor**: создание/редактирование задач, комментарии
- **Reviewer**: просмотр, ревью задач
- **Mentor**: просмотр, консультации, ревью
- **Stakeholder**: просмотр, комментарии

**Структура команды:**
```json
{
  "case_id": "uuid",
  "team_members": [
    {
      "user_id": "uuid",
      "role": "owner|contributor|reviewer|mentor|stakeholder",
      "joined_at": "timestamp",
      "contribution_hours": "number"
    }
  ]
}
```

**Интеграции:**
- ORY (4.14) — контроль доступа
- CRM (4.7) — координация с клиентами
- Club (4.3) — привлечение участников из сообщества

#### 4. Progress Tracking (Треккинг прогресса)

**Назначение:** Мониторинг прогресса по кейсам и задачам.

**Функции:**
- **Dashboard** — дашборды для кейсов и команд
- **Metrics Calculation** — расчёт метрик (velocity, cycle time, burndown)
- **Milestone Tracking** — отслеживание вех
- **Risk Detection** — обнаружение рисков и блокеров
- **Progress Reports** — генерация отчётов о прогрессе
- **Time Budget** — бюджет времени на день/неделю

**Ключевые метрики:**
- **Velocity** — скорость выполнения задач (tasks/week)
- **Cycle Time** — время от начала до завершения задачи
- **Lead Time** — время от создания до завершения задачи
- **Burndown** — оставшийся объём работы vs время
- **Completion Rate** — процент завершённых задач
- **Blocker Rate** — процент заблокированных задач
- **Time Invested** — учтённое время выполнения (Pomodoro)

**Индикаторы здоровья кейса:**
- **On Track** — прогресс соответствует плану (зелёный)
- **At Risk** — есть риски отставания (жёлтый)
- **Off Track** — серьёзное отставание (красный)
- **Blocked** — критические блокеры (красный мигающий)

**Интеграции:**
- Activity Hub (4.6) — данные активности
- Posthog — передача данных для аналитики
- BI Tools — визуализация метрик

#### 5. Time Management (Управление временем)

**Назначение:** Планирование и треккинг времени выполнения задач.

**Функции:**
- **Time Budgeting** — составление бюджета времени на день/неделю
- **Pomodoro Tracker** — встроенный Pomodoro таймер
- **Time Logging** — логирование времени выполнения
- **Time Reports** — отчёты по инвестированному времени
- **Leaderboards** — рейтинги по учтённому времени

**Pomodoro Integration:**
- Встроенный таймер для задач (25 min work + 5 min break)
- Автоматический учёт времени
- История Pomodoro-сессий
- Статистика продуктивности

**Time Budget:**
- Плановое время для задач
- Бюджет времени на день/неделю (на сессии стратегирования)
- Сравнение плана vs факта
- Корректировка планов на основе истории

**Leaderboards:**
- Рейтинг по учтённому времени (день, неделя, месяц, год)
- Параллельно с рейтингом активных действий
- Поощрение за систематичность

**Интеграции:**
- Activity Hub (4.6) — логирование времени
- Tokenomics (4.10) — начисление токенов за время

#### 6. Knowledge Capture (Захват знаний)

**Функции:**

**Типы артефактов:**

**Чек-листы:**

#### 7. Notifications & Reminders (Уведомления и напоминания)

**Назначение:** Автоматические напоминания о задачах и работах.

**Функции:**
- **Daily Reminders** — ежедневные уведомления о работах на день
- **Weekly Digest** — еженедельный дайджест предстоящих работ
- **Deadline Alerts** — напоминания о приближающихся дедлайнах
- **Telegram Integration** — отправка сообщений в Telegram
- **In-app Notifications** — уведомления в интерфейсе системы

**Логика напоминаний:**
- Ежедневно утром — напоминание о работах на день
- Раз в неделю — дайджест на неделю (в начале недели)
- За 1 день до дедлайна — предупреждение о близком дедлайне
- Персонализация: время напоминаний настраивается пользователем

**Формат сообщений:**
- Краткое описание работы/задачи
- Ссылка на план работ на день
- Ссылка на задачу в системе

**Интеграции:**
- Telegram Bot — отправка сообщений
- Email — альтернативный канал уведомлений

#### 8. Gamification & Rewards (Геймификация и награды)

**Назначение:** Мотивация участников через геймификацию и вознаграждения.

**Функции:**
- **Achievement Tracking** — отслеживание достижений
- **Badges & Milestones** — бейджи за достижения
- **Leaderboards** — таблицы лидеров
- **Contribution Scoring** — оценка вклада
- **Reward Calculation** — расчёт вознаграждений
- **Points Program** — начисление баллов за активные действия

**Типы достижений:**
- **Task Completion** — завершение задач
- **Case Leadership** — успешное ведение кейса
- **Peer Review** — качественное ревью
- **Mentorship** — менторство участников
- **Innovation** — создание нового решения
- **Consistency** — систематичность выполнения задач

**Scoring модель:**
```python
def calculate_contribution_score(user_id, case_id):
    tasks_completed = count_tasks(user_id, case_id, status='done')
    reviews_done = count_reviews(user_id, case_id)
    mentoring_hours = count_mentoring(user_id, case_id)
    artifacts_published = count_artifacts(user_id, case_id)
    time_invested = get_total_time(user_id, case_id)

    score = (
        tasks_completed * 10 +
        reviews_done * 5 +
        mentoring_hours * 3 +
        artifacts_published * 20 +
        time_invested * 0.5
    )

    return score
```

## Процессы

### П1. Создание и запуск образовательного проекта

**Назначение:** Запуск потока обучения по курсу.

**Роль:** Деканат

**Этапы:**
1. **Create Educational Project** — деканат создаёт проект «Обучение группы по курсу X»
2. **Define Schedule** — составление расписания занятий
3. **Assign Instructor** — назначение преподавателя на курс
4. **Add Mandatory Tasks** — преподаватель добавляет обязательные задачи для стажёров
5. **Enroll Students** — добавление стажёров в проект
6. **Kickoff** — начало обучения, уведомления участникам
7. **Monitoring** — отслеживание прогресса стажёров

**Интеграции:**
- LMS (4.4) — синхронизация курсов
- Club (4.3) — публикация расписания
- Personal Path (4.5) — интеграция с маршрутами стажёров

### П2. Создание личного проекта стажёром

**Назначение:** Стажёр создаёт личный или рабочий проект.

**Роль:** Стажёр

**Этапы:**
1. **Idea** — стажёр формулирует идею проекта
2. **Create Project** — создание проекта (тип: personal или work)
3. **Define Goals** — определение целей и критериев успеха
4. **Add Tasks** — добавление задач с приоритетами и сроками
5. **Set Time Budget** — установка планового времени для задач (на сессии стратегирования)
6. **Start Working** — начало работы над задачами
7. **Track Progress** — использование Pomodoro tracker для учёта времени
8. **Complete Project** — завершение проекта, ретроспектива

**Интеграции:**
- Personal Path (4.5) — проект как часть маршрута
- Digital Twin (4.11) — обновление опыта

### П3. Выполнение задачи с Pomodoro tracker

**Назначение:** Структурированное выполнение задачи с треккингом времени.

**Роль:** Стажёр/Сотрудник

**Этапы:**
1. **Receive Notification** — получение уведомления о задаче на день
2. **Start Task** — стажёр берёт задачу в работу (status: in_progress)
3. **Start Pomodoro** — запуск Pomodoro таймера (25 min)
4. **Work** — выполнение задачи
5. **Break** — перерыв после Pomodoro (5 min)
6. **Log Time** — автоматическое логирование времени
7. **Submit for Review** — отправка на ревью (если требуется)
8. **Complete** — завершение задачи (status: done)
9. **Activity Logging** — логирование в Activity Hub

**Интеграции:**
- Activity Hub (4.6) — логирование работы
- Tokenomics (4.10) — начисление токенов за время

### П4. Руководитель проекта назначает задачи сотрудникам

**Назначение:** Управление задачами команды.

**Роль:** Руководитель проекта (Служба МИМ)

**Этапы:**
1. **Create Project** — создание рабочего проекта
2. **Define Goals** — определение целей и вех
3. **Add Tasks** — добавление задач с описанием и ресурсами
4. **Assign Tasks** — назначение задач сотрудникам
5. **Set Deadlines** — установка дедлайнов
6. **Monitor Progress** — мониторинг прогресса через дашборд
7. **Adjust Plan** — корректировка плана при необходимости
8. **Review Completion** — приёмка завершённых задач

**Интеграции:**
- CRM (4.7) — если проект для клиента
- Activity Hub (4.6) — логирование работы команды

### П5. Сессия стратегирования (планирование недели)

**Назначение:** Еженедельное планирование времени и задач.

**Роль:** Стажёр/Сотрудник

**Этапы:**
1. **Review Last Week** — анализ прошлой недели (что сделано, что нет)
2. **Collect Tasks** — сбор всех задач (постоянные, временные, разовые)
3. **Prioritize** — определение приоритетов на неделю
4. **Estimate Time** — оценка времени для каждой задачи
5. **Create Time Budget** — составление бюджета времени на неделю
6. **Schedule Tasks** — распределение задач по дням недели
7. **Set Reminders** — настройка напоминаний
8. **Commit** — фиксация плана на неделю

**Бюджет времени:**
```python
def create_weekly_budget():
    available_hours = 40  # часов в неделю

    permanent_tasks_hours = 10  # постоянные (сессии, обучение)
    temporary_tasks_hours = 20  # временные (курс)
    one_time_tasks_hours = 10   # разовые (проекты)

    total_planned = permanent_tasks_hours + temporary_tasks_hours + one_time_tasks_hours

    if total_planned > available_hours:
        print("Warning: Overcommitted! Adjust priorities.")

    return {
        'permanent': permanent_tasks_hours,
        'temporary': temporary_tasks_hours,
        'one_time': one_time_tasks_hours,
        'total': total_planned,
        'available': available_hours
    }
```

**Интеграции:**
- Personal Path (4.5) — рекомендации по задачам
- Digital Twin (4.11) — контекст для планирования

### П6. Peer review и приёмка работы

**Назначение:** Контроль качества работы через peer review.

**Роль:** Reviewer

**Этапы:**
1. **Submit** — участник отправляет задачу на ревью
2. **Assign Reviewer** — назначение ревьюера (автоматически или вручную)
3. **Review** — ревьюер проверяет работу по чек-листу
4. **Feedback** — ревьюер оставляет комментарии и рекомендации
5. **Decision** — принятие (approve) или запрос доработок (request changes)
6. **Rework (if needed)** — доработка по замечаниям
7. **Final Approval** — финальное принятие
8. **Reward** — начисление вознаграждения за ревью

**Чек-лист ревью:**
- Соответствие критериям приёмки (Definition of Done)
- Качество работы (код, документ, модель)
- Наличие документации
- Отсутствие явных ошибок/проблем
- Воспроизводимость результата

**Интеграции:**
- Proof-of-Impact (4.17) — вознаграждение за ревью
- Tokenomics (4.10) — начисление токенов
- Epistemic Graph (4.16) — обновление статуса артефакта

### П7. Завершение кейса и ретроспектива

**Назначение:** Формальное завершение кейса с фиксацией уроков.

**Роль:** Команда кейса

**Этапы:**
1. **All Tasks Done** — все задачи завершены
2. **Final Review** — финальная проверка целей и критериев
3. **Retrospective** — ретроспективная встреча команды
4. **Lessons Learned** — фиксация уроков и улучшений
5. **Artifact Publishing** — публикация артефактов кейса (в Club)
6. **Contribution Calculation** — расчёт вклада участников
7. **Rewards Distribution** — распределение вознаграждений
8. **Case Closure** — закрытие кейса (status: completed)
9. **Archive** — архивирование кейса

**Ретроспектива (формат):**
- Что было сделано хорошо? (keep)
- Что можно улучшить? (improve)
- Что нужно начать делать? (start)
- Что нужно перестать делать? (stop)
- Action items для следующих кейсов

**Интеграции:**

## Хранилище данных

### Схема данных

**Таблицы (PostgreSQL):**

```sql
-- Кейсы
CREATE TABLE Case (
  id UUID PRIMARY KEY,
  title VARCHAR NOT NULL,
  description TEXT,
  type VARCHAR,  -- learning, research, product, client, ops, community
  status VARCHAR,  -- ideation, proposal, approved, in_progress, review, completed, archived
  owner_id UUID NOT NULL,
  goals TEXT[],
  acceptance_criteria TEXT[],
  tags VARCHAR[],
  start_date DATE,
  target_end_date DATE,
  actual_end_date DATE,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  metadata JSONB
);

-- Задачи
CREATE TABLE Task (
  id UUID PRIMARY KEY,
  case_id UUID REFERENCES Case(id),
  title VARCHAR NOT NULL,
  description TEXT,
  status VARCHAR,  -- todo, in_progress, review, done, blocked, cancelled
  priority VARCHAR,  -- critical, high, medium, low
  recurrence VARCHAR,  -- permanent, temporary, one_time
  assignee_id UUID,
  reviewer_id UUID,
  estimated_hours DECIMAL,
  actual_hours DECIMAL,
  due_date DATE,
  dependencies UUID[],  -- task IDs
  tags VARCHAR[],
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  completed_at TIMESTAMP
);

-- Команды кейсов
CREATE TABLE CaseTeam (
  id UUID PRIMARY KEY,
  case_id UUID REFERENCES Case(id),
  user_id UUID NOT NULL,
  role VARCHAR,  -- owner, contributor, reviewer, mentor, stakeholder
  joined_at TIMESTAMP,
  contribution_hours DECIMAL DEFAULT 0,
  contribution_score INT DEFAULT 0
);

-- Вехи (milestones)
CREATE TABLE Milestone (
  id UUID PRIMARY KEY,
  case_id UUID REFERENCES Case(id),
  name VARCHAR NOT NULL,
  description TEXT,
  due_date DATE,
  status VARCHAR,  -- pending, in_progress, completed
  completed_at TIMESTAMP,
  created_at TIMESTAMP
);

-- Артефакты
CREATE TABLE Artifact (
  id UUID PRIMARY KEY,
  case_id UUID REFERENCES Case(id),
  task_id UUID REFERENCES Task(id),
  type VARCHAR,  -- code, document, model, presentation, tutorial
  title VARCHAR NOT NULL,
  description TEXT,
  url VARCHAR,
  epistemic_status VARCHAR,  -- draft, peer_checked, accepted, superseded
  created_by UUID,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

-- Комментарии
CREATE TABLE Comment (
  id UUID PRIMARY KEY,
  case_id UUID,
  task_id UUID,
  user_id UUID NOT NULL,
  content TEXT NOT NULL,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

-- Логи времени (Pomodoro)
CREATE TABLE TimeLog (
  id UUID PRIMARY KEY,
  task_id UUID REFERENCES Task(id),
  user_id UUID NOT NULL,
  hours DECIMAL NOT NULL,
  date DATE,
  description TEXT,
  pomodoro_count INT,  -- количество Pomodoro сессий
  created_at TIMESTAMP
);

-- Чек-листы
CREATE TABLE Checklist (
  id UUID PRIMARY KEY,
  task_id UUID REFERENCES Task(id),
  title VARCHAR NOT NULL,
  items JSONB,  -- список элементов чек-листа
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

- **Чек-листы:** Бессрочно

### Алерты

**Критические:**
- Case Management system unavailable
- Database connection lost
- Critical blocker >3 days (no resolution)
- Case deadline missed (high priority)
- Notification service down

**Предупреждения:**
- Case at risk (>20% behind schedule)
- High blocker rate (>15%)
- Low velocity trend (decline >20%)
- Tasks overdue (>5 days)
- Low review rate (<50%)
- Team member overload (>40 hours/week)
- Low Pomodoro usage (<5 sessions/day per active user)
