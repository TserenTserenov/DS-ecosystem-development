# Предложение: 3-слойная архитектура цифрового двойника

> **Статус:** На согласование с архитектором
> **Source-of-truth:** [DP.ARCH.003](https://github.com/TserenTserenov/PACK-digital-platform/blob/main/pack/digital-platform/02-domain-entities/DP.ARCH.003-digital-twin-architecture.md) — полное описание в Pack
> **Контекст:** Возникло при работе над WP-55 (Principles Curriculum) — нужно записывать результаты тестов принципов в ЦД, но в текущей метамодели нет места для этого

---

## 1. Суть предложения

Заменить текущую 4-слойную метамодель ЦД (declarative / collected / derived / generated) на **3-слойную архитектуру Event Sourcing**:

```
Events (неизменяемый лог)  →  State (проекции)  →  Views (генерируемые)
```

**Ключевой принцип: каждое действие пользователя = событие.** Ни одно взаимодействие не проходит мимо.

---

## 2. Проблемы текущей модели

| # | Проблема | Пример |
|---|---------|--------|
| 1 | **Действия теряются.** Слой `2_collected` содержит 5 индикаторов из запланированных десятков. Сырые события (ответы, клики, чаты) не сохраняются | Пользователь ответил на 50 вопросов марафона — что записалось? Ничего, кроме «прошёл шаг» |
| 2 | **Нельзя добавить аналитику ретроспективно.** Без лога событий новая метрика начинает считаться только «с сегодня» | Решили считать P(known) по принципам — исторических данных нет |
| 3 | **Искусственные границы.** Деление «declarative vs collected» не архитектурное — это разный confidence одних и тех же событий | Самооценка ZP.1=4 и тестовый ответ ZP.1=correct — оба обновляют профиль, с разным весом |
| 4 | **Нет модели забывания.** Навык, не повторённый 6 месяцев, показан так же, как вчерашний | P(known)=0.8 вечно, хотя реально 0.3 |
| 5 | **Derived вычисляется из чего?** 36 производных индикаторов, но для большинства нет формулы и нет первичных данных | IND.3.2.04 «Развитие мировоззрения» — checklist по стадиям, без объективного расчёта |

## 3. Предлагаемая архитектура

### 3 слоя

```
┌──────────────────────────────────────────────────┐
│  Events (слой 1)                                  │
│  Append-only лог всех действий из всех систем     │
│  Каждое с: event_type, source, payload,           │
│            confidence, skill_ids, timestamp        │
│  Никогда не удаляется, не изменяется              │
└────────────────────┬─────────────────────────────┘
                     │ on_event() / rebuild()
                     ▼
┌──────────────────────────────────────────────────┐
│  State (слой 2)                                   │
│  N независимых проекций из одного потока:         │
│  ├─ skill_mastery   (BKT → P(known) по навыкам)  │
│  ├─ memory_decay    (HLR → полураспад навыков)    │
│  ├─ engagement      (streak, regularity)          │
│  ├─ misconceptions  (карта ошибок)                │
│  └─ qualifications  (стадия, индексы)             │
│                                                   │
│  Из одного потока можно строить N разных проекций │
└────────────────────┬─────────────────────────────┘
                     │ по запросу
                     ▼
┌──────────────────────────────────────────────────┐
│  Views (слой 3)                                   │
│  Генерируемые, не хранятся:                       │
│  Рекомендации, прогнозы, отчёты, сравнения        │
│  Open Learner Model (профиль для пользователя)    │
└──────────────────────────────────────────────────┘
```

### Почему 3 слоя, а не 4

| Было (4 слоя) | Стало (3 слоя) | Почему |
|---|---|---|
| declarative (кто: пользователь) | Events (confidence=low) | Различие в `confidence`, не в слое |
| collected (кто: система) | Events (confidence=high) | То же событие, другой источник |
| derived (вычисляемое) | State (проекции) | Любой уровень агрегации — это проекция |
| generated (по запросу) | Views | Без изменений |

## 4. Что получаем и откуда

### 4.1 Каталог событий (слой Events)

| event_type | Источник | Пример payload | Confidence |
|---|---|---|---|
| `test_answered` | Бот, LMS | `{questionId, skillId, correct, responseMs}` | 0.9 |
| `homework_submitted` | LMS | `{homeworkId, skillIds[], score}` | 0.9 |
| `marathon_step` | Бот | `{lessonId, stepId, completed, timeSpentMs}` | 0.9 |
| `feed_answered` | Бот | `{questionId, answer, correct}` | 0.9 |
| `ai_chat` | Бот | `{sessionId, message, extractedSkills[]}` | 0.5 |
| `reflection` | Бот | `{text, extractedInsights[]}` | 0.5 |
| `self_assessed` | Бот | `{skillId, selfRating}` | 0.3 |
| `goal_set` | Бот | `{goal, targetSkill, targetDepth}` | 1.0 |
| `session_start` | Бот | `{platform}` | 1.0 |
| `note_created` | Бот | `{text, tags[]}` | 0.7 |
| `strategy_session` | Стратег | `{weekPlan, rpCount}` | 1.0 |

### 4.2 Проекции и методы (слой State)

| Проекция | Метод | Входные события | Что считает |
|---|---|---|---|
| **Skill Mastery** | Bayesian Knowledge Tracing (BKT) | test_answered, homework, feed | P(known) по каждому навыку: 0.0–1.0 → Bloom depth 0–5 |
| **Memory Decay** | Half-Life Regression (HLR, Duolingo) | Те же | Полураспад навыка, `next_review_date` |
| **Engagement** | Скользящие агрегаты | Все события | Streak, regularity, avg session, preferred hour |
| **Misconceptions** | Паттерн-матчинг + LLM (Haiku, async) | test_answered (wrong), ai_chat | Карта конкретных заблуждений по навыкам |
| **Qualifications** | Threshold rules (поверх других проекций) | Из skill_mastery + engagement | Стадия (Random→Proactive), индексы |

### 4.3 Три типа характеристик по вычислимости

| Тип | Описание | Confidence | Доля характеристик |
|---|---|---|---|
| **A. Прямо вычислимые** | Формула из событий | Высокий | ~40% (streak, часы, P(known), recovery) |
| **B. Косвенно выводимые** | Proxy-сигналы | Средний | ~30% (эмоц. стабильность, misconceptions) |
| **C. Только декларация** | Пользователь сообщает сам | Низкий | ~30% (мемы, калибр личности, окружение) |

Для каждой характеристики метамодели нужна **карточка метода**: из каких событий, какой формулой, с какой достоверностью.

## 5. Как это участвует в работе платформы

### 5.1 Замыкание цикла (Intervention Loop)

```
Пользователь отвечает на вопрос
      ↓
Event: test_answered {skill: ZP.3, correct: false}
      ↓
State: P(known ZP.3) обновляется: 0.23 → 0.19
      ↓
View: Диагност видит bottleneck → рекомендует ZP.3
      ↓
Бот предлагает материал по ZP.3 (из curriculum)
      ↓
Пользователь изучает → отвечает → Event → ...
```

### 5.2 Использование по системам

| Система | Что читает из State | Что делает |
|---------|-------|-----------|
| **Бот: Консультация** | skill_mastery + misconceptions | Адаптирует глубину объяснения |
| **Бот: Марафон** | engagement + skill_mastery | Подбирает следующий урок |
| **Бот: Лента** | memory_decay (P(recall) < 0.5) | Spaced repetition вопросы |
| **Бот: /mydata** | Все проекции | Open Learner Model |
| **Бот: Тест принципов** | skill_mastery | Адаптивный тест (начинает с P≈0.5) |
| **Стратег** | qualifications + engagement | Учёт стадии при планировании |
| **Проводник** | skill_mastery + misconceptions | Адаптация промпта |

### 5.3 Для пользователя (Open Learner Model)

Пользователь видит свой профиль:

```
Профиль принципов:
  ZP.1 Аксиоматичность: ████████░░ 78% (Применение)
  ZP.2 Структура:       ████░░░░░░ 42% (Понимание)
  ZP.3 Многомасштабность:██░░░░░░░░ 23% (Запоминание) ← bottleneck

Рекомендация: ZP.3 — ограничивает ZP.4 и FPF.Holon
Навыки для повторения: ZP.1 (через 3 дня без повторения)
Streak: 12 дней | Регулярность: 5/7 | Сессия: ~25 мин
```

## 8. План внедрения от текущих ИТ-систем

### Текущее состояние

| Система | Статус | Роль в новой архитектуре |
|---------|--------|------------------------|
| AIST Bot (Railway + Neon) | Production | Основной источник событий |
| LMS (Aisystant) | Внешняя | Источник events: homework, courses |
| MCP digital-twin (CF Workers) | Production | API для чтения/записи State |
| MCP knowledge-mcp (CF Workers) | Production | Без изменений |
| Neon DB | Production | Хранилище Events + State |

### Фазы

#### Фаза 0: Event Store (1 день)

- Таблица `user_events` в Neon (append-only)
- Функция `log_event()` в боте
- P0-точки: session_start, ai_chat, marathon_step
- Простая проекция engagement (SQL)

**Результат:** каждое действие в боте записывается. Через неделю — данные для анализа.

#### Фаза 1: Skill Mastery (1 неделя, зависит от WP-55)

- Ячейки curriculum со skill_ids (WP-55)
- Тестовые вопросы по навыкам
- BKT-модуль: P(known) при каждом ответе
- Проекция skill_mastery в Neon

**Результат:** бот знает P(known) по принципам для каждого пользователя.

#### Фаза 2: Memory + Recommendations (2 недели)

- HLR: half-life, next_review_date
- Spaced repetition в ленте
- Open Learner Model в /mydata
- Bottleneck-рекомендация

#### Фаза 3: LLM Extraction + Full Profile (позже)

- Async Haiku extraction из чатов
- Misconception map
- Карточки методов для всех характеристик (A, B, C)
- Qualifications проекция

#### Фаза 4: LMS Integration (когда будет API)

- Webhook для приёма событий из LMS
- Маппинг курсов на skill_ids

## 9. Вопросы для архитектора

1. **Event Store в Neon vs отдельный EventStoreDB/Kafka?** При текущем масштабе (<1K пользователей) Neon достаточен. Переход на Kafka — при >10K concurrent.

2. **Совместимость с DS-twin MCP.** Предлагается расширить API (log_event, get_skill_profile), а не заменять. Миграция данных — через replay. Нужно согласовать формат.

3. **Авторизация событий.** Бот пишет события по chat_id. LMS — по user_id через ORY. Нужен единый маппинг.

4. **Retention policy.** Хранить события навсегда? Или снапшотить + агрегировать старые (>1 год)?

5. **Порядок:** Фаза 0 (event store в боте) — можно начинать немедленно без изменений в DS-twin?
