# WP-215: Концепция разделения инфраструктуры Россия и мир

> **Источник:** ИТ-встреча 5 (9 апр). Принципиальное решение: полное разделение финансов по юрлицам, дублирование технической инфраструктуры, общее сообщество без границ.

---

## 1. Принятые решения (9 апр)

1. **Финансы полностью разделены** по юрлицам: российское юрлицо принимает оплаты в России, международное — за рубежом
2. **CRM = отдельный сервис** (не часть LMS), пишет в единую базу Neon
3. **Neon пока одна база** за границей; мониторить канал RU→Neon, при проблемах — прокси/туннель
4. **Сообщество = общее**, без границ государств (баллы, ЦД, контент)
5. Инфраструктура за границей — обсудить с Андреем в воскресенье 13 апр

## 2. Финансовые и юридические вопросы

### Два юрлица

| Аспект | Российское юрлицо | Международное юрлицо |
|--------|-------------------|---------------------|
| **Платёжные системы** | ЮKassa, монета, карта «Мир» | Stripe, PayPal, международные карты |
| **Пользователи** | Резиденты РФ, оплата в рублях | Все остальные, оплата в USD/EUR |
| **Данные** | Хранение PII россиян в РФ (152-ФЗ) | GDPR, данные в EU |
| **Налоги** | Российское налогообложение | По юрисдикции международной компании |
| **Инвесторы** | — | Не должно быть карты «Мир» и российских атрибутов на сайте |

## 3. Технические вопросы: разделённая инфраструктура

### Архитектура двух контуров

```
┌─────────────────────────┐     ┌─────────────────────────┐
│   КОНТУР «РОССИЯ»       │     │   КОНТУР «МИР»          │
│                         │     │                         │
│ Бот (@aist_ru)          │     │ Бот (@aist_me)          │
│ Ory (RU self-hosted)    │     │ Ory Cloud (EU)          │
│ CRM-сервис (RU)         │     │ CRM-сервис (EU/cloud)   │
│ Neon RU / self-hosted PG│     │ Neon EU                 │
│ Сервер в РФ             │     │ Railway / облако         │
└────────────┬────────────┘     └────────────┬────────────┘
             │                               │
             └───────────┬───────────────────┘
                         │
              ┌──────────▼──────────┐
              │  ОБЩИЙ СЛОЙ         │
              │                     │
              │ Сообщество (баллы)  │
              │ Цифровой двойник    │
              │ Knowledge base      │
              │ Контент (курсы)     │
              └─────────────────────┘
```

### Что дублируется (по контурам)

| Компонент | Россия | Мир | Примечание |
|-----------|--------|-----|------------|
| **Бот** | Свой экземпляр, конфиг → RU backend'ы | Свой экземпляр, конфиг → EU backend'ы | Один код, разные env |
| **Ory** | Self-hosted в РФ | Ory Cloud (EU) | Разные identity stores |
| **CRM-сервис** | Экземпляр в РФ | Экземпляр за границей | Пишут в разные/одну базу — решить |
| **Платёжный шлюз** | ЮKassa, монета | Stripe | Полностью раздельные |
| **Neon / PG** | Self-hosted PG в РФ | Neon EU | Вопрос: общая или раздельные? |

### Что остаётся общим

| Компонент | Почему общее | Как реализовать |
|-----------|-------------|-----------------|
| **Сообщество** | Границы государств не должны делить людей | Единая система баллов, один user graph |
| **Баллы** | Общий учёт вклада в сообщество | Единый points engine, агрегация из обоих контуров |
| **Цифровой двойник** | Один человек = один ЦД, независимо от страны | Git-репо пользователя — где хостить? |
| **Knowledge base** | Знания не имеют гражданства | Общие эмбеддинги, общий RAG |
| **Контент** | Курсы, материалы — общие | Один источник контента |

## 4. Ключевые противоречия для проработки

### 4.1. Цифровой двойник

Человек переезжает → его ЦД должен остаться с ним. Но:
- Если ЦД хранится в РФ → доступ может отвалиться из-за границы
- Если ЦД хранится за рубежом → нарушение 152-ФЗ для россиян
- **Вариант:** Git-репо как portable контейнер. Экспорт/импорт при смене контура

### 4.2. Биллинг и баллы

- Финансы раздельные, но баллы — общие
- Баллы начисляются за оплату → как считать баллы из разных юрлиц?
- **Вариант:** баллы = нефинансовая метрика (вклад в сообщество), не привязаны к конкретному юрлицу

### 4.3. Единый аккаунт vs два аккаунта

- Один Ory в РФ, другой Ory в EU → два аккаунта?
- Или один аккаунт (по ory_id) с привязкой к обоим контурам?
- **Вариант:** федерация identity — один ory_id, но данные хранятся в юрисдикции пользователя

### 4.4. Связность RU ↔ мир

Из транскрипта: доступ из РФ к зарубежным сервисам нестабилен (блокировки, замедления). Решения:
- Туннель/прокси между серверами (Паша)
- Между дата-центрами фильтрации обычно нет
- Но нужен мониторинг и байпас на случай проблем

## 5. Принцип общего сообщества

> «Мне нету такого, что разделение границ, какая разница, где он родился, какой у него паспорт. Я не хочу разделять так.» — из транскрипта встречи

**Принцип:** финансы и юридика — разделены (вынуждены). Сообщество и знания — общие (наш выбор).

**Что это значит практически:**
1. **Один форум / один чат** — участники из всех стран
2. **Общая система баллов** — вклад считается независимо от юрлица оплаты
3. **Общий рейтинг и прогресс** — ЦД показывает полную картину
4. **Общий контент** — курсы, материалы доступны всем (доступ определяется подпиской, не географией)
5. **Общие команды (Б5)** — участники из разных стран в одной команде

**Аналогия:** как open-source проект. Код общий, контрибьюторы из любой страны. Но компания, которая продаёт support — может быть в разных юрисдикциях.

### 6.1. Ф0.5 — Исследование: разделение инфраструктуры, сообщества и финансов

> Перед проработкой концепции (Ф1) и юридической консультацией (Ф3) — собрать опыт компаний, решавших аналогичные задачи.

**Цель:** Понять паттерны разделения для платформ, работающих одновременно в России и в мире, чтобы не изобретать велосипед.

#### A. Опыт Telegram (приоритет)

Telegram — ближайший аналог по проблематике:
- **Юрисдикция:** зарегистрирован в ОАЭ (Dubai), ранее в Британии и Сингапуре. Не в России и не в ЕС
- **Финансы:** TON/Telegram Stars — своя платёжная система, обходящая привязку к конкретной юрисдикции. Telegram Premium — оплата через App Store / Google Play (ещё один уровень абстракции от юрисдикции)
- **Данные:** хранение распределённое по странам, ключи шифрования — в другой юрисдикции от серверов. 152-ФЗ — формально не исполняет (заблокирован в 2018, разблокирован в 2020 без выполнения требований)
- **Сообщество:** единое, без разделения по странам. Каналы, группы — глобальные
- **Вопросы для исследования:**
  - [ ] Как Telegram решает конфликт между 152-ФЗ и GDPR?
  - [ ] Структура юрлиц: Telegram FZ-LLC (ОАЭ), Telegram Group Inc, TON Foundation — как разделены?
  - [ ] Как работают платежи Stars/Premium с точки зрения юрисдикций?
  - [ ] Последствия блокировки 2018-2020: какие технические решения помогли?

#### B. Другие релевантные кейсы

| Компания | Что исследовать | Почему релевантно |
|----------|----------------|-------------------|
| **Notion** | Как работает в РФ без локализации данных | SaaS с PII, не локализует, но доступен |
| **GitLab** | Self-hosted vs SaaS, федерация инстансов | Аналогичная задача двух контуров |
| **Miro** | Два юрлица (RU + US), единый продукт | Точный аналог нашей ситуации |
| **JetBrains** | Чехия + глобальный, работает в РФ через отдельное юрлицо | Разделение юрлиц при едином продукте |
| **Wise (TransferWise)** | Мультиюрисдикционные финансы | Разные юрлица в разных странах, единый UX |

#### C. Юридические паттерны

- [ ] **Модель «зонтичной» компании:** головная → дочерние в юрисдикциях. Revenue sharing между ними
- [ ] **Модель «франшизы»:** независимые юрлица, лицензионное соглашение на технологию
- [ ] **Модель open-source фонда:** сообщество = Foundation (некоммерческое), коммерция = отдельные компании (Red Hat / SUSE модель)
- [ ] **Трансграничная передача данных:** стандартные договорные условия (SCC), адекватность юрисдикции, binding corporate rules

#### D. Технические паттерны разделения

- [ ] **Federation:** два инстанса, единый протокол обмена (как Matrix/ActivityPub)
- [ ] **Multi-tenant с geo-routing:** один код, данные маршрутизируются по юрисдикции пользователя
- [ ] **Data mesh:** домены владеют данными, контракты на обмен между контурами
- [ ] **Portable identity:** один аккаунт, миграция данных при смене контура (как номер телефона при смене оператора)

#### Результат Ф0.5

Документ-исследование (~3-5 стр.) с:
1. Таблица паттернов: что работает, что нет, применимость к нашему случаю
2. Рекомендация по модели юрлиц
3. Рекомендация по технической архитектуре разделения
4. Список рисков и митигаций

## 7. Контекст из сессии (для проработки)

### 7.1. Позиции участников встречи (из транскрипта)

**Церен (владелец):**
- Финансы — разделить полностью, тут нет сомнений
- Технически — тоже согласен разделить, если можно сделать достаточно быстро
- Сообщество — категорически против разделения. Баллы, ЦД, контент = общее
- Готов сам проработать концепцию, потом прийти к архитектору для обсуждения
- Два бота (RU и мир) — «хрен с ним, можно будет сделать»

**Андрей (архитектор):**
- Предлагает полное разделение: «по-другому как будто бы не сильно можно»
- Сервис платежей = обязательно отдельный, не часть LMS
- Один код бота, два экземпляра с разными конфигами (RU backend'ы vs EU backend'ы)
- Пользователь из РФ в международном боте → отказать, направить к RU-боту
- Его платформа (сервер в РФ) = для AI-агентов. Activity Hub и Langfuse — не AI-агенты, нужно другое решение
- Предостерегает: делать автоматизацию на старой LMS = потерянная работа

**Ильшат (операционный менеджер):**
- Дима начинает CRM-сервис как приоритет на 2 недели
- Марафон Алёны (приём оплат) — отодвинуть, не делать на старой схеме
- Доступ из регионов РФ к зарубежным серверам нестабилен (не только Москва)
- Канал RU→Neon нужно мониторить, иметь байпас

**Олег Царёв (внешний консультант, легал):**
- Рекомендует «бросить Россию и делать на мир»
- Карта «Мир» на сайте = отпугивает зарубежных инвесторов
- Церен не согласен с этой позицией: «мы за объединение»

### 7.2. Ключевые цитаты из транскрипта

> «Коммерческие штуки вынести отдельно, компаний может быть несколько в разных юрисдикциях, а комьюнити и результаты работы этого комьюнити — отдельная от коммерции штука, и кто угодно может в ней принять участие»

> «Мы строим новое для международных пользователей. А на Россию потом делаем копию»

> «Финансы разделить полностью. А третье — сообщество — я всё-таки против того, чтобы полностью разделять»

> «Делаем отдельный сервис [CRM]. Давай на 2 недели. Церен и Дима как разработчики, Ильшат — операционная деятельность»
