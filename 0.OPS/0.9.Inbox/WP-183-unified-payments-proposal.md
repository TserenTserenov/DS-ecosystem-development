### Архитектурные решения

> **Онтология:** Payment Registry (SYS.011) -- это **система** (подсистема платформы), которая принимает, записывает и отдаёт данные об оплатах. **Реестр оплат** -- это **документ** (набор записей в таблицах PostgreSQL), который Payment Registry ведёт. Система создаёт и обновляет документ; потребители (Directus, Metabase, бот Aist) читают данные из документа через систему.

| # | Решение | Обоснование |
|---|---------|-------------|
| Р1 | **Выделяем подсистему Payment Registry (SYS.011)** из Aisystant. Payment Registry -- система учёта всех оплат. Payment Registry ведёт единый документ **Реестр оплат** как набор записей (таблицы) -- источник правды: кто, когда, сколько, через какой канал заплатил. Реестр оплат сейчас реализован в виде таблиц PostgreSQL Aisystant, где уже хранятся данные об оплатах (каналы 1-5). Все каналы оплаты регистрируют транзакции через Payment Registry в Реестре оплат. Бот Aist (@aist_me_bot) и Directus -- потребители данных из Реестра оплат | Один источник, не два. Бот Aist не дублирует данные. Выделение подсистемы -- первый шаг к отдельному Billing Service |
| Р2 | **Две системы: Payment Registry (SYS.011) и Billing Service (SYS.010).** Payment Registry = система учёта транзакций (уже существует де-факто, выделяем). Billing Service = новая система с бизнес-логикой (тарифы, подписки, баллы, revenue sharing, юнит-экономика). На Фазе B Payment Registry мигрирует в Billing Service как модуль | Разделение факта оплаты и бизнес-логики. Позволяет строить Billing Service поэтапно, не ломая текущий учёт |
| Р3 | **Webhook: Payment Registry -> бот Aist** (не сайт -> бот Aist напрямую). Оплата на сайте МИМ или через витрину Tilda/Ecwid регистрируется в Payment Registry, затем Payment Registry шлёт webhook боту Aist | Один канал, один контракт, одна точка интеграции |
| Р4 | **Привязка tg_id <-> email <-> aisystant_id -- через Ory (Фаза B).** Каждая система знает только свою связку с Ory (звезда), не между собой (mesh). Временный костыль до Ory не делаем | Правильная архитектура identity. Mesh = O(n^2) интеграций |
| Р5 | **Два варианта допуска в чат:** (A) бот Aist генерирует одноразовую ссылку (`createChatInviteLink`, `member_limit=1`), (B) админ создаёт одноразовую ссылку через интерфейс Telegram. Бот Aist узнаёт о вступлении в обоих случаях (`chat_member` update, поле `invite_link.creator`) | Вариант A предпочтительнее (автоматический учёт). Вариант B -- запасной (ручной допуск админом, бот Aist помечает `source: admin_invite`) |
| Р6 | **Бот МИМ (@SystemsSchool_bot) тоже принимает оплаты.** Оба бота (Aist и МИМ) -- каналы приёма оплат. Оплаты через бот МИМ регистрируются в Payment Registry (система) так же, как оплаты через бот Aist. Payment Registry шлёт webhook обоим ботам при необходимости | Бот МИМ обслуживает существующую аудиторию. Отказ от приёма оплат через него = потеря канала |

### Решения по миграции оплат (7 апр, встреча 5 — команда)

| # | Решение | Обоснование |
|---|---------|-------------|
| М1 | **Q-consolidation: гибридная схема (dual-write + read).** Каналы 1-5: Aisystant пишет в свою PG как сейчас + дублирует каждый платёж/списание в Neon. Каналы 6-8: пишут сразу в Neon (как сейчас). Aisystant получает **read-only доступ к Neon** для проверки оплат (баны, подписки). Данные из Aisystant НЕ удаляются до полной миграции | Дима: «неохота дублировать [в обратном направлении]. Организовать доступ к Neon на чтение для Aisystant. На первом этапе будет проверять: есть ли платёж в текущей базе ИЛИ в Neon» |
| М2 | **Монета — без автоинтеграции пока.** Оплаты вносятся вручную (Гиляна) через **Directus → Neon**. Церен разворачивает Directus + форму ручного ввода. Интеграция с Монетой — отдельная задача | Дима: «Монеты здесь нет. Всё вручную.» Ильшат: интеграцию обсудим следующим шагом |
| М3 | **Биллинг-логика остаётся в Aisystant.** Управление подписками, добавление в стажировки, продление — существующие модули поверх `payment` + `charge_off`. Не переносить сейчас. Церен учитывает это при проектировании схемы Neon (совместимость) | Дима: «В Aisystant есть модуль поверх — после платежа продлевает подписку, добавляет в группу. Он работает, пока не трогаем» |
| М4 | **`charge_off` — основа для принятия решений**, не `payment`. `payment` содержит ВСЕ платежи включая неуспешные. `charge_off` = только успешные списания. Церен реплицирует обе таблицы в Neon, бот Aist и Directus используют `charge_off` для проверки доступов | Дима: «charge_off добавляются только если платёж был успешно проведён. Принимать решение для биллинга нужно на основе списаний» |
| М5 | **Neon API для всех подсистем.** Церен делает API (или прямой PG-доступ), чтобы Aisystant и другие подсистемы могли читать оплаты из Neon | Дима: «Если доступ от сервера бэка до Neon физически есть, пользоваться тем, что есть в Neon, я не вижу проблем» |
| М6 | **Индивидуальные ссылки на оплату с сайта.** API для YooKassa уже есть в Aisystant (подписки, стажировки). Церен интегрирует в новую схему для семинаров на сайте. Для Монеты — добавлять позже | Дима: «API на бэке есть, для трёх платёжных систем уже работает. Фронт делает запрос → бэк взаимодействует с ЮКассой → возвращает индивидуальную ссылку» |
| М7 | **Церен готовит схему в Neon, Дима валидирует.** Дима подключается к Neon через средства разработки, проверяет поля, пишет замечания | Дима: «Как только готова схема в Neon и будет доступ, смогу подключиться и валидировать» |
| М8 | **Деплой Aisystant — вечером.** Монолит, обновлять лучше когда меньше пользователей | Дима: «Не мгновенно, в пределах дня, лучше поздно обновлять» |

> **Ключевое изменение:** Q-consolidation решён НЕ как вариант A (полная миграция) и НЕ как вариант B (два хранилища + Directus). Решение — **гибрид**: Aisystant дублирует в Neon (write), Aisystant читает из Neon (read). Neon постепенно становится единым источником.

### Переходный период: гибридная схема dual-write

> Три состояния системы: AS-IS → ПЕРЕХОДНЫЙ ПЕРИОД → TO-BE.
> Переходный период длится до полного перехода на Neon (Фаза B+). В это время **оба хранилища активны**, но с чёткими ролями.

**AS-IS (сейчас):**
```
Каналы 1-5 → Aisystant PG (единственное хранилище)
Каналы 6-7 → Neon (единственное хранилище)
Канал 8    → нигде (вручную)

Aisystant НЕ видит каналы 6-8.
Бот Aist НЕ видит каналы 1-5 (кроме API подписок с cache 5мин).
```

**ПЕРЕХОДНЫЙ ПЕРИОД (Фаза A):**
```
Каналы 1-5 → Aisystant PG (как сейчас) → dual-write → Neon
Каналы 6-7 → Neon (как сейчас)
Канал 8    → Directus → Neon

Aisystant PG:
  - ПИШЕТ: каналы 1-5 (без изменений, как сейчас)
  - ДУБЛИРУЕТ: каждый save(payment) + save(charge_off) → Neon
  - ЧИТАЕТ: свою PG + Neon (для банов, подписок, доступов)
  - НЕ получает данные из Neon (без обратного дублирования)

Neon:
  - ПИШЕТ: каналы 6-8 (бот + Directus)
  - ПОЛУЧАЕТ: дубликаты каналов 1-5 от Aisystant
  - = единый Реестр оплат (все 8 каналов)

Биллинг-логика Aisystant (подписки, стажировки, продление):
  - Работает на СВОИХ таблицах (Aisystant PG) — без изменений
  - Дополнительно: проверяет Neon на наличие оплат каналов 6-8
  - Решения принимает на основе charge_off (М4)

Directus + Metabase:
  - Работают ТОЛЬКО с Neon
  - Видят все 8 каналов в одном месте

Бот Aist:
  - ПИШЕТ в Neon (каналы 6-7, как сейчас)
  - ЧИТАЕТ из Neon (все каналы)
  - Автодопуск, маршрутка — по данным из Neon
```

**Инварианты переходного периода:**
1. **Aisystant ничего не теряет.** Его PG содержит те же данные, что и раньше. Биллинг работает
2. **Neon = надмножество.** Содержит данные всех 8 каналов (свои + дубликаты из Aisystant)
3. **Дублирование одностороннее:** Aisystant → Neon. НЕ Neon → Aisystant. Дима: «дублировать обратно не хочу, это не нужно»
4. **Read из Neon для Aisystant:** при проверке оплат (баны, подписки) Aisystant проверяет и свою PG, и Neon. Если платёж есть в Neon (каналы 6-8) — он учитывается
5. **Сверка:** cron-задача сравнивает записи Aisystant PG и Neon за последние 24ч. Расхождение = алерт

**Критерии выхода из переходного периода (→ TO-BE):**
1. Все каналы 1-5 стабильно дублируются в Neon (0 расхождений за 2 недели)
2. Aisystant успешно читает из Neon и принимает решения на основе данных оттуда
3. Ory развёрнут (WP-187) → identity linking через `ory_id`, не через `suser_id`
4. Биллинг-логика Aisystant переведена на чтение из Neon (вместо своей PG)

**TO-BE (после выхода из переходного периода):**
```
Все каналы → Neon (единственный источник)
Aisystant = потребитель (read-only из Neon, не пишет в свою PG)
Или: Billing Service (SYS.010) заменяет биллинг-модуль Aisystant
```

<summary><b>2.1. Lifecycle семинара</b></summary>

Семинар проходит два режима. Переключение -- вручную или автоматически по дате завершения.

### Режим 1: Live (семинар идёт)

- Многоразовая пригласительная ссылка («маршрутка»)
- Все оплатившие вступают по этой ссылке
- **Накануне семинара (или за пару часов):** бот Aist кикает неоплативших из чата
- После кика -- можно вступить по ссылке только после оплаты (тот же механизм, что при продаже записи: оплата -> одноразовая ссылка)

### Режим 2: Запись (семинар завершился)

- **Отключить** для всех кроме админов возможность добавлять/приглашать новых участников и генерировать вступительные ссылки
- Вступление **только по одноразовой ссылке** (после оплаты)
- Многоразовая ссылка деактивируется

### Переключение режима

| Событие | Действие бота Aist |
|---------|--------------|
| Семинар завершился (дата или ручная команда) | 1. Отозвать многоразовую ссылку. 2. Отключить приглашения для не-админов (`setChatPermissions`). 3. Пометить семинар `mode: recording` |
| Новый семинар запланирован | Создать многоразовую ссылку, включить приглашения, `mode: live` |

### Два варианта допуска (оба рабочие)

| Вариант | Кто | Как | Бот Aist знает? |
|---------|-----|-----|-----------|
| **A. Через бот Aist** | Покупатель оплачивает -> бот Aist генерирует одноразовую ссылку (`createChatInviteLink`, `member_limit=1`) | Автоматически | Да -- бот Aist создал ссылку, видит вступление |
| **B. Через админа** | Админ создаёт одноразовую ссылку через интерфейс Telegram -> отправляет покупателю | Вручную | Да -- бот Aist получает `chat_member` update с `invite_link.creator` = админ. Помечает `source: admin_invite` |

**Предпочтительный вариант: A** (автоматический учёт). Вариант B -- для исключительных случаев (ручной допуск).

<summary><b>3. Целевая схема (to-be, с Ory)</b></summary>

**Принципы:**
- **Не дублировать данные** -- каждый source-of-truth остаётся на месте
- **Ory = единый аккаунт** -- один аккаунт связывает Telegram, email и аккаунт Aisystant (заменяет ручную склейку `/link`)
- **Ory = единое управление доступом** -- платформа решает, кто к чему имеет доступ (чаты, функции, контент)

### Единый аккаунт (Ory)

Сейчас один человек может иметь 3 отдельных «аккаунта»: Telegram, email на сайте, Aisystant. Склейка -- вручную через `/link` в боте.

С Ory -- **один аккаунт на человека**:
- Регистрация на сайте -> Ory создаёт аккаунт (email)
- `/start` в боте Aist -> Ory связывает telegram_id с тем же аккаунтом
- Привязка Aisystant -> миграция из LMS в тот же аккаунт

Все каналы оплаты видят одного человека. Нет путаницы «оплатил на сайте, но бот Aist не знает».

### Архитектура

```
                         Ory (аккаунты + доступы)
                                |
                    +-----------+-----------+
                    v           v           v
          Payment Registry     Neon        Directus
          (система SYS.011,  (семинары   (CRM UI --
           Реестр оплат в    бота Aist)  окно в документ
           PostgreSQL                    Реестр оплат)
           Aisystant)
                    |           |           |
                    +-----+-----+           |
                          v                 |
                    Metabase (аналитика) <---+
                          |
                          |-> Гиляна: все оплаты, сверка
                          |-> Алёна: воронка T0->T4, конверсия, когорты
                          |-> Юля: наполняемость потоков и семинаров
                          +-> Преподаватели: списки участников

              Activity Hub (WP-109)
                          |
                          +-> Billing adapter: платёж = событие
                              -> engagement -> ЦД -> аналитика
```

### Webhook-контракт (Payment Registry -> бот Aist)

Когда человек платит через Tilda/Ecwid/Монету/сайт, Payment Registry (система) записывает транзакцию в Реестр оплат (документ) и уведомляет бот Aist:

```
POST https://aistmebot-production.up.railway.app/webhook/workshop-payment
Header: X-Webhook-Secret: {секрет}

{
  "ory_id": "uuid",              // единый аккаунт (когда Ory развёрнут)
  "telegram_id": 123456789,      // до Ory -- Telegram ID (если известен)
  "tg_username": "username",     // опционально, для отправки в личку
  "email": "user@example.com",   // email покупателя
  "purpose": "SEMINAR",          // WORKSHOP | SEMINAR | INTERNSHIP
  "seminar_code": "SE-2026.2-T", // код семинара
  "stream_code": "INT-2026.3",   // код потока (если применимо)
  "amount": 5000,
  "currency": "RUB",
  "payment_id": "yookassa_xxx",  // защита от дублей
  "source": "tilda"              // tilda | moneta | yookassa | stripe | paybox | manual
}
```

**До Ory:** бот Aist принимает `telegram_id` и/или `tg_username`. **После Ory:** бот Aist получает `ory_id` и сам находит telegram_id через Ory.

### Обратный webhook (бот Aist -> Payment Registry)

Когда человек вступает в чат по одноразовой ссылке, бот Aist уведомляет Payment Registry:

```
POST https://aisystant.com/api/webhook/chat-join
Header: X-Webhook-Secret: {секрет}

{
  "telegram_id": 123456789,
  "chat_id": -100123456789,
  "seminar_code": "SE-2026.2-T",
  "invite_source": "bot" | "admin_invite",
  "joined_at": "2026-04-04T12:00:00Z"
}
```

Payment Registry (система) записывает факт вступления в Реестр оплат (документ) и использует данные для связки tg_id с аккаунтом (до Ory -- по email, если известен; после Ory -- через Ory).

### Воронка входа (для маркетинга)

```
T0 (анонимный)
  | /start в боте Aist или заход на сайт
  | -> Ory создаёт аккаунт
  v
T1 (зарегистрирован, без подписки)
  | Trial 30 дней -- полный доступ
  | -> бесплатные семинары, марафон
  v
T2 (подписка БР)
  | Оплата: YooKassa / Stripe / Монета / Stars
  | -> доступ: лента, консультации, заметки, планы, Сообщество IWE
  v
T3 (ЦД подключён)
  | Бесплатно -- заполнить цифровой двойник
  v
T4 (экзокортекс)
  | Бесплатно -- setup.sh / Gateway URL
```

**Metabase считает по Ory:** сколько на каждом тире, конверсия T0->T1->T2, удержание, отток.
