# WP-227 — Identity-архитектура платформы

> **Назначение документа:** полное описание предложения «один ЦД и общий identity-регистр» для встречи 13 апр. Короткая версия — в повестке встречи (`WP-73-architect-agenda-next.md → Решения на утверждение → Р4`). Этот документ содержит обоснования, схемы, жизненный цикл UUID, UX-последствия, процесс matching, связи с другими решениями.
> **Читать перед встречей:** стороны обсуждения должны быть знакомы с документом, чтобы сама встреча шла по существу, а не по первичному ознакомлению.

## Контекст: почему этот вопрос сейчас

WP-227 (миграция `digital_twins` из БД `aist_bot` в отдельную БД `digitaltwin`) вскрыл более глубокий вопрос об identity-модели платформы. Сейчас таблица `digital_twins` физически живёт в БД бота — это артефакт старого дизайна, когда `calculate_derived()` считался кодом бота. После вынесения calculator'а в standalone profiler (WP-218 сессия A, 10 апр) стало очевидно, что ЦД — общая платформенная подсистема, не часть бота, и должен лежать в своей БД.

Но как только начали физический перенос — встал вопрос:

> **Если `digital_twins` выносим, должна ли `users` (identity-регистр) тоже лежать в `digitaltwin` БД, или identity остаётся в каждой системе своё? А если свой в каждой системе — как устроены связи между системами? Как ЦД знает «это один человек» для пользователя, который есть в боте, LMS и Клубе?**

Ответы на эти вопросы — не техническая деталь одной миграции. Они определяют:
- Какая архитектура identity у платформы в целом (одна централизованная или распределённая).
- Как появляются пользователи в ЦД — сразу (независимо от Ory) или только после OAuth.
- Как LMS, Клуб, CRM, бот связываются между собой.
- Что делать с 93% пользователей бота, у которых сейчас нет Ory (105 из 113).
- Как работает оплата семинаров без регистрации (UX-решение «быстро купить»).

Эти вопросы выходят далеко за WP-227. Это фундамент WP-73 (Architecture Phase 2), WP-212 (RLS policies), WP-214 (PII), WP-183 (CRM), WP-215 (разделение РФ/мир).

## Текущее положение дел (факт, как работает сейчас)

- `aist_bot.users` (113 записей) — identity-регистр бота: `id UUID PK`, `telegram_id`, `ory_id` (NULL для T0 = 105 из 113), `name`, ...
- На `users.id` ссылаются 10+ таблиц бота с ~70k рядами: `user_events`, `user_state`, `identity_map`, `learning_history`, `user_integrations`, `channel_monitors`.
- `digital_twins` сейчас физически лежит в БД `aist_bot`, ключ `user_id` = либо `users.id` (для T0), либо `ory_id` (для T1+). Из-за этого получаются дубли T1+ пользователей (4 случая), симптом `/twin=66` вместо 96.2 у Тсерена.
- LMS (aisystant.com) имеет свой `suser` с email/ФИО/Ory sub — отдельный регистр, отдельная БД.
- Discourse (Клуб) имеет свой `user` — отдельный регистр, Ory SSO.
- CRM (`payment-registry` → Directus) — свои `customers` в Neon БД `directus`.
- **В каждой системе — свой внутренний ID** и своя identity-таблица. Между ними нет общего ключа, кроме `ory_id` (но он есть только у T1+).

## Принципы (то, что предлагаем утвердить)

### Принцип 1. Один физический ЦД на всю платформу

Таблица `digital_twins` — в одной БД `digitaltwin` в Neon. Ни бот, ни LMS, ни Клуб, ни CRM **не имеют своих локальных копий** ЦД. Все системы читают и пишут в общий ЦД через API или прямые SQL-запросы.

Это архитектурно правильно: «ЦД — общая платформенная подсистема», а не часть какой-либо конкретной системы. Это фундаментальное разделение слоёв, без которого все остальные принципы не работают.

### Принцип 2. У ЦД свой глобальный `digitaltwin.users.id` (UUID)

Генерируется через `gen_random_uuid()`. Независим от Ory, telegram_id, email, LMS suser.id, Discourse user.id.

**Почему именно свой UUID, а не один из существующих идентификаторов:**

- **`telegram_id`** — это про бота. Если человек никогда не заходил в бота, у него нет `telegram_id`. ЦД должен уметь работать для LMS-пользователей, которых бот не знает.
- **`ory_id`** — появляется **только** после OAuth (или при eager-эскалации через email, см. принцип 5). До этого момента у человека его нет. Если привязать ЦД к `ory_id` — человек, который установил бота и пока не готов проходить OAuth, **не имеет ЦД-записи**. Это противоречит смыслу цифрового двойника: «мы видим человека с первого касания, собираем данные, показываем ему прогресс».
- **`email`** — зависит от сценария (оплата сайта через форму). Бот без email. Telegram-платёжка без email. Не универсален.
- **Собственный UUID** — стабилен, не зависит от внешних систем, генерируется платформой, доступен для всех систем как общая точка склейки.

### Жизненный цикл `digitaltwin.users.id`

> **Ключевой вопрос:** когда UUID появляется? Как долго действует? Когда исчезает?

**Когда появляется:**

UUID создаётся, когда платформа впервые хочет записать что-то о человеке, которого ещё не знает ни по одному ключу. Формально: любая система при записи в ЦД сначала ищет человека по доступным ключам в порядке приоритета:

1. Если есть `ory_id` → `SELECT ... FROM users WHERE ory_id = $1`
2. Если есть `email` → `SELECT ... FROM users WHERE email = $1` (триггерит ручной matching, если нашли)
3. Если есть `telegram_id` → `SELECT ... FROM users WHERE telegram_id = $1`
4. Если ничего не нашли → `INSERT INTO users (id, ...) VALUES (gen_random_uuid(), ...)` → получили новый UUID → записываем действие

Триггеры создания новой записи:

| Сценарий | Первый момент | Что известно о человеке |
|---|---|---|
| Бот: `/start` в Telegram | Первое сообщение боту | `telegram_id`, возможно `name` из Telegram-профиля |
| Сайт: оплата семинара | Успешная оплата с формой email | `email`, `ФИО` + автоматически `ory_id` (eager через Kratos registration) |
| Сайт: регистрация в LMS | Пользователь проходит Ory OAuth | `email`, `ory_id`, `ФИО` |
| Клуб: первый вход | Ory SSO в Discourse | `ory_id`, `email`, `ФИО` |
| CRM: ручной ввод оператором | Добавление клиента руками | Что ввёл оператор (минимум email или телефон) |
| Telegram-платёжка | Успешная оплата через Telegram | `telegram_id` + `payment_id`, БЕЗ email/ФИО |

**Сколько действует:**

**Навсегда. UUID не меняется никогда.**

Сценарии, где UUID **остаётся стабильным** (новых данных прибавляется, ID не меняется):
- Человек прошёл Ory OAuth через месяц после первого /start → `ory_id` добавляется в существующую запись, `users.id` не меняется.
- Человек ввёл email впервые на сайте → `email` добавляется в существующую запись, `users.id` не меняется.
- Человек связал LMS-аккаунт с ботом → `digitaltwin_user_id` FK синхронизируется в обеих системах, указывает на ту же запись.
- Человек удалил бота и через год вернулся → если мы узнаём его по `telegram_id` (бот помнит) или по Ory → та же запись, тот же UUID.

**Когда UUID «исчезает»** (редкие сценарии):

- **Merge через ручной matching** (см. принцип 6): оператор обнаруживает, что `users.id = U1` (из бота) и `users.id = U2` (из сайта оплаты) — это один человек. Делается merge: все FK на `U2` перепривязываются на `U1`, запись `U2` удаляется (или помечается `merged_into = U1`). **`U2` перестаёт существовать, `U1` остаётся.** С точки зрения систем это выглядит как «наш `digitaltwin_user_id` поменялся с U2 на U1» — они обновляют свою FK-колонку.
- **Удаление по запросу пользователя** (GDPR right to erasure / 152-ФЗ): запись удаляется (soft или hard delete). Редкая операция, не плановый lifecycle.

**Временные UUID не нужны.** Нет кейсов «сгенерировать на 15 минут и удалить». Если ЦД создал запись — это потому, что человек реально коснулся платформы, и мы хотим помнить его. «Призрачные» записи (установил бота один раз, больше не вернулся) остаются навсегда с минимумом данных — они не мешают и могут стать полезными, если человек вернётся.

**Cleanup «призрачных» записей** — отдельный процесс retention policy, не часть WP-73. Связан с WP-214 (persistence) и юридическими требованиями (152-ФЗ / GDPR на хранение). Вопрос к юристам.

### Принцип 3. В каждой системе — свой внутренний ID

Мы **не удаляем** существующие PK в локальных системах:

- `aist_bot.users.id` — остаётся. На него ссылаются 10+ таблиц бота с 67k+ рядов (`user_events`, `user_state`, `identity_map`, `learning_history`, ...).
- LMS `suser.id` — остаётся. PK базы aisystant.
- Discourse `user.id` — остаётся. PK Клуба.
- CRM `customers.id` — остаётся.

Они продолжают играть роль внутренних идентификаторов своих систем. **Дополнительно** в каждой identity-таблице появляется колонка `digitaltwin_user_id UUID` — FK на `digitaltwin.users.id`.

Когда боту нужны данные ЦД для обработки команды `/twin`:
1. `SELECT id, digitaltwin_user_id FROM aist_bot.users WHERE telegram_id = $1` → получили `digitaltwin_user_id`.
2. `SELECT data FROM digitaltwin.digital_twins WHERE user_id = $2` → получили ЦД.

Два запроса, но оба дешёвые (одна БД Neon, один регион). Никаких cross-database FK-конструкций, никакой lock contention.

### Принцип 4. Ory — универсальная связка, но не PK ЦД

Когда человек прошёл OAuth хотя бы в одной системе, у него появляется `digitaltwin.users.ory_id`. С этого момента все системы могут **найти одного и того же человека по `ory_id`** и связать его каналовые identity (`telegram_id` + `suser.id` + Discourse `user.id` + `customers.id`).

До этого момента человек существует в `digitaltwin.users` под собственным UUID, но его присутствие в других системах неизвестно.

**Важно: Ory — не PK ЦД.** `digitaltwin.users.id` остаётся своим UUID. `ory_id` — атрибут (колонка), который может быть NULL или заполнен.

Это правильное DDD-разделение: **identity provider (Ory) ≠ entity identifier (`digitaltwin.users.id`)**. Ory — сервис аутентификации. ЦД — агрегат данных о человеке. У них разная семантика, разная эволюция, разная жизнь. Не должны ключеваться одним и тем же ID.

### Принцип 5. Eager Ory при вводе email (не «Ory только где обязательно»)

**Формулировка: любой ввод email = создание Ory identity.**

Где применяется:
- **Оплата через сайт** (форма с email + ФИО + оплата): на backend **автоматически** вызывается Ory Kratos `POST /self-service/registration` с этим email → Ory создаёт identity → возвращает `ory_id` → записываем в `digitaltwin.users.ory_id`. Человек не видит никакого OAuth-flow — он купил семинар, всё работает. Но технически с этого момента у него есть Ory identity, которую можно использовать для LMS/Клуба/других систем.
- **LMS** (aisystant.com): уже требует Ory — без изменений.
- **Клуб** (Discourse): уже требует Ory — без изменений.

Где **не** применяется (осознанно):
- **Бот** (Telegram-бот): Ory **не требуется**. Человек запустил бота → создаётся `digitaltwin.users` с `telegram_id` без `ory_id`. Бот работает из коробки. Низкое трение onboarding = высокая конверсия. Если попросить человека сначала пройти OAuth, потом «теперь можешь написать /start» — потеря 80% пользователей.
- **Оплата через Telegram-платёжку** (inline-кнопка в боте): человек платит прямо в Telegram, никакой email не запрашивается. Ory не создаётся. Известен только `telegram_id`. Это сознательный UX-выбор: хочешь купить один семинар — вот тебе кнопка, всё работает.

**UX-принцип:** каждый сценарий просит у пользователя **ровно столько**, сколько нужно для этого сценария. Не больше. Бот и Telegram-оплата — ничего. Сайт оплаты — email. LMS — полная регистрация.

ЦД при этом работает для всех одинаково — у каждого человека есть запись в `digitaltwin.users` с момента первого касания, независимо от уровня регистрации.

### Принцип 6. Matching нескольких каналов одного человека — ручной, по email

**Это отдельный процесс, не автоматическая дедупликация.**

Пример сценария:
1. Человек купил семинар через сайт с email `x@y.com` → появился `digitaltwin.users` с `id=U1, ory_id=ORY1, email=x@y.com` + LMS `suser.digitaltwin_user_id=U1`.
2. Через неделю этот же человек установил бота — бот его не узнаёт (не знает `email`, только `telegram_id`) → создаёт новую запись `digitaltwin.users` с `id=U2, ory_id=NULL, telegram_id=42, email=NULL`.
3. Через месяц человек в боте в каком-то контексте даёт email `x@y.com` (например, «пришли мне конспект» → форма с email).
4. Бот замечает: «у нас уже есть `digitaltwin.users` с этим email, id=U1». **Это триггер для matching, но не автоматический merge.**

**Merge делается вручную оператором**, не по эвристике. Оператор (админ платформы или менеджер поддержки) видит два `digitaltwin.users` с общим email, сравнивает их (ФИО, история событий, каналы), нажимает «это один человек». Система делает merge: записи `U1` и `U2` объединяются, все FK из бота/LMS/Клуба перепривязываются на единый ID. История событий склеивается. Audit-log фиксирует факт merge.

**Почему ручной, а не автоматический:**
- Автоматическая дедупликация по email — слишком агрессивна. Email может быть общим (семейный), корпоративным, ошибочным. Автоматический merge = риск склеить **разных людей**, что хуже чем не склеить вообще.
- Автоматическая дедупликация по ФИО — ненадёжна. «Иван Иванов» + «Ivan Ivanov» + «Иванов И.» могут быть разными или одним человеком.
- Ручной matching — дорогая операция в смысле времени оператора, но она **безопасна**. И кейсов мало: большинство пользователей появляются сразу с email через сайт (принцип 5), у них matching не нужен.

## Сводная схема БД

```
Neon project aisystant:

БД digitaltwin (глобальный регистр + ЦД):
  public.users                         ← новая таблица
    id UUID PK                         ← глобальный user_id ЦД, gen_random_uuid()
    ory_id UUID UNIQUE NULL            ← появляется при eager Ory (email) или явном OAuth
    email TEXT                         ← ключ для ручного matching
    name TEXT                          ← ФИО
    telegram_id BIGINT UNIQUE NULL     ← атрибут (если бот знает)
    jurisdiction TEXT                  ← RU/EU/US (связь с WP-215)
    consents JSONB                     ← согласия PII (WP-214)
    merged_into UUID NULL              ← если запись смержена в другую (ручной matching)
    created_at, updated_at

  public.digital_twins                 ← основная таблица ЦД
    user_id UUID FK → users.id
    data jsonb                         ← 2_collected + 3_derived
    updated_at

БД aist_bot (локальная БД бота):
  public.users                         ← остаётся, рекомендуется переименовать в bot_users
    id UUID PK                         ← internal PK бота
    telegram_id BIGINT                 ← канал-специфичный
    digitaltwin_user_id UUID           ← НОВАЯ колонка, FK → digitaltwin.users.id
    name, ory_id, ...                  ← существующие поля
  public.user_events (67k+), user_state, learning_history, user_integrations
    — FK остаются на aist_bot.users.id

LMS (БД aisystant на отдельном PG):
  public.suser                         ← как сейчас
    + digitaltwin_user_id UUID         ← НОВАЯ колонка

Discourse (Клуб):
  user + custom_fields.digitaltwin_user_id

CRM (payment-registry → Directus):
  customers + digitaltwin_user_id
```

## UX-последствия (ключевое, не потерять)

Эта архитектура **явно** отстаивает UX-принцип «не нагружать человека данными без нужды»:

- **Бот** работает без Ory, без email, без регистрации — потому что для простого диалога с ботом это излишне.
- **Telegram-платёжка** работает без Ory и без email — чтобы купить единичный семинар было быстро и без трения.
- **Сайт с формой оплаты** — просит email, потому что всё равно нужен для чека, и заодно создаёт Ory (eager).
- **LMS и Клуб** — полная регистрация, потому что это долгосрочное обучение и сообщество, где профиль реально важен.

Каждый сценарий просит у пользователя **ровно столько**, сколько нужно для этого сценария. ЦД при этом работает для всех одинаково — у каждого человека есть ЦД-запись с момента первого касания, независимо от уровня регистрации.

## Ручной matching как процесс (для будущей реализации)

На эту встречу **не** принимаем решение по реализации matching UI — только утверждаем принцип, что matching ручной. Но для полноты картины, что именно нужно будет сделать:

- **UI для оператора:** список кандидатов на merge (пары `digitaltwin.users` с общим email или с другими сигналами «возможно, это один человек»). Оператор видит обе записи рядом, историю событий каждой, и нажимает «merge» или «skip».
- **Механика merge:** перепривязать все FK в `digital_twins`, `bot_users.digitaltwin_user_id`, LMS `suser.digitaltwin_user_id`, Discourse, CRM — с `U2` на `U1`. Пометить `U2.merged_into = U1` (не удаляем физически — для audit). Историю событий склеить по timestamp.
- **Audit-log:** запись в `digitaltwin.users.merges` (отдельная таблица): кто (оператор), когда, какие ID, на основании чего. Связь с WP-214 (согласия, трассируемость).
- **Роль оператора:** админ платформы (сейчас Тсерен) или менеджер поддержки.
- **Правила конфликтов:** если ФИО разные — что делаем (спрашиваем у пользователя? предпочитаем тот, что из LMS?). Если истории платежей конфликтуют — как объединять? Вырабатывается вместе с реализацией.

## Что просим от команды на встрече

1. **Утвердить 6 принципов** (один ЦД / свой UUID `digitaltwin.users.id` / локальные ID + FK `digitaltwin_user_id` / Ory как связка / eager Ory при email / ручной matching по email) — да/нет по каждому, с правками если есть.
2. **Дима, команда aisystant:** согласие на реализацию **eager Ory при вводе email** в flow оплаты сайта. Это новый hook — когда приходит успешная оплата с email, на backend вызывается Ory Kratos `POST /self-service/registration` и результат сохраняется. Оценка объёма работы — нужна Диме.
3. **Паша:** подтверждение, что Ory Kratos `POST /self-service/registration` можно вызывать серверно без пользовательского flow (без redirect, без подтверждения). Это необычный паттерн для Ory — нужно сказать «да, поддерживается» или «нет, нужен альтернативный путь».
4. **Андрей, Олег:** согласие на архитектурный принцип «`digitaltwin.users` — новый слой с собственным UUID», понимание что это затрагивает WP-212, WP-214, WP-215.
5. **Ресурс на UI оператора matching:** кто и когда? Не блокер на эту встречу — можно до первого реального кейса merge жить без UI (делать через SQL вручную). Но поднять вопрос явно, чтобы не забыть.

## Связи с другими решениями встречи

- **Р1 (Режим E `pg_session_jwt`):** `digitaltwin.users` с колонкой `ory_id` используется как источник для RLS-политик через `auth.user_id()`. Принцип 4 (Ory как связка) делает это естественным — RLS проверяет `ory_id` в JWT против `digitaltwin.users.ory_id`.
- **Р2 (две дорожки roadmap):** Р4 попадает в **Дорожку A** (данные и безопасность) как отдельный трек. Не блокирует WP-212 B4.22, но должен быть учтён в B4.23 при раскатке RLS по таблицам — `digitaltwin.users` — одна из первых таблиц для RLS-политик.
- **Р3 (три сценария юрисдикций, S3):** если S3 — в `digitaltwin.users` появляется колонка `jurisdiction`, RLS по юрисдикции раскатывается поверх. Community Layer в S3 = псевдонимизация по `digitaltwin.users.id` без PII.
- **WP-214 (PII):** `digitaltwin.users` становится владельцем PII (email, ФИО, телефон). Это меняет классификацию домена: `digitaltwin` — PII-домен, нужен отдельный access tier. Связь с принципом S3 Regional-Split «Community Layer без PII, только UUID» — значит email/ФИО хранить **только** в `digitaltwin.users`, а в `digital_twins.data` — только `digitaltwin.users.id` как FK.
- **WP-183 (CRM):** CRM сейчас строится на платежах, не на identity. После Р4 CRM `customers.digitaltwin_user_id` связывает платежи с глобальным человеком — можно строить когортный анализ, LTV, retention по всем каналам сразу.

## Что не решаем на этой встрече

**Scope реализации (как именно и за какой бюджет делать).** На встрече утверждаем **принципы**, не implementation plan. План реализации (полный рефакторинг vs поэтапный vs компромиссный) — это отдельный РП после встречи, со своим ArchGate по объёму работ, бюджету, порядку систем. Это зависит от того, какие именно принципы команда примет и с какими поправками.

## Если не утвердим

Identity-модель платформы остаётся неопределённой. WP-227 (миграция `digital_twins`) зависает — без ясности про `users` невозможно решить, куда класть ключ `digital_twins.user_id` и что делать с дублями. WP-212 B4.23 (RLS roll-out) начинается без опорной таблицы `digitaltwin.users`. WP-214 (Cedar conditions) не знает, на какой PK ссылаться в `principal`. WP-183 (CRM) не может строить cross-channel аналитику. Каждая система продолжает владеть своей identity, и расходящиеся identity-решения становится всё дороже унифицировать в будущем.

**Цена отсрочки:** каждая неделя без принципов = ещё одна система, которая строит свою локальную логику идентификации и будет требовать отдельного рефакторинга когда-нибудь потом.

## Источник

Сессия обсуждения 11 апр между Тсереном и Claude при подготовке WP-227. Тсерен:

> «Я вообще не хочу, чтобы бот и другие системы имели свою базу по ЦД, они все пишут в общий ЦД. ORY является центральной частью и каждая система связывается с другой системой через ORY. Собственный ID ЦД для меня — это просто необходимость, чтобы записывать действия, когда нет ORY, но при этом не зависеть от телеграм. Matching — отдельный процесс, нужно всё учесть. Он ручной и по мейлу. В каждой системе есть свой ID.»
