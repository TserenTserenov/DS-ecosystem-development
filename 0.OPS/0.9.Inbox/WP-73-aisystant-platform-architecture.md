Платформа Aisystant — ядро экосистемы развития интеллекта. Текущая архитектура: **Discourse (клуб) + LMS Aisystant (обучение) + Telegram-бот (быстрый канал)**. ### 1.5. Терминология платформы

> **Источник:** WP-63 (Стандарт терминологии) + DP.D.033 (Role-centric архитектура). Здесь — рабочий реестр терминов, применяемых в этом документе.

| Термин (UL) | Определение | НЕ путать с |
|-------------|-------------|-------------|
| **Роль (Role)** | Функция в системе: Стратег (R1), Экстрактор (R2) и т.д. Исполняется человеком или ИИ-системой | Агент, Исполнитель |
| **Исполнитель (Holder)** | Тот, кто исполняет роль. Один исполнитель → несколько ролей. Claude = исполнитель роли Стратег (R1) | Роль |
| **ИИ-система** | Автономная программная система (бот, MCP-сервер, агентная система). НЕ синоним «ИИ-роли» | Агент (полисемичный), Роль |
| **Система** | Bounded context с чётким назначением, интерфейсами и данными | Подсистема, Модуль |
| **Сервис** | Описание метода = именованная точка входа (entry point). Каталог: DP.MAP.002 | Процесс (полисемичный) |
| **Поверхность (Surface)** | Thin client, интерфейс пользователя: Web App, бот, CLI | Система |
| **Тир (Tier)** | Ось Entitlement в модели доступов. Определяет «что ДОСТУПНО» по подписке | Роль (что можно ДЕЛАТЬ) |
| **Позиция (Preset)** | Именованный бандл: Role + Tier + Scope. Назначается одним действием | Должность (у нас сообщество, не компания) |

> **Связь с WP-63:** ID `DP.AGENT.xxx` подлежат миграции (Phase 3 WP-63). В данном документе используется корректная терминология: «роль», а не «агент», для обозначения функций.

### 2.9. Биллинг и лояльность

**Source:** `DS-ecosystem-development/.../Описание системы биллинга 3.2.md`
**Тарифная политика:** [Тарифы и пакеты 2.1](../../../A.Systems-Builder/A2.Systems-Builder/A2.1.Meaning/2.1.1.%20Продуктовые%20предложения/Тарифы%20и%20пакеты%202.1.md) — подписка (T1-T4) + события (on-demand) + система баллов

**5 функций:**
1. Управление тарифами (freemium, subscription, pay-per-course, семейные/корпоративные)
2. Управление доступами (feature gating, quota, auto-отключение)
3. Система баллов (начисление за активные действия, оплата программных продуктов баллами)
4. Распределение доходов (Revenue Sharing: platform 30%, author 50%, instructor 15%, curator 5%)
5. Финансовая отчётность (MRR, ARR, LTV, CAC, churn)

**Каналы оплаты (P12 — две юрисдикции, единый биллинг):**
- **Россия 🇷🇺:** YooKassa, Paybox, TG Stars, system-school.ru (подписка БР + отдельные курсы)
- **Мир 🌍:** Stripe, PayPal, TG Stars
- **Единый Billing Service** с адаптерами по юрисдикции (Strategy pattern). Подписки, тарифы, баллы — общие

**Система баллов (вместо скидок и токенов):**

Принцип: вместо фиксированных скидок по тирам — начисление баллов за активные действия. Баллами можно полностью или частично оплачивать программные продукты (практикумы, резидентуры, семинары).

Начисление (множитель по квалификации: T2 x1, T3 x1.5, T4 x2):
- Завершённый слот обучения: 5-10 баллов
- Завершённый модуль: 20-50 баллов
- Перенос в практику (артефакт): 20-100 баллов
- Peer-review чужой работы: 10-50 баллов
- Вклад в сообщество (пост, кейс): 10-30 баллов
- Проверка ДЗ (наставник): 5-20 баллов (фиксировано)
- Проект / рабочий продукт: 50-500 баллов

Трата (1 балл ≈ $1):
- Практикум ($300-500): 300-500 баллов
- Резидентура ($800-2000): 800-2000 баллов
- Семинар (₽5000-12000): 50-120 баллов
- Менторская сессия: 80-150 баллов

Ключевое: чем выше квалификация участника, тем больше множитель начисления. Это мотивирует апгрейд тира и активное участие. Баллы — признание вклада, а не покупаемая валюта.

### 3.4. Двух-юрисдикционная архитектура (P12)

> **Принцип:** Одно сообщество. Две юрисдикции. Раздельные сайты и железо — всё остальное общее.

Экосистема работает в двух правовых пространствах: **Россия** (152-ФЗ, санкционные ограничения, блокировки сервисов) и **мир** (GDPR, международные платёжные системы, свободный доступ к сервисам). При этом сообщество **едино**: участник из России и участник из Берлина видят друг друга, обмениваются знаниями, работают в одних командах, имеют общие баллы и репутацию.

```
                         ПОЛЬЗОВАТЕЛИ
                    RU Россия    World Мир
                        |           |
               +--------v--+  +-----v--------+
               |  Web App  |  |   Web App     |

                              |
               +--------------v--------------+
               |       API Gateway           |
               |     (ORY SSO + RBAC)        |
               |  Routing по юрисдикции      |
               +--------------+--------------+
                              |
    +-------------------------+-------------------------+
    |         ЕДИНЫЙ ЛОГИЧЕСКИЙ СЛОЙ (бэкенд)           |
    |                                                     |

    +-------------------------+---------------------------+

    |  YooKassa, Paybox, TGS  |  Stripe, PayPal, TGS     |
    +-------------------------+---------------------------+
    |    DB + PII: RU node    |    DB + PII: EU node      |
    |  (Hetzner RU / VK Cloud)|  (Neon EU Frankfurt)      |
    +-------------------------+---------------------------+
```

**Ключевое:** Два фронтенда — один код (Next.js), разный деплой. API Gateway определяет юрисдикцию по аккаунту и маршрутизирует к нужному платёжному адаптеру и PII-хранилищу. Вся бизнес-логика, знания, баллы, сообщество — общие.

**Billing:** Единый Billing Service с адаптерами (Strategy pattern). Пользователь привязывается к юрисдикции по стране регистрации.

**Данные — две ноды, один логический слой:**
- **РФ-нода:** PostgreSQL в российском ЦОД — ПД граждан РФ (ФИО, email, телефон, адрес) + реплика операционных данных для низкой латентности
- **EU-нода:** Neon EU Frankfurt — ПД международных пользователей + мастер-данные (знания, контент, баллы)
- **Синхронизация:** Event Bus обеспечивает eventual consistency между нодами
- Нужна юридическая консультация по точному разграничению

**ORY:** Единый ORY. Если доступ из РФ заблокирован — self-hosted ORY на РФ-ноде с федерацией к основному.

### 3.7. Самообучение ИИ-агентов (принцип #21)

> **Архитектурный механизм AI Self-Learning Flywheel.** Платформа самоулучшается: артефакты учащихся обучают ИИ, а обученный ИИ улучшает качество обучения.

```
УЧАСТНИК                       ПЛАТФОРМА                        ИИ-АГЕНТЫ
────────                       ─────────                        ──────────
Учится, создаёт артефакт ─────► ЦД фиксирует прогресс
                               Наставник / peer-review
                               проверяет артефакт
                                        │
                               Epistemic status
                               подтверждён (≥ peer-reviewed)
                                        │
                               ┌────────▼────────┐
                               │  AI Training    │
                               │  Pipeline (S61) │
                               │  weight = role  │
                               │  × quality      │
                               └────────┬────────┘
                                        │
                                        ├────────────────────► ДЗ-чекер улучшает рубрики
                                        ├────────────────────► Консультант расширяет базу
                                        ├────────────────────► Проводник точнее маршруты
                                        └────────────────────► Оценщик калибрует критерии
                                        │
                               Улучшенные ИИ-агенты
                               лучше помогают учиться ──────► Участник учится эффективнее
                                                              создаёт ещё лучшие артефакты
                                                                        ↺ (маховик)
```

**Механизм:**

| Этап | Что происходит | Компонент |
|------|---------------|-----------|
| 1. Создание | Участник создаёт артефакт (ДЗ, кейс, проект, модель) | Рабочая тетрадь / IWE |
| 2. Валидация | Наставник / peer-review подтверждает качество | Панель наставника / ESG |
| 3. Классификация | Система присваивает epistemic status и вес автора | ESG + ЦД |
| 4. Обучение | Артефакт включается в тренировочный набор ИИ-агентов | AI Training Pipeline (S61) |
| 5. Улучшение | ИИ-агенты обновляют свои модели / промпты / рубрики | ИИ-системы (Zone A) |
| 6. Обратная связь | Улучшенные агенты дают более качественную помощь | Все поверхности |

**Множители веса артефакта в обучении:**

| Ролевая ступень автора | Множитель | Обоснование |
|------------------------|-----------|-------------|
| Ученик | ×1.0 | Базовые артефакты |
| Интеллектуал | ×1.5 | Структурированные артефакты |
| Профессионал | ×2.0 | Практические артефакты |
| Исследователь | ×3.0 | Фронтирные артефакты |
| Просветитель | ×4.0 | Методологические артефакты |

**Epistemic gates (что попадает в обучение):**

| Epistemic status | Попадает? | Пример |
|-----------------|-----------|--------|
| Предположение | Нет | Черновик ДЗ |
| Проверено наставником | Да (weight ×0.5) | Принятое ДЗ |
| Peer-reviewed | Да (weight ×1.0) | Кейс после рецензии |
| В production | Да (weight ×1.5) | Работающий проект |

**Реализация (фазы roadmap):**
- **Фаза 2:** Сбор и классификация артефактов (ESG, epistemic status). Ручная курация тренировочных наборов
- **Фаза 3:** Автоматический AI Training Pipeline (S61). Fine-tuning / RAG на артефактах. A/B-тестирование улучшенных агентов

**Безопасность:** PII удаляется из артефактов перед обучением. Участник может opt-out (артефакты не используются). Consent при регистрации (GDPR Art. 6).

#### ADR-014: Трёхосевая модель доступов + Позиции (DP.D.034)

> **Source-of-truth:** [`DP.D.034-access-control-model.md`](../../../PACK-digital-platform/pack/digital-platform/DP.D.034-access-control-model.md)

**Три оси:** Entitlement (тир) × Role (роль) × Scope (область). `Permission = Entitlement ∩ Role ∩ Scope`.

**Позиция** — именованный бандл (Role + Tier + Scope), назначаемый одним действием. В коде: `preset`.

| Позиция | Role | Tier | Scope |
|---------|------|------|-------|
| Наставник потока | Наставник | TM2 | `cohort:X/group:Y` |
| Куратор программы | Администратор + Автор | T0 | `program:Z` |
| Стажёр-наставник | Наставник | TM1 | `cohort:X/group:Y` (read-only) |
| Амбассадор | Амбассадор + Участник | T2+ | `referrals:*` |

**Реализация:** ORY Kratos (identity) + JWT claims (tier + roles) + ORY Keto / Neon (scope, ReBAC Zanzibar) + таблица `positions` / `user_positions`.

#### ADR-015: Образованные агенты — fine-tuning на ZP/FPF

> **Статус:** 🔶 предлагается
> **Связь:** P21 (AI Self-Learning Flywheel), §3.7, S61

**Контекст.** Anthropic, OpenAI и другие провайдеры создают минимальных агентов — системы с общим образованием уровня «выпускник школы». Они умеют рассуждать, но не владеют методологической грамотностью: смешивают систему и описание, роль и исполнителя, уровень и контур.

Платформа Aisystant уже обучает людей нулевым (ZP) и первым (FPF) принципам. Те же принципы можно использовать для fine-tuning моделей — создавая **методологически грамотных агентов**, которые «из коробки» владеют корректным мышлением.

**Решение.** Платформа выпускает не только образованных людей, но и образованных агентов:

| Уровень образования | Что получает агент | Механизм |
|---------------------|-------------------|----------|
| Минимальный агент (Anthropic/OpenAI) | Общие способности | Pre-training провайдера |
| Методологически грамотный агент (Aisystant) | ZP + FPF: различения, принципы корректности, онтологическая дисциплина | Fine-tuning на синтетических данных из ZP/FPF |
| Доменный агент (пользователь) | Знания предметной области (Pack) | Context engineering (MCP, промпты) |
| Рабочий агент (пользователь) | Конкретные задачи (DS) | Постановка задач (промпты) |

**Источники обучающих данных:**

1. **ZP (30 ячеек).** Каждая ячейка содержит формализованное различение. Из каждого — 15-20 вариаций (бытовые, технические, управленческие контексты). Итого: 450-600 примеров.
2. **FPF.** Принципы корректности, типология (A.7), онтологические паттерны. Ещё 300-500 примеров.
3. **Бот-тренажёр (@aist_me_bot).** Уже содержит вопросы и ответы с оценками (pass/fail) — готовый верифицированный датасет.

**Генерация обучающих данных с помощью ИИ.** Принципы формализованы — у каждого различения есть чёткая граница правильного и неправильного. Это позволяет генерировать синтетические обучающие примеры:

1. Подать принцип/различение из ZP или FPF на вход модели
2. Сгенерировать разнообразные ситуации нарушения и корректного применения
3. Верифицировать через бота-тренажёра (автоматическая проверка)
4. Собрать в обучающий датасет (JSONL)

**Конкурентное преимущество.** Ни одна другая образовательная платформа не имеет формализованных принципов мышления в виде, пригодном для fine-tuning. ZP и FPF — уникальный актив, который можно конвертировать в образованных агентов.

**Связь с P21 (AI Self-Learning Flywheel):**
- P21 описывает маховик: артефакты учащихся обучают ИИ → ИИ улучшает обучение
- ADR-015 описывает **начальное состояние** агентов в этом маховике: они начинают не с нуля, а уже методологически грамотными
- P21 обогащает агентов доменными знаниями (артефакты). ADR-015 даёт им фундамент корректного мышления (ZP/FPF)

**Реализация (фазы roadmap):**
- **Фаза 2:** Генерация синтетического датасета из ZP (30 ячеек × 20 вариаций). Пилотный fine-tuning через OpenAI API (gpt-4o-mini). A/B-тестирование: fine-tuned модель vs модель с ZP в контексте
- **Фаза 3:** Расширение на FPF. Fine-tuning Claude (при доступности Anthropic fine-tuning API). Интеграция в AI Training Pipeline (S61)

**Безопасность:** обучающие данные — синтетические, на основе публичных принципов. PII отсутствует. Модель не содержит персональных данных учащихся.
