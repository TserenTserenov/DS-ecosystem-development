### 1.2. Текущая платформа

Платформа Aisystant — ядро экосистемы развития интеллекта. Текущая архитектура: **Discourse (клуб) + LMS Aisystant (обучение) + Telegram-бот (быстрый канал)**. Назрела потребность в новой архитектуре с Web App как каноническим интерфейсом, цифровым двойником, биллингом, программой лояльности и ИИ-системами.

- **UX:** «банковское приложение для развития» + дашборды по ролям (учащийся, наставник, администратор, куратор)
- **Сводные матрицы:** сценарии × роли, сценарии × предметные области × системные уровни

### 1.5. Терминология платформы

> **Источник:** WP-63 (Стандарт терминологии) + DP.D.033 (Role-centric архитектура). Здесь — рабочий реестр терминов, применяемых в этом документе.

| Термин (UL) | Определение | НЕ путать с |
|-------------|-------------|-------------|
| **Роль (Role)** | Функция в системе: Стратег (R1), Экстрактор (R2) и т.д. Исполняется человеком или ИИ-системой | Агент, Исполнитель |
| **Исполнитель (Holder)** | Тот, кто исполняет роль. Один исполнитель → несколько ролей. Claude = исполнитель роли Стратег (R1) | Роль |
| **ИИ-система** | Автономная программная система (бот, MCP-сервер, агентная система). НЕ синоним «ИИ-роли» | Агент (полисемичный), Роль |
| **Система** | Bounded context с чётким назначением, интерфейсами и данными | Подсистема, Модуль |
| **Сервис** | Описание метода = именованная точка входа (entry point). Каталог: DP.MAP.002 | Процесс (полисемичный) |
| **Поверхность (Surface)** | Thin client, интерфейс пользователя: Web App, бот, CLI | Система |
| **Тир (Tier)** | Ось Entitlement в модели доступов. Определяет «что ДОСТУПНО» по подписке | Роль (что можно ДЕЛАТЬ) |
| **Позиция (Preset)** | Именованный бандл: Role + Tier + Scope. Назначается одним действием | Должность (у нас сообщество, не компания) |

> **Связь с WP-63:** Миграция ID `DP.AGENT.xxx → DP.ROLE.xxx` завершена (WP-63 done, 2026-03-10). В документе используется корректная терминология: «роль», а не «агент», для обозначения функций.

### 2.9. Биллинг и лояльность

**Source:** `DS-ecosystem-development/.../Описание системы биллинга 3.2.md`
**Тарифная политика:** [Тарифы и пакеты 2.1](../../../A.Systems-Builder/A2.Systems-Builder/A2.1.Meaning/2.1.1.%20Продуктовые%20предложения/Тарифы%20и%20пакеты%202.1.md) — подписка (T1-T4) + события (on-demand) + система баллов

**5 функций:**
1. Управление тарифами (freemium, subscription, pay-per-course, семейные/корпоративные)
2. Управление доступами (feature gating, quota, auto-отключение)
3. Система баллов (начисление за активные действия, оплата программных продуктов баллами)
4. Распределение доходов (Revenue Sharing: platform 30%, author 50%, instructor 15%, curator 5%) _(пожелание: детализировать пропорции — маркетинг ~10%, наставники+авторы суммарно ~50%; уточнить в спецификации Billing Phase 2)_
5. Финансовая отчётность (MRR, ARR, LTV, CAC, churn)

**Каналы оплаты (P12 — две юрисдикции, единый биллинг):**
- **Россия 🇷🇺:** YooKassa, Paybox, Монета/PayAnyWay, TG Stars, system-school.ru (подписка БР + отдельные курсы)
- **Мир 🌍:** Stripe, TG Stars
- **Единый Billing Service** с адаптерами по юрисдикции (Strategy pattern). Подписки, тарифы, баллы — общие
- **Детали:** [WP-183-unified-payments-proposal.md](WP-183-unified-payments-proposal.md)

**Система баллов (вместо скидок и токенов):**

Принцип: вместо фиксированных скидок по тирам — начисление баллов за активные действия. Баллами можно полностью или частично оплачивать программные продукты (практикумы, резидентуры, семинары).

Начисление (множитель по квалификации: T2 x1, T3 x1.5, T4 x2):
- Завершённый слот обучения: 5-10 баллов
- Завершённый модуль: 20-50 баллов
- Перенос в практику (артефакт): 20-100 баллов
- Peer-review чужой работы: 10-50 баллов
- Вклад в сообщество (пост, кейс): 10-30 баллов
- Проверка ДЗ (наставник): 5-20 баллов (фиксировано)
- Проект / рабочий продукт: 50-500 баллов

Трата (1 балл ≈ $0.01):
- Практикум ($300-500): 300-500 баллов
- Резидентура ($800-2000): 800-2000 баллов
- Семинар (₽5000-12000): 50-120 баллов
- Менторская сессия: 80-150 баллов

Ключевое: чем выше квалификация участника, тем больше множитель начисления. Это мотивирует апгрейд тира и активное участие. Баллы — признание вклада, а не покупаемая валюта.

### 2.9.1. Анализ процесса оплаты (as-is, 2026-03-16)

> **Источник:** верификация WP-73, Phase 2. Проверены: `handlers/buy.py`, `handlers/subscription.py`, `core/access.py`, `clients/aisystant.py`, `core/tier_detector.py`.

**Архитектура оплаты — pull-модель:**

```
Пользователь → /buy | /subscription
  → Бот создаёт платёж через Aisystant API (create_subscription_payment / create_internship_payment)
  → Aisystant возвращает confirmationUrl (YooKassa)
  → Пользователь переходит по ссылке и оплачивает
  → YooKassa → webhook → Aisystant LMS (Java) → запись подписки/доступа в БД Aisystant
  → Бот при следующем действии: has_active_subscription() → Aisystant API → True
  → Tier detection: T1 → T2 → UI обновляется
```

**Матрица автоматизации:**

#### 3.1.1a. Ритм дня и сессия саморазвития (конкретизация P21)

> **WP-98.** ОРЗ-фрактал: паттерн Open → Work → Close применяется на двух масштабах — День и Сессия.

**Ритм дня (Day Rhythm):**

| Масштаб | Открытие | Работа | Закрытие |
|---------|----------|--------|----------|
| **День** | Вход в контекст дня (план, саморазвитие, мир) | Сессии с перерывами (помидорки) | Рефлексия: что узнал, похвала, задел |
| **Сессия** | WP Gate, определение роли | Capture, Pre-action Gates | Коммит, captures, статус |

**Сессия саморазвития** — специальный тип сессии ОРЗ:
- **Открытие:** агент показывает текущее руководство, где остановился, список черновиков
- **Работа:** мышление письмом, работа с руководством, черновики
- **Закрытие:** стандартный Close + captures в Pack
- **Связь с DayPlan:** Слот 1 = саморазвитие. Если нет — агент явно сообщает пользователю

**Требования к агенту (экзоскелет):**
- **Помидорки:** 25 мин работа → 5 мин перерыв. После 4 циклов — длинный перерыв 15-30 мин
- **Напоминание о перерыве:** если сессия >50 мин без перерыва → агент напоминает
- **«Что нового узнал»:** при закрытии дня агент подсвечивает captures, инсайты, новые знания — помогает пользователю увидеть свой прогресс (экзоскелет, не протез)
- **Похвала:** обязательный шаг закрытия дня — рефлексия достижений

**День стратегии:** Один день в неделю — сессия стратегирования (пользователь выбирает день). DayPlan этого дня содержит напоминание.

```

                    RU Россия    World Мир
                        |           |

               +--------+--+  +-----+---------+
                        |           |
                        +-----+-----+
                              |
               +--------------v--------------+
               |       API Gateway           |
               |     (ORY SSO + RBAC)        |
               |  Routing по юрисдикции      |
               +--------------+--------------+
                              |

    |         ЕДИНЫЙ ЛОГИЧЕСКИЙ СЛОЙ (бэкенд)           |
```

**Ключевое:** Два фронтенда — один код (Vue/Nuxt.js), разный деплой. API Gateway определяет юрисдикцию по аккаунту и маршрутизирует к нужному платёжному адаптеру и PII-хранилищу. Вся бизнес-логика, знания, баллы, сообщество — общие.

**Billing:** Единый Billing Service с адаптерами (Strategy pattern). Пользователь привязывается к юрисдикции по стране регистрации.

**Данные — две ноды, один логический слой:**
- **РФ-нода:** PostgreSQL в российском ЦОД — ПД граждан РФ (ФИО, email, телефон, адрес) + реплика операционных данных для низкой латентности
- **EU-нода:** Neon EU Frankfurt — ПД международных пользователей + мастер-данные (знания, контент, баллы)
- **Синхронизация:** Event Bus обеспечивает eventual consistency между нодами
- Нужна юридическая консультация по точному разграничению

**ORY (уточнено 29 мар):** Два инстанса ORY: self-hosted в РФ + ORY Network EU. **Репликация** (не федерация). Механизм синхронизации identity между инстансами — через Event Bus.

### 3.7. Самообучение ИИ-агентов (P-SLF, ADR-015)

> **Архитектурный механизм AI Self-Learning Flywheel.** Платформа самоулучшается: артефакты учащихся обучают ИИ, а обученный ИИ улучшает качество обучения.

```
УЧАСТНИК                       ПЛАТФОРМА                        ИИ-АГЕНТЫ
────────                       ─────────                        ──────────
Учится, создаёт артефакт ─────► ЦД фиксирует прогресс
                               Наставник / peer-review
                               проверяет артефакт
                                        │
                               Epistemic status
                               подтверждён (≥ peer-reviewed)
                                        │
                               ┌────────▼────────┐
                               │  AI Training    │
                               │  Pipeline (S61) │
                               │  weight = role  │
                               │  × quality      │
                               └────────┬────────┘
                                        │
                                        ├────────────────────► ДЗ-чекер улучшает рубрики
                                        ├────────────────────► Консультант расширяет базу
                                        ├────────────────────► Проводник точнее маршруты
                                        └────────────────────► Оценщик калибрует критерии
                                        │
                               Улучшенные ИИ-агенты
                               лучше помогают учиться ──────► Участник учится эффективнее
                                                              создаёт ещё лучшие артефакты
                                                                        ↺ (маховик)
```

**Механизм:**

| Этап | Что происходит | Компонент |
|------|---------------|-----------|
| 1. Создание | Участник создаёт артефакт (ДЗ, кейс, проект, модель) | Рабочая тетрадь / IWE |
| 2. Валидация | Наставник / peer-review подтверждает качество | Панель наставника / ESG |
| 3. Классификация | Система присваивает epistemic status и вес автора | ESG + ЦД |
| 4. Обучение | Артефакт включается в тренировочный набор ИИ-агентов | AI Training Pipeline (S61) |
| 5. Улучшение | ИИ-агенты обновляют свои модели / промпты / рубрики | ИИ-системы (Zone A) |
| 6. Обратная связь | Улучшенные агенты дают более качественную помощь | Все поверхности |

**Множители веса артефакта в обучении:**

| Ролевая ступень автора | Множитель | Обоснование |
|------------------------|-----------|-------------|
| Ученик | ×1.0 | Базовые артефакты |
| Интеллектуал | ×1.5 | Структурированные артефакты |
| Профессионал | ×2.0 | Практические артефакты |
| Исследователь | ×3.0 | Фронтирные артефакты |
| Просветитель | ×4.0 | Методологические артефакты |

**Epistemic gates (что попадает в обучение):**

| Epistemic status | Попадает? | Пример |
|-----------------|-----------|--------|
| Предположение | Нет | Черновик ДЗ |
| Проверено наставником | Да (weight ×0.5) | Принятое ДЗ |
| Peer-reviewed | Да (weight ×1.0) | Кейс после рецензии |
| В production | Да (weight ×1.5) | Работающий проект |

**Реализация (фазы roadmap):**
- **Фаза 2:** Сбор и классификация артефактов (ESG, epistemic status). Ручная курация тренировочных наборов
- **Фаза 3:** Автоматический AI Training Pipeline (S61). Fine-tuning / RAG на артефактах. A/B-тестирование улучшенных агентов

**Безопасность:** PII удаляется из артефактов перед обучением. Участник может opt-out (артефакты не используются). Consent при регистрации (GDPR Art. 6).

### 3.8. MCP Hub — центр MCP-серверов (ADR-018)

> **АрхГейт 8.6/10 — пройден.** MCP Hub = централизованный реестр и runtime-прокси для всех MCP-серверов экосистемы. Заменяет planned «Apps SDK & Маркетплейс» (#29) — MCP и есть платформенный SDK.

#### 3.8.1. Проблема

Текущее состояние: 4 платформенных MCP-сервера (Knowledge, Guides, Digital Twin, Composer) настраиваются вручную в `claude_desktop_config.json` каждого клиента. Проблемы:

1. **Онбординг:** пользователю нужно знать адреса серверов, вручную прописывать конфигурацию
2. **Контроль доступа:** нет единой точки авторизации — каждый MCP реализует свою (или никакую)
3. **Расширяемость:** пользователь не может создать свой MCP для своих Pack/DS и опубликовать для других
4. **Клиент-агностичность:** конфигурация привязана к конкретному клиенту (Claude Desktop, Cursor, etc.)

#### 3.8.2. Решение: MCP Hub

MCP Hub — подсистема Layer 2 (Zone B), выполняющая три функции:

```
MCP Hub = Registry + Gateway + Provisioner

┌──────────────────────────────────────────────────────────────────┐
│                          MCP Hub                                  │
│                                                                    │
│  ┌─────────────┐  ┌─────────────────┐  ┌──────────────────────┐  │
│  │  Registry    │  │  Gateway        │  │  Provisioner         │  │
│  │  (каталог)   │  │  (прокси +      │  │  (создание user      │  │
│  │             │  │   авторизация)  │  │   MCP из шаблона)    │  │
│  └─────────────┘  └─────────────────┘  └──────────────────────┘  │
│         │                 │                       │                │
│  ┌──────▼──────┐   ┌──────▼──────┐        ┌──────▼──────┐        │
│  │ Platform    │   │ User MCP    │        │ Template    │        │
│  │ MCP (4+)    │   │ (per-user)  │        │ Gallery     │        │
│  └─────────────┘   └─────────────┘        └─────────────┘        │
└──────────────────────────────────────────────────────────────────┘
```

**Три типа MCP-серверов:**

| Тип | Владелец | Доступ | Примеры |
|-----|----------|--------|---------|
| **Platform MCP** | Платформа (L2) | По тиру (T1+ или публичный) | Knowledge, Guides, Digital Twin, Composer |
| **User MCP** | Пользователь (L4) | Только владелец + явный share | MCP для личного Pack, MCP для личного DS |
| **Community MCP** | Автор (L4, опубликовано) | По тиру + установка | MCP для предметной области (Pack-X), утилиты |

#### 3.8.3. Компоненты MCP Hub

**A. Registry (каталог)**

Neon DB, таблица `operations.mcp_registry`:

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | PK |
| `owner_id` | UUID (FK → users) | NULL = платформенный |
| `name` | VARCHAR | Уникальное имя (namespace/name) |
| `type` | ENUM | platform / user / community |
| `endpoint` | VARCHAR | URL MCP-сервера |
| `transport` | ENUM | sse / streamable-http / stdio |
| `schema_snapshot` | JSONB | Кешированная MCP schema (tools, resources, prompts) |
| `access_tier` | VARCHAR | Минимальный тир для доступа (NULL = публичный) |
| `status` | ENUM | active / disabled / pending_review |
| `created_at` | TIMESTAMP | — |

RLS: пользователь видит `type='platform'` + `type='community' AND status='active'` + свои (`owner_id = current_user`).

**B. Gateway (прокси + авторизация)**

Cloudflare Worker, единый endpoint: `mcp-hub.aisystant.com`

Протокол:
1. Клиент (Claude Desktop, Cursor, etc.) подключается к MCP Hub по SSE/Streamable HTTP
2. MCP Hub проверяет ORY OAuth2 токен → определяет user_id, тир
3. Клиент вызывает `tools/list` → Hub агрегирует tools из всех доступных MCP пользователя
4. Клиент вызывает конкретный tool → Hub маршрутизирует к нужному MCP-серверу, передаёт auth context

```
Клиент (Claude/Cursor/etc.)
    │
    │ SSE / Streamable HTTP
    │ Authorization: Bearer <ory_token>
    │
    ▼
MCP Hub Gateway (CF Worker)
    │
    ├── Verify ORY token → user_id, tier
    ├── SELECT * FROM mcp_registry WHERE accessible(user_id, tier)
    ├── tools/list → aggregate from all MCPs
    │
    ├── tool call: "knowledge.search" → route to Knowledge MCP
    ├── tool call: "twin.read" → route to Digital Twin MCP (+ pass user_id)
    └── tool call: "my-pack.query" → route to User MCP (pack-personal)
```

**Преимущество:** пользователь прописывает ОДИН endpoint в своём клиенте. Все MCP доступны через Hub.

**C. Provisioner (создание пользовательского MCP)**

Пользователь (T3+) может создать MCP для своих Pack/DS.

**Принцип BYOB (Bring Your Own Backend):** данные пользователя хранятся на **его** ресурсах, не на платформенных. Платформа предоставляет код (шаблоны) и L2-знания, но не хостит пользовательские данные. Это снимает GDPR-ответственность с платформы и даёт пользователю полный контроль.

1. **Шаблон:** Платформа предоставляет шаблоны MCP-серверов:
   - `pack-mcp-template` — MCP для чтения Pack-репозитория (search, read entity, list methods)
   - `ds-mcp-template` — MCP для DS-репозитория (search docs, read processes)
   - `custom-mcp-template` — пустой MCP с boilerplate (auth, transport, tools scaffold)

2. **Knowledge Gateway (шаблон, L3):** Локальный прокси на машине пользователя. Объединяет L2 (платформенный, remote) и L4 (личный, local/remote) в единый MCP-endpoint. Пользователь подключает **один** MCP и не думает, откуда приходит ответ.

```
Клиент (Claude Code / Cursor / etc.)
    │  один MCP-endpoint
    ▼
Knowledge Gateway (локальный процесс)
    │
    ├── L2: Platform Knowledge MCP (remote, наш)
    │   ZP, FPF, SPF, платформенные Pack, courses, guides
    │   URL: knowledge-mcp.aisystant.workers.dev
    │
    └── L4: Personal Knowledge MCP (local или remote пользователя)
        Пользовательские Pack-*, DS-*, личные заметки
        Backend: Neon (его) / Supabase (его) / SQLite (локально)

    merge результатов по score → единый ответ
```

3. **Хранилище данных (BYOB):** Пользователь выбирает backend при setup:

| Backend | Бесплатный tier | Embeddings | Сложность setup | Для кого |
|---------|----------------|------------|-----------------|----------|
| **Neon** (PostgreSQL + pgvector) | 0.5 GB, 1 project | Через ingest (CF AI) | Низкая (уже обкатано) | Большинство |
| **Supabase** (PostgreSQL + pgvector) | 500 MB | Через ingest (CF AI) | Низкая | Альтернатива Neon |
| **SQLite** (локальный + sqlite-vss) | Безлимит | Через ingest (CF AI) | Нулевая (ничего не регистрировать) | Без доверия к облаку |

Общий интерфейс `KnowledgeBackend`:
- `search(query, embedding, limit) → SearchResult[]`
- `ingest(docs) → void`
- `listSources() → Source[]`

4. **Setup для пользователя** (в `setup.sh` шаблона IWE):

```bash
# Секция Knowledge Gateway в setup.sh
echo "=== Настройка Knowledge Gateway ==="
echo "Ваши Pack-данные хранятся у ВАС, не на нашем сервере."
echo "Выберите хранилище:"
echo "  1) Neon (рекомендуется, бесплатно до 0.5GB)"
echo "  2) Supabase (бесплатно до 500MB)"
echo "  3) Локальное (SQLite, без регистрации)"
read -p "Выбор [1]: " choice
# → записывает L4_BACKEND, L4_DATABASE_URL в .exocortex.env (секция T3+, WP-180 Ф8)
# → запускает init-schema.ts (создание таблиц)
# → запускает ingest.ts (индексация Pack-папок)
```

5. **Синхронизация:** Sync-agent (R8 Синхронизатор) при Close или по cron вызывает `selective-reindex.sh` → ingest обновлённых Pack в L4 backend пользователя. L2 обновляется платформой независимо.

6. **Данные:** User MCP имеет доступ ТОЛЬКО к данным владельца:
   - Pack: индексируются из локальных файлов пользователя (не из GitHub)
   - DS: то же
   - ЦД: только свой digital twin (RLS по user_id в платформенном Neon)

7. **Деплой Knowledge Gateway:**
   - **Рекомендуемый:** Локальный процесс (`npx knowledge-gateway`), запускается через `.mcp.json` как stdio MCP
   - **Альтернативный (T4+):** Self-hosted CF Worker, регистрируется в Hub
   - **Managed deploy пользовательских данных — НЕ предоставляется.** Платформа не хранит чужие данные

8. **IWE-режим (SC-17):** Knowledge Gateway — ключевой enabler для режима «IWE как сервис». Участник T3+ получает Claude Code с загруженными Pack через BYOB-архитектуру. Это новая поверхность платформы наряду с ботом, LMS и Web App. См. [WP-74 SC-17](WP-74-platform-concept-of-use.md). Открытые вопросы:
   - **Модель оплаты Claude API:** (A) участник платит сам (Anthropic Pro $20/мес или API), (B) платформа проксирует (нужен API gateway + биллинг), (C) гибридная (L2-знания бесплатны, compute — на участнике)
   - **Тир:** T3 (персонализация) или T4 (созидание)? Связь с Q5 выше
   - **Онбординг:** `setup.sh` в `FMT-exocortex-template` уже покрывает клонирование и MCP — нужно добавить подключение к L2 (авторизация по Ory-токену)

9. **Distribution Pipeline (SC-18, 25 мар):** Обобщение SC-17 на уровень организации/партнёра. Aisystant = «Ubuntu для интеллекта» (DECK v0.6). Платформа поставляет **дистрибутив** — готовое окружение (Pack + шаблоны + MCP + AI-конфигурация), разворачиваемое поверх любого AI-провайдера.
   - **Distribution Pipeline:** `fork template` → `configure` → `provision MCP` → `connect AI provider` → `validate` → working IWE
   - **Knowledge Gate** (аналог Quality Gate в CI/CD): Pack-compliance, FPF-conformance, MCP-доступность, AI-провайдер
   - **AI Provider Abstraction:** vendor-agnostic интерфейс к LLM (Claude / GPT / Gemini / локальные). Принцип #23 (DP.ARCH.001). Кандидат ADR-019
   - **Data Portability:** Full Export API, открытые форматы (Git + Markdown + YAML), стандартизованный формат дистрибутива
   - **Infrastructure as External Service:** инфра-провиженинг = external (партнёрство с SRE-платформой), не in-house. Кандидат ADR-020
   - **Три контура:** L4 (деятель) / L2 (организация) / L3 (партнёр-fork)
   - Связь: ADR-018 (BYOB), WP-145 DECK v0.6, WP-73 context «design input дистрибутив». См. [WP-74 SC-18](WP-74-platform-concept-of-use.md)

#### 3.8.7. Принцип #6 — Platform-space vs User-space + BYOB

MCP Hub реализует принцип отчуждаемости + принцип BYOB (Bring Your Own Backend):

- **Platform-space (L2):** Knowledge, Guides, Digital Twin, Composer — платформа управляет, обновляет, гарантирует SLA. Данные на платформенном Neon
- **User-space (L4):** User MCP — пользователь создаёт, управляет, может удалить. **Данные хранятся на ресурсах пользователя** (его Neon/Supabase/SQLite). Платформа предоставляет только код (шаблоны) и L2-знания, но не хостит и не имеет доступа к пользовательским данным
- **Community-space:** опубликованные User MCP. Автор управляет, платформа модерирует (review перед публикацией)

**Следствия BYOB:**

| Аспект | Платформа отвечает | Пользователь отвечает |
|--------|-------------------|----------------------|
| L2 данные (платформенные Pack, FPF, SPF) | Хранение, индексация, SLA | — |
| L4 данные (пользовательские Pack, DS) | — | Хранение, бэкапы, оплата |
| Код (gateway, ingest, шаблоны MCP) | Разработка, обновления | Установка, запуск |
| Embeddings API | Предоставление (CF AI, бесплатно) | Использование |
| GDPR / PII | Только за платформенные данные | За свои данные сам |

#### 3.8.9. ADR-018: MCP Hub как единая точка доступа к MCP (v2 — BYOB)

| Поле | Значение |
|------|----------|
| **Статус** | 🔶 Предлагается |
| **Контекст** | 4 MCP-сервера настраиваются вручную. Нет каталога, нет user MCP, нет единой авторизации. Хранение данных пользователей на платформе = GDPR-ответственность + trust barrier |
| **Решение** | MCP Hub: Registry (Neon) + Gateway (CF Worker) + Provisioner (шаблоны). **BYOB для L4**: данные пользователя на его ресурсах (Neon/Supabase/SQLite). Knowledge Gateway (локальный) объединяет L2 (платформа, remote) + L4 (личный, BYOB) в единый endpoint. Пользователь не знает, какой MCP отвечает |
| **Следствия** | Один endpoint для всех клиентов. Пользователи создают свои MCP. Apps SDK (#29) поглощается. Платформа НЕ хранит пользовательские данные |
| **Альтернативы** | (1) Оставить как есть (ручная конфигурация) — не масштабируется. (2) Multi-tenant Neon (все данные на платформе) — ЭМОГССБ 8.7, но Б=7 (GDPR). (3) Только Registry без Gateway — пользователю нужно прописывать endpoints |
| **Оценка** | ЭМОГССБ 9.1/10 (v2, улучшение с 8.6 за счёт BYOB) |

#### ADR-014: Трёхосевая модель доступов + Позиции (DP.D.034)

> **Source-of-truth:** [`DP.D.034-access-control-model.md`](../../../PACK-digital-platform/pack/digital-platform/DP.D.034-access-control-model.md)

**Три оси:** Entitlement (тир) × Role (роль) × Scope (область). `Permission = Entitlement ∩ Role ∩ Scope`.

**Позиция** — именованный бандл (Role + Tier + Scope), назначаемый одним действием. В коде: `preset`.

| Позиция | Role | Tier | Scope |
|---------|------|------|-------|
| Наставник потока | Наставник | TM2 | `cohort:X/group:Y` |
| Куратор программы | Администратор + Автор | T0 | `program:Z` |
| Стажёр-наставник | Наставник | TM1 | `cohort:X/group:Y` (read-only) |
| Амбассадор | Амбассадор + Участник | T2+ | `referrals:*` |

**Реализация:** ORY Kratos (identity) + JWT claims (tier + roles) + **ORY Keto** (scope, ReBAC Zanzibar, ✅ подтверждено 29 мар) + таблица `positions` / `user_positions`. План внедрения: тестирование на боте → потом на Aisystant.

#### ADR-015: Образованные агенты — fine-tuning на ZP/FPF

> **Статус:** 🔶 предлагается
> **Связь:** P-SLF (AI Self-Learning Flywheel, §3.1.2), §3.7, S61

**Контекст.** Anthropic, OpenAI и другие провайдеры создают минимальных агентов — системы с общим образованием уровня «выпускник школы». Они умеют рассуждать, но не владеют методологической грамотностью: смешивают систему и описание, роль и исполнителя, уровень и контур.

Платформа Aisystant уже обучает людей нулевым (ZP) и первым (FPF) принципам. Те же принципы можно использовать для fine-tuning моделей — создавая **методологически грамотных агентов**, которые «из коробки» владеют корректным мышлением.

**Решение.** Платформа выпускает не только образованных людей, но и образованных агентов:

| Уровень образования | Что получает агент | Механизм |
|---------------------|-------------------|----------|
| Минимальный агент (Anthropic/OpenAI) | Общие способности | Pre-training провайдера |
| Методологически грамотный агент (Aisystant) | ZP + FPF: различения, принципы корректности, онтологическая дисциплина | Fine-tuning на синтетических данных из ZP/FPF |
| Доменный агент (пользователь) | Знания предметной области (Pack) | Context engineering (MCP, промпты) |
| Рабочий агент (пользователь) | Конкретные задачи (DS) | Постановка задач (промпты) |

**Источники обучающих данных:**

1. **ZP (30 ячеек).** Каждая ячейка содержит формализованное различение. Из каждого — 15-20 вариаций (бытовые, технические, управленческие контексты). Итого: 450-600 примеров.
2. **FPF.** Принципы корректности, типология (A.7), онтологические паттерны. Ещё 300-500 примеров.
3. **Бот-тренажёр (@aist_me_bot).** Уже содержит вопросы и ответы с оценками (pass/fail) — готовый верифицированный датасет.

**Генерация обучающих данных с помощью ИИ.** Принципы формализованы — у каждого различения есть чёткая граница правильного и неправильного. Это позволяет генерировать синтетические обучающие примеры:

1. Подать принцип/различение из ZP или FPF на вход модели
2. Сгенерировать разнообразные ситуации нарушения и корректного применения
3. Верифицировать через бота-тренажёра (автоматическая проверка)
4. Собрать в обучающий датасет (JSONL)

**Конкурентное преимущество.** Ни одна другая образовательная платформа не имеет формализованных принципов мышления в виде, пригодном для fine-tuning. ZP и FPF — уникальный актив, который можно конвертировать в образованных агентов.

**Связь с P-SLF (AI Self-Learning Flywheel, §3.1.2):**
- P-SLF описывает маховик: артефакты учащихся обучают ИИ → ИИ улучшает обучение
- ADR-015 описывает **начальное состояние** агентов в этом маховике: они начинают не с нуля, а уже методологически грамотными
- P-SLF обогащает агентов доменными знаниями (артефакты). ADR-015 даёт им фундамент корректного мышления (ZP/FPF)

**Реализация (фазы roadmap):**
- **Фаза 2:** Генерация синтетического датасета из ZP (30 ячеек × 20 вариаций). Пилотный fine-tuning через OpenAI API (gpt-4o-mini). A/B-тестирование: fine-tuned модель vs модель с ZP в контексте
- **Фаза 3:** Расширение на FPF. Fine-tuning Claude (при доступности Anthropic fine-tuning API). Интеграция в AI Training Pipeline (S61)

**Безопасность:** обучающие данные — синтетические, на основе публичных принципов. PII отсутствует. Модель не содержит персональных данных учащихся.

#### ADR-016: JWT-based доступ к семинарам и вебинарам

> **Статус:** 🔶 предлагается
> **Связь:** Подсистема #28 (Семинары/Вебинары), ADR-014 (трёхосевая модель доступов), Billing Service (S45), Notification Service (S51)

**Контекст.** Текущий сценарий платных семинаров (100+ человек):

1. Регистрация → ссылка на Zoom отправляется в Telegram-группу
2. Прогрев — все зарегистрированные в группе
3. Перед платной частью — бот кикает неоплативших из TG-группы
4. Без TG-группы = без ссылки на Zoom

**Проблемы текущего подхода:**

| Проблема | Следствие |
|----------|-----------|
| Привязка TG-аккаунта — блокирующее требование | Пользователь без Telegram не может участвовать |
| Ссылка на Zoom видна всем в группе до отсечки | Может утечь (переслать, скопировать) |
| Бот кикает N человек через TG API | Rate limits при 100+ участниках |
| Telegram-группа = единственный gatekeeper | Не масштабируется на другие каналы (web, email) |

**Решение.** Платформа контролирует доступ напрямую через персональные JWT-ссылки:

```
1. Пользователь регистрируется на семинар (web или бот)
2. Платформа создаёт Scope: event:seminar-{id}
3. Оплата → Billing Check (S45) → присвоение: role:participant + scope:event:seminar-{id}
4. TG-канал (read-only) — прогрев для ВСЕХ зарегистрированных (ссылки на встречу нет)
5. За 10 мин до платной части:
   → Event Access Service (S62) генерирует персональные JWT-ссылки ТОЛЬКО оплатившим
   → Notification Service (S51) доставляет ссылку: TG-бот / email / личный кабинет
6. Видеоплатформа принимает ТОЛЬКО участников с валидным JWT
```

**Mapping на модель доступа (ADR-014, DP.D.034):**

```
Permission = Entitlement ∩ Role ∩ Scope

Entitlement: T2+ (оплатившие семинар)
Role:        participant
Scope:       event:seminar-{id}

→ Результат: может получить персональную JWT-ссылку на встречу
```

**Telegram-привязка — стимул, не барьер:**

| Без привязки TG | С привязкой TG |
|-----------------|----------------|
| JWT-ссылка на email или в личном кабинете | JWT-ссылка в TG-бот + push-уведомление |
| Прогрев — только web (страница события) | Прогрев — TG-канал + web |
| Работает, но UX хуже | Полный UX |

**Видеоплатформа — заменяемый компонент:**

| Платформа | Механизм JWT-доступа | Self-hosted | Примечание |
|-----------|---------------------|-------------|------------|
| **Zoom** | Registration API: `POST /meetings/{id}/registrants` → персональный `join_url` | Нет | Проверенный, платный |
| **Jitsi Meet** | JWT-токен в URL: `https://meet.example.com/room?jwt={token}` | Да | Полный контроль, бесплатный |
| **LiveKit** | Room token с claims (identity, room, permissions) | Да | Современный WebRTC, low-latency |

Рекомендация: **Jitsi Meet** для self-hosted (полный контроль, JWT из коробки) или **Zoom Registration API** если остаёмся на Zoom.

**Сервис S62 — Event Access Service:**

| Параметр | Значение |
|----------|----------|
| Триггер | Расписание (за N минут до события) или on-demand (администратор) |
| Вход | `event_id` + список `user_id` с `Permission = Entitlement ∩ Role ∩ Scope` |
| Выход | Персональные JWT-ссылки (по одной на участника) |
| Доставка | Через Notification Service (S51): TG-бот, email, in-app |
| Аудит | Event: `access_link_generated { user_id, event_id, channel, timestamp }` |

**ЭМОГССБ:**

| Характеристика | Оценка | Обоснование |
|----------------|--------|-------------|
| Эволюционируемость | **9** | Видеоплатформа заменяема (Zoom → Jitsi → LiveKit). Каналы доставки расширяемы через S51 |
| Масштабируемость | **9** | При 10x — просто не генерируем ссылку. Нет операции кикания на каждого неоплатившего |
| Обучаемость | **7** | Нужно настроить JWT + видеоплатформу. Но API простой (один сервис S62) |
| Генеративность | **9** | Паттерн «персональная ссылка по Permission» работает для любого контента: семинары, вебинары, записи, файлы |
| Скорость | **9** | Генерация JWT — миллисекунды. Доставка через S51 — секунды. Нет polling/кикания |
| Современность | **9** | JWT-based access, Zero Trust (не доверяем групповым ссылкам). Jitsi/LiveKit — SOTA |
| Безопасность | **8** | Персональный JWT нельзя переиспользовать (exp + sub). Нет утечки через группу. Аудит-лог |
| **Среднее** | **8.6** | Проходит АрхГейт (порог ≥8) |

**Обратная связь Характеристика → Принцип:**
- Обучаемость = 7 (слабое покрытие) → митигация: пошаговая документация + готовый Docker-образ Jitsi с JWT

**Безопасность (§8.5):**
- JWT с `exp` (время жизни = длительность события + 15 мин буфер)
- JWT с `sub` (user_id) — одноразовый, привязан к пользователю
- HTTPS only для доставки ссылок
- Аудит-лог: кто получил ссылку, через какой канал, когда вошёл

**Реализация (фазы roadmap):**
- **Фаза 3 (roadmap 3.6):** Event Access Service (S62), интеграция с Jitsi/Zoom, UI управления событиями
- **Временное решение (до Фазы 3):** Zoom Registration API через скрипт — платформа регистрирует оплативших через API, без ручной работы

| Навигация, меню, настройки, SM | — | Нет | **Работает** |
| Выбор тем, расписание | — | Нет | **Работает** |

> **Вывод:** ~80% пользовательского опыта бота зависит от LLM. Без fallback outage Claude = деградация основных функций.

**Решение.** Shared library `llm_client` с per-component fallback. НЕ отдельный сервис (gateway), а подключаемая библиотека.

**Два типа LLM-использования — два подхода:**

| Тип | Пример | Fallback | Почему |
|-----|--------|----------|--------|
| **Chat Completion** | Генерация, оценка, Q&A, tool_use | Claude → OpenAI (GPT-4o) → degraded mode | Модели совместимы по capability |
| **Embeddings** | guides-mcp, embeddings.py | **Без fallback** | Эмбеддинги несовместимы между провайдерами (разная размерность, разное пространство). Переключение = полная переиндексация |

**Архитектура:**

```
llm_client/              ← shared library (Python-пакет)
├── base.py              ← интерфейс: ask(prompt, tools?) → response
├── adapters/
│   ├── anthropic.py     ← адаптер Claude (Sonnet, Haiku)
│   └── openai.py        ← адаптер OpenAI (GPT-4o, GPT-4o-mini)
├── router.py            ← fallback chain + retry (3x exponential backoff)
└── config.py            ← приоритеты провайдеров, таймауты

Потребители:
  Бот:        from llm_client import ask    ← единый интерфейс
  hw-checker: from llm_client import ask
  translate:  from llm_client import ask
```

**Fallback chain:**

```
1. Claude API (primary) — retry 3x (1s, 2s, 4s)
   ↓ 529/503/timeout после 3 retry
2. OpenAI API (secondary) — retry 3x
   ↓ fail
3. Degraded mode: "ИИ-консультант временно недоступен"
   (навигация работает: меню, настройки, выбор тем. Генерация контента — нет: уроки, вопросы, оценка, Q&A требуют LLM)
```

**Tool Use:** формат различается между провайдерами. Адаптер транслирует Anthropic tool_use ↔ OpenAI function_calling. Единый internal формат в `base.py`.

**ЭМОГССБ:**

| Характеристика | Оценка | Обоснование |
|----------------|--------|-------------|
| Эволюционируемость | **9** | +провайдер = +адаптер. Единый интерфейс стабилен |
| Масштабируемость | **9** | Распределение нагрузки между провайдерами при 10x |
| Обучаемость | **8** | Shared library, не gateway. Разработчик видит код, не конфигурирует сервис |
| Генеративность | **8** | Паттерн переиспользуется всеми компонентами платформы |
| Скорость | **9** | Primary path — без overhead. Fallback — +2-4 сек на timeout |
| Современность | **9** | LiteLLM, aisuite — стандартный паттерн 2024-2025 |
| Безопасность | **8** | +1 API key (OpenAI). Секреты вне git (P10). Server-to-server |
| **Среднее** | **8.8** | Проходит АрхГейт (порог ≥8) |

**Обратная связь Характеристика → Принцип:**
- Все характеристики ≥8. Слабых нет.

**Безопасность:** промпты отправляются двум провайдерам (server-to-server). PII пользователей минимален (только имя + контекст обучения). API keys — в env vars, не в git.

**Реализация:**
- **Фаза 1 (текущая):** retry с exponential backoff в существующем `claude_client.py`
- **Фаза 2:** shared library `llm_client` + адаптер OpenAI + интеграция в бот
- **Фаза 3:** подключение hw-checker и утилит

<summary><b>7. Knowledge Gateway + BYOB (предложение)</b></summary>

> **Контекст:** ADR-018 (MCP Hub) описывает единую точку доступа к MCP-серверам. Это предложение уточняет модель данных для пользовательских MCP — как хранить знания пользователя и как объединять платформенные + пользовательские источники в один endpoint.

<details>
<summary><b>1. Проблема</b></summary>

Сейчас knowledge-mcp — монолит: 11 источников (4 платформенных Pack + SPF/FPF + DS + guides + личные) в одной Neon DB. Проблемы:

1. **Пользователь не может подключить свой Pack.** У каждого пользователя IWE будут собственные Pack-репо (их предметные области). Текущий knowledge-mcp — это наш сервер, пользователь не может добавить туда свои источники.

2. **Смешаны контуры.** Платформенные знания (ZP, FPF, SPF, PACK-digital-platform) и личные знания (DS-Knowledge-Index-Tseren, aist-bot-docs) — в одной базе. Нет изоляции.

3. **Масштаб.** При 100 пользователях × 1000 docs = 100K записей в одном Neon. Платформа платит за хранение чужих данных.

4. **Доверие.** Pack пользователя = конфиденциальное доменное знание (его бизнес). Хранение на нашем сервере = trust barrier.

5. **GDPR.** Если храним данные пользователей — мы data processor со всеми обязательствами.

</details>
<details>
<summary><b>2. Предлагаемое решение: BYOB (Bring Your Own Backend)</b></summary>

**Принцип:** платформа даёт код и L2-знания. Данные пользователя — на **его** ресурсах. Пользователь не думает, какой MCP спрашивать — один endpoint, за кулисами fan-out по источникам.

### 2.1. Два контура

| Контур | Что индексирует | Где живёт | Кто владеет |
|---|---|---|---|
| **L2 Platform** | ZP, FPF, SPF, платформенные Pack, courses, guides, FMT-docs | Remote (наш CF Worker + Neon) | Платформа |
| **L4 Personal** | Пользовательские Pack-*, DS-*, личные заметки | BYOB: его Neon / Supabase / SQLite | Пользователь |

### 2.2. Knowledge Gateway (локальный прокси)

```
Клиент (Claude Code / Cursor / бот)
    │  один MCP-endpoint (stdio или SSE)
    ▼
Knowledge Gateway (локальный процесс на машине пользователя)
    │
    ├── L2: Platform Knowledge MCP (remote, наш)
    │   URL: knowledge-mcp.aisystant.workers.dev
    │   Источники: ZP, FPF, SPF, PACK-DP, PACK-education, courses, guides
    │
    └── L4: Personal Knowledge MCP (BYOB)
        Backend: Neon (его) / Supabase (его) / SQLite (локально)
        Источники: PACK-my-domain-1, PACK-my-domain-2, DS-*

    Алгоритм:
    1. search(query) → параллельно в L2 и L4
    2. Merge результатов по score (cosine similarity)
    3. Единый ответ клиенту
```

### 2.3. Backend Interface

Общий контракт для любого L4-хранилища:

```typescript
interface KnowledgeBackend {
  search(query: string, embedding: number[], limit: number): Promise<SearchResult[]>
  ingest(docs: Document[]): Promise<void>
  listSources(): Promise<Source[]>
}

// Реализации:
class NeonBackend implements KnowledgeBackend { ... }
class SupabaseBackend implements KnowledgeBackend { ... }
class SqliteBackend implements KnowledgeBackend { ... }
```

### 2.4. Варианты хранилища для пользователя

| Backend | Бесплатный tier | Vector search | Setup |
|---------|----------------|---------------|-------|
| **Neon** (pgvector) | 0.5 GB | pgvector cosine | Низкая — обкатано |
| **Supabase** (pgvector) | 500 MB | pgvector cosine | Низкая |
| **SQLite** (sqlite-vss) | Безлимит | sqlite-vss cosine | Нулевая |

### 2.5. Setup для пользователя

В `setup.sh` шаблона IWE добавляется секция:

```bash
echo "=== Настройка Knowledge Gateway ==="
echo "Ваши Pack-данные хранятся у ВАС."
echo "Выберите хранилище:"
echo "  1) Neon (рекомендуется, бесплатно до 0.5GB)"
echo "  2) Supabase (бесплатно до 500MB)"
echo "  3) Локальное (SQLite, без регистрации)"
read -p "Выбор [1]: " choice
# → записывает L4_BACKEND, L4_DATABASE_URL в .exocortex.env (секция T3+, WP-180 Ф8)
# → init-schema.ts (создание таблиц)
# → ingest.ts (первичная индексация Pack-папок)
```

### 2.6. Синхронизация

Sync-agent (R8 Синхронизатор) при Close или по cron:
- `selective-reindex.sh` → ingest обновлённых Pack в L4 backend пользователя
- L2 обновляется платформой независимо (weekly full + selective on commit)

<summary><b>§3.9. Ролевое разделение агентов (верификация)</b></summary>

> Добавлено: 2026-03-17 (WP-122). Source-of-truth: PACK-verification (VR).

### Проблема

Один агент с одним контекстом не может проверить собственную работу — воспроизводит те же ошибки и слепые пятна (VR.FM.001).

### Решение: роли с разным контекстом

Разделение не по агентам, а по **ролям с разным контекстом загрузки**. Один Claude Code, но два вызова с разными промптами и данными.

```
                    Pack (один, общий — source of truth)
                   /                  \
          Creator role             Verifier role
          ├─ промпт: «создай»     ├─ промпт: «найди ошибки»
          ├─ контекст: задание     ├─ контекст: результат + эталон
          ├─ DS: рабочий репо      ├─ DS: тот же репо (read-only)
          └─ модель: Opus          └─ модель: Sonnet/Haiku
```

### Ключевые принципы

1. **Context isolation** (VR.SOTA.002): верификатор НЕ видит промежуточные рассуждения создателя
2. **Graduated verification** (VR.SOTA.003): глубина проверки пропорциональна стоимости ошибки
3. **CLAUDE.md — один**: общие правила одинаковые. Различие — в ролевых инструкциях и загружаемом контексте
4. **Pack — один**: верификатор и создатель читают одни и те же доменные знания

### Четыре роли верификации (PACK-verification)

| Роль | Что делает | Модель | Триггер |
|------|-----------|--------|---------|
| VR.R.001 Верификатор | Сравнивает с эталоном | Sonnet/Opus | После создания артефакта |
| VR.R.002 Аудитор | Проверяет полноту | Haiku/Sonnet | Стратегирование, перед релизом |
| VR.R.003 Рецензент | Экспертная оценка | Opus | Перед публикацией |
| VR.R.004 Приёмщик | Решение pass/fail | Человек | Закрытие РП, мерж в прод |

### Кросс-провайдерность

Роли могут исполняться разными агентами: Claude Code, ChatGPT, Gemini. Требование — не конкретный агент, а **загрузка правильного контекста** (эталон из Pack + ролевая инструкция).

### Связь с ОРЗ

| Стадия | Масштаб | Роли верификации |
|--------|---------|-----------------|
| Работа | Сессия | VR.R.001 на рубежах |
| Закрытие | Сессия | VR.R.001 + VR.R.004 |
| Открытие | День | VR.R.002 (аудит плана) |
| Закрытие | День | VR.R.002 + VR.R.004 |
| Стратегирование | Неделя | VR.R.002 (полный аудит) |

<summary><b>§3.10. Типология событий Activity Hub (ADR-009)</b></summary>

> Добавлено: 2026-04-05 (WP-73 Phase 2). Разблокирует Roadmap 0.5 и WP-109 Ф3.
> Принцип: DP.ARCH.003 §4.4 (Channel-Agnostic Events). Событие описывает ЧТО ПРОИЗОШЛО, не ГДЕ.

### Назначение

Activity Hub (ADR-009) -- единый event store платформы. Все системы пишут в `development.user_events`. Типология нужна для:
1. **Проекции ЦД** -- какие события питают какие проекции (BKT, HLR, engagement, qualifications)

3. **Подключение новых систем** -- контракт: event_type + payload schema
4. **Nudge Engine** -- на какие события реагировать уведомлениями

```sql
confidence   FLOAT         -- 0.0-1.0
skill_ids    TEXT[]         -- связь с навыками
```

### Retention Policy

| Категория | Retention | Обоснование |
|---|---|---|
| **Learning events** (marathon_*, test_*, text/task_submitted, homework_*, guide_progress) | Бессрочно | Питают BKT/HLR. Нужны для replay проекций. Невосстановимы |
| **Qualification events** (completed, subscription_*, wp_completed, knowledge_extracted) | Бессрочно | Формируют квалификационный профиль |
| **Financial events** (payment_*, subscription_*) | Бессрочно | Требование ФЗ (бухучёт), невосстановимы |
| **Engagement events** (ai_chat, commit_created, coding_time, post_created) | 1 год | Нужны для проекций, но не критичны для replay |
| **Metadata events** (settings_changed, mode_changed, page_viewed) | 30-90 дней | Метаданные, не влияют на проекции |
| **Observability events** (error_shown, dt_collect_snapshot) | 30-90 дней | Служебные, для диагностики |

**Механизм:** `pg_cron` job еженедельно: `DELETE FROM development.user_events WHERE created_at < NOW() - retention_interval AND event_category = ?`. Перед удалением: агрегация в `development.user_events_monthly` (счётчики по event_type per user per month).

### Подключение новой системы (контракт)

Чтобы новая система писала в Activity Hub:

1. Определить event_type по формату `{категория}_{действие}` (snake_case)
2. Определить source (уникальный идентификатор системы)
3. Реализовать INSERT с обязательными полями: user_id/user_uuid, event_type, source, payload, external_id (для dedup)
4. external_id формат: `{source}-{user_id}-{event_type}-{unique}` (наносекунды или UUID)
5. Добавить в эту типологию (секция §3.10)

**Варианты записи:**
- **Прямой INSERT** (рекомендуется для систем с Neon-доступом) -- ADR-IWE-005
- **HTTP webhook** (для внешних систем) -- WP-109 Ф3
- **Адаптер** (для систем с API: LMS, клуб) -- dt-collect cron
