Архитектурный принцип учёта персональных данных в AI-среде: триада **Учёт / Доступ / Аудит** = **N доменных агентов + 1 координатор + 1 верификатор**. Координатор маршрутизирует, но не хранит сырьё; верификатор изолирован, проверяет результат, не может менять данные.

#### Г.1. Что нам нужно от платформы (требования)

**Слой 1 — Инфраструктура (без AI, базовый):**

- Управление окружениями: создание sandbox'ов под задачу с контролируемым доступом к секретам и сетям. Запуск/остановка без следов.
- Управление секретами: единое хранилище, доступ только из явно описанных окружений, ротация.
- Склейка готовых инструментов (Daytona-like, но юзерфрендли): описание окружения декларативно, поднятие воспроизводимо.
- Сервер не вырождается в snowflake — конфигурация в Git, инфраструктура как код.
- **Принцип:** работает и без AI. AI — это слой над инфра, не часть её.

**Слой 2.1 — Шаблон окружения / «мастерская под задачу»:**

- Декларативное описание (yaml/cli): список репозиториев, набор инструментов (Claude Code, Kimi, Python runner, …), переменные окружения, доступ к секретам.
- Один шаблон → N параллельных мастерских (для распараллеливания задач или сравнения подходов на одной задаче).
- Жизненный цикл: создание / вход / выход / уничтожение, без накопления мусора.
- Интерфейс — для Тсерена и других пользователей платформы, не только для агентов.

**Слой 2.2 — Разграничение агентов (Парламент-модель):**

- Агенты с **заранее определённой зоной ответственности** по репозиториям/доменам: агент A → только repo X, агент B → только repo Y. Это аналог «ведомства» из триады Учёт.
- **Координатор** маршрутизирует запросы, но не имеет права хранить сырые данные доменов или решать вместо доменного агента. Видит только агрегаты/витрины.
- **Верификатор** в изоляции (sub-agent context isolation, как уже используется для VR.R.001/R.002). Проверяет результат — не имеет write-доступа к данным.
- **Аудит-журнал:** все межагентские вызовы и доступы к секретам логируются. Доступ к журналу — Тсерен и владелец данных.
- **Авторизация между агентами:** на уровне платформы (OAuth/JWT/сертификаты — реализация на усмотрение Андрея). Принцип — least-privilege.
- **Условие архитектуры:** разграничение работает как гарантия, а не как рекомендация. Если агент по ошибке/намеренно попросил данные не своего домена — отказ на уровне платформы, не на уровне промпта.

**Слой 2.3 — Оркестратор + декомпозиция метода:**

- Не чат-сессия с накапливающимся контекстом, а **сборка контекста под один шаг → один вызов LLM на дешёвой модели → проверка → откат/фиксация**.
- Метод декомпозируется до элементарных шагов (тех, что можно отдать дешёвой модели). Аналог: бизнес-процесс внутри одной операции.
- **Командный режим как UX:** не «беседа», а «команда + результат». Поддержка асинхронных команд (можно отправить ещё одну, не дожидаясь предыдущей).
- Андрей уже пробует Aider; самописная этапная итерация через LiteLLM + DeepSeek даёт лучший результат и дешевле.

**Слой 2.4 — Token Observability:**

- Учёт **реальных токенов per task / per WP / per agent**, не суммы подписки.
- Метрика стоимости workflow: «сколько эта задача стоила в токенах × тариф модели».
- Граф вызовов: «эта подзадача потратила X токенов на модели Y».
- Возможность сравнения: какой подход на какой задаче дешевле (Kimi vs Claude vs DeepSeek; чат vs оркестратор).
- **Назначение:** дать прозрачность экономики AI, чтобы решения о выборе модели/подхода принимались на числах, а не на ощущении «подписки хватает».

**Слой 2.5 — Light LLM / переключатель моделей** (из повестки 16 п.9):

- Единый переключатель: Claude / OpenRouter / local Qwen / distilled Opus / Kimi / DeepSeek.
- На уровне шаблона окружения: «этому шагу метода — дешёвую модель, тому — мощную».
- Связь с Token Observability — измерения дают основание переключать.

### З. Новая платформа — LMS, мобильное приложение, интерфейсы, табло, моделер, биллинг, личная база знаний (P1, ~15 мин)

> **Контекст.** Параллельно с Track B и Agent Workshop Platform назревает потребность в комплексной новой платформе для участников: единое окно обучения с новым LMS, мобильным приложением, персональным табло (dashboard участника), инструментом моделирования (моделер), биллингом и личной базой знаний. Требуется архитектурная рамка, чтобы не получить набор несвязанных сервисов.

#### З.1. Архитектурные вопросы

16. **Единая платформа vs набор сервисов.** Новый LMS, мобильное приложение, табло, моделер, биллинг и личная база знаний — это один зонтичный продукт с общим ядром (identity, данные пользователя, единый UI-фреймворк) или автономные сервисы, интегрируемые через API/Gateway?
17. **Мобильное приложение.** Нативное (iOS/Android) или кроссплатформенное (Flutter/React Native/PWA)? Какое соотношение офлайн/онлайн функциональности? Мобильное приложение = основной канал или дополнение к вебу?
18. **Новый LMS.** Чем принципиально отличается от текущего (Discourse + самописное)? Какие функции критичны: треки/курсы, тестирование, сертификация, адаптивные learning paths, геймификация? Интеграция с существующей базой знаний (Pack + цифровой двойник) или автономный контент?
19. **Табло для участника (Dashboard).** Что показываем: прогресс обучения, активности, метрики развития (индикаторы), задачи, уведомления, финансы (биллинг)? Единый dashboard или роль-based views (участник, ментор, админ)?
20. **Моделер.** Интеграция с существующим инструментом моделирования (System Modeling Tool / DS-ecosystem) или встроенный упрощённый редактор прямо в платформе? Форматы совместимости (XML, JSON, PNG/SVG экспорт)?
21. **Биллинг.** Интеграция со Stripe (Track B) и/или российскими платёжками (Track A)? Единый биллинг-сервис для обоих треков или раздельный? Какие сущности: подписки, разовые платежи, кредиты, купоны, возвраты?
22. **Личная база знаний.** Это расширение текущего Pack/exocortex или отдельный продукт? Как граница между личной базой знаний участника и публичной/проектной базой? Синхронизация с цифровым двойником (Neon)?
23. **Единый identity.** Ory (текущий) или новый провайдер? Единый профиль участника across LMS, табло, моделер, биллинг, база знаний? SSO между всеми компонентами?
24. **Интеграция с существующей экосистемой.** Как новая платформа связывается с текущими сервисами: event-gateway, projection-worker, activity-hub, цифровой двойник, бот? Через тот же MCP Gateway или отдельный шина?
25. **UI/UX единообразие.** Дизайн-система: одна общая (DS Kit) или каждый компонент сам по себе? Кто отвечает за дизайн-систему?
26. **Данные и миграция.** Существующие данные участников (прогресс, посты, сессии, квалификации) — мигрируем в новый LMS или оставляем в старом с read-only доступом? Пользователь сам переносит (как Р-TrackB-4) или автоматическая миграция?
27. **Таймлайн и приоритеты.** Что из перечисленного — MVP новой платформы (минимально жизнеспособный набор), а что — последующие фазы? Как это соотносится с Track B (мировая инфраструктура) — запускаем на Track B сразу или сначала на Track A?
28. **Владелец.** Кто архитектурный owner этого направления — Андрей, Тсерен, или выделяется отдельный product lead?
