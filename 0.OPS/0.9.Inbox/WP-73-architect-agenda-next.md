## 2. Как бот проверяет подписку?

### Проблема

Бот должен при вызове T2+ функций (Навигатор, Диагност, Knowledge Gateway) проверить наличие активной подписки. Сейчас механизма нет -- бот не знает тир пользователя.

### Что изменилось (4 апр, WP-183 unified-payments)

Принятые решения упрощают картину:
- **Р2:** Выделены две подсистемы: **Payment Registry (SYS.011)** -- единый реестр всех транзакций (уже существует де-факто), и **Billing Service (SYS.010)** -- бизнес-логика (тарифы, подписки, баллы). Payment Registry = source-of-truth по оплатам.
- **Р3:** Webhook: Payment Registry → бот Aist. Один канал, один контракт. Бот получает уведомление об оплате.
- **Р4:** Привязка `tg_id ↔ email ↔ aisystant_id` -- через Ory (Фаза B). Звезда, не mesh.
- **Фазирование:** Фаза A (~16h) работает БЕЗ Ory. Фаза B (~10h) добавляет Ory для управления доступами.

### Варианты (уточнённые)

| # | Вариант | Суть | Плюсы | Минусы |
|---|---------|------|-------|--------|
| А | Payment Registry напрямую | Бот при T2+ функции делает SQL-запрос в Payment Registry (SYS.011): «есть ли активная подписка у tg_id X?» | Уже есть данные (Реестр оплат). Не зависит от Ory. Работает на Фазе A | Source-of-truth по оплатам ≠ source-of-truth по доступам. Бизнес-логика (тариф → набор функций) размазана |
| Б | Ory Keto (Фаза B) | Billing Service при успешной оплате записывает relation tuple в Keto. Бот проверяет: `POST /check` -- «есть ли у user X permission T2?» | Чистое разделение: Payment Registry = факт оплаты, Keto = факт доступа. Соответствует решению А5 (29 мар) | Keto -- Фаза B. Нужна привязка identity (Р4). DE-34 в Backlog |
| В | Billing Service API | Billing Service (SYS.010) вычисляет тир на основе Payment Registry и отдаёт бот-у через API: `GET /access/{tg_id}` | Бизнес-логика в одном месте. Легко расширять (баллы, revenue sharing -- Фаза C) | Ещё один сервис. Billing Service пока не существует |

### Что предлагается

**Фаза A (сейчас):** Вариант А -- бот читает Payment Registry напрямую. Данные уже есть, webhook от Payment Registry (Р3) оповещает о новых оплатах.

**Фаза B (после Ory):** Миграция на вариант Б (Keto) или В (Billing Service API). Зависит от:
1. **Готов ли Keto?** DE-34 (привязка `ory_id ↔ lms_user_id`) в Linear Backlog, исполнитель `@t4k_developer`. Без привязки Keto не работает.
2. **Нужен ли Billing Service как отдельный сервис?** Или бизнес-логика остаётся в боте (модуль)?

**Нужно решение:** подтвердить вариант А для Фазы A + целевой вариант для Фазы B (Keto или Billing Service API).

## 3. Merge-стратегия L2 + L4

### Проблема

Gateway (ADR-IWE-003, принят 3 апр) объединяет результаты из нескольких MCP-бэкендов: L2 Knowledge (платформенные знания), L4 Personal Knowledge (репо пользователя), Digital Twin. Каждый бэкенд возвращает результаты с `score [0.0, 1.0]`.

ADR-IWE-003 уже формализовал:
- ✅ Нормализация score в диапазон `[0.0, 1.0]` -- обязательна для всех бэкендов
- ✅ Дедупликация по ключу `source:filename` -- при совпадении берётся max score
- ✅ Knowledge Gate KG-11 -- калибровка score при подключении нового бэкенда

Остаётся два открытых вопроса, отмеченных в ADR-IWE-003 как **риск** (строки 269-272):

### Открытые вопросы

**3а. Порог score (cutoff):** ниже какого значения результат не показывать пользователю? Сейчас не определён -- Gateway возвращает всё. Слишком низкий порог = шум (нерелевантные фрагменты). Слишком высокий = потеря хороших результатов, особенно для L4 (личные заметки могут иметь другое распределение scores).

**3б. Сопоставимость scores между моделями — уже актуально.** На платформе сейчас две embedding-модели:
- `knowledge-mcp` и `personal-knowledge-mcp` — OpenAI `text-embedding-3-small` (1024d)
- `guides-mcp` — Cloudflare Workers AI `@cf/baai/bge-m3` (1024d)

Scores от разных моделей имеют разное распределение -- score 0.7 от OpenAI не эквивалентен 0.7 от BGE-M3. Если Gateway объединяет результаты из обоих, ранжирование будет некорректным. KG-11 (калибровка) упомянут в ADR-IWE-003, но конкретный механизм не описан.

### Что предлагается

Для MVP:
- Фиксированный порог `0.3` + top-K (например, K=20) -- просто и предсказуемо
- Нормализация scores между OpenAI и BGE-M3: нужен калибровочный набор запросов для определения коэффициентов пересчёта (или per-backend percentile normalization)
- Альтернатива: не объединять результаты guides-mcp с остальными в один ранжированный список, а возвращать отдельным блоком

**Нужно решение:** как именно нормализовать scores между двумя моделями, или разделить выдачу по источникам.

## 4. Community MCP: как автор публикует знания для других?

### Проблема

В текущей архитектуре хостинг MCP-сервера автором не нужен:
- **Данные** (Pack, markdown) живут в GitHub-репо автора (BYOB, ADR-IWE-004)
- **Embeddings** хранятся на платформе (Neon, RLS по user_id, ADR-IWE-001)
- **MCP-сервер** -- платформенный (personal-knowledge-mcp за Gateway)
- **Ingestion** -- платформа делает через GitHub App webhook на push (ADR-IWE-004)

Автору не нужно ничего деплоить. Вопрос не в хостинге, а в **механизме публикации**: как автор разрешает платформе отдавать его embeddings другим пользователям?

### Открытые вопросы

**4а. Permission-модель публикации.** Сейчас RLS ограничивает embeddings по `user_id` -- только владелец видит свои данные. Для Community MCP нужно: автор помечает репо/раздел как "публичный" → Gateway включает эти embeddings в выдачу для подписчиков. Где хранить этот флаг? В Backend Registry? В GitHub-репо (meta-файл)?

**4б. Гранулярность публикации.** Автор может хотеть опубликовать один Pack, но не весь GitHub-репо. Нужна ли фильтрация на уровне папок/тегов при ingestion?

**4в. Модерация контента.** Knowledge Gate (KG-01..KG-13) проверяет технический контракт MCP-бэкенда, но не содержание. Нужна ли модерация опубликованных знаний? Кто отвечает за качество?

**4г. Монетизация.** Если автор публикует Pack для других -- это бесплатно (Community) или платно (marketplace)? Влияет на архитектуру: нужен ли entitlement-check при доступе к embeddings автора?

### Что предлагается

Для MVP: флаг `public: true` в Backend Registry. Автор включает через UI/API. RLS расширяется: `WHERE user_id = $1 OR public = true`. Без модерации и монетизации на первом этапе.

**Нужно решение:** permission-модель публикации + гранулярность.

## 5. Репликация RU↔EU: перечень таблиц

### Проблема

Решение принято (встреча 29 мар, Б1): два Neon-инстанса (RU + EU) + репликация (не федерация). Ory self-hosted в РФ для 152-ФЗ. Но не определено, какие конкретно данные где живут. Это блокирует:
- Activity Hub (WP-109): куда писать события, если пользователь из РФ, а контент из EU?
- Синхронизацию identity между двумя инстансами Ory (в WP-73 строка 624 упоминается Event Bus для этого)
- Billing: YooKassa (РФ) и Stripe (мир) пишут в разные базы -- как агрегировать?

### Два принципа разделения

| Принцип | Суть | Плюсы | Минусы |
|---------|------|-------|--------|
| По юрисдикции пользователя | Данные российских пользователей в RU, остальных в EU | 152-ФЗ покрыт точно. Один пользователь -- одна нода | Сложная маршрутизация (нужно знать юрисдикцию при каждом запросе). Что если пользователь переезжает? |
| По типу данных | PII (identity, contacts, платежи) в RU для россиян, контент (курсы, знания) в EU для всех | Проще маршрутизация -- тип данных определяет ноду. Контент не дублируется | Нужна юридическая проверка: достаточно ли хранить только PII в РФ? |

### Текущая гипотеза

| Категория | RU нода | EU нода | Комментарий |
|-----------|---------|---------|-------------|
| ПД россиян (152-ФЗ) | master | нет | identity, contacts |
| ПД международных | нет | master | |
| Events / Activity Hub | **?** | **?** | Содержат user_id (PII?) + действие (не PII). Дуальная природа |
| Knowledge (контент) | реплика (read) | master | Курсы, Pack-знания -- не PII |
| Billing | YooKassa (RU) | Stripe (EU) | Разные платёжные системы |
| Digital Twin | **?** | **?** | Личные данные пользователя -- PII? |

### Что предлагается

Определить перечень таблиц по нодам. Это Q13 из WP-73 -- TODO с 29 марта. Ключевой вопрос: **Activity Hub и Digital Twin -- это PII?** Если да -- маршрутизация по юрисдикции пользователя. Если нет -- всё в EU, RU только для identity.

**Нужно решение:** принцип разделения + классификация Activity Hub и Digital Twin.

## 6. Новый сервер -- что на него переезжает?

### Проблема

Сервер куплен 1 апреля. Нет плана: что деплоим, в каком порядке, какие зависимости. Без плана рискуем: (а) деплоить не то первым, (б) создать зависимости между Railway и сервером без сетевой связности, (в) дублировать сервисы без понимания кто master.

### Кандидаты на переезд

| Сервис | Зачем на сервер | Зависимости | Приоритет |
|--------|----------------|-------------|-----------|
| Ory self-hosted (Kratos + Hydra + Keto) | 152-ФЗ: identity россиян должна быть в РФ. Решение Б1 (29 мар) | Нужна БД (Postgres). Нужен домен + TLS. Репликация с Ory Network EU | Высокий -- блокирует всю RU-ноду |
| Neon RU-нода | 152-ФЗ: PII россиян. Но Neon -- managed service, self-hosted невозможен | Альтернатива: self-hosted Postgres | Зависит от решения по Neon vs self-hosted PG |
| Activity Hub | Высокая нагрузка на запись (все события платформы). Railway дорогой для write-heavy | Зависит от Neon/PG (пишет в БД). pg_notify -- внутренний транспорт | Средний -- можно начать на Railway |
| Gateway (mcp.aisystant.com) | Latency-sensitive для пользователей из РФ | Зависит от Ory (авторизация). Зависит от бэкендов (L2, L4, DT) | Низкий для MVP -- CF Workers могут проксировать |

### Что предлагается

Определить:
1. **Что первым?** Предположение: Ory self-hosted (РФ) -- это prerequisite для RU-ноды и 152-ФЗ
2. **Сетевая связность:** как сервер связывается с Railway (бот) и Neon EU? VPN? Public endpoints + mTLS?
3. **Что точно НЕ переезжает?** Бот остаётся на Railway? Knowledge-mcp остаётся на Vercel/CF?

**Нужно решение:** порядок деплоя + сетевая архитектура сервер ↔ Railway ↔ Neon.
