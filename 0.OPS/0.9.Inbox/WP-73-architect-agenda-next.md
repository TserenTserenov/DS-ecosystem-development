### 0. Концепция разделения РФ/мир: архитектура трёх уровней (~20 мин)

**Контекст:** После встречи 5 проведена полная архитектурная проработка трёхуровневого разделения: экономика и компания, инфраструктура и сервисы, сообщество и культура. Исследованы паттерны Telegram, Miro, JetBrains, Wise, Linux Foundation.

**Суть решения:** Финансы разделяем полностью (два юрлица: ООО + Delaware Inc, уже существует). Инфраструктуру дублируем по контурам (два бота, два Ory, два CRM). Сообщество сохраняем единым (баллы, ЦД, знания — общие через псевдонимизацию по ory_id). Ключевой технический ход — в Community Layer нет PII, только UUID, что снимает требования 152-ФЗ и GDPR для общего слоя.

**Документы (GitHub):** [Regional-Split/](https://github.com/aisystant/DS-ecosystem-development/tree/main/C.IT-Platform/C2.IT-Platform/C2.2.Architecture/Regional-Split/) — 5 документов:
- [00-concept.md](https://github.com/aisystant/DS-ecosystem-development/blob/main/C.IT-Platform/C2.IT-Platform/C2.2.Architecture/Regional-Split/00-concept.md) — **обзорная концепция (начинать отсюда)**
- 01 — экономическая модель (три модели юрлиц, Delaware, эволюция)
- 02 — инфраструктура (варианты A/B/C разделения данных, identity federation, outbox)
- 03 — сообщество и культура (баллы, ЦД, портируемость, псевдонимизация)
- 04 — исследование паттернов (7 компаний, сводная таблица)

**Вопросы для обсуждения:**
1. Вариант данных: единая Neon EU (A) vs Community Layer API (C) — что закладываем сейчас?
2. Identity federation при двух Ory — canonical_ory_id через identity_links или другой подход?
3. Outbox-паттерн для RU→EU — реализация (PG, Redis, файл)?
4. Пересекается с п.1 ниже (инфраструктура за границей) — серверы для контура РФ

### 1. Инфраструктура за границей (~15 мин)

**Контекст (из встречи 5):** Принято решение разделять инфраструктуру Россия/мир. Решение по серверу в России принято (платформа Андрея). **Решение по инфраструктуре за границей — не принято.**

| Сервис | Зачем | Статус |
|--------|-------|--------|
| Activity Hub | Write-heavy, Railway дорого | Андрей: не AI-агент → не его платформа. Нужен отдельный сервер |
| Langfuse | Трейсинг prod AI-вызовов | Аналогично — не AI-агент |
| Agent Runner | Серверные агенты (Scout, ночные) | Подходит под платформу Андрея |

**Вопросы:**
1. Инфраструктура за границей: облако (какое?) или свой сервер?
2. Activity Hub + Langfuse — куда, если не на платформу Андрея?
3. Сетевая связность: RU-сервер ↔ зарубежный сервер ↔ Neon EU

### 3. Безопасность: архитектурные решения (~15 мин)

**Не обсуждалось.** Вопросы остаются:
1. **Encryption at rest:** pgcrypto / app-level / Vault?
2. **Inter-service auth:** стандартизуем?
3. Сертификации — SOC 2, ISO 27001?
4. **NEW (9 апр): Neon Authorize + Gateway + CI — согласование концепции мультитенантной изоляции данных.** Инцидент: утечка персональных данных через fallback в personal-knowledge-mcp. Подготовлена концепция Defense-in-Depth (3 слоя), АрхГейт пройден (7✅/0⚠️). Связи: WP-212, WP-214, WP-215, WP-187, WP-121. → [Концепция](../../C.IT-Platform/C2.IT-Platform/C2.2.Architecture/Data-Governance/neon-authorize-rls-concept.md)
5. **NEW (9 апр): Ory JWT vs opaque tokens — нужно ли переключать?** Neon Authorize требует JWT. Сейчас Ory выдаёт opaque tokens (валидация через `/userinfo`). Альтернатива: ручной RLS (SET LOCAL) работает с opaque без изменений Ory/Gateway/бота, но разработчик может забыть SET LOCAL. Вопрос: переключаем Ory на JWT (чище, но меняем auth flow) или начинаем с ручного RLS (быстрее, без рисков для инфраструктуры)?

### 5. Учёт персональных данных: концепция WP-214 (~15 мин)

**Контекст:** Проведено исследование (WP-214 Ф0-Ф2b): госинфраструктура 5 стран (X-Road, India Stack, СМЭВ, GAIA-X, Сингапур) + SOTA ИИ-агентов (Cedar, Keto, Mem0, MCP). Результат — концепция распределённого учёта данных в IWE, основанная на аналогии «парламент вместо президента»: данные в доменах, агенты изолированы, доступ по протоколу.

**Ключевые решения для обсуждения:**
1. **Протокол межагентного доступа (Ф3):** Cedar-политики vs Keto vs гибрид? Зависит от Ory JWT strategy (WP-187) — без формата токена нельзя писать policy conditions
2. **Реестр доменов:** 5 текущих доменов (знания, ЦД, стратегия, платформа, руководства) + 5 будущих (здоровье, финансы, дети, соцсвязи, карьера). Какие домены приоритетны для формализации?
3. **Формула N+2:** минимум агентов = домены + 1 междоменный + 1 инфраструктурный. Согласны ли с моделью физической изоляции (а не policy-only)?
4. **Consent layer:** нужен ли consent artifact (как India DEPA) или достаточно декларативных policies?

**Связи:** WP-73 (Keto permissions), WP-212 (Neon Authorize), WP-187 (Ory JWT), WP-215 (разделение Россия/мир — два контура усиливают потребность в протоколе)

**Документы:** [Data-Governance — папка с исследованием и концепцией](https://github.com/aisystant/DS-ecosystem-development/tree/main/C.IT-Platform/C2.IT-Platform/C2.2.Architecture/Data-Governance)
