<summary><b>3. Безопасность: Режим E (pg_session_jwt) на утверждение (~15 мин) — РЕШЕНИЕ Р1</b></summary>

**Изменение с 9 апр:** Ory JWT DONE (B4.21), Neon Authorize упразднён для asyncpg, проведён повторный АрхГейт v3. **Это больше не «согласование концепции» — это утверждение конкретного режима.**

**Почему Режим E (pg_session_jwt, extension 0.4.0 Neon):**

| Критерий ЭМОГССБ | Оценка | Комментарий |
|------------------|--------|-------------|
| **Э** (эволюционируемость) | ✅ | open source, portable (не vendor-lock Neon), extension доступен в 0.4.0 |
| **М** (модульность) | ✅ | TCB = Ory + DB. Gateway не участвует в RLS-решениях (принцип Г2: прозрачный прокси) |
| **О** (операционность) | ⚠️ | обвязка JWKS cache + retry (kid=""-вопрос к Паше). Принято: +2h в B4.22 |
| **Г** (governance) | ✅ | policy внутри БД (SQL RLS), review через Neon branch |
| **С** (совместимость) | ✅ | asyncpg работает, pool init hook для установки сессии |
| **С** (сохранность) | ✅ | TCB меньше, чем при Data API (меньше компонентов, меньше поверхность) |
| **Б** (безопасность) | ⚠️ | раскатка по таблицам через Neon branch, audit-log отложен осознанно. Принято |

**Итог АрхГейта v3:** 5✅ / 2⚠️ (оба приняты с обоснованием). Крит. характеристики (Безопасность + Эволюционируемость) закрыты.

<summary><b>4. Один ЦД и общий identity-регистр (~15 мин) — СРЕДИ ПЕРВЫХ ВОПРОСОВ, ОБСУЖДЕНИЕ</b></summary>

**Контекст:** WP-227 (миграция `digital_twins` из БД `aist_bot` в отдельную БД `digitaltwin`) вскрыл более глубокий вопрос. Сейчас таблица `digital_twins` физически живёт в БД бота как артефакт старого дизайна, когда `calculate_derived()` считался кодом бота. После WP-218 сессии A (10 апр) calculator вынесен в standalone profiler, бот стал чистым collector'ом. Осталось привести физическое хранение в соответствие с логикой: **ЦД — общая платформенная подсистема, не часть бота**. Но как только мы начали это делать — встал вопрос: если таблицу `digital_twins` мы выносим в `digitaltwin` БД, то таблица `users` (identity-регистр) тоже должна там жить? Или identity остаётся в каждой системе своё?

**Текущее положение дел (факт):**
- `aist_bot.users` (113 записей) — identity-регистр бота: `id UUID PK`, `telegram_id`, `ory_id` (NULL для T0), `name`, ...
- На `users.id` ссылаются 10+ таблиц бота с ~70k рядами: `user_events`, `user_state`, `identity_map`, `learning_history`, `user_integrations`, `channel_monitors`.
- LMS (Aisystant) имеет свой `suser` с email/ФИО/Ory sub — отдельный регистр.
- Discourse (Клуб) имеет свой `user` — отдельный регистр, ключ Ory.
- **В каждой системе — свой внутренний ID** и своя identity-таблица.

**Предложение пользователя (для обсуждения):**

1. **Один физический ЦД.** Никаких локальных таблиц digital_twins в боте, LMS, Клубе. Все системы пишут в **одну** БД `digitaltwin`. Это архитектурно правильно и это WP-227 в текущем виде.

2. **Собственный `user_id` ЦД, независимый от Ory/Telegram/email.** ЦД генерирует свой UUID для каждого человека. Назначение: записывать действия, когда у человека ещё нет Ory, но при этом не привязываться к одному каналу (telegram_id). То есть в `digitaltwin.users` есть колонка `id` — это **глобальный ID человека в ЦД**, не зависящий ни от чего.

3. **В каждой системе есть свой локальный ID** (`aist_bot.users.id`, LMS `suser.id`, Discourse `user.id`). Они не заменяются `digitaltwin.users.id` — они остаются как внутренние PK своих систем. Но каждая система знает свою связь с `digitaltwin.users.id` (через атрибут или маппинг).

4. **Ory как центральная связка между системами.** Системы «узнают друг друга» через `ory_id`. Если человек прошёл OAuth хотя бы где-то — появляется `digitaltwin.users.ory_id`, и этот человек однозначно идентифицируем во всех системах. Если не прошёл — живёт в ЦД под собственным `user_id`, но только в рамках одной системы, где его знают.

5. **Эскалация до Ory как только появляется email.** Позиция пользователя: «как только вводит email — Ory появляется сразу». Это значит: оплата через сайт (где есть форма с email) = автоматическое создание Ory identity. LMS / Клуб — уже требуют Ory (как сейчас). Бот и Telegram-платёжка — **остаются без Ory** (онбординг мгновенный, не нагружаем человека).

6. **Matching нескольких каналов одного человека — ручной, по email.** Если человек сначала купил семинар через сайт (email → Ory появился), потом установил бота (telegram_id, Ory не появился автоматически), потом в LMS зарегистрировался (email тот же) — задача matcher'а понять, что это один человек. Позиция пользователя: **matching ручной, ключ — email**. Не автоматическая дедупликация по эвристикам, а процедура «оператор смотрит на два профиля и подтверждает, что это один человек».

**Сводная схема:**

```
digitaltwin БД:
  users            ← глобальный регистр людей ЦД
    id UUID PK     ← собственный user_id, независимый от всего
    ory_id UUID    ← появляется, когда человек прошёл Ory OAuth (обязательно для LMS/Клуба/оплаты через сайт)
    email          ← ключ для ручного matching
    ФИО            ← атрибут
    telegram_id    ← атрибут (если бот его знает)
    ...
  digital_twins    ← ЦД конкретного человека
    user_id FK → users.id
    data jsonb     ← 2_collected + 3_derived

aist_bot БД (внутренняя):
  bot_users        ← локальный регистр ботa (если вообще нужен)
    id UUID PK     ← свой internal PK
    telegram_id    ← чем ключует Telegram
    digitaltwin_user_id UUID  ← связь с глобальным ЦД
  user_events, user_state, ...  ← локальные FK на bot_users.id

LMS БД (aisystant):
  suser            ← как сейчас, свой внутренний регистр
    + digitaltwin_user_id  ← связь с глобальным ЦД

Discourse (Клуб):
  user             ← как сейчас
    + digitaltwin_user_id  ← связь с глобальным ЦД
```

**Что это меняет архитектурно:**

- Бот больше не владеет identity. Он владеет своей связью с глобальным человеком (`bot_users`) и своей локальной логикой (учёт состояний, событий, messaging).
- `digitaltwin.users` — источник истины для «кто этот человек» на уровне платформы. LMS/Клуб/бот — вкладчики, читатели, каналы.
- Ory становится **универсальной связкой** между системами (как сейчас заявлено), но не PK — PK у ЦД свой (`users.id`).
- Для T0 пользователей (только бот, нет Ory) — они **всё равно есть в `digitaltwin.users`**, под собственным `user_id`, но без `ory_id` и без доступа из LMS/Клуба. Это честно: человек есть, мы его видим, но только через один канал.

**Q6: Eager vs lazy Ory.**
- Позиция пользователя (из обсуждения): «как только вводит email — Ory появляется сразу». То есть при оплате через сайт автоматически создаётся Ory identity.
- Это **eager Ory для каналов с email**, но **lazy Ory для бота без email**. Это гибридная стратегия — не текущая (lazy везде), но и не полностью eager.
- Следствие: нужно API / hook в flow оплаты сайта, который вызывает Ory Kratos `POST /self-service/registration` с `email + telegram_id + ФИО` и сохраняет созданный `ory_id` в `digitaltwin.users`.
- Это реализуемо, но затрагивает текущий flow оплаты — кто его владелец? Дима / команда aisystant?

## Решения на утверждение

> Три конкретных решения, по которым нужно «да/нет» или «да, с поправками». Не «обсудим» — а именно утвердить. Альтернативы рассмотрены и обоснованно отвергнуты до встречи.

<details>
<summary><b>Р1. Режим E (pg_session_jwt) — основной путь RLS-автоматизации</b></summary>

**Что утверждаем:** для защиты данных мультитенантных таблиц в Neon используем extension `pg_session_jwt` (0.4.0). Приложение (бот, Gateway) при старте сессии к БД вызывает функцию, которая принимает Ory JWT и устанавливает `auth.user_id()`. RLS-политики таблиц опираются на `auth.user_id()` — Postgres сам отрезает чужие строки.

**Альтернативы (рассмотрены в АрхГейте v3, 10 апр):**
- **D. Data API (Neon Authorize как HTTP-посредник):** упразднён. Custom JWKS в Neon Authorize работает **только** через HTTP Data API, который ломает asyncpg (наш основной драйвер). Переписывать весь data-access на HTTP = неприемлемый объём + регресс производительности.
- **F. Режим B (JWT в claim вместо session):** хуже по operability — JWT нужно прокидывать в каждый запрос, hook в пул не работает, легко забыть. Тот же класс ошибок, что и «забыть `SET LOCAL`».
- **G. Keto-only (без RLS на уровне БД):** нарушает Defense-in-Depth. Если баг в приложении — данные утекают без второго барьера. Инцидент 9 апр (утечка через fallback в personal-knowledge-mcp) — ровно этот сценарий.

**АрхГейт ЭМОГССБ v3:** 5✅ / 2⚠️. Оба ⚠️ (operability + рассчитанный риск неполной раскатки) приняты осознанно: первый закрывается обвязкой в B4.22, второй — поэтапной раскаткой по таблицам в B4.23 через Neon branch.

**Что это значит для реализации (WP-212 B4.22, 4h):**
1. `CREATE EXTENSION pg_session_jwt` в миграции (применить через Neon branch)
2. JWKS cache + retry в Python-обвязке (обход `kid=""` в Ory — **зависит от фикса Паши**, но резервный monkey-patch игнорирует kid)
3. Pool init hook в asyncpg — автоматическая установка JWT при взятии соединения из пула, сброс при возврате
4. Smoke-тест end-to-end: два пользователя, две RLS-политики, проверка что каждый видит только свои строки

**Что просим от команды на встрече:**
- **Андрей, Олег:** согласие на Режим E как путь (не «давайте посмотрим ещё варианты»)
- **Паша:** (а) фикс `kid=""` в Ory JWKS — блокер нормального JWKS cache (оценка: <1h); (б) процесс review RLS-политик через Neon branch — кто апрувит, где чеклист
- **Порядок раскатки таблиц (B4.23):** начинаем с `digital_twins` и `user_sources` (малые, high-value, hit от инцидента 9 апр). Дальше `payments`, `user_progress`, …. Согласовать порядок или оставить на усмотрение разработчика?

**Если не утвердим сегодня:** WP-212 B4.22 блокируется ещё на неделю, раскатка RLS сдвигается → Defense-in-Depth не закрыт → риск нового инцидента утечки. Цена отсрочки реальная.

Русскоговорящие пользователи из РФ работают как международные | Простота (одна инфраструктура, одно право, один compliance). Никаких 152-ФЗ | Потеря российской аудитории (для многих РФ-хостинг = доверие). Банковские блокировки. Риск санкций на стороне провайдеров (Neon EU для .ru) | Если российский рынок стратегически не ключевой, и проще «уйти начисто» |
| **S2. Только РФ** | Полностью в России: ООО, 152-ФЗ, российский хостинг, российские платежи. Международные пользователи работают через российскую инфру | Полный контроль, простота compliance в одном контуре | Невозможность привлечения иностранных инвесторов (DECK Слайд 12 — инвесторы из США/EU не вкладываются в РФ-юрлица после 2022). Отсутствие доступа к SOTA-инструментам (OpenAI, Anthropic — ограничены для РФ-юрлиц) | Если инвесторы не ищутся, фокус — внутренний рынок |
| **S3. Две юрисдикции + общий core без PII** | Два юрлица, два Ory, два CRM, два бота. **Community Layer** (баллы, ЦД без имени, знания) — единый, работает только с `ory_id` без PII, что снимает 152-ФЗ и GDPR-требования для общего слоя | Сохраняет и РФ-аудиторию, и возможность инвестиций. Псевдонимизация через ory_id — техническое решение правового противоречия. Паттерн Telegram/Wise/JetBrains | Двойные операционные расходы (инфра×2, compliance×2). Сложность identity federation. Outbox RU→EU — netto-новый код. Тонкая грань «что PII, а что нет» требует юруточнения | Если хотим сохранить обе стороны: рынок + инвесторы. **Это сценарий, на котором построена концепция Regional-Split/** |

**Предложение:** утвердить **S3** как рабочую гипотезу. Основание: (1) инвесторы — стратегический приоритет (DECK, WP-145 Олег Царёв), без них модель масштабирования не работает; (2) российская аудитория — существующая база (LMS, клуб, бот), бросить её = потерять маховик; (3) псевдонимизация по ory_id — единственный найденный технический механизм совместить (1) и (2) без нарушения compliance; (4) концепция Regional-Split/ (7 документов) уже детально это прорабатывает, к встрече есть материал для обсуждения.

**Альтернатива на столе:** если кто-то из команды считает, что S3 не взлетает (слишком дорого operationally или юридически нестабильно), — нужна альтернатива **до** встречи, иначе обсуждение деградирует в «нам не нравится, но лучше не предложим».

**Что утверждаем на встрече:**
- **S3 как рабочая гипотеза** — да/нет
- **Главные риски S3**, которые нужно проверить в юрконсультации: что именно считается PII в 152-ФЗ/GDPR в контексте ory_id-based Community Layer? Outbox с какими полями допустим?
- **Список юристов/ресурсов** для Q9 (152-ФЗ) и Q11 (санкции) — Олег уже предупреждал что вопросы нетривиальные

**Если не утвердим сегодня:** WP-215 Ф3-Ф5 остаются без опорной точки, WP-186 юрлица оформляются вслепую (под какой из 3 сценариев?), концепция Regional-Split/ висит недописанная.
