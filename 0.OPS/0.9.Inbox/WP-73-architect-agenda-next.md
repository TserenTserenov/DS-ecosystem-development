**А. РФ/мир — синхронизация Ory между двумя контурами** *(перенесено с 12 апр, главный вопрос)*

Три варианта открыты:
- **Вариант 1: Полная копия** — реплицируем Ory RU → EU. Проблема: Ory хранит PII (email, имя) → 152-ФЗ/GDPR-конфликт при репликации.
- **Вариант 2: Независимые** — два Ory, нет связки. Проблема: один человек = два профиля, ЦД не узнаёт «это тот же человек».
- **Вариант 3: Гибрид** — два Ory + identity bridge только по ory_id (без PII). Community Layer работает только с ory_id, PII остаётся в своём контуре.

**Решение нужно, потому что:** WP-215 Ф3-Ф5 (dual-contour спецификация) и WP-186 (оформление юрлиц) ждут рабочей гипотезы.

<details>
<summary><b>Р3. Рабочая гипотеза юрисдикций для WP-215 (до юрконсультации)</b></summary>

**Что утверждаем:** S3 (две юрисдикции + общий core без PII) как рабочую гипотезу — от которой отталкиваемся при разработке дизайна до юрзаключения.

**Три сценария:**

| Сценарий | Суть | Плюсы | Минусы |
|----------|------|-------|--------|
| **S1. Только EU** | Сворачиваем РФ-контур | Простота, один compliance | Потеря РФ-аудитории, банковские блокировки |
| **S2. Только РФ** | Всё в России | Контроль | Нет иностранных инвесторов после 2022, нет SOTA-инструментов |
| **S3. Две юрисдикции + Community Layer без PII** | Два юрлица, два Ory. Community Layer только по ory_id | РФ-аудитория + возможность инвестиций. Паттерн Telegram/Wise/JetBrains | Инфра×2, compliance×2. Тонкая грань PII требует юруточнения |

**Предложение:** S3. Документы: [Regional-Split/](https://github.com/aisystant/DS-ecosystem-development/tree/main/B.Aisystant-Ecosystem/B3.Ecosystem-Builder/B3.1.Meaning/Regional-Split/).

**Что утверждаем:** S3 как рабочая гипотеза — да/нет. Главные риски для проверки у юриста: что считается PII в контексте ory_id-based Community Layer? Outbox с какими полями допустим?

</details>
