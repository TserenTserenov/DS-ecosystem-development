### 3. Безопасность: Режим E (pg_session_jwt) на утверждение (~15 мин) — РЕШЕНИЕ

**Изменение с 9 апр:** Ory JWT DONE (B4.21), Neon Authorize упразднён для asyncpg, проведён повторный АрхГейт v3. **Это больше не «согласование концепции» — это утверждение конкретного режима.**

**Что решаем:** из 4 оставшихся кандидатов (D Data API / E pg_session_jwt / F Режим B / G Keto-only) — **утвердить Режим E** как основной путь RLS-автоматизации.

**Почему Режим E (pg_session_jwt, extension 0.4.0 Neon):**

| Критерий ЭМОГССБ | Оценка | Комментарий |
|------------------|--------|-------------|
| **Э** (эволюционируемость) | ✅ | open source, portable (не vendor-lock Neon), extension доступен в 0.4.0 |
| **М** (модульность) | ✅ | TCB = Ory + DB. Gateway не участвует в RLS-решениях (принцип Г2: прозрачный прокси) |
| **О** (операционность) | ⚠️ | обвязка JWKS cache + retry (kid=""-вопрос к Паше). Принято: +2h в B4.22 |
| **Г** (governance) | ✅ | policy внутри БД (SQL RLS), review через Neon branch |
| **С** (совместимость) | ✅ | asyncpg работает, pool init hook для установки сессии |
| **С** (сохранность) | ✅ | TCB меньше, чем при Data API (меньше компонентов, меньше поверхность) |
| **Б** (безопасность) | ⚠️ | raskatka по таблицам через Neon branch, audit-log отложен осознанно. Принято |

**Итог АрхГейта v3:** 5✅ / 2⚠️ (оба приняты с обоснованием). Крит. характеристики (Безопасность + Эволюционируемость) закрыты.

**Что нужно от команды:**
1. **Утверждение Режима E** как решения (альтернативы рассмотрены, не пересобираем)
2. **Паша: фикс `kid=""` в Ory JWKS** — блокер pg_session_jwt JWKS cache. Без этого extension будет падать на `kid lookup`. Оценка: Паше — менее часа
3. **Паша: review-процесс политик RLS** — через Neon branch, кто апрувит, где чеклист
4. **Раскатка по таблицам (B4.23):** начинаем с `digital_twins` и `user_sources` (небольшие, high-value). Порядок остальных — обсудить

**Старые вопросы (остаются фоном, не блокер встречи):**
- Encryption at rest: pgcrypto / app-level / Vault — отложить на отдельную встречу
- Inter-service auth: стандартизуем? — отложить
- Сертификации SOC 2 / ISO 27001 — привязаны к WP-215 юрисдикциям, отдельная тема

**Связи:** WP-212 B4.22 (4h реализация extension + JWKS cache), B4.23 (6h раскатка по таблицам), B4.24 (2h упрощён — asyncpg остаётся). WP-214 §3 (Cedar conditions теперь с реальным sub=ory_id). WP-187, WP-215, WP-121.

### 5. Учёт персональных данных: концепция WP-214 (~15 мин)

**Контекст:** Проведено исследование (WP-214 Ф0-Ф2b): госинфраструктура 5 стран (X-Road, India Stack, СМЭВ, GAIA-X, Сингапур) + SOTA ИИ-агентов (Cedar, Keto, Mem0, MCP). Результат — концепция распределённого учёта данных в IWE, основанная на аналогии «парламент вместо президента»: данные в доменах, агенты изолированы, доступ по протоколу.

**Ключевые решения для обсуждения:**
1. **Протокол межагентного доступа (Ф3):** Cedar-политики vs Keto vs гибрид? Зависит от Ory JWT strategy (WP-187) — без формата токена нельзя писать policy conditions
2. **Реестр доменов:** 5 текущих доменов (знания, ЦД, стратегия, платформа, руководства) + 5 будущих (здоровье, финансы, дети, соцсвязи, карьера). Какие домены приоритетны для формализации?
3. **Формула N+2:** минимум агентов = домены + 1 междоменный + 1 инфраструктурный. Согласны ли с моделью физической изоляции (а не policy-only)?
4. **Consent layer:** нужен ли consent artifact (как India DEPA) или достаточно декларативных policies?

**Связи:** WP-73 (Keto permissions), WP-212 (Neon Authorize), WP-187 (Ory JWT), WP-215 (разделение Россия/мир — два контура усиливают потребность в протоколе)

**Документы:** [Data-Governance — папка с исследованием и концепцией](https://github.com/aisystant/DS-ecosystem-development/tree/main/C.IT-Platform/C2.IT-Platform/C2.2.Architecture/Data-Governance)
