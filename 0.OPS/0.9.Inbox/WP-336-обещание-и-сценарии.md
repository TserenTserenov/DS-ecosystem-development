# WP-336: Обещание пользователю и сценарии использования

> **Связанный документ:** [Архитектура платформы IWE](WP-336-архитектура-платформы-iwe.md)

<details open>
<summary><b>Обещание платформы</b></summary>

Aisystant IWE помогает специалисту работать системнее и расти профессионально — не через абстрактные курсы, а через поддержку в реальной работе.

**Что платформа гарантирует:**

- **Ты работаешь быстрее.** IWE помнит паттерны твоих ошибок и предлагает правильный метод в нужный момент — не после, а во время работы.
- **Ты видишь свой рост.** Платформа отслеживает реальные действия и показывает, как меняется мастерство по шести измерениям (ритм, системное мышление, навыки, инструменты, интеграция, инициатива).
- **Ты контролируешь свои данные.** Ни один агент не имеет доступа ко всему сразу. Каждый сервис работает только в своей зоне. Ты можешь экспортировать, удалить или ограничить любой доступ в любой момент.
- **Интерфейс растёт вместе с тобой.** На первых ступенях Ученика — одна кнопка и рецепт. На верхних — полная конфигурация, собственные инструменты, участие в развитии методов.
- **Ты не теряешь контекст.** IWE помнит, что ты делал в прошлой сессии, какие методы освоил, что шло сложно. Следующая сессия начинается не с нуля.

**Что платформа НЕ делает:**

- Не заменяет мышление — усиливает его (экзоскелет, не автопилот)
- Не хранит переписку и личные данные в агентах
- Не навязывает темп — каждый сам решает, сколько и когда

<details>
<summary><b>Новичок — присматривается</b></summary>

**Роль:** человек, который ещё не принял решение о подписке. Изучает, что такое платформа.

### Сценарий Н.1 — Первое знакомство

1. Заходит на сайт, читает лендинг
2. Видит публичную часть Knowledge Index — статьи, гайды, примеры из сообщества
3. Пробует демо-режим: одно упражнение без регистрации
4. Видит, как работает дашборд прогресса (с заглушкой «ваши данные появятся здесь»)
5. Принимает решение: подписаться или уйти

### Сценарий Н.2 — Вопрос перед покупкой

1. Пишет в чат поддержки: «А это подходит для менеджера проектов?»
2. Менеджер поддержки отвечает с примером сценария для его роли
3. Новичок записывается на демо-звонок

**Ценность:** платформа должна снижать барьер входа — публичный контент, демо без регистрации, быстрый ответ на вопрос.

</details>

<details>
<summary><b>Пользователь — подписчик, использует платформу</b></summary>

**Роль:** участник сообщества с активной подпиской. Использует платформу для обучения и применения методов в работе. Квалификация по МИМ: уровни 1–3.

### Сценарий П.1 — Начало работы на платформе

1. Открывает IWE (браузер или VS Code)
2. Дашборд показывает: текущую ступень Ученика, bottleneck (слабое звено), рекомендованные задачи на неделю
3. Выбирает задачу → работает → коммитит артефакт
4. В конце дня: `day_close` — 3 минуты рефлексии
5. Ступень и показатели пересчитываются автоматически

### Сценарий П.2 — Застрял на задаче

1. Пишет в IWE: «Не знаю с чего начать — проект слишком большой»
2. IWE предлагает чек-лист: «Кто владелец проблемы? Какой артефакт нужен на выходе?»
3. Пользователь отвечает на 3 вопроса → появляется первый конкретный шаг
4. Результат зафиксирован → метод отмечен как применённый

### Сценарий П.3 — Работа с двух устройств

1. Работает в VS Code — IWE накапливает контекст сессии
2. Закрывает сессию → контекст синхронизируется с облаком
3. Открывает браузерную версию на другом устройстве
4. IWE: «В прошлой сессии ты работал над X. Продолжаем?»

**Ценность:** платформа снижает когнитивную нагрузку и делает прогресс видимым — не через самооценку, а через реальные данные.

</details>

<details>
<summary><b>Разработчик — подписчик, создаёт инструменты</b></summary>

**Роль:** участник сообщества с активной подпиской и техническими навыками. Создаёт сервисы, руководства, расширения для платформы. Квалификация по МИМ: уровни 1–3.

### Сценарий Р.1 — Создание пользовательского сервиса

1. Обнаружил, что стандартный чек-лист не подходит для его работы
2. Открывает SDK платформы, создаёт `my-review-service`
3. Регистрирует в Coordinator с явным scope: `["day_close", "iwe_session_end"]`
4. Сервис начинает работать в его IWE
5. Решает поделиться → публикует в Marketplace с описанием и лицензией

### Сценарий Р.2 — Форк и адаптация существующего сервиса

1. Находит в Marketplace сервис «Weekly metrics summary»
2. Форкает, меняет под свои метрики
3. Тестирует — работает
4. Оставляет себе или возвращает улучшение автору через PR

### Сценарий Р.3 — Участие в развитии методов

1. Замечает, что его метод работы с большими проектами даёт высокий success_rate
2. Платформа: «Хочешь оформить это как руководство?»
3. Оформляет метод в Pack-формат через IWE
4. Метод публикуется в Knowledge Index — другие пользователи находят и применяют

**Ценность:** разработчики расширяют платформу изнутри, а платформа обеспечивает им аудиторию и attribution.

</details>

<details>
<summary><b>Член сообщества — социализация и участие</b></summary>

**Роль:** активный участник клуба сообщества (systemsworld.club или внутренняя площадка). Ценит общение, обмен опытом, совместную работу.

### Сценарий С.1 — Обсуждение в клубе

1. Пишет пост в сообществе: «Как вы справляетесь с большим WIP?»
2. Получает 5 ответов с разными подходами
3. Лучший ответ оформляет в capture → добавляется в его IWE

### Сценарий С.2 — Участие в групповом проекте

1. Организатор создаёт группу для работы над совместным артефактом
2. Член сообщества присоединяется, получает свою роль и задачи
3. Работает через общий IWE-workspace
4. Артефакт группы публикуется в Knowledge Index

### Сценарий С.3 — Нетворкинг на событии

1. Платформа анонсирует онлайн-встречу по теме
2. Участник регистрируется, приходит
3. После встречи — краткое резюме автоматически добавляется в его IWE как capture

**Ценность:** платформа поддерживает не только индивидуальный рост, но и социальный капитал участника.

</details>

<details>
<summary><b>Наставник — проверка ДЗ и контакт с участниками</b></summary>

**Роль:** опытный участник (или сотрудник), который проверяет домашние задания, даёт обратную связь, поддерживает новичков.

### Сценарий Н.1 — Проверка домашнего задания

1. Открывает очередь заданий на проверку в LMS-интерфейсе
2. Видит работу участника: артефакт + самооценка + контекст ступени
3. Оставляет структурированный комментарий: что сделано хорошо, что улучшить
4. Выставляет оценку → событие `assignment_reviewed` → у участника обновляется cp_skl

### Сценарий Н.2 — Индивидуальная сессия

1. Участник запросил помощь: «Застрял на задаче X»
2. Наставник открывает контекст участника: его ступень, bottleneck, последние артефакты
3. Проводит 30-минутную сессию разбора
4. Фиксирует краткое резюме → участник видит в своём IWE

### Сценарий Н.3 — Эскалация проблемы

1. Наставник видит: участник не активен 2 недели (нет events)
2. Платформа: «Участник X показывает признаки оттока»
3. Наставник пишет личное сообщение
4. Если нет ответа → эскалация менеджеру поддержки

**Ценность:** наставник работает не вслепую — у него есть контекст каждого участника на момент взаимодействия.

</details>

<details>
<summary><b>Разработчик руководств — создание контента</b></summary>

**Роль:** создаёт образовательный контент платформы: курсы, гайды, руководства (универсальные и персональные).

### Сценарий КР.1 — Создание нового руководства

1. Получает задачу: «Нужно руководство для ступени 2 по теме "Декомпозиция задач"»
2. Открывает Pack — читает доменные знания (source-of-truth)
3. Создаёт проекцию для ступени 2: упрощённый язык, конкретные примеры, чек-лист
4. Проходит ревью у наставника
5. Публикуется в LMS с тегами ступени и темы

### Сценарий КР.2 — Обновление устаревшего руководства

1. Платформа сигнализирует: «Руководство R-045 не обновлялось 3 месяца, Pack изменился»
2. Разработчик открывает diff Pack vs руководство
3. Актуализирует за 1-2 часа
4. Коммитит → руководство автоматически обновляется у всех пользователей

### Сценарий КР.3 — Персональное руководство под пилота

1. Получает запрос: создать персональное руководство для пилота с bottleneck cp_iwe
2. Читает его профиль: ступень, domain, стиль
3. Создаёт адаптированную версию с примерами из его области
4. Доставляется в managed-канал пилота

**Ценность:** контент создаётся системно — из Pack (source-of-truth), с версионированием, с привязкой к ступеням.

</details>

<details>
<summary><b>Менеджер поддержки</b></summary>

**Роль:** первая линия контакта с пользователями. Отвечает на вопросы, решает технические проблемы, сопровождает адаптацию.

### Сценарий МП.1 — Вопрос нового участника

1. Новый участник пишет: «Не понимаю, как начать»
2. Менеджер открывает его профиль: зарегистрирован вчера, ступень не определена, 0 events
3. Отправляет персонализированный стартовый гайд (выбирает из шаблонов)
4. Ставит follow-up через 3 дня: «Как дела?»

### Сценарий МП.2 — Технический сбой

1. Пользователь сообщает: «Не загружается дашборд»
2. Менеджер видит в системе: у пользователя 0 events за последние 24 часа при обычной активности
3. Передаёт в техподдержку с контекстом: user_id, timeline, последние events
4. Информирует пользователя о статусе

### Сценарий МП.3 — Отток под угрозой

1. Алерт: «У пользователя X заканчивается подписка через 7 дней, активность снизилась»
2. Менеджер пишет письмо с конкретным предложением (не шаблонный маркетинг)
3. Если есть незавершённый курс — предлагает продление с фокусом на него

**Ценность:** менеджер поддержки работает с данными, а не вслепую — видит контекст каждого обращения.

</details>

<details>
<summary><b>Маркетолог</b></summary>

**Роль:** отвечает за привлечение новых участников, контент-маркетинг, воронку от новичка до подписчика.

### Сценарий М.1 — Контент из реальных историй

1. Запрашивает у платформы: «Какие трансформации были у участников за квартал?»
2. Платформа (с согласия участников) показывает агрегированные истории роста
3. Маркетолог выбирает 2-3 истории, оформляет как кейсы для публикации
4. Публикуется в клубе, telegram, habr

### Сценарий М.2 — A/B тест онбординга

1. Предлагает протестировать два варианта первого экрана
2. Согласовывает с командой, запускает через платформу
3. Смотрит данные: конверсия в первый `day_close` — ключевая метрика
4. По итогам — один вариант побеждает, второй архивируется

### Сценарий М.3 — Реферальная программа

1. Участник хочет пригласить коллегу
2. Получает реферальную ссылку в профиле
3. Коллега регистрируется → обоим начисляются бонусные баллы
4. Маркетолог видит воронку: клик → регистрация → первый платёж

**Ценность:** маркетинг работает с реальными данными платформы, а не догадками о пользователях.

</details>

<details>
<summary><b>Финансист-бухгалтер — биллинг и юнит-экономика</b></summary>

**Роль:** ведёт биллинг подписок, начисляет выплаты наставникам и разработчикам руководств, отслеживает юнит-экономику.

### Сценарий Ф.1 — Ежемесячный биллинг

1. Платформа автоматически формирует список: кто продлил, кто отменил, кто новый
2. Финансист проверяет нестандартные случаи (промо-коды, заморозки, споры)
3. Утверждает → платёжная система обрабатывает
4. Отчёт по биллингу за месяц в одном экране

### Сценарий Ф.2 — Начисление выплат наставникам

1. За месяц: Наставник А проверил 47 заданий, Наставник Б — 23
2. Платформа рассчитывает выплаты по ставке (из контракта)
3. Финансист проверяет, утверждает
4. Выплата через внешний сервис, запись в accounting

### Сценарий Ф.3 — Отчёт по юнит-экономике

1. Финансист запрашивает: «Юнит-экономика за апрель»
2. Платформа формирует: CAC, LTV, Churn по когортам, ARPU
3. Экспорт в Excel / Google Sheets для CFO
4. Сравнение с предыдущим месяцем и планом

### Сценарий Ф.4 — Скидки и промо

1. Маркетолог запрашивает: «Дать скидку 30% первым 50 новым»
2. Финансист настраивает промо-код с лимитом и датой истечения
3. Видит в реальном времени: использовано 34/50
4. По истечении — автоматически деактивируется

**Ценность:** биллинг полностью автоматизирован, финансист работает только с исключениями и аналитикой.

</details>

<details>
<summary><b>Организатор групп (Деканат)</b></summary>

**Роль:** управляет учебными группами, потоками, расписанием, записью участников на программы.

### Сценарий ОГ.1 — Открытие нового потока

1. Создаёт поток «Программа ЛР, Поток 7, сентябрь 2026»
2. Указывает: программа, наставник, максимум участников (30), даты
3. Открывает запись — участники видят в личном кабинете
4. При достижении лимита — закрывает запись автоматически

### Сценарий ОГ.2 — Перевод участника между группами

1. Участник пишет: «Не могу в это время, можно перевестись?»
2. Организатор видит свободные места в других потоках
3. Переводит в один клик — участник получает уведомление
4. Наставник нового потока видит нового участника со всей историей

### Сценарий ОГ.3 — Отчёт по успеваемости потока

1. В конце месяца — отчёт: у скольких участников завершены ключевые задания
2. Видит: у 8 из 30 — риск отставания (нет events за 10+ дней)
3. Передаёт список наставнику для работы с этими участниками
4. На следующей неделе — повторный срез: ситуация улучшилась у 5 из 8

**Ценность:** организатор управляет потоками системно — видит данные по каждому участнику, а не собирает их вручную.

</details>
