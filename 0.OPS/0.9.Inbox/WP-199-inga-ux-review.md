## Что сейчас есть в боте

Сейчас онбординг упрощён до 4 шагов:
1. Имя — берётся автоматически из Telegram
2. Длительность занятия — кнопки 5 / 15 / 25 мин
3. Время занятий — пользователь вводит HH:MM
4. Дата старта — кнопки Сегодня / Завтра / Послезавтра

Профиль (род занятий, интересы, мотивация, цели) — собирается отдельно через команду `/profile`, не во время онбординга. Диагностики и определения состояния пока нет. Развилки 1д/14д нет — все идут на 14 дней.

Возвращающийся пользователь (повторный `/start`) видит: приветствие + текущий режим + статистика (дней активности, streak) + меню кнопок.

В новой версии онбординга к этим шагам добавятся диагностика и персонализация из твоих макетов.

### Бот после онбординга

Для большинства пользователей бот — **основной инструмент**. После онбординга они остаются в боте: занимаются по расписанию, задают вопросы, смотрят прогресс. Онбординг — это главный сценарий, и твои макеты покрывают именно его.

Для продвинутых пользователей (кто дорос до полного IWE — claude.ai, Cursor, GitHub) бот дополнительно поможет подключить эти инструменты. Но это отдельный сценарий, он не влияет на текущие макеты онбординга.

## Что корректируем

### Четыре состояния (Хаос / Тупик / Поворот / Развитие)

Эти состояния — часть нашей доменной модели, ты их правильно использовала. Уточняем, как они связаны с внутренней системой:

- **Хаос** — пользователь без ритма и целей (внутри = ступень 0)
- **Тупик** — есть ритм, но нет прогресса (внутри = ступень 1)
- **Поворот** — осознание проблемы, начало изменений (внутри = переход между ступенями)
- **Развитие** — устойчивый рост (внутри = ступень 2 и выше)

**Для макетов это значит:** внутренние термины (ступень 0, ступень 2) пользователю не показываем. Но вопрос по формулировкам для UI: «Хаос» и «Тупик» могут звучать жёстко для первого контакта. Если у тебя есть идеи для более мягких формулировок — предложи (см. вопрос 4 ниже).

### Развилка 1 день / 14 дней

Решили так: **пользователь выбирает сам**, но мы даём рекомендацию на основе диагностики.

- Если диагностика показала Хаос → рекомендуем «Погрузись за 14 дней»
- Если Тупик или выше → рекомендуем «Попробуй за 1 день»
- Финальный выбор всегда за пользователем

**Для макетов:** на экране после диагностики — две кнопки с подсказкой «Мы рекомендуем: ...». Можешь выделить рекомендованный вариант визуально.

### «Сохранить прогресс» = регистрация через Ory

Если в flow есть кнопка «Сохранить прогресс» — это технически OAuth-регистрация через Ory (единый вход). Пользователь создаёт аккаунт одним нажатием. Текст кнопки «Сохранить прогресс» — хороший, оставляем. Просто учитывай: после нажатия пользователь попадёт на экран OAuth (ввод email или вход через Google), а потом вернётся в бота.

### Диагностика вместо Навигатора

В текущем боте есть роль Навигатора (помогает выбрать программу). Ты заменила его диагностикой в онбординге — это правильное решение. Диагностика на входе лучше, чем отдельная команда, которую пользователь может не найти.

### Что записывается после онбординга

По результатам диагностики + профиля в систему записывается:
- Состояние (Хаос/Тупик/Поворот/Развитие)
- Предпочтения обучения (длительность занятия, время, дата старта)
- Профиль (род занятий, интересы, мотивация, цели)
- Выбранный путь (экспресс 1д / полный 14д)

Пользователь увидит своё состояние в боте по команде `/status`. Позже — в отдельном дашборде.

## Что нужно от тебя

### 1. Учёт текущего бота

Сейчас в боте онбординг = 4 шага: имя (авто), длительность занятия, время, дата старта. Вопрос: **учитывала ли ты эти шаги в своих макетах?** Если нет — нужно вписать их в flow, чтобы новый онбординг включал и эти шаги.

### 2. Экран возвращающегося пользователя

Пользователь прошёл онбординг, ушёл на 2 недели, вернулся. Что он видит?

Варианты:
- a) Дашборд прогресса (дней активности, streak, текущее состояние)
- b) Мягкая повторная диагностика («Как дела с тех пор?»)
- c) Продолжение с последней точки

Нужен макет или описание — какой вариант лучше с точки зрения UX?

### 3. Точки замера KPI в flow

У нас целевые метрики воронки:
- Step0→Step3: ≥60% доходят
- Step3→Step6: ≥80% доходят
- OFFER→выбор сценария: ≥40%
- DAY1→DAY2: ≥25–35% возвращаются

Нужно понять: **какие экраны в твоём flow соответствуют каким шагам?** Просто пометь на flow chart или в Figma — где Step0, где Step3, где Step6 и т.д.

### 4. Формулировки состояний для интерфейса

Сейчас в модели: Хаос / Тупик / Поворот / Развитие. Для внутренней логики это точно, но для пользователя на первом экране «Хаос» и «Тупик» могут отпугнуть.

Вопрос: **есть ли у тебя идеи для более мягких/вовлекающих формулировок?** Например, «Старт» вместо «Хаос», «Пауза» вместо «Тупик»? Или считаешь, что честные названия работают лучше?

### 5. Приоритеты экранов для MVP

20+ экранов — для пилота на 5–10 пользователей нужен минимальный набор.

Вопрос: **какие экраны обязательны для первого запуска, а какие можно добавить позже?** Раздели на две группы: «MVP» и «потом».
