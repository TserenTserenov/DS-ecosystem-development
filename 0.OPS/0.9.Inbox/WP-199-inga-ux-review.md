## Часть 1: Решаем сами (доменная модель + архитектура)

> Эти вопросы не для Инги — мы фиксируем решения и передаём ей как входные данные.

### 1. Маппинг состояний на ступени Pack

Четыре состояния (Хаос / Тупик / Поворот / Развитие) → ступени мастерства из PACK-personal (Случайный 0 → Практикующий 1 → Систематический 2 → Дисциплинированный 3 → Проактивный 4).

**Предложение:**
- Хаос → ступень 0 (Случайный)
- Тупик → ступень 1 (Практикующий, но застрял)
- Поворот → переход между ступенями
- Развитие → ступень 2+ (Систематический и выше)

**Решение:**
- Хаос → ступень 0 (Случайный)
- Тупик → ступень 1 (Практикующий, но застрял)
- Поворот → переход между ступенями (не отдельная ступень, а процесс)
- Развитие → ступень 2+ (Систематический и выше)
- В UI показываем пользователю его термины (Хаос/Тупик/Поворот/Развитие). Маппинг на ступени 0–4 — внутренний, для ЦД и аналитики.

### 2. Какие данные диагностики → ЦД

Диагностика в боте собирает 5–7 ответов. Нужно определить, какие поля ЦД (WP-151) заполняются по результатам онбординга.

Диагностическая карта R28 (5 уровней): состояния → убеждения → привычки → среда → характеристики.

**Решение:**
Из онбординга в ЦД записываются:
- `IND.3.11.01` Диагностическое состояние → одно из 4 значений (chaos/stuck/turn/development)
- `IND.3.4` Ступень ученика → определяется по результатам диагностики (0–4)
- `IND.1.5` Стиль подачи → study_duration + schedule_time (уже собираются в текущем онбординге)
- `IND.2` Собранные данные → occupation, interests, motivation, goals (текстовые поля из онбординга)

Не пишем в ЦД: промежуточные ответы на диагностические вопросы (хранятся в логах бота, не в ЦД).

### 3. Критерий развилки 1 день / 14 дней

В flow есть развилка: короткий путь (1 день) и длинный (14 дней). Нужно определить бизнес-логику: по какому критерию пользователь попадает на каждый путь.

**Решение:**
- Выбор пользователя, не автоматика. На экране после диагностики: «Попробуй за 1 день» (экспресс) или «Погрузись за 14 дней» (полный курс).
- Критерий подсказки: если диагностика показала ступень 0 (chaos) → рекомендуем 14 дней. Ступень 1+ → можно 1 день. Но финальный выбор всегда за пользователем.
- Для аналитики: записываем выбор в ЦД как `onboarding_path: express | full`.

### 4. Связь онбординга с ЦД

Данные из онбординга должны попасть в ЦД пользователя (WP-151). Какие поля? Видит ли пользователь своё «состояние» в ЦД после прохождения?

**Решение:**
- Запись в ЦД происходит при `confirming_profile` (последний шаг онбординга).
- Поля: см. решение п.2 (IND.3.11.01, IND.3.4, IND.1.5, IND.2).
- Пользователь видит своё состояние (Хаос/Тупик/Поворот/Развитие) на экране результата. Термины ступеней (0–4) не показываем — только пользовательские метафоры.
- После онбординга пользователь может посмотреть своё состояние в ЦД-табло (когда оно будет готово, Ф6). До Ф6 — только в боте по команде `/status`.

## Часть 2: Вопросы для Инги (UX-решения)

> Эти вопросы требуют UX-экспертизы Инги.

### 5. Экран возвращающегося пользователя

Пользователь уже проходил онбординг, возвращается через 2 недели. Какой экран он видит? Варианты: повторная диагностика, дашборд прогресса, продолжение с последней точки.

**Вопрос для Инги:** Как выглядит экран возврата? Есть ли он в текущих макетах?

### 6. Точки замера KPI в flow

Целевые метрики: Step0→Step3 ≥60%, Step3→Step6 ≥80%, OFFER→сценарий ≥40%, DAY1→D2 ≥25–35%.

**Вопрос для Инги:** Где в текущем flow расставлены точки замера? Какие экраны соответствуют каким шагам воронки?

### 7. Приоритеты экранов для MVP

20+ экранов в Figma. Для Ф3 (пилот-бот) нужен минимальный набор.

**Вопрос для Инги:** Какие экраны критичны для MVP (пилот на 5–10 пользователей), а какие можно отложить?
