## Контекст

Инга подготовила flow chart онбординга (Ф0) и Figma-макеты на 20+ экранов (Ф1). Сейчас фаза Ф2 — согласование модели с Pack. Нужно выровнять UX-решения с доменной моделью платформы.

## Открытые вопросы (ждём ответ Инги)

### Маппинг состояний (пп. 7–10)

**7. Четыре состояния пользователя в UX → маппинг на доменную модель:**

Инга предложила 4 состояния: Хаос / Тупик / Поворот / Развитие.
В Pack (PACK-personal) модель ступеней: Случайный (0) → Практикующий (1) → Систематический (2) → Дисциплинированный (3) → Проактивный (4).

**Вопрос:** Как соотносятся? Предложение:
- Хаос → ступень 0 (Случайный)
- Тупик → ступень 1 (Практикующий, но застрял)
- Поворот → переход между ступенями
- Развитие → ступень 2+ (Систематический и выше)

**8. Экран диагностики — какие данные собираем?**

В Figma диагностика = 5–7 вопросов. Нужно увязать с диагностической картой R28 (5 уровней: состояния → убеждения → привычки → среда → характеристики).

**Вопрос:** Достаточно ли вопросов? Какие уровни диагностической карты покрывает текущий flow?

**9. Развилка 1 день / 14 дней — критерий выбора?**

В flow есть развилка: короткий путь (1 день) и длинный (14 дней).

**Вопрос:** По какому критерию пользователь попадает на каждый путь? Это его выбор или автоматика на основе диагностики?

**10. Фиксация результата — что видит пользователь после онбординга?**

**Вопрос:** Что именно показывается на «экране результата»? Связь с ЦД-табло (WP-151)?

### Дополнительные вопросы (пп. 12–13)

**12. Экран возвращающегося пользователя:**

Пользователь уже проходил онбординг, возвращается через 2 недели.

**Вопрос:** Какой экран он видит? Повторная диагностика или дашборд прогресса?

**13. Связь с ЦД (цифровым двойником):**

Данные из онбординга должны попасть в ЦД пользователя (WP-151).

**Вопрос:** Какие поля ЦД заполняются по результатам онбординга? Видит ли пользователь своё «состояние» в ЦД после прохождения?

## Замечания от автора (Церен)

1. **Терминология в UI:** В интерфейсе бота использовать термины пользователя (Хаос/Тупик/Поворот/Развитие), а не доменные (ступень 0–4). Маппинг — внутренний.
2. **KPI воронки:** Step0→Step3 ≥60%, Step3→Step6 ≥80% — эти метрики заложены в flow? Где точки замера?
3. **Figma → реализация:** 20+ экранов — нужны приоритеты. Что в MVP (Ф3 pilot), что позже?
