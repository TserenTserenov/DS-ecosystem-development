## Контекст: что такое Knowledge Gateway

**L2 (Platform)** — централизованный MCP-сервер платформы. ZP, FPF, SPF, курсы, руководства, платформенные Pack. Хостинг Aisystant, CF Workers. Уже работает (`knowledge-mcp.aisystant.workers.dev`, 5400+ docs).

**L4 (Personal)** — пользовательский MCP. Личные Pack-*, DS-*, заметки. Данные **у пользователя** (BYOB). Backend: Neon / Supabase / SQLite.

**Knowledge Gateway** — локальный прокси, объединяющий L2 + L4 в единый MCP-endpoint для AI-клиента (Claude Code, Cursor и др.). Принцип: один endpoint → один MCP для всего.

```
Клиент (Claude Code / Cursor / etc.)
    │ один MCP-endpoint
    ▼
Knowledge Gateway (локальный процесс)
    ├── L2: Platform Knowledge MCP (remote, наш)
    │   ZP, FPF, SPF, платформенные Pack, courses, guides
    └── L4: Personal Knowledge MCP (local или remote пользователя)
        Pack-*, DS-*, личные заметки
        Backend: Neon / Supabase / SQLite
    merge результатов по score → единый ответ
```

**MCP Hub** (ADR-018) — надстройка: Registry + Gateway + Provisioner. Единый endpoint `mcp-hub.aisystant.com` вместо ручной конфигурации каждого MCP. Поглощает «Apps SDK & Маркетплейс».

## Блок В: АРХИТЕКТУРНЫЕ — нужно решение

### В1. Gateway — архитектура процесса

**Вопрос:** Knowledge Gateway — отдельный процесс или встроен в personal-mcp?

**Контекст:**
- Сейчас personal-mcp (Composer) работает отдельно на машине пользователя
- Gateway нужно: (a) запрашивать L2, (b) запрашивать L4, (c) merge результатов
- Вариант «встроен» = одна точка входа, меньше частей
- Вариант «отдельный» = независимый деплой, можно обновлять без касания personal-mcp

**Наша рекомендация:** встроен в personal-mcp (меньше частей для пользователя).

**Вопрос архитектору:** согласен? Есть ли аргументы за отдельный процесс?

### В2. Embeddings L4: где вычислять?

**Вопрос:** Где генерировать embeddings для личного Pack пользователя при индексации?

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **CF AI** (централизованно) | Бесплатно. Единая модель. Уже используем для L2 | Пользователь отправляет данные на CF. Противоречит BYOB-принципу? |
| B | Локально (llama / nomic-embed-text) | Полная приватность. Оффлайн | Нужен GPU или Ollama. Сложный setup |
| C | OpenAI / Anthropic API | Качество. Единый стандарт | Платно. Зависимость от вендора |
| D | Neon pgvector + pgai (pgsql function) | Всё в БД. Один стек | Ограниченная модель. Нужен compute |

**Наша рекомендация:** CF AI для MVP (единообразие с L2, бесплатно).

**Вопрос архитектору:** CF AI подходит если пользователь использует SQLite-backend (локальный)? Нужен ли отдельный путь для тех, кто не хочет отправлять данные на CF?

### В3. Merge стратегия L2 + L4

**Вопрос:** Как объединять результаты поиска из L2 и L4?

| # | Стратегия | Описание | Плюсы | Минусы |
|---|-----------|----------|-------|--------|
| A | **Простой merge по score** | Объединить, sort by relevance score | Просто, работает для MVP | L4 может перебить L2 (разные модели embedding) |
| B | Weighted merge | L2 weight × L4 weight, настраиваемые | Балансировка источников | Нужна калибровка весов |
| C | Cascade | Сначала L4, если нет — L2. Либо L2 + L4 независимо | Персональное имеет приоритет | Может пропустить релевантный L2-контент |
| D | Re-rank | Оба источника → LLM re-ranking → топ-K | Лучшее качество | Доп. LLM-вызов. Цена |

**Наша рекомендация:** простой merge для MVP (A), затем re-rank при масштабировании.

**Вопрос архитектору:** какой порог score считать «хорошим результатом»? Есть ли практика нормализации scores между разными embedding-моделями?

### В4. Подключение Gateway к клиенту: транспорт

**Вопрос:** Как клиент (Claude Code, Cursor, Claude Desktop) подключается к Gateway?

| # | Транспорт | Описание | Плюсы | Минусы |
|---|-----------|----------|-------|--------|
| A | **stdio** | `npx knowledge-gateway` — локальный процесс | Стандарт Claude Code. Нет сетевого стека. Просто | Только локально. Каждый клиент запускает свой |
| B | SSE | HTTP/SSE сервер на localhost | Несколько клиентов одновременно | Нужен порт. Ещё один сервис на машине |
| C | Через MCP Hub (remote) | Hub проксирует к пользовательскому Gateway | Единый endpoint | Gateway должен быть публично доступен или VPN |

**Наша рекомендация:** stdio для MVP (стандарт Claude Code, минимальная сложность). SSE для multi-client сценариев позже.

**Вопрос архитектору:** подходит ли stdio если пользователь работает на нескольких машинах? Как это обычно решается?

### В5. Миграция knowledge-mcp: сразу или поэтапно?

**Вопрос:** Текущий `knowledge-mcp.aisystant.workers.dev` — мигрировать в L2 Platform MCP сразу или поэтапно?

**Контекст:**
- Сейчас: один монолитный MCP (9 sources, 5400+ docs)
- Цель: knowledge-mcp становится L2 в MCP Hub
- Зависимость: DS-Knowledge-Index-Tseren и aist-bot-docs как источники — их перевод в L4 = breaking change для текущих сессий Claude Code

**Вариант A (сразу):** knowledge-mcp = L2, весь текущий контент туда, DS-Knowledge-Index-Tseren → L4.
- Риск: breaking change для всех текущих пользователей

**Вариант B (поэтапно, рекомендуем):**
- Ф1: Выделить L2 из текущего knowledge-mcp (платформенный контент: ZP, FPF, SPF, курсы)
- Ф2: Добавить L4 + Gateway для пользовательского контента
- DS-Knowledge-Index-Tseren мигрирует в L4 после Ф2

**Вопрос архитектору:** согласен с поэтапным? Есть ли риск несовместимости schema между Ф1 и Ф2?

### В7. Community MCP: публикация Pack другим пользователям

**Вопрос:** Может ли пользователь A опубликовать свой Pack-MCP для пользователя B?

**Контекст (ADR-018):**
- User MCP: доступ только владельцу + явный share
- Community MCP: автор публикует → доступно другим по тиру + установке

**Открытые вопросы реализации:**
1. Где хранятся данные Community MCP? На инфраструктуре автора (BYOB) или платформы?
2. Как платформа гарантирует SLA для Community MCP, который живёт на машине автора?
3. Модерация перед публикацией или after-the-fact?

**Варианты:**

| # | Вариант | Описание | Плюсы | Минусы |
|---|---------|----------|-------|--------|
| A | **Автор хостит** (чистый BYOB) | Community MCP = URL к серверу автора, зарегистрирован в Registry | Нет нагрузки на платформу. BYOB-консистентно | Нет гарантий uptime. Автор может удалить |
| B | Платформа хостит (snapshot) | При публикации — snapshot контента → платформенный Neon | Гарантированный доступ | Дублирование данных. Платформа хранит чужое |
| C | Гибрид: snapshot L2 + live L4 | Платформа кешит L2-знания, L4 = live от автора | Баланс | Сложнее |

**Наша рекомендация:** Вариант A для MVP (минимальная инфраструктура), затем snapshot при проблемах uptime.

**Вопрос архитектору:** подводные камни Варианта A? Как другие решают проблему uptime User-hosted Community MCP?

### В5. Knowledge Gateway — с какого тира?

> Это продуктовое решение, не архитектурное. Выносим на обсуждение для согласованности.

**Варианты:**
- **T3 (Персонализация):** Gateway = инструмент персонализации обучения. T3 — тир, где появляется Digital Twin read/write.
- **T4 (Созидание):** Gateway = инструмент создателя. Нужен GitHub, BYOB-бэкенд.

**Наша склонность:** T3 (чем раньше пользователь получает персонализированный контекст, тем выше ценность), но без платёжного барьера для T3 MVP.

**Наша склонность:** CLI для MVP, Web App settings позже (зависит от готовности Web App).
