## Мировая платформа (Track B)

### Ключевое архитектурное решение: Neon не нужен для Track B

CF Workers могут подключаться к любому Postgres (включая Cloud SQL и Vultr Managed Postgres) напрямую через `postgres.js` + **Cloudflare Hyperdrive** (TCP через `cloudflare:sockets`, добавлено Cloudflare в 2023).

| | Старый подход | Новый подход (Track B) |
|--|---------------|------------------------|
| Драйвер в CF Workers | `@neondatabase/serverless` (HTTP-транспорт) | `postgres.js` (TCP через cloudflare:sockets) |
| Пулинг соединений | Neon pooler | **Hyperdrive** (Cloudflare, входит в Paid план $5/мес) |
| База данных | Только Neon (нужен HTTP API) | Любой Postgres: Cloud SQL, Vultr, и т.д. |
| Neon нужен? | Да | **Нет** |

**Neon (12 БД) остаётся только в Track A** — там существующие данные, ничего не трогаем.
Track B стартует с нуля: новые таблицы сразу в Cloud SQL или Vultr Managed Postgres.

### ✅ Вариант 2 — GKE Standard (EU) [ВЫБРАНО — встреча 14, 7 мая]

> Google Managed Kubernetes + Cloud SQL. База — в том же Google (Cloud SQL), это логично и нет egress между K8s и БД.
> SOC2 / GDPR attestation — важно для YC и корпоративных клиентов.
> **GKE Standard, не Autopilot** — Autopilot имеет ограничение 0.5 vCPU min per pod (дорого при нашем MVP). Standard = ручное управление нодами, без ограничения CPU per pod.
> **$300 кредита / 90 дней** — платформа запускается за $0 первый квартал.
> **Старт 18 мая**: Паша создаёт кластер, Cloud SQL, приватная сеть + Cloud Router + LB + API Gateway. Деплой через Werf (изучить до 18 мая). Отдельная GitHub-организация для Track B репозиториев.

#### ⭐ Бесплатные первые 3 месяца

Google предоставляет $300 кредита на 90 дней при первой регистрации GCP аккаунта ([cloud.google.com/free](https://cloud.google.com/free)).
При MVP стоимости ~$90–120/мес кредита хватает на весь первый квартал — платформа запускается за $0.
Кто активирует: нужен аккаунт без истории GCP billing. Зарегистрировать до запуска кластера.

#### Autopilot vs Standard — два режима GKE

| | GKE Autopilot | GKE Standard |
|--|---------------|--------------|
| **Управление нодами** | Google управляет автоматически | Вы управляете нодами вручную |
| **Минимум CPU на под** | **0.5 vCPU** ⚠️ — даже если под использует меньше | Нет ограничения — определяется нодой |
| **Стоимость при 4–6 малых подах** | 4 пода × 0.5 vCPU = 2 vCPU min = ~$84/мес только CPU | 1 нода e2-standard-2 (2vCPU, 8GB) = ~$48/мес, все поды на ней |
| **Масштабирование** | Автоматическое ✅ | Ручное (добавлять ноды) |
| **Подходит для** | Много нагрузки, нужен autoscaling | MVP с небольшим числом сервисов |

**Вывод по Андрею:** Autopilot выгоден при высокой и переменной нагрузке. При нашем MVP (4–6 маленьких подов) GKE Standard на одной ноде e2-standard-2 дешевле примерно на $36/мес.

**Рекомендация:** GKE Standard для MVP → переключиться на Autopilot при росте >1K пользователей.

### Изменения в коде CF Workers для Track B

Track B CF Workers пишутся с нуля (новые сервисы). Использовать `postgres.js` вместо `@neondatabase/serverless`:

```typescript
// Track A (старый код — оставляем как есть)
import { neon } from '@neondatabase/serverless';
const sql = neon(env.DATABASE_URL);

// Track B (новый код)
import postgres from 'postgres';
const sql = postgres(env.DATABASE_URL); // Hyperdrive автоматически подхватывает
```

Hyperdrive настраивается в `wrangler.toml` как binding — CF Workers его подхватывает автоматически, код не меняется при добавлении/смене пула.

Track A CF Workers (`@neondatabase/serverless`) — **не трогаем**. Они подключены к Neon и работают.

### Hetzner tsekh-1 — вспомогательная роль (не Track A)

Hetzner tsekh-1 — это личный выделенный сервер Тсерена в Германии (NixOS + ZFS mirror). Это **не российская инфраструктура** и не обязательный компонент Track A. Ильшат не будет им управлять.

Что на него можно перенести — вспомогательные задачи:

| Что перенести на Hetzner | Зачем | Вместо чего |
|--------------------------|-------|-------------|
| **GitHub Actions self-hosted runner** | Сборка Docker-образов и деплой в VK Cloud K8s — быстрее и дешевле платных минут GitHub | GitHub-hosted runner (платные минуты) |
| **Резервные копии Neon** | pg_dump по расписанию → хранение на ZFS (версионированные снапшоты) | Только бэкапы Neon (PITR 7 дней) |
| **Staging-среда Track A** | Docker Compose — тестировать перед деплоем в VK Cloud K8s | Railway preview или ручное тестирование |

**Итог по Hetzner:** не нужен для работы Track A, но полезен как CI/CD-узел и хранилище бэкапов. Стоимость (~€44/мес) уже оплачена Тсереном независимо от платформ.

### Сценарий S4: Русский пользователь переходит на мировую платформу

**Профиль:** Был на Track A (Russian), хочет перейти на Track B (World). Платит сейчас рублями, хочет переплатить через Stripe.

| Аспект | Решение | Конфликт | Техническое решение |
|--------|---------|----------|---------------------|
| **Экспорт из Track A** | Пользователь нажимает «Экспорт» | ✅ Его данные | Ф6 WP-285: endpoint экспорта в Track A (все события, заметки, баллы, достижения) → ZIP/JSON |
| **Миграция аккаунта** | Новый аккаунт в Track B (Ory EU) | 🟡 Новый user_id | RLS требует привязки к `user_id`. Old user_id из Track A не валиден в Track B. Новый аккаунт нужен |
| **Слияние identity** | Привязать его email к обоим? | 🟡 Сложно | Возможно: настроить federated identity (email — точка привязки, но из двух разных Ory-инстансов). Требует настройки OAuth |
| **Импорт данных** | Загрузить экспортированное в Track B | ✅ Через импорт-endpoint (Ф6) | Идемпотентность: повторная загрузка игнорирует дубли по `external_id` |
| **Платёж YooKassa → Stripe** | Как перейти на новый метод оплаты? | 🟡 Не automatic | Текущая подписка в YooKassa лучше отменить вручную (иначе двойное списание). Track B платит в Stripe (новый subscription_grants) |
| **Баллы / Points** | Переносятся ли баллы? | ❌ POLICY: не переносятся | Points = игровая валюта, привязана к платформе. Новая платформа = новая валюта. Пользователь может потратить баллы на Track A перед уходом, или они остаются там как archive |
| **Наставник** | Тот же наставник на обеих платформах? | 🟡 Проверить | Если наставник зарегистрирован на Track A, ему нужен отдельный аккаунт на Track B. Нет синхронизации наставников между платформами |

**Решение:** Ф6 WP-285 — стандартный формат экспорт/импорт без синхронизации.

### Сценарий S5: Резидент РФ, но хочет участвовать в резидентуре (мировой программе на английском)

**Профиль:** Живёт в России, русский язык, но хочет учиться в мировой программе (Track B), английском языке, платит рублями через VPN или ещё как.

| Аспект | Решение | Конфликт | Техническое решение |
|--------|---------|----------|---------------------|
| **Платёж** | Как платить рублями за Track B (мировую платформу)? | 🟡 Stripe не принимает RUB | Вариант 1: пользователь платит в USD/EUR через конвертер (Wise, Revolut). Вариант 2: Track B добавляет YooKassa (но зачем, если есть Stripe?). Вариант 3: Bridge-платёж (Track A платит за Track B пользователя) |
| **Политические риски** | Может ли Stripe работать с RU-резидентом? | 🔴 LEGAL | Санкции, OFAC — Stripe блокирует RU-резидентов. Даже через VPN → блокировка по IP/карте. **Требуется юрконсультация (О-6 из WP-285)** |
| **Альтернатива** | Telegram Stars (рубль-equivalent)? | 🟡 Fallback | Stars = в-приложение валюта. Пользователь покупает Stars → платит Stars в Track B. Но это не масштабируется для всех платежей |
| **Подписка** | Где хранится? | ✅ subscription_grants (#1, Track B) | Независимо от способа платежа, подписка на Track B = запись в subscription_grants мировой платформы |
| **Язык программы** | Английский материал + русский наставник? | 🟡 Найти наставника из мировой базы, который говорит по-русски | Маркер в teacher-profile: languages = ['ru', 'en']. Поиск в world-DB |

**Решение:** Требуется уточнение юрисдикции (WP-285 Ф4). Технически: Track B поддерживает несколько платежей, но политические риски управляются юристами, не инфраструктурой.

### Сценарий S6: Наставник, ведущий учеников на обеих платформах

**Профиль:** Наставник (например, Тсерен или другой сотрудник), ведёт кого-то на Track A (Russian) и кого-то на Track B (World). Нужна синхронизация расписания / платежа / отчётов?

| Аспект | Решение | Конфликт | Техническое решение |
|--------|---------|----------|---------------------|
| **Два аккаунта?** | Один наставник = два login'а (Track A + Track B) | ✅ Да | Ory A (RU) + Ory B (EU) — разные инстансы, не federated. Два отдельных аккаунта |
| **Единое расписание** | Где хранить расписание наставника? | 🟡 Разделено | Две БД, нет синхронизации. Наставник должен синхронизировать вручную (или через Telegram calendar-bot) |
| **Платежи наставнику** | Зарплата / бонусы за обе платформы | 🟡 Учёт разделён | финансовый отчёт генерируется отдельно по Track A (YooKassa-доход) и Track B (Stripe-доход). Бухгалтерия складывает вручную |
| **Отчёты о прогрессе учеников** | Доступ к данным обоих платформ одновременно? | 🟡 Нельзя | Нет общего cockpit-а. Наставник должен открывать обе платформы отдельно. Данные не объединены |
| **Решение (будущее)** | Unified Dashboard для мультиплатформенных наставников? | ❓ WP-TBD | Требуется отдельный РП: синхронизация данных наставника между платформами через API (не автоматическая, а через явный экспорт-импорт типа Ф6) |

**Вывод:** Bridge-синхронизация наставников (WP-281 отказывает: R-3 говорит «полностью независимы») ⇒ ручная работа наставников или будущий RП.

### Сценарий S7: Организация (компания) покупает подписки для своих сотрудников

**Профиль:** Company (RU-based или international) платит за годовые подписки своих сотрудников на Track B (World). Сотрудники разные: английский, русский, немецкий.

| Аспект | Решение | Конфликт | Техническое решение |
|--------|---------|----------|---------------------|
| **Платёж** | Кто платит: организация или сотрудник? | ✅ Organization | Stripe → invoice на company_email. subscription_grants (Stripe) → bulk grant для employees |
| **Язык** | Разные языки в team | 🟡 Выбор по умолчанию | Интерфейс Aisystant = English. Требуется: (а) многоязычность UI, (б) контент на других языках (пока только EN + RU) |
| **Наставник** | Кто наставляет? | 🟡 Company's internal или Aisystant? | Если internal coach → Aisystant = платформа, но без наставников. Если Aisystant nastavnik → нужны multi-language teachers |
| **Отчёты** | Company хочет видеть прогресс сотрудников | ❌ Нет API | Сейчас: каждый сотрудник сам видит прогресс. Организационный cockpit = будущий РП (B7.3 Analytics API) |
| **Region Lock** | Company в Европе, но работает с русскоговорящими удалённо | ✅ Технически OK | Track B = EU-based (Cloud SQL EU). Все работают через Ory EU. Нет geofencing |

**Решение:** Track B в ранней версии не рассчитана на организационные team-подписки. Требуется РП (B3.3 Team Billing + Analytics).

### Сценарий S8: Контент-создатель публикует курс на обе платформы

**Профиль:** Creator написал курс, хочет продать его на Track A (русская аудитория, YooKassa) и Track B (мировая аудитория, Stripe).

| Аспект | Решение | Конфликт | Техническое решение |
|--------|---------|----------|---------------------|
| **Контент** | Один курс или две версии? | 🟡 Две версии | Content таблица = разные record'ы на Track A и Track B. Синхронизация контента = ручная (РП Ф6 или будущий content-sync API) |
| **Цена** | Одна цена (RUB на Track A, USD на Track B) или разные? | 🟡 Creator решает | Pricing table (content_pricing, #9) хранится в каждой платформе отдельно. Creator устанавливает разные цены (обычно разные в разных валютах) |
| **Продажи / Royalty** | Creator видит доход с обоих платформ? | 🟡 Отдельно | Finance report на Track A (YooKassa) и на Track B (Stripe) не объединены. Creator должен складывать вручную (или делать выписку бухгалтерии) |
| **Студенты** | Кто может купить курс на Track A vs Track B? | ✅ Зависит от подписки | Student с подпиской на Track A может купить курс на Track A. Student на Track B — на Track B. Нет кросс-платформенных покупок |
| **Обновление контента** | Creator обновил курс на Track A, как синхронизировать на Track B? | 🟡 Ручное копирование | Нет автоматической синхронизации. Creator должен обновить курс в обоих местах (или использовать Ф6 экспорт-импорт с перезаписью) |

**Решение:** Content Sync API (будущий РП в B3.3 или C2.3) для creators.
