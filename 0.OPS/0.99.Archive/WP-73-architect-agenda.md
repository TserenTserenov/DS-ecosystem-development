## Блок А: КРИТИЧЕСКИЕ — нужно решение архитектора

> Без этих решений невозможно перейти к реализации. Нужно обсудить варианты и выбрать.

### А1. Изоляция данных в Neon — RLS ✅ решено 29 мар

**Источник:** [WP-73 §5.1 Q1](WP-73-aisystant-platform-architecture.md) | **Блокирует:** CRM, Billing, ЦД, Knowledge Gateway, масштабирование

**Решение (29 мар):** **RLS (Row-Level Security)**. Администраторам — доступ ко всем строкам определённых таблиц. Студентов ограничиваем на уровне сервисов (не RLS, а бизнес-логика сервиса определяет, что студент видит).

**Обоснование:** Одна БД, стандарт PostgreSQL, Directus/Metabase работают нативно. Двойной барьер: RBAC Directus + RLS Neon.

<details>
<summary><b>Рассмотренные варианты</b></summary>

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **RLS (Row-Level Security)** ✅ | Одна БД, одна схема. Directus/Metabase работают нативно. Стандарт PostgreSQL | Сложность отладки RLS-политик. Один баг = утечка данных |
| B | Schema per tenant | Полная изоляция на уровне схемы | 1000 tenant = 1000 schemas. Directus не поддерживает dynamic schema switching |
| C | DB per tenant | Максимальная изоляция | Neon per-project pricing. Cross-tenant query невозможен (Metabase). Слишком дорого |

</details>

### А2. Event Bus: механизм взаимодействия между системами ✅ направление 29 мар

**Источник:** [WP-73 §5.1 Q4](WP-73-aisystant-platform-architecture.md) | **Блокирует:** CRM↔бот, Billing→[Activity Hub](WP-109-activity-hub-lms-integration-proposal.md)→Points Engine, все новые системы

**Решение (29 мар):**
1. **Фиксировать ВСЕ события** на платформе (не только межсистемные — полный журнал событий)
2. **Реализация:** pg_notify + Neon (подтверждена)
3. **TODO:** сделать типологию входов и выходов по каждой системе — определить, сколько хранить каждое событие (retention policy)

**Следующий шаг:** Типология событий по системам → спецификация Event Bus (отдельная сессия).

<details>
<summary><b>Рассмотренные варианты</b></summary>

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **Outbox + pg_notify** ✅ | Нулевая новая инфра (всё в Neon). Transactional outbox = надёжность | pg_notify теряет сообщения если слушатель отключён. Нужен poller как fallback |
| B | RabbitMQ (сразу) | Гарантированная доставка. Dead letter queues. Готовые паттерны | +$5-10/мес. Ops overhead. Ещё один сервер |
| C | NATS | Легковесный. JetStream для persistence | Менее зрелая экосистема |

</details>

### А3. LMS Aisystant: не трогаем, строим рядом ✅ решено 29 мар

**Источник:** [WP-73 §5.1 Q7, §2.4](WP-73-aisystant-platform-architecture.md) | **Блокирует:** Web App MVP (WP-65), все LMS-зависимые сценарии, CRM↔LMS

**Решение (29 мар):** **Не трогаем LMS.** Новую архитектуру строим рядом для **новых пользователей**. Существующий LMS продолжает работать для текущей аудитории. Связь между LMS и новой архитектурой — через Event Bus (pg_notify) и общую Neon БД.

**Обоснование:** Нет миграции legacy. Чистый старт на новом стеке. Текущие пользователи не затрагиваются. Постепенная конвергенция — по мере переноса функций.

<details>
<summary><b>Рассмотренные варианты</b></summary>

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | Обёртка (API) на 12 мес | Быстро. Сохраняет работающее | Lock-in в legacy на год+. API может быть ограничен |
| B | Постепенная замена (strangler fig) | Новые фичи в новом стеке | Два стека параллельно. Сложность синхронизации |
| C | Полная замена | Чистый стек | 1000+ часов. Нереалистично |
| **D** | **Новая архитектура на новой аудитории** ✅ | Нет миграции legacy. Чистый старт | Два мира параллельно |

</details>

### А4. Web App hosting: Vue + раздельные хостинги ✅ решено 29 мар

**Источник:** [WP-73 §5.1 Q3](WP-73-aisystant-platform-architecture.md) | **Блокирует:** WP-65 Web App MVP

**Решение (29 мар):**
1. **Раздельные хостинги** — Россия и Мир (соответствует P12)
2. **Vue** (остаёмся на Vue, не переходим на React)

**Обоснование:** Vue — текущий стек команды, переход на React не оправдан. Мобильное приложение — отдельный вопрос (сценарии не определены).

<details>
<summary><b>Рассмотренные варианты</b></summary>

| Вариант хостинга | Плюсы | Минусы |
|---------|-------|--------|
| **Vercel** | SSR + Edge + AI SDK. Next.js native. CDN | Vendor lock-in. Цена при росте. React-ориентирован |
| **Railway** | Уже используем (бот). Единая платформа | Нет Edge, AI SDK, ISR |
| **CF Pages** | Бесплатно. Edge. Рядом с MCP Workers | Ограниченный SSR |

| Фреймворк | Плюсы | Минусы |
|-----------|-------|--------|
| React (Next.js) | Экосистема, путь к React Native | Полный переход стека, команда на Vue |
| **Vue (Nuxt.js)** ✅ | Текущий стек, команда знает | Путь к мобильному менее прямой |

</details>

### А5. Access Management ✅ решено 29 мар

**Источник:** [WP-183 Q1](WP-183-crm-billing-architecture-proposal.md) | **Блокирует:** архитектура доступов

**Решение (29 мар):** Отдельная подсистема **ORY Keto** (Google Zanzibar) для управления доступом. Billing ≠ Access Management — это разные подсистемы.

**План внедрения:**
1. Тестирование на боте (после запуска ORY)
2. Распространение на Aisystant (LMS)

**Связь:** ADR-014 (трёхосевая модель доступов) подтверждена. Keto = реализация ReBAC из ADR-014.

**Источник:** [WP-183 Q7](WP-183-crm-billing-architecture-proposal.md)
