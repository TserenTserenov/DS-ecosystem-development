# Архитектура общих баллов при разделении контуров РФ/мир

> **Контекст:** Встреча 5 (9 апр) — принято решение о полном разделении финансов по юрлицам и дублировании технической инфраструктуры. Сообщество (баллы, ЦД, контент) — общее, без границ.
>
> **Задача:** как реализовать единую систему баллов при двух раздельных контурах?
>
> **Полная проработка разделения:** [Regional-Split/ (B3.1.Meaning)](../../../../B.Aisystant-Ecosystem/B3.Ecosystem-Builder/B3.1.Meaning/Regional-Split/) — экономика, инфраструктура, сообщество, compliance, исследование паттернов.

---

## 1. Юридическая модель: баллы ≠ деньги

Баллы — **нефинансовая метрика вклада в сообщество**, а не валюта.

| Свойство | Деньги | Баллы |
|----------|--------|-------|
| Юридический статус | Платёжное средство | Программа лояльности / gamification |
| Владелец | Юрлицо в юрисдикции | Сообщество (платформа как оператор) |
| Трансграничная передача | Регулируется (валютный контроль) | **Не регулируется** — это не деньги |
| 152-ФЗ / GDPR | Применяется к PII | Баллы — не PII (но привязка к user — да) |
| Налогообложение | Да | Нет, пока не конвертированы в скидку |

### Юридическая конструкция

1. **Баллы = бонусная программа** платформы (аналог: мили Аэрофлота, баллы Яндекс.Плюс)
2. Оператор программы — **одно юрлицо** (международное), либо **отдельный Foundation/ассоциация**
3. Правила программы описаны в оферте платформы, которую принимает каждый участник
4. Конвертация баллов в скидку происходит **в момент покупки**, в юрлице покупки

**Аналогия: авиамили.** United Airlines (USA) и ANA (Япония) в Star Alliance — общие мили, разные юрлица. Участник копит мили в одной программе, тратит в любой. Финансы авиакомпаний разделены полностью.

### Ключевой юридический ход: псевдонимизация

В `user_events` и `point_transactions` **нет PII**. Только `ory_id` (UUID), `event_type`, `points`, `timestamp`. Это **псевдонимизированные данные** — и по 152-ФЗ, и по GDPR они не требуют локализации, если ключ деанонимизации (ory_id → ФИО) хранится **в контуре**.

```
КОНТУР РФ                    ОБЩИЙ СЛОЙ                 КОНТУР МИР
┌────────────────┐     ┌──────────────────┐     ┌────────────────┐
│ ory_id → ФИО   │     │ ory_id → баллы   │     │ ory_id → ФИО   │
│ ory_id → email  │     │ ory_id → events  │     │ ory_id → email  │
│ ory_id → телефон│     │ ory_id → ЦД      │     │ ory_id → адрес  │
│ (PII, 152-ФЗ)  │     │ (псевдоним, ок)  │     │ (PII, GDPR)    │
└────────────────┘     └──────────────────┘     └────────────────┘
```

### Юридические требования по юрисдикциям

| Требование | РФ (152-ФЗ) | ЕС (GDPR) | Решение |
|-----------|-------------|-----------|---------|
| Локализация PII | ФИО, email, телефон — в РФ | PII — в ЕС | PII в контурных БД. Баллы — по ory_id (псевдоним) |
| Право на удаление | Есть | Есть (Art. 17) | Удаление PII в контуре. Баллы → анонимизация (ory_id → null) |
| Трансграничная передача | Согласие + адекватность | SCC / adequacy | Баллы = не PII. Events содержат ory_id, не ФИО |
| Согласие на обработку | Да | Да | В оферте платформы: «участие в программе лояльности» |
| Налогообложение скидки | Скидка >4000₽/год — НДФЛ (спорно) | Зависит от юрисдикции | Caps по квалификации ограничивают сумму |

## 2. Технические варианты

### Вариант A: Единая база (рекомендуемый на старте)

```
КОНТУР РФ                     КОНТУР МИР
┌──────────────┐              ┌──────────────┐
│ Бот-RU       │              │ Бот-EU       │
│ CRM-RU       │              │ CRM-EU       │
│ Ory-RU       │              │ Ory-EU       │
│ ЮKassa       │              │ Stripe       │
└──────┬───────┘              └──────┬───────┘
       │ ingest_event                │ ingest_event
       │                             │
       └─────────────┬───────────────┘
                     ▼
           ┌─────────────────┐
           │  Neon EU (общая) │
           │                  │
           │ user_events      │  ← append-only, все контуры
           │ point_rules      │  ← единые правила
           │ point_transactions│ ← единый журнал
           │ point_balances   │  ← единый баланс
           └─────────────────┘
                     │
              Points Engine
              (один инстанс)
```

| + | - |
|---|---|
| Простота, консистентность, один источник правды | Зависимость от канала RU→Neon EU |
| Один Points Engine, одна кодовая база | При блокировке Neon из РФ — нужен буфер |
| Нет проблем с consistency | |

**Митигация:** outbox + retry в RU-контуре. События доставляются eventually. Баллы = вычисляемая проекция (replay по events) — задержка на часы/дни не проблема.

### Вариант B: Федерация (при разделении баз)

```
КОНТУР РФ                     КОНТУР МИР
┌──────────────┐              ┌──────────────┐
│ PG-RU        │              │ Neon-EU      │
│ user_events  │◄────────────►│ user_events  │
│ point_txns   │  CDC / sync  │ point_txns   │
│ point_balance│              │ point_balance │
└──────────────┘              └──────────────┘
       │                             │
  Points Engine-RU            Points Engine-EU
       │                             │
       └─────────────┬───────────────┘
                     ▼
            Reconciliation Job
            (сверка балансов)
```

| + | - |
|---|---|
| Автономность контуров | CDC, конфликты, eventual consistency |
| Нет зависимости от канала | Двойной код, двойные миграции |
| | Сложность сверки для финансово-подобных данных |

**Когда переходить:** если канал RU→Neon станет ненадёжным ИЛИ 152-ФЗ потребует локализации event-данных.

### Вариант C: Community Layer (целевой)

```
КОНТУР РФ           COMMUNITY LAYER           КОНТУР МИР
┌───────────┐     ┌──────────────────┐     ┌───────────┐
│ Финансы   │     │ Points Engine    │     │ Финансы   │
│ CRM-RU    │     │ ЦД (profiles)    │     │ CRM-EU    │
│ Ory-RU    │     │ Knowledge base   │     │ Ory-EU    │
│ PG-RU     │     │ Форум/чат        │     │ Neon-EU   │
└─────┬─────┘     │ Контент (курсы)  │     └─────┬─────┘
      │           └────────┬─────────┘           │
      │                    │                     │
      └────────────────────┴─────────────────────┘
         события              события
```

Community Layer как отдельный сервис/база:
- Принимает события из обоих контуров
- Хранит баллы, ЦД, knowledge — общие данные сообщества
- **Не хранит** финансовые и PII-данные
- Идентификация — по `ory_id` (псевдонимизация)

| + | - |
|---|---|
| Чистое разделение (финансы ↔ сообщество) | Дополнительный инфраслой |
| Юридически прозрачно | Нужен на масштабе |
| Масштабируется на N контуров | |

## 3. Рекомендация: эволюционный путь

| Этап | Архитектура | Триггер перехода |
|------|-------------|------------------|
| **Сейчас** | **Вариант A** — одна Neon, один Points Engine | — |
| При проблемах с каналом | A + outbox/buffer в RU | Neon недоступна из РФ >1% времени |
| При масштабе (500+ активных) | **Вариант C** — Community Layer | Юридическое требование или 2 базы |

**Почему не B:** федерация двух Points Engine — это CRDT/eventual consistency для финансово-подобных данных. Сложность не оправдана. Лучше сразу Community Layer (C), где данные общие по определению.

## 4. Сценарии

### 4.1. Участник из РФ платит баллами

```
Участник (ory_id: abc) → Бот-RU → /buy_with_points "Семинар X"
  → CRM-RU проверяет: GET point_balances WHERE ory_id='abc'
  → Баланс 500, семинар стоит 200 баллов
  → CRM-RU: INSERT point_transactions (type='spent', -200, source='crm-ru')
  → CRM-RU: выдаёт доступ
  → Финансовая проводка: 200 баллов × $0.01 = $2 скидка по РФ-юрлицу
```

Скидка оформляется как бонус программы лояльности РФ-юрлица. Баллы начислены глобально, скидка — локальная.

### 4.2. Участник переехал из РФ за границу

1. Баллы НЕ переносятся — они уже глобальные (ory_id = тот же)
2. Меняется: платёжный контур (ЮKassa → Stripe)
3. Меняется: юрлицо оплаты
4. НЕ меняется: баланс баллов, история, ЦД, knowledge

### 4.3. Событие из RU-контура при недоступном Neon

```
Бот-RU → ingest_event → Neon EU недоступна
  → outbox (локальный PG-RU или файл)
  → retry с exponential backoff
  → событие доставлено eventually
  → Points Engine пересчитывает при replay
```

### 4.4. Два Ory → два ory_id?

Если два инстанса Ory (RU + EU) — у одного человека могут быть два разных ory_id.

**Решение: identity federation.**
- Один Ory = **primary** (EU), второй = **federated**
- Либо: `identity_links` таблица: `ory_id_ru ↔ ory_id_eu`
- Для Points Engine: привязка к **canonical ory_id** (из primary Ory)
- При создании аккаунта в RU-Ory — автоматически создаётся link к EU-Ory

Пока Ory одна (EU) — проблемы нет. Решение при появлении RU-Ory — через ADR-008 (ory_id = единственный ID).

## 5. Подготовка к эволюции (закладываем сейчас)

| Что заложить в Варианте A | Зачем |
|--------------------------|-------|
| `source_region` поле в `point_transactions` (`'ru'` / `'world'`) | Аудит: откуда событие. Разделение при переходе к C |
| `source_system` поле в `user_events` (уже есть) | Трассировка до источника |
| Outbox-паттерн в RU-адаптерах | Устойчивость к недоступности Neon |
| `canonical_ory_id` в identity_links | Готовность к федерации Ory |
