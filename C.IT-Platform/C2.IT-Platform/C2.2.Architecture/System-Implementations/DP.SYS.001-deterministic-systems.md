## 5. Заметочник (SYS.015)

**Назначение:** Быстрый захват мыслей и заметок из любого интерфейса (Telegram, CLI, Web) в единое хранилище.

**Механика:**
- Пользователь вводит `.` + текст в любом интерфейсе → система сохраняет текст в inbox (`fleeting-notes.md`)
- Детерминированная маршрутизация: распознаёт префикс, извлекает текст, записывает в файл
- LLM не требуется → прямой путь Интерфейс (слой 4) → MCP (слой 2), минуя ИИ-систему

**Жизненный цикл заметки:**
```
Ввод (". текст") → Захват (SYS.015) → Inbox → Обработка (strategy session) → Архив или План
```

**Глобальные префиксы интерфейса:**

| Префикс | Маршрут | Система |
|---------|---------|---------|
| `.` | Заметка → inbox | SYS.015 Заметочник |
| `?` | Вопрос → ИИ-консультант | AISYS.009 Консультант |

**Расширения (будущее):**
- Авто-тегирование заметок (→ переход в ИИ-систему)
- Группировка по темам
- Срок жизни (исчезающие = автоудаление после обработки)

## 6. MCP-архитектура знаний

### 6.1. Принцип: 3 MCP по типу владения

| MCP-сервер | Данные | Хостинг | Auth | Пользователи |
|------------|--------|---------|------|---------------|
| **Knowledge MCP** (SYS.017) | Pack + guides (публичные знания) | Cloudflare Workers (aisystant) | Нет (публичный) | Все |
| **Digital Twin MCP** (SYS.001) | Профиль, роли, прогресс | Cloudflare Workers (aisystant) | OAuth2 per-user | Все (каждый свои данные) |
| **Personal MCP** (будущее) | Стратегия, заметки, личные DS | Self-hosted / personal CF | API key | Только владелец |

**Обоснование разделения:** Три типа данных имеют разные модели владения, хостинга и аутентификации. Объединение ломает принцип наименьших привилегий и усложняет без выигрыша.

### 6.2. Knowledge MCP (SYS.017)

**Назначение:** Единый MCP-сервер для гибридного поиска (semantic + keyword) по доменным знаниям (Pack) и учебному контенту (guides).

**Стек:** Cloudflare Workers AI + Neon PostgreSQL (pgvector + pg_trgm + tsvector) + bge-m3 embeddings (1024d)

**Поиск (DP.D.024):** keyword-first routing + vector fallback. Коды сущностей (`DP.ROLE.001`) → keyword path (pg_trgm ILIKE + tsvector FTS, ~5ms). Естественный язык → vector path (cosine similarity, ~300ms). Keyword miss → автофолбэк на vector.

**Tools:**

| Тул | Назначение | Ключевые параметры |
|-----|-----------|-------------------|
| `search` | Гибридный поиск (keyword-first + vector fallback) | `query`, `source?`, `source_type?`, `limit?` |
| `get_document` | Получить документ по имени | `filename`, `source?` |
| `list_sources` | Список доступных баз знаний | `source_type?` |

**Source types:**

| Тип | Что содержит | Примеры |
|-----|-------------|---------|
| `pack` | Доменные сущности (source-of-truth) | PACK-digital-platform, PACK-personal, PACK-MIM |
| `guides` | Учебный контент | Курсы Aisystant |
| `ds` | Процессы, реестры (не код) | DS-ecosystem-development |

**Схема документа в Neon PostgreSQL:**
```
documents {
  filename: text           // относительный путь
  content: text            // полный текст
  source: text             // имя источника (репо)
  source_type: text        // pack | guides | ds
  embedding: vector(1024)  // bge-m3 1024d
  hash: text               // SHA-256 для инкрементальной индексации
  search_vector: tsvector  // полнотекстовый индекс
}
```

**Data flow:**
```
Pack/DS repo → ingest.ts (скрипт) → CF Workers AI (bge-m3) → Neon pgvector ← Knowledge MCP ← Bot/Claude
```

### 6.3. Personal MCP (будущее)

Компонент экзокортекса пользователя. Разворачивается при форке template-репо (FMT-exocortex-template). Содержит личные данные: стратегию, планы, заметки. Доступ по API-ключу.

**Реализация:** knowledge-mcp fork с отдельной Neon базой и API-key auth.

## 7. Синхронизатор (SYS.016)

**Назначение:** Watcher + dispatcher экзокортекса. Единая точка наблюдения за изменениями и уведомления о работе агентов.

**Паттерн:** `Watch → Detect → Route → Notify`

**Процессы:**

| Процесс | Описание | Триггер | Статус |
|---------|----------|---------|--------|
| file-sync | Точечная синхронизация файлов remote → local | launchd (2 мин) | ✅ Active |
| notify | Dispatch уведомлений в Telegram от имени агентов | Вызов из agent runner | ✅ Active |
| code-scan | Ночное сканирование Downstream → captures | scheduler (00:00) | ✅ Active |
| pack-project | Pack → Projection → Downstream (YAML read model) | Ночной (после code-scan) / ручной | ✅ Active |
| time-trigger | Вызов агентов по расписанию (scheduler.sh) | launchd (10 моментов/день) | ✅ Active |
| inbox-watch | Обнаружение нового контента в captures.md | Будущее | 📋 |
| code-watch | Обнаружение коммитов в Pack-репо → trigger | Будущее | 📋 |

**Паттерны:**

1. **Watch → Detect → Route → Notify** — основной паттерн, наблюдение + dispatch
2. **Pack → Projection → Downstream** (pack-project) — проецирование знаний из Pack в read-only YAML-проекции для Downstream-систем. По Хононову (DDD): Single Writer (Pack) → Projected Read Model (CQRS-lite) → Conformist (Downstream). Конфигурация системы: `configs/systems/<id>.yaml` (generic, одна система = один конфиг).

**Notify dispatch:**
- Все агенты (Стратег, Экстрактор) делегируют уведомления Синхронизатору
- Шаблоны по агентам: `templates/<agent>.sh` (build_message + build_buttons)
- Канал: Telegram API (HTML + inline keyboard)
- Env: `~/.config/aist/env`

**Реализация:** [DS-ai-systems/synchronizer](https://github.com/TserenTserenov/DS-ai-systems)

## 8. WakaTime (SYS.018)

**Назначение:** Автоматический трекинг рабочего времени в VS Code. Фиксирует активность по проектам, языкам, файлам и Git-веткам без участия пользователя.

**Тип:** SaaS (wakatime.com) + VS Code extension. Исполнитель: I10 (Grade 0 — чистый инструмент).

**ai_role:** none — полностью детерминированная система, LLM не задействован.

### 8.1. Роли системы (DP.D.033)

Одна система — несколько ролей в зависимости от контекста потребления:

| Роль | Контекст | Потребитель | Что делает в этой роли |
|------|----------|-------------|----------------------|
| **Хронометрист** | Непрерывная фоновая работа | Пользователь (через дашборд) | Пассивное измерение времени: по проекту, языку, файлу, ветке. Heartbeat каждые 2 мин при активности |
| **Поставщик метрик** | Week Review, Morning Check | R1 Стратег, R10 Статистик (через API) | Отдаёт структурированные данные (JSON) для аналитики и принятия решений |
| **Зеркало продуктивности** | Ретроспектива, планирование | Пользователь, R1 Стратег | Визуализация паттернов работы: пики, провалы, переключения между проектами |

### 8.2. Рабочие продукты

| РП | Формат | Потребитель |
|----|--------|-------------|
| Дашборд активности | HTML (wakatime.com/dashboard) | Человек |
| API метрик | JSON (wakatime.com/api/v1/) | Агенты (Стратег, Статистик) |
| Еженедельный email-отчёт | Email (авто, Пн) | Человек |

### 8.3. Интеграция в IWE

```
VS Code (heartbeats) → WakaTime Extension → wakatime.com API
                                                    ↓
                                        Dashboard (человек)
                                        API (агенты: Week Review, Morning Check)
```

**Установка:** VS Code → Extensions → «WakaTime» → Install → [API key](https://wakatime.com/settings/api-key).

**Tier:** T4+ (Созидание / IWE). Рекомендуется при установке экзокортекса.

## 9. CRM (SYS.003)

**Назначение:** Единое окно для просмотра всех оплат, участников и доступов. Подключается к двум БД (Aisystant PG read-only + Neon read-write).

**Реализация:** Directus (Railway, Docker). Introspects обе БД. RBAC: Бухгалтер (все данные), Маркетолог (воронка, когорты), Декан (наполняемость), Преподаватели (свои потоки).

**Потребители:** Гиляна (бухгалтерия), Алёна (маркетинг), Юля (декан), преподаватели.

**Аналитика:** Metabase (Railway) — MRR, LTV, CAC, churn, воронка T0→T4, наполняемость потоков.

**Связи:** SC.114, WP-183.

**Статус:** proposal создан, Phase 0 не начата.

## 10. Payment Registry (SYS.011)

**Назначение:** Журнал всех транзакций из всех каналов. Единый источник правды: кто, когда, сколько, через какой канал заплатил. 8 каналов (YooKassa, Paybox, Stripe, Монета, Tilda, YooKassa бот, TG Stars, Manual). Три получателя: ИП (РФ), Aisystant Corp (USA), Телеграм (физ.лицо).

**As-is:** де-факто существует внутри монолита Aisystant (таблицы оплат в Aisystant PG, каналы 1-5) + Neon (каналы 6-7, семинары бота). Не выделена как отдельная подсистема.

**To-be:** Directus (Фаза A) даёт единое окно в оба источника. На Фазе B Payment Registry мигрирует в Billing Service как модуль.

**Связи:** SYS.010 (Billing Service), SYS.003 (CRM), SC.112, WP-183.

## 11. Billing Service (SYS.010)

**Назначение:** Бизнес-логика работы с деньгами: тарифы, подписки, баллы (WP-121), revenue sharing (Platform 30% / Author 50% / Instructor 15% / Curator 5%), юнит-экономика, управление доступами.

**As-is:** не существует как отдельная система. Логика размазана: бот (витрина, Strategy pattern, адаптеры YooKassa/Stripe/TG Stars/Баллы/Manual), Aisystant LMS (подписки, потоки).

**To-be:** новая система поверх Payment Registry (SYS.011) и Ory (SYS.009). Строится поэтапно: Фаза A (единый учёт), Фаза B (Ory + единый аккаунт), Фаза C (баллы, revenue sharing, юнит-экономика).

**Автоматизация (целевая):**
```
Оплата → Payment Registry → Billing Service → invite в чат / выдача доступа → Activity Hub событие
```

**Связи:** SYS.011 (Payment Registry), SYS.003 (CRM), SYS.009 (Ory), SC.112, WP-183, WP-121 (баллы).

**Статус:** Витрина семинаров реализована (бот). Единый учёт — [proposal](../../../../0.OPS/0.9.Inbox/WP-183-unified-payments-proposal.md).
