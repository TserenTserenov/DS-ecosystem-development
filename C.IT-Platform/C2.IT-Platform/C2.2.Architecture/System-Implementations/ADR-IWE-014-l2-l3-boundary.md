# ADR-IWE-014: Граница L2/L3 — open-core IWE-платформа

## 1. Контекст

22 апр 2026 в WP-250 Ф-F прошёл ArchGate продуктового решения «двухуровневая open-core платформа: L2 общая платформа vs L3 персональный IWE». Профиль ЭМОГССБ: ✅ Масштабируемость / Скорость / Современность, ⚠️ Эволюционируемость / Обучаемость / Генеративность / Безопасность, 0 ❌. Вето-фильтр не сработал — решение принято.

Среди 4 митигаций критическая — **Ф-F.1: формально зафиксировать границу L2/L3** (митигация ⚠️ Эволюционируемости + ⚠️ Безопасности). Без явной фиксации:

- **Эволюционируемость:** при росте ~15 сервисов (Gateway, Knowledge-MCP, Health DB, ContentPipeline, Digital-Twin, aist-bot, captures-flow, Pack-personal/platform, Persona Git, Activity-hub, Metabase, Langfuse, Ory, LMS-sync) команда не может сходу ответить «этот сервис где живёт?». При найме нового разработчика (Г-О1, к 1 июня) онбординг буксует. При создании Plugin API L2 (WP-258) непонятно, какие функции платформы открыть наружу.
- **Безопасность:** PII и payment_credentials (DP.ARCH.004 §2 П6.1) распределяются между L2 (Neon-стороны: payments, subscription, persona) и L3 (Git пользователя: captures, Lifework). Без явной классификации сервис×слой нельзя написать checklist для аудита.
- **Коммуникация наружу:** Алёна (служба продвижения) делает landing'и под холодную аудиторию — нужна таблица «что открыто / что платно». Без границы нельзя сказать «это бесплатно навсегда» / «это часть подписки».

Этот ADR фиксирует решение и матрицу сервис×слой как канонический источник для downstream-РП (Ф-F.2 «Что открыто / что платно», Ф-F.3 «L2 Data Classification Map», WP-258 Plugin API L2, WP-262 Бот тонкий клиент, WP-188 Ф10.0 Landing).

## 2. Рассмотренные варианты

ArchGate Ф-F (22 апр) проработал только **вариант A — двухуровневая open-core**. Альтернативы B/C/D зафиксированы как «не сравнивались, открыто к пересмотру по триггеру»:

| Вариант | Описание | Когда пересматривать |
|---------|----------|----------------------|
| **A. Двухуровневая open-core** | L2 — общая платформа (runtime), L3 — у пользователя (Git). FPF/ZP/Pack-platform публично открыты. Монетизация на runtime-доступе к L2. | — (выбран) |
| B. Моноуровневая SaaS | Всё на платформе (как Notion AI / ChatGPT Team). Данные пользователя — внутри платформы. | Регуляторное требование локализации в РФ + impossibility поддерживать L3 экспорт |
| C. Push-only L3 | Платформа только генерирует контент в L3 пользователя (как GitHub Copilot пишет в репо). Нет runtime-сервисов, кроме Gateway-моста. | Конкурент с эквивалентом + критическое падение CAC модели подписки |
| D. Трёхуровневая L1+L2+L3 | L1 — публичный open-core (FPF/ZP), L2 — runtime для подписчиков, L3 — личный. Уже фактически есть, но L1 не выделен как самостоятельный слой. | Сообщество требует self-hostable L1 (выкатить ZP/FPF в self-host без ограничений) |

**Триггеры пересмотра решения A:**
- CAC > 3000 ₽ в органике (Г-К4 правило отказа) → возможно проблема в позиционировании двухслойности.
- Конкурент (OpenAI/Anthropic/Notion) выкатывает Pack-эквивалент с адаптивной персонализацией → ускорение защиты через вертикальность + B2B2C.
- Регуляторное требование РФ: PII/payment_credentials исключительно на территории РФ → пересмотр L2-инфры (см. WP-215, заморожен до подтверждения гипотезы).

## 3. Решение

**Принят вариант A — двухуровневая open-core IWE-платформа.**

### 3.1. Определения слоёв

**L2 = общая платформа.** Runtime-сервисы, обслуживающие всех подписчиков. Writer = платформа (автоматически), owner = инфраструктура команды (Neon / Cloudflare Workers / Railway). Состав:

- Pack-platform публичная часть (FPF, ZP, SOTA, базовые методы) — открыта на GitHub public.
- MCP/Gateway, Knowledge-агенты (Помощник, Навигатор, Диагност, Портной) — runtime под подпиской.
- Память пользователя в Neon (события, платежи, подписки, расчёты/baseline/индикаторы) — append-only, owner: команда.
- Identity & access (Ory Hydra, Cedar policies, Keto).

**L3 = персональный IWE.** Данные конкретного пользователя. Writer = пользователь (или агент по его acceptance), owner = его Git-репозиторий + локальная установка. Состав:

- PACK-personal, DS-my-strategy, Lifework, captures, fleeting-notes — у пользователя в Git.
- Preferences (CLAUDE.md, extensions, IWE Template настройки).
- История решений, ноты, замыслы — всё, что пользователь записал.

### 3.2. Матрица «сервис → слой» (~15 ключевых сервисов)

**Граничные случаи:**
- **Pack-platform приватная часть** (внутренние черновики методов до публикации) — **L2 внутренний**, не публичный (writer: Tseren+ИТ-команда; owner: GitHub приватные репо команды). Не путать с L3 — это не пользователь.
- **Knowledge-индексы personal-коллекций** (когда пользователь даёт пакам своего Git проиндексироваться через personal-knowledge-mcp) — **L2 projection of L3**: writer = платформа (индексирует), но source-of-truth = L3 (Git пользователя). При удалении из L3 → projection в L2 устаревает (re-index).
- **OAuth tokens, JWT** — **L2 ephemeral**: писатель платформа, время жизни < 1 час, не хранятся. Не отдельная сущность.

### 3.3. Правила границы

**R1. Direction of writes (one-way).** L2 пишет в L2, L3 пишет в L3. Платформа **не пишет** в L3 пользователя без явного acceptance flow (Captures-flow, Personal-knowledge-mcp `personal_write`). Пользователь **не пишет** в L2 напрямую — только через UI / API платформы (бот / VS Code / web), где платформа выступает прокси.

**R2. Read direction.** L3 → L2 запросы возможны (Gateway / Knowledge-MCP отдают L2 данные пользователю). L2 → L3 чтение возможно только с явного согласия (`github_connect` flow выдаёт installation token, ограниченный конкретным репо).

**R3. Source-of-truth никогда не дублируется.** Если факт принадлежит L3 (например, captures пользователя) — L2 хранит только projection (индексы). При конфликте L3 побеждает.

**R4. Open-core инвариант (Манифест §10).** Pack-platform публичная часть (FPF, ZP, SOTA, базовые методы) **никогда** не закрывается за платной стеной. Изменение этого правила = изменение Манифеста, не изменение ADR. См. Lifework/Защита Дела созидателей.md опора 7.

**R5. Reversibility инвариант (Манифест §10, опора 5).** Экспорт L3 пользователя работает за ≤10 минут в открытых форматах. Регулярный smoke-test (квартально) — child-РП в WP-260.

**R6. PII routing (DP.ARCH.004 §2 П6.1, расширяется через WP-212 B7.3.1 / Ф-F.3).** PII живёт в L2 Neon (с Cedar policies) или в L3 Git пользователя. Логирование PII запрещено (включая маскированное). Payment_credentials — строже PII: только в зашифрованном виде в `payment` БД, доступ по service account с аудитом. Cross-user consent — только через явный acceptance flow.

### 3.4. Open-core экономика

**Бесплатно навсегда (L2 публично + L3):**
- Pack-platform публичная часть (FPF/ZP/SOTA/методы), базовые курсы
- L3 пользователя (его Git-репозитории, captures, Lifework)
- IWE Template (FMT-exocortex-template, open-source)
- Knowledge-MCP `knowledge_*` + `get_instructions` (с Ory JWT, без подписки)
- Ory identity (декларации Персоны)

**По подписке (L2 runtime):**
- Gateway полный: `dt_*`, `personal_*`, `search`, `github_*`, `create_repository`
- Digital-Twin (Память.Derived)
- Активные агенты (Помощник, Навигатор, Диагност, Портной)
- ContentPipeline (Q3+ multi-tenant)
- Бот (триал → платная подписка)

**Никогда не закрываем (инвариант Манифеста §10):**
- Pack-platform публичная часть
- Экспорт L3
- Базовые курсы методологии

## 4. Защита от вендоров (структурная)

> **Канонический источник аргументов:** [Lifework/Защита Дела созидателей.md](../../../../../DS-my-strategy/Lifework/Защита%20Дела%20созидателей.md). Здесь — привязка границы L2/L3 к опорам.

Граница L2/L3 — не только архитектурная конвенция, она работает на структурную защиту Дела от глобальных ИИ-вендоров (OpenAI, Anthropic, Google, Notion). Привязка к четырём опорам из канонического документа:

| Опора защиты | Как граница L2/L3 её реализует |
|--------------|-------------------------------|
| **5. Обратимость как принцип** | L3 у пользователя (его Git) — естественный switching cost через накопленный Pack/captures, **без технического lock-in**. Экспорт всегда открыт, это инвариант Манифеста. R5 закрепляет smoke-test ≤10 мин. Вендоры удерживают тем, что разговоры теряются при уходе. Мы — тем, что не хочется уходить. |
| **6. Вертикальность: специализация под созидателя** | L2-агенты (Помощник/Навигатор/Диагност/Портной) — специализированы под роль созидателя, не универсальная ИИ-платформа. Вендор структурно не спускается в вертикаль без потери горизонтальности. L2 — наша игра, на которой вендор не играет. |
| **7. Открытость как стратегия** | Pack-platform публичная часть в L2 + FPF/ZP открыты на GitHub public (инвариант R4 + Манифест §10). Монетизация на runtime-доступе к L2-сервисам, не на закрытых курсах. Мы доплачиваем за распространение методологии, не запираем её. |
| **1. Букварь, написанный живыми носителями** | L2-агенты — инструмент доставки методологии, не замена живым носителям (Левенчук + сподвижники с 2015). Линия передачи методологии живёт в L3 (Git пользователя, captures, сессии) через взаимодействие с сообществом. Граница L2/L3 формализует, что L2 — букварь, L3 — живой опыт пользователя. |

**Что вендор скопирует и что нет:** OpenAI/Anthropic могут скопировать каждый L2-сервис по отдельности (Gateway, Knowledge-агенты, Digital-Twin). Но **не могут скопировать связку**: Pack-platform (10+ лет методологии) + сообщество ~13 500 (опора 2) + репутация русскоязычного методологического ядра (опора 3) + распределённый конвейер эволюции через L3-пользователей (опора 4) + L3 как естественный switching cost (опора 5). Граница L2/L3 — конструкция, делающая эту связку нескопируемой.

## 5. Последствия

### Плюсы

- **Чёткая ответственность.** Любой сервис в матрице → понятно: writer, owner, PII-класс, open-core статус. Снимает 80% вопросов «это где живёт?».
- **Onboarding ускоряется.** Новый разработчик читает ADR + матрицу → понимает топологию за 30 минут (Г-О1 митигируется).
- **Безопасность checklist'абельна.** Аудит «PII не утекает в L3 без consent» → проход по матрице с фильтром PII-класс. Ф-F.3 (L2 Data Classification Map в DP.ARCH.004) разрабатывается с этой матрицей как entry-point.
- **Landing «Что открыто / что платно»** (Ф-F.2 для Алёны) собирается из §3.4 как drop-in.
- **Plugin API L2 (WP-258)** имеет точку входа: расширения в L2 — публичные / приватные / подписочные? Ответ — по матрице.
- **WP-262 (бот тонкий клиент)** разблокирован: бот = handler TG в L2 (с FSM-state ephemeral), оркестрация выносится в отдельный L2-сервис.
- **Защита от вендоров** становится явной — секция §4 цитируется в холодных каналах.

### Минусы

- **Гранулярность 15 сервисов недостаточна для деталей.** Каждый сервис сам по себе сложен (Gateway имеет ~10 sub-функций). При росте — потребуется sub-карта на ключевые сервисы. Не блокер сейчас, ремарка для Q3.
- **Граница «projection of L3» в L2** (knowledge-индексы personal-коллекций) — нюанс, который при невнимательности интерпретируется как «L2 владеет данными пользователя». R3 + конкретное правило в §3.2 граничные случаи фиксируют это, но требует обучения команды (Q2-Q3).
- **Альтернативы B/C/D не сравнивались количественно.** Если триггер пересмотра сработает (CAC > 3000 ₽ или конкурент с эквивалентом), ArchGate B/C/D придётся делать «с нуля». Принимаемый сейчас риск.

## 6. Критерии отмены ADR

- **Регуляторное требование:** РФ потребует локализации всех PII на территории РФ → пересмотр L2-инфры (WP-215 размораживается, возможен переход к варианту B или D).
- **CAC > 3000 ₽ стабильно (Г-К4 правило отказа):** проблема в позиционировании двухслойности → пересмотр через Ф-F.2 (карта «открыто/платно»).
- **Конкурент (OpenAI/Anthropic/Notion) выкатил Pack-эквивалент с адаптивной персонализацией:** ускорение защиты через B2B2C (Г-К5) + усиление вертикальности (опора 6); возможен переход на D (выделение L1) для self-hosted пакета.
- **Сообщество требует self-hostable L2-runtime (для оффлайн / приватного облака):** возможен переход на D (L1+L2+L3), где L1 = публичный open-core, выкатываемый для self-host.
