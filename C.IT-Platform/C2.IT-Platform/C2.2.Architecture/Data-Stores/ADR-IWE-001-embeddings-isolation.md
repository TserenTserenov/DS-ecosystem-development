# ADR-IWE-001: Изоляция embeddings личных Pack в multi-tenant среде

## Контекст

При реализации Gateway-архитектуры IWE (DP.IWE.003) embeddings личных Pack пользователей хранятся на платформе Aisystant. Необходимо обеспечить изоляцию между пользователями: поиск по embeddings пользователя A не должен затрагивать данные пользователя B.

## Рассмотренные варианты

| Вариант | Описание | ~30к users | Изоляция |
|---------|---------|-----------|----------|
| A. Namespace per user (фильтр) | `WHERE user_id = $1` в pgvector | ✅ | ✅ |
| B. Schema per user | Отдельная схема БД на пользователя | ❌ дорого | ✅✅ |
| C. Партиционирование по user_id | Отдельный HNSW индекс на партицию | ✅ | ✅ |
| D. Qdrant, collection per user | Отдельная коллекция с HNSW | ✅ | ✅✅ |

## Решение

**Принят вариант A** — namespace per user в pgvector (Neon).

Фильтр: `WHERE user_id = $1`, где `user_id` берётся из JWT-токена (Ory), не из параметра запроса пользователя.

## Обоснование

- Остаёмся в Neon (нет нового вендора, нет новой инфраструктуры)
- Достаточно для целевой аудитории до ~30к пользователей × ~1к векторов = ~30М строк
- Простота реализации и дебаггинга
- Безопасность: SQL-инъекция исключена (user_id из JWT, не из запроса)

## Порог пересмотра

При достижении **30к активных пользователей** — инициировать ADR-IWE-002 по миграции на Qdrant (collection per user). Признаки деградации раньше: p95 latency поиска >500ms при нагрузке.

## Последствия

- **Положительные:** минимальная сложность, работает на текущем стеке
- **Отрицательные:** глобальный HNSW индекс растёт со всеми пользователями, recall может снижаться при масштабировании
- **Риск:** при ошибке в WHERE-фильтре — утечка данных. Митигация: integration tests на изоляцию (тест: пользователь A не видит векторы пользователя B)
