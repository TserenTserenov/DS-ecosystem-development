# B7.2 — Ответы на 6 открытых вопросов

> Источник вопросов: B7.2-stride-threat-model.md §6 (статус: draft → переходит в reviewed после ревью Паши).

## 1. Backend trust model — Variant B vs E

**Вопрос:** переходим на Variant E (JWT claim) до 1 мая или после? Если после — какой interim mitigation?

**Ответ (8 мая):** **После 1 мая** — Variant E реализуется когда Паша допишет Hydra token hook (статус 14 апр: deployed на тестовом аккаунте, has_subscription=false → 403 на personal-mcp ✅, ждём миграции на боевой Hydra).

**Interim mitigation (текущий — Variant B):**
- Gateway = единственная точка subscription check (✅ работает с 14 апр)
- Backends (dt-mcp, personal-mcp) делают `verifyJwtLocally` через JWKS, доверяют Gateway что подписка проверена
- **Допущение:** прямой доступ к dt-mcp/personal-mcp в обход Gateway невозможен (CF Workers без публичного DNS bypass'а)
- **Митигация допущения:** B4.21 JWT-верификация в каждом MCP — даже при bypass'е Gateway без валидного JWT не пройдут

**Срок миграции на E:** при готовности Hydra hook на боевой инстанс (приоритет — после WP-253 пилота 11 мая).

## 2. YooKassa idempotency

**Вопрос:** реализована ли в текущем коде (`aist_bot/payment.py` мигрирующем в payment-receiver)?

**Ответ:** **Аудит до пилота 11 мая.** Сейчас payment-receiver (CF Worker) — приоритет 1 для аудита. Чек:
- HMAC signature timing-safe verification (`crypto.timingSafeEqual`)
- Idempotency key: проверка `order_id` уже обработан → возврат cached response, не повторное начисление
- Source IP allow-list YooKassa (документированный список IP)

**Действие:** добавить в WP-246 backlog как Ф-AUDIT (1h, до пилота). Если не реализовано — критический фикс до cut-over WP-246.

## 3. OpenAI embedding consent

**Вопрос:** где правильно показать (онбординг бота? /privacy? явный prompt при первом personal-doc index)?

**Ответ:** **Все три уровня (defence in depth):**

1. **Privacy Policy (B8.0):** общий пункт «Личные документы индексируются через OpenAI embeddings. Текст отправляется в OpenAI API (US, DPA с пользователем)»
2. **/start бота (онбординг):** галочка «Я согласен на обработку личных репозиториев через OpenAI» — обязательная для активации /scaffold
3. **Первый /scaffold для нового user_id:** явный prompt в Telegram «Ты подключаешь репо X. Содержимое будет проиндексировано через OpenAI. Продолжить? [Да] [Нет]»

**Storage решения:** `users.consent_openai_embedding_at` (timestamp). Без этого поля — `/scaffold` возвращает 403.

**Бюджет реализации:** ~3h (UI бот + поле в БД + middleware в personal-mcp). Привязать к B8.0 + B3.9.

## 4. Tier escalation security

**Вопрос:** был ли когда-либо аудит «может ли пользователь дописать в свой `tier_event` / `subscription_grant`»?

**Ответ:** **Не было формального аудита.** Текущая защита (косвенная):
- `subscription_grants` пишет только webhook-receiver (YooKassa) и Directus (admin only) — не пользовательский путь
- `tier_event` пишет только `user_state` middleware при тарификации (server-side)
- RLS на `subscription_grants` — заблокирована dep WP-231 Ф4 → **gap**

**Действие (приоритет: после WP-231 Ф4):**
- B4.23 пр.1 — RLS на `subscription_grants`: пользователь может только READ свои, INSERT/UPDATE заблокированы для всех ролей кроме `payment_writer` и `admin`
- E2E тест: попытка пользователя через app middleware вызвать INSERT на `subscription_grants` → 403
- Audit log (WP-237 trigger) — любое изменение `subscription_grants` логируется с актором

## 5. External webhook IP allow-list

**Вопрос:** для каких sources доступен (GitHub IP ranges публично, остальные?)?

**Ответ (8 мая):**

| Source | IP allow-list | Статус |
|--------|--------------|--------|
| GitHub webhooks | https://api.github.com/meta — `hooks` array | ✅ доступно публично |
| YooKassa | https://yookassa.ru/developers/using-api/webhooks#ip-addresses — фиксированный список 4 IP | ✅ доступно публично |
| Telegram | https://core.telegram.org/bots/webhooks — широкий range CIDR | ✅ доступно (но широкий range — менее эффективно) |
| WakaTime | в публичной API доке нет — **gap** | 🔴 запросить у Wakatime support |
| Aisystant LMS | внутренний — Hetzner static IP | ✅ известен |

**Действие:** добавить allow-list в `event-gateway` middleware. Запрос Wakatime support (5 мин). Бюджет реализации: ~2h.

## 6. STRIDE coverage — какие сервисы добавить

**Вопрос:** status-proxy, fsm-mcp, observability-webhook были опущены как low-risk; согласен?

**Ответ:**

- **status-proxy:** ✅ согласен пропустить (read-only proxy для Hydra `/userinfo`, нет PII вне ответа Hydra, нет DB-доступа)
- **fsm-mcp:** 🟡 **добавить** — хранит state пользовательских сессий (FSM context может содержать историю консультаций, чувствительно). Приоритет: средний.
- **observability-webhook:** ✅ согласен пропустить — receives только internal events, никакого PII в payload по контракту
- **payment-receiver:** уже в scope ✅
- **event-gateway:** уже в scope ✅

**Новый low-risk кандидат:** **knowledge-mcp/concepts API** — read-only доступ к `concept_graph.concepts` (платформенная таблица). Без user_id фильтрации (намеренно, концепты публичны). Stride: только Information Disclosure категория, фактически ноль PII.

**Действие:** добавить fsm-mcp в B7.2 §1 scope при следующей ревизии (предложение R24 Аудитору внести при ежемесячном Month Close).
