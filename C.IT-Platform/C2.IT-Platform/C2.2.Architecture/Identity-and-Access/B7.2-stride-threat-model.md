# B7.2 STRIDE Threat Model

> **Назначение.** Структурированный анализ угроз по 6 категориям STRIDE для 8 ключевых сетевых сервисов IWE. Цель — единый источник для (а) ArchGate §Б при добавлении нового сервиса, (б) подготовки B7.4 (внешний human audit), (в) ввода в Ф10 комплексное ревью.
>
> **Метод.** STRIDE = Spoofing / Tampering / Repudiation / Information disclosure / Denial of service / Elevation of privilege. Per-service таблица: угроза × текущая митигация × gap × приоритет. Не претендует на исчерпывающий аудит — это **first-pass** карта для планирования; глубокий audit приходит на Ф10.
>
> **Принцип фильтрации:** включаются только реалистичные угрозы (`probability × impact ≥ medium`). Теоретические (например «утечка через ЭМ-излучение») опущены.
>
> **Связь с B7.3.1:** STRIDE-анализ опирается на классификацию данных (PII / payment_credentials / secrets / public) — это позволяет приоритизировать угрозы по классу затронутых данных.

## 1. Scope: 8 сервисов

| # | Сервис | Размещение | Trust boundary | Класс данных (B7.3.1) |
|---|--------|-----------|----------------|----------------------|
| 1 | **gateway-mcp** | Cloudflare Worker | OAuth in (Ory) → MCP out (бэкенды) | secrets (JWT) + PII (user_id) |
| 2 | **knowledge-mcp** | Cloudflare Worker | Gateway → OpenAI + Neon + GitHub | PII (личные docs) + secrets (GitHub PAT) |
| 3 | **digital-twin-mcp** | Cloudflare Worker | Gateway → Neon `indicators` | PII (Память.Derived) |
| 4 | **personal-knowledge-mcp** | Cloudflare Worker | Gateway → GitHub App + Neon | PII (личные репо) + secrets (installation tokens) |
| 5 | **payment-receiver** | Cloudflare Worker | YooKassa webhook → Neon `payment` | payment_credentials + PII |
| 6 | **multi-domain-projection-worker** (WP-270) | Railway | Neon `learning.domain_event` → 4 проекции | PII (events) |
| 7 | **bot @aist_me_bot** | Railway | Telegram → Gateway + Neon `aist_bot` | PII (messages, FSM, ory_tokens) |
| 8 | **event-gateway** | Cloudflare Worker | External webhooks → Neon `journal.raw_event` | PII (event payload) |

**Не в scope (отдельный аудит):**
- LMS Aisystant (legacy, READ-ONLY ⛔, мигрирует через WP-268)
- Activity-hub (миграция в event-gateway, scope WP-228)
- Observability-webhook (B5.7-pre-commit ✅, прочие угрозы низкий-medium)
- Status-proxy, fsm-mcp, guides-mcp, tailor-mcp (вспомогательные, низкий риск)

## 2. STRIDE-категории в контексте IWE

| Кат | Что значит у нас | Главные защитные слои |
|-----|------------------|----------------------|
| **S — Spoofing** | Выдача себя за пользователя / сервис / источник события | JWT verification (Ory), HMAC signatures (YooKassa, GitHub webhooks), OAuth scopes |
| **T — Tampering** | Изменение данных in-transit или at-rest | TLS, Neon RLS (B9.x), application-side validation, append-only event log |
| **R — Repudiation** | Отказ от факта действия (нет audit trail) | audit.log (B8.7-B8.10), trigger-based logging (WP-237), B7.3.1 §3.1 audit logging |
| **I — Information disclosure** | Утечка PII / payment_credentials / secrets | RLS, encryption at-rest, B7.3.1 §3.1 logging policy, B7.7a/b hooks |
| **D — Denial of service** | Перегрузка / отказ обслуживания | Rate limiting (B4.3-B4.4), graceful degradation (B4.20), CF DDoS protection |
| **E — Elevation of privilege** | Получение чужого доступа / выход за scope | Tier-based ACL (T0-T4), API-level RBAC (B4.1 pending), subscription gate (DP.SC.112) |

## 3. Per-service анализ

### 3.1. `gateway-mcp` (Cloudflare Worker)

**Роль:** trust boundary между внешним миром и приватными MCP. JWT verify, subscription check, routing.

| Кат | Угроза | Текущая митигация | Gap | Приоритет |
|-----|--------|-------------------|-----|-----------|
| S | Подделка JWT (signing key compromise / `kid=""` exploit) | JWT verify через Ory JWKS (B4.21 ✅), `kid=""` починен Пашей 13 апр | Нет rotation policy для Ory keys, нет alert при изменении JWKS | средний |
| S | Spoofing `X-User-Id` header при прямом обращении к MCP минуя Gateway | JWKS verify в каждом MCP (B4.21 ✅), backend doesn't trust X-User-Id alone | ⚠️ Backend trust model: digital-twin-mcp + personal-mcp полагаются на Gateway → если Gateway скомпрометирован, header можно подделать | высокий |
| T | Modification request body in-transit | TLS обязательно (CF managed) | — | низкий |
| R | Нет audit log кто что вызвал через Gateway | Wrangler tail логи, Langfuse traces (опт-аут) | ⚠️ Постоянный structured audit-log не пишется (B4.22 audit-log минимум отложен) | высокий |
| I | Subscription status leak через 403 messaging | Generic error «requires subscription» (B4.10 ✅) | — | низкий |
| I | Tier leak через response timing (T0 быстрее T4) | Не измерялось | Side-channel timing analysis — теоретический риск | низкий |
| D | DDoS на /mcp endpoint | Rate limiting 60 req/60s sliding window (B4.4 ✅), CF DDoS protection | Нет per-tenant rate limit (только глобальный); один пользователь может exhaust лимит | средний |
| D | Slow loris через медленные MCP-tool вызовы | CF Worker timeout 30s | — | низкий |
| E | Subscription bypass через прямой обращение к backend MCP | JWKS verify + checkSubscription в подписочных MCP (Variant E) — pending фикса Hydra hook | ⚠️ Текущая модель: backends trust Gateway, без double-check. Если backend exposed напрямую — bypass | высокий |
| E | Bypass через freemium tier для подписочных tools | DP.SC.112 freemium = только `knowledge_*` + `get_instructions`; остальное проверяется | ✅ |

### 3.6. `multi-domain-projection-worker` (Railway, WP-270 новый)

**Роль:** consume `learning.domain_event` → upsert в 4 проекции (persona, subscription, indicators, learning). Per-domain cursor.

| Кат | Угроза | Текущая митигация | Gap | Приоритет |
|-----|--------|-------------------|-----|-----------|
| S | Forge event в `learning.domain_event` (write путь) | Только writers с правами на schema; service-account credentials | Если service account leaked — можно писать fake events | высокий |
| T | Tampering payload event | Append-only по дизайну, no UPDATE | ✅ |
