## A. Аутентификация и авторизация

| # | Проверка | Ожидание | Источник | Статус |
|---|---------|---------|---------|--------|
| A1 | JWT-верификация в каждом MCP | Каждый CF Worker делает `verifyJwtLocally` через JWKS, не доверяет X-User-Id без токена | B4.21 (ADR-IWE-012) | ✅ done 14 апр |
| A2 | Gateway = единственная точка subscription check | Backends (dt-mcp, personal-mcp) не делают своих проверок, доверяют Gateway | B4.21 Вариант B | ✅ done 14 апр |
| A3 | Fallback без spoofing | `/scaffold` и `/reindex` не принимают X-User-Id если есть Authorization | B4.22 security fix | ✅ done 14 апр |
| A4 | OAuth state tokens в PostgreSQL | GitHub/Linear/Ory/Google/WakaTime state → Postgres, не in-memory | B4.11-B4.12 | ✅ done 12 апр |
| A5 | Rate limiting активен | Бот: 20 msg/60s; Gateway: 60 req/60s | B4.3-B4.4 | ✅ done 12 апр |
| A6 | Subscription claim в JWT (Variant E) | Gateway hook передаёт `has_subscription` в JWT — backends верифицируют без DB-запроса | B4.21 Variant E | 🔄 pending Паша |
| A7 | API-level RBAC | Endpoints проверяют роли, не только tier | B4.1 | ❌ pending |

**Команды для аудитора:**
```bash
# Проверить что X-User-Id без Authorization отклоняется:

# Ожидание: 401 Unauthorized

# Проверить JWT-верификацию:
curl -X POST https://gateway.workers.dev/mcp -H "Authorization: Bearer invalid.jwt.here"

```

## Протокол R24 Аудитор — ежемесячный цикл

```
ВХОДНЫЕ УСЛОВИЯ: Month Close триггер (первый Пн месяца)
РОЛЬ: R24 Аудитор (Sonnet, context isolation)
БЮДЖЕТ: ~1h

ШАГ 1. Загрузить артефакты:
  - security-posture.md (этот дашборд)
  - WP-212 context (inbox/WP-212-product-security-program.md) — §Метрики прогресса
  - B7.2-stride-threat-model.md — §Open questions статус

ШАГ 2. Пересчитать метрики:
  - Прогресс фаз: grep '\- \[x\]' и '\- \[ \]' в WP-212
  - Open vulns: проверить §3 дашборда — какие закрыты с прошлого аудита?

ШАГ 3. Раздел за разделом через чеклист (A-F):
  - Статус изменился? → обновить ячейку
  - Новая уязвимость? → добавить в §3 дашборда
  - Критический gap → добавить в §7 топ-3

ШАГ 4. Обновить security-posture.md:
  - §2 (% прогресса), §3 (open vulns count), §6 (новая строка истории)
  - Пересчитать Security Maturity Score §1
  - updated + next_audit в frontmatter

ШАГ 5. Отчёт пользователю (≤1/2 экрана):
  - Что изменилось с прошлого аудита (закрытые риски)
  - Топ-3 текущих риска
  - Рекомендуемые действия следующего месяца

ШАГ 6. Коммит в DS-ecosystem-development:
  git commit -m "docs(WP-212): security audit YYYY-MM — R24"
```
