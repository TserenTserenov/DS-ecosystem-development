# B6.1 Backup & Disaster Recovery Policy

## Neon PostgreSQL — управляемый backup

### Что Neon гарантирует (managed)

| Параметр | Free/Launch | Scale |
|---------|-------------|-------|
| **Point-in-Time Recovery (PITR)** | 7 дней | 30 дней |
| **Continuous WAL archiving** | ✅ | ✅ |
| **Granularity восстановления** | до минуты | до секунды |
| **Geo-redundancy** | single-region | single-region |
| **RTO (recovery time objective)** | ~5-15 мин (branch restore) | ~5-15 мин |
| **RPO (recovery point objective)** | ~1 мин | ~1 с |

> **Текущий план:** уточнить в Neon console (проект: aisystant / platform).
> **Регион:** уточнить (предположительно `aws-eu-central-1` или `aws-us-east-1`).

### Процедура восстановления через Neon Branch

```bash
# 1. Создать ветку из точки восстановления (через Neon CLI или Console)
neonctl branch create \
  --project-id <PROJECT_ID> \
  --name recovery-2026-04-14 \
  --timestamp "2026-04-14T10:00:00Z"

# 2. Получить connection string ветки
neonctl connection-string recovery-2026-04-14

# 3. Проверить данные
psql "CONNECTION_STRING" -c "SELECT COUNT(*) FROM digital_twins;"

# 4. При подтверждении — переключить сервисы на новую ветку
# (обновить DATABASE_URL через wrangler secret put)
```

### Ограничения Neon

- **Нет multi-region failover** — один регион. Если AWS EU-CENTRAL-1 упадёт — база недоступна.
- **Soft deletes не защищают от DROP TABLE** — команда выполняется немедленно; PITR = единственная защита.
- **Ветки не бесплатны** — платный план нужен для PITR > 7 дней.

## Disaster Recovery Runbook (B6.3 — краткий)

### Сценарий: полная потеря базы данных

1. **T+0**: Обнаружение — мониторинг Railway/Cloudflare показывает 500 ошибки
2. **T+5**: Создать Neon branch из последней точки (PITR через Neon Console)
3. **T+10**: Проверить данные на ветке (SELECT COUNT на критических таблицах)
4. **T+15**: Обновить DATABASE_URL во всех сервисах:
   - 5 Cloudflare Workers (wrangler secret put)
   - Railway aist_bot (env var update)
   - GitHub Actions secrets (payment-registry)
5. **T+20**: Smoke test каждого сервиса
6. **T+30**: Уведомить пользователей если downtime > 10 мин

### Сценарий: случайный DROP TABLE

1. Немедленно создать Neon branch из точки **до** DROP
2. Проверить объём потерянных данных
3. Если < 5 мин — прямое восстановление через PITR
4. Если > 5 мин — частичное восстановление + reconciliation
