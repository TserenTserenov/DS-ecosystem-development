# Спецификация ротации криптографических ключей

> **Назначение.** Ответ на открытый вопрос Павла из переписки 29 апр: «ротацию вот только понять бы как делать». Зафиксировать **процедуру** ротации для трёх ключевых классов криптографии IWE с гарантией zero-downtime и отсутствия cascade-сбоев.
>
> **Отличие от B2.3.** B2.3 даёт общую политику ротации всех secrets (расписание, журнал, базовая процедура per-storage). Этот документ — узкая спецификация для **трёх криптографических ключей**, у которых ротация имеет специфику dual-key window и взаимные зависимости.

---

## 1. Какие ключи в скоупе

| Тип | Конкретные секреты | Где хранится | Влияние при компрометации |
|-----|--------------------|--------------|---------------------------|
| **Fernet master key** (column-level encryption) | `FERNET_MASTER_KEY_V{N}` (current + previous) | CF Worker secrets (gateway-mcp, personal-mcp) + Railway env (aist_bot) | Decrypt всех PII / OAuth-токенов в БД при утечке |
| **OAuth client_secret** (gateway-mcp DCR + Hydra) | `ORY_CLIENT_SECRET`, `GITHUB_CLIENT_SECRET`, `GOOGLE_CLIENT_SECRET` | CF Worker secrets, Railway env | Выдача себя за наш сервис при OAuth flow → MITM на пользовательских токенах |
| **JWT signing key** (Ory Hydra + gateway JWT verify) | `JWT_SIGNING_KEY` (Hydra), `JWT_PUBLIC_KEY` (gateway/MCP-сервисы verify) | Ory Hydra managed; public key — JWKS endpoint | Подделка JWT → bypass auth во всех сервисах |

**Не в скоупе** (закрыты другими процессами):
- API ключи провайдеров (Anthropic/OpenAI/Linear/Railway/CF) — B2.3 §Расписание ротации, q90d.
- DB credentials (Neon connection strings) — B2.3, q180d.
- GitHub PAT — B2.3, q180d.
- Webhook secrets — B2.3, q180d.

## 2. Fernet master key

### 2.1. Параметры

| Параметр | Значение |
|----------|----------|
| Алгоритм | Fernet (AES-128-CBC + HMAC-SHA256), стандарт `cryptography` (Python) и `fernet` (TypeScript для CF Worker) |
| Длина | 32 base64-encoded байта |
| Частота плановой ротации | **q90d** (квартально) |
| Триггер immediate ротации | (а) подозрение на утечку; (б) увольнение/смена доступа admin'а; (в) GitHub secret scanning alert; (г) compromise в alerter (B7.3.3 правило A4 unknown actor) |
| Dual-key window | **30 дней** (current + previous version параллельно) |
| Auto-revoke previous | По истечении 30 дней через scheduler job |

### 2.2. Структура keying material

```
FERNET_MASTER_KEY_CURRENT  = "v{N}:base64_encoded_key"
FERNET_MASTER_KEY_PREVIOUS = "v{N-1}:base64_encoded_key"  # only during dual-key window
```

Все зашифрованные строки в БД хранят префикс `v{N}:` — приложение знает каким ключом расшифровывать. Версионирование = key_version в `pii_access_log` (B7.3.3 §1).

### 2.3. Процедура плановой ротации (зашита в runbook)

> **Время:** 30-45 мин координации + 30 дней soak. **Downtime:** нет.

```
1. Подготовка
   1.1. Проверить B7.3.3 alerter rules чистые (нет активных incident'ов)
   1.2. Сгенерировать новый ключ:
        python -c "from cryptography.fernet import Fernet; print(f'v{N+1}:{Fernet.generate_key().decode()}')"
   1.3. Записать в журнал ротаций (B2.3 §Журнал) — дата, кто инициировал, причина "плановая q90d"

2. Деплой PREVIOUS (старая ставится previous)
   2.1. CF Worker secrets:
        echo "<v{N}_value>" | wrangler secret put FERNET_MASTER_KEY_PREVIOUS -c DS-MCP/gateway-mcp/wrangler.toml
        echo "<v{N}_value>" | wrangler secret put FERNET_MASTER_KEY_PREVIOUS -c DS-MCP/personal-knowledge-mcp/wrangler.toml
   2.2. Railway env (aist_bot):
        railway env set FERNET_MASTER_KEY_PREVIOUS=<v{N}_value> --service aist_bot

3. Деплой CURRENT (новый ключ как current)
   3.1. CF Worker secrets:
        echo "<v{N+1}_value>" | wrangler secret put FERNET_MASTER_KEY_CURRENT -c DS-MCP/gateway-mcp/wrangler.toml
        echo "<v{N+1}_value>" | wrangler secret put FERNET_MASTER_KEY_CURRENT -c DS-MCP/personal-knowledge-mcp/wrangler.toml
   3.2. Railway env:
        railway env set FERNET_MASTER_KEY_CURRENT=<v{N+1}_value> --service aist_bot

4. Smoke test
   4.1. POST /test-decrypt {"encrypted": "<known_v{N}_string>"} — должно вернуть plaintext (читает по PREVIOUS)
   4.2. POST /test-encrypt {"plaintext": "test"} — должно зашифровать в v{N+1} (пишет CURRENT)
   4.3. Проверить логи: 0 ошибок decrypt в течение 5 мин

5. Re-encryption job (background, 30 дней soak)
   5.1. Запустить worker `re-encrypt-fernet-rows` (читает все строки с префиксом v{N}, расшифровывает PREVIOUS, перешифровывает CURRENT, обновляет row)
   5.2. Прогресс в `health.rotation_status` (per-table counter)
   5.3. Цель: 100% rows с префиксом v{N+1} к +30 дням

6. Auto-revoke PREVIOUS (через 30 дней)
   6.1. Scheduler проверяет `health.rotation_status`: если 100% rows = CURRENT version
   6.2. Удалить PREVIOUS:
        wrangler secret delete FERNET_MASTER_KEY_PREVIOUS -c DS-MCP/gateway-mcp/wrangler.toml
        railway env delete FERNET_MASTER_KEY_PREVIOUS --service aist_bot
   6.3. Записать в журнал ротаций "PREVIOUS revoked"
```

### 2.4. Процедура immediate ротации (compromise)

Сокращённая версия §2.3:
- Шаг 1.2 generate.
- Шаг 3 деплой CURRENT **без** PREVIOUS (без dual-key window — risk: failed decrypt всех old rows; принимаем, потому что compromise хуже).
- **Параллельно:** запустить re-encryption job синхронно (блокирующий) — все rows расшифровать ПОСЛЕДНИМ известным ключом и сразу перешифровать новым.
- Альтернатива (предпочтительно): шаги 2-3-5-6 как обычно, но dual-key window сжатое до 24-48h, с принудительным форсированием re-encryption-job.

### 2.5. Disaster recovery

- Master key BACKUP — печатный лист в физическом сейфе (B6.1 backup-dr-policy). Без backup'а потеря всех CURRENT/PREVIOUS = **полная потеря данных под Fernet**.
- Restore process: восстановить значение из backup → ставить как PREVIOUS → ротация по §2.3.

## 3. OAuth client_secret

### 3.1. Параметры

| Секрет | Где используется | Частота ротации | Dual-secret window |
|--------|------------------|-----------------|---------------------|
| `ORY_CLIENT_SECRET` | gateway-mcp ↔ Ory Hydra OAuth flow | q90d (центральный OAuth для всей платформы; обоснование выше B2.3 «высокие q180d») | до 7 дней (provider-зависимо, см. ниже) |
| `GITHUB_CLIENT_SECRET` | personal-knowledge-mcp ↔ GitHub OAuth | q180d | до 7 дней (provider-зависимо) |
| `GOOGLE_CLIENT_SECRET` | google-calendar integration ↔ Google OAuth | q180d | 7+ дней (Google поддерживает до явного revoke) |

**Особенность OAuth:** RFC 6749 §10.3 рекомендует поддержку 2 активных secret'ов, но **не обязывает провайдеров реализовывать**. Реальная поддержка различается:

| Провайдер | Dual-secret | Подтверждение | Если не поддерживает |
|-----------|-------------|---------------|----------------------|
| **Ory Hydra** | ⚠️ Не подтверждено — `client_secret` в OAuth2 client object — single field; обновление через `PATCH /clients/{id}` перезатирает старый. **Проверить**: либо `hydra create oauth2-client --secret <new>` без revoke старого, либо двух-клиентов pattern (новый client_id с тем же audience, старый удаляется через 7д). | Hydra Admin API docs + smoke-test перед каждой ротацией |  Fallback: cutover-style ротация с микро-downtime (≤30 сек) во время `secret swap` |
| **GitHub OAuth Apps** | ⚠️ Не подтверждено — есть UI «Generate new client secret», но точная семантика grace-period для старого secret требует сверки с docs (`https://docs.github.com/en/apps/oauth-apps/maintaining-oauth-apps`) | Проверить перед ротацией | Fallback: координированная ротация с redeploy personal-mcp синхронно |
| **Google OAuth (GCP)** | ✅ Подтверждено — Cloud Console позволяет создать новый client_secret, старый остаётся валидным до явного revoke | GCP Console UI (стандартный flow) | — |

**Действие:** перед первой плановой ротацией каждого secret'а провести verification smoke-test «два secret'а активны параллельно?», обновить эту таблицу с результатами. До тех пор `ORY_CLIENT_SECRET`/`GITHUB_CLIENT_SECRET` ротируем **через cutover** (sub-минутный downtime на координированный swap), не через dual-secret window.

### 3.2. Процедура ротации (Ory Hydra пример, остальные аналогично)

```
1. Подготовка
   1.1. Hydra Admin API: создать новый secret для existing client
        curl -X PATCH .../clients/<client_id> --data '{"client_secret": "<new>"}'
   1.2. Записать BOTH секрета (новый и старый) в журнал

2. Soft transition (7 дней)
   2.1. Деплой нового secret в gateway-mcp:
        echo "<new>" | wrangler secret put ORY_CLIENT_SECRET -c DS-MCP/gateway-mcp/wrangler.toml
   2.2. Старый secret остаётся валидным в Ory (rotate later)
   2.3. Smoke test: gateway успешно делает client_credentials flow

3. Revoke old (через 7 дней)
   3.1. Hydra Admin API: удалить старый secret
   3.2. Smoke test: попытка с старым secret → 401 unauthorized
   3.3. Записать в журнал "old revoked"
```

**Особенность для Dynamic Client Registration (DCR)** в gateway-mcp: каждый MCP-клиент получает per-registration client_secret. Ротация на этом уровне не централизована — клиент сам перерегистрируется при инвалидации (`401 invalid_client` → re-register flow). Эта спецификация покрывает только "корневой" `ORY_CLIENT_SECRET` для самого gateway.

## 4. JWT signing key (Ory Hydra)

### 4.1. Параметры

| Параметр | Значение |
|----------|----------|
| Алгоритм | RS256 (RSA-SHA256, 2048-bit) — Hydra default |
| Управление | Ory Hydra managed (public/private key пара через Hydra Admin API) |
| Verification | gateway + все MCP читают public key через JWKS endpoint `https://hydra.aisystant.com/.well-known/jwks.json` |
| Частота плановой ротации | **q180d** (полугодовая) |
| Dual-key window | По умолчанию Hydra держит N последних public keys в JWKS (configurable, default 7 дней) |
| Triggered ротация | (а) compromise; (б) увольнение admin'а Hydra; (в) Ory security advisory |

### 4.2. Процедура ротации

```
1. Подготовка
   1.1. Проверить, что все MCP-сервисы корректно читают JWKS (B7.7c smoke-test endpoint)
   1.2. Проверить cache TTL у MCP — не должен превышать 1h (см. B4.21)

2. Генерация нового key pair (Hydra Admin API)
   curl -X POST .../keys/hydra.openid.id-token --data '{
     "alg": "RS256",
     "use": "sig",
     "kid": "<new-kid>"
   }'
   # Hydra сразу публикует новый public key в JWKS, но private key используется только для НОВЫХ JWT

3. Soft transition (default 7 дней)
   3.1. Existing JWT (с прошлым kid) остаются валидными до их exp
   3.2. Новые JWT подписываются новым private key, kid=<new>
   3.3. MCP-сервисы при verify читают JWKS, находят kid из header → верифицируют соответствующим public key

4. Revoke old (через 7 дней)
   4.1. Hydra Admin API: удалить старый key
        curl -X DELETE .../keys/hydra.openid.id-token/<old-kid>
   4.2. Старые JWT с этим kid становятся НЕвалидны → forced re-auth
   4.3. Smoke test: gateway-mcp /me с старым JWT → 401

5. Координация с MCP сервисами
   5.1. Обновить JWKS cache TTL в каждом MCP до 1h (если ещё не сделано)
   5.2. Проверить retry-on-signature-fail: при первом fail верификации re-fetch JWKS (см. B4.21)
```

**Безопасность Hydra master key**: сам ключ, которым Hydra шифрует свои keys в БД (`SECRETS_SYSTEM`) — отдельный класс, ротация через Hydra config + DB re-encrypt (q90d, координация с DBA Neon, downtime до 5 мин).

## 5. Координационная таблица

> **Расхождение с B2.3 разрешено:** B2.3 §Расписание ротации классифицирует OAuth secrets как «высокие q180d». Здесь `ORY_CLIENT_SECRET` повышен до **q90d**, потому что это **центральный** OAuth для всей платформы (gateway-mcp как trust boundary): компромет = compromise всех downstream MCP. Остальные OAuth secrets (`GITHUB_*`, `GOOGLE_*`) остаются q180d по B2.3. Календарь B2.3 строка 36 (`ORY_CLIENT_SECRET 2026-10-14`) обновить — следующая ротация **2026-07-14**, не октябрь.

| Класс ключа | Ротация | Downtime | Координация с | Кто инициирует |
|--------------|---------|----------|---------------|-----------------|
| Fernet master | q90d + immediate | Нет (dual-key 30 дней) | aist_bot, gateway-mcp, personal-mcp | Owner (Tseren) |
| ORY_CLIENT_SECRET | **q90d** (повышено относительно B2.3 — обоснование выше) | До 30 сек cutover (если dual-secret не подтверждён) | gateway-mcp + Ory Hydra | Owner |
| GITHUB_CLIENT_SECRET | q180d | До 30 сек cutover | personal-mcp + GitHub OAuth app | Owner |
| GOOGLE_CLIENT_SECRET | q180d | Нет (dual-secret подтверждён) | bot/google-calendar + Google Cloud Console | Owner |
| JWT signing key | q180d + immediate | Нет (JWKS soft-transition 7 дней) | Hydra + все MCP-сервисы | Owner |
| Hydra SECRETS_SYSTEM | q90d | Зависит от deploy-модели Hydra: single-instance ~5 мин, HA-cluster — rolling restart (≈0); **проверить текущую конфигурацию `mcp.aisystant.com` перед первой ротацией** | Hydra + Neon DB | Owner + DBA |

## 7. Open questions для Павла

1. **Sec gate Павла** при чтении decrypted PII (B7.3.6): ротация Fernet требует, чтобы и его инструмент знал dual-key window и читал PREVIOUS на старых rows.
