# B6.3 Disaster Recovery Runbook

> Подробный runbook восстановления после аварии. Краткая версия: B6.1.

## Сценарий 1: Полная потеря базы данных (DROP DATABASE / потеря проекта Neon)

**RTO цель:** 30 минут | **RPO цель:** 1 минута

```
T+0   Обнаружение
      - Railway: 500 ошибки на /health бота
      - Cloudflare: MCP Workers возвращают 500/503
      - Триггер: алерт или пользователь сообщает

T+5   Диагностика
      $ neonctl projects list
      $ neonctl branches list --project-id <PROJECT_ID>
      → Если проект существует — восстановление через PITR
      → Если проект удалён — обратиться в Neon support (enterprise recovery)

T+10  Создание recovery branch
      $ neonctl branch create \
          --project-id <PROJECT_ID> \
          --name recovery-$(date +%Y%m%d-%H%M) \
          --timestamp "$(date -u -v-5M +%Y-%m-%dT%H:%M:%SZ)"
      # Проверить: datetime должен быть ДО аварии

T+15  Верификация данных
      $ neonctl connection-string recovery-YYYYMMDD-HHMM > /tmp/recovery-conn.txt
      $ psql "$(cat /tmp/recovery-conn.txt)" -c "SELECT COUNT(*) FROM digital_twins;"
      $ psql "$(cat /tmp/recovery-conn.txt)" -c "SELECT COUNT(*) FROM subscription_grants WHERE revoked_at IS NULL;"
      $ psql "$(cat /tmp/recovery-conn.txt)" -c "SELECT COUNT(*) FROM users;"
      # Ожидаемо: digital_twins ~113+, subscription_grants ~771+, users ~100+

T+20  Переключение сервисов (все одновременно)
      RECOVERY_URL=$(cat /tmp/recovery-conn.txt)

      # Cloudflare Workers
      for svc in knowledge-mcp digital-twin-mcp personal-knowledge-mcp guides-mcp; do
        echo "$RECOVERY_URL" | wrangler secret put DATABASE_URL -c ~/IWE/DS-MCP/$svc/wrangler.toml
      done
      echo "$RECOVERY_URL" | wrangler secret put DATABASE_URL -c ~/IWE/DS-MCP/gateway-mcp/wrangler.toml

      # Railway aist_bot — через Railway UI или CLI:
      # railway variables set DATABASE_URL="$RECOVERY_URL" --service aist_bot

      # GitHub Actions payment-registry — через GitHub UI:
      # Settings → Secrets → NEON_AIST_BOT_URL → update

T+25  Smoke tests
      curl https://digital-twin-mcp.aisystant.workers.dev/health
      curl https://knowledge-mcp.aisystant.workers.dev/health
      # Ожидаемо: {"status":"ok",...}

T+30  Уведомление
      - Если downtime > 10 мин → уведомить пользователей в боте
      - Зафиксировать инцидент в DS-ecosystem-development/C2.3.Operations/Incidents/
```

## Сценарий 2: Случайный DROP TABLE / массовый DELETE

**RTO цель:** 20 минут | **RPO цель:** 5 минут

```
T+0   Обнаружение симптома (пустые ответы, 0 результатов поиска)

T+3   Определить время события
      $ psql "$DATABASE_URL" -c "SELECT MAX(updated_at) FROM digital_twins;"
      # Если значительно раньше текущего — подтверждение потери данных

T+5   PITR на точку ДО аварии
      $ neonctl branch create --name recovery-drop --timestamp "<время ДО DROP>"

T+10  Экспорт потерянных данных
      $ pg_dump "$(neonctl connection-string recovery-drop)" \
          --table=<affected_table> \
          --format=plain \
          --column-inserts \
          > /tmp/recovery-<table>.sql

T+15  Импорт в prod
      $ psql "$DATABASE_URL" < /tmp/recovery-<table>.sql

T+20  Верификация + закрытие
```

## Сценарий 3: Cloudflare Worker падает (deploy failure)

```
T+0   Обнаружение: Cloudflare dashboard показывает deploy failed

T+2   Откат к предыдущей версии
      # Cloudflare UI: Workers → <worker-name> → Deployments → Rollback

T+5   Или из CLI:
      wrangler rollback --message "Emergency rollback WP-212"

T+10  Smoke test + фиксация инцидента
```

## Сценарий 4: Railway bot недоступен

```
T+0   Обнаружение: бот не отвечает

T+2   Railway dashboard → Deployments → статус

T+5   Если OOM/crash → перезапустить деплой
      railway up --service aist_bot

T+10  Если DB connection issue → проверить DATABASE_URL secret
      railway variables --service aist_bot | grep DATABASE

T+15  Откат если новый деплой сломал:
      railway rollback --service aist_bot
```

# B6.4 Neon SLA и Multi-region

## Текущий SLA Neon

| Параметр | Значение |
|---------|---------|
| **Plan** | уточнить в console.neon.tech (Free/Launch/Scale) |
| **Uptime SLA** | Scale plan: 99.95% | Launch: best-effort |
| **Region** | Single region (предположительно aws-eu-central-1 или aws-us-east-1) |
| **PITR** | Free: 7 дней | Launch/Scale: 30 дней |
| **Support** | Scale: <1h response | Launch: community |

## Ограничения текущей архитектуры

| Риск | Вероятность | Impact | Митигация |
|------|------------|--------|----------|
| AWS регион недоступен | низкая (~0.01%/год) | Полный outage платформы | Neon failover на другой AZ (автоматически). Cross-region — требует Scale plan |
| Neon проект удалён случайно | очень низкая | Полная потеря данных | PITR + Neon support recovery |
| Партиционирование не работает | низкая | Рост БД без очистки | Monitoring + pg_partman alerts |

## Рекомендации

1. **Краткосрочно:** убедиться что используется Launch/Scale план (не Free) — для 30-дневного PITR
2. **Среднесрочно:** настроить Neon alerts на storage usage > 80%
3. **Долгосрочно (при росте >1000 активных пользователей):** рассмотреть Neon Scale с multi-AZ или резервный snapshot в S3 через pg_dump cron
