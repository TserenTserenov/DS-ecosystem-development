# Концепция разделения экосистемы развития интеллекта: Россия и мир

> **Для кого:** архитектор, владелец продукта, инвестор — любой, кто хочет понять, как экосистема Aisystant будет работать в двух юрисдикциях одновременно.
>
> **Детали:** [01 — Экономика](01-economic-model.md) · [02 — Инфраструктура](02-infrastructure-and-services.md) · [03 — Сообщество](03-community-and-culture.md) · [04 — Исследование](04-research-patterns.md) · [05 — Compliance](05-compliance.md)

---

## 1. Проблема

Мы строим **экосистему развития интеллекта** — систему из нескольких юридических лиц (ООО в РФ, Inc в Delaware), глобального сообщества практиков, образовательной платформы (LMS), цифровой инфраструктуры (боты, CRM, Identity, Knowledge Base) и экономики вклада (Points Engine). IWE (Intellectual Work Environment) — часть этой экосистемы, платформа Aisystant.

Экосистема работает с пользователями в России и за рубежом. Три давления одновременно:

| Давление | Суть |
|----------|------|
| **Юридическое** | Россия требует хранить ПД граждан в РФ (152-ФЗ). ЕС требует GDPR. Два режима несовместимы в одной инфраструктуре |
| **Финансовое** | Российские и международные платёжные системы изолированы (ЮKassa/Мир vs Stripe). Инвесторы не хотят видеть карту «Мир» на международном сайте |
| **Инфраструктурное** | Доступ из РФ к зарубежным сервисам нестабилен (блокировки, замедления). Neon EU может стать недоступна из РФ |

**Противоречие:** финансы и данные вынуждают разделить. Сообщество и культура требуют объединить. Единое сообщество — архитектурный инвариант, потому что экосистема создаёт созидателей, а среда развития неделима ([обоснование →](#a-почему-единое-сообщество--архитектурный-инвариант)).

## 2. Как это делают другие

> Подробный разбор: [04-research-patterns.md](04-research-patterns.md)

Все успешные глобальные платформы (Telegram, JetBrains, Wise, Linux Foundation, GitHub, Stack Overflow, Wikipedia) следуют одному паттерну: **коммерция локальна, сообщество глобально, персональные данные отделены от платформы вклада.** Ни одна из них не разделена по юрисдикциям в части вклада и знаний.

Мы реализуем этот же паттерн:

1. **Юрлица и платежи — по регионам.** ООО (РФ) + Inc (Delaware). Каждое принимает оплаты в своей юрисдикции. Пользователь не думает, в каком контуре он — маршрутизация прозрачна (как Wise)
2. **PII — на границе, в региональных Ory.** Персональные данные хранятся в юрисдикции пользователя. Community Layer содержит только UUID — не является персональными данными ни по 152-ФЗ, ни по GDPR (как GitHub отделяет billing от contributions)
3. **Сообщество и вклад — единые, без границ.** Один клуб, одна система баллов, один knowledge base. Псевдонимизация позволяет соблюсти законы и сохранить глобальное сообщество (как Linux Foundation: фонд = сообщество, компании = коммерция)

## 3. Решение: три уровня разделения

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  УРОВЕНЬ 1: ЭКОНОМИКА И КОМПАНИЯ                               │
│  Разделяем полностью                                            │
│                                                                 │
│  ООО (Россия)  ◄── лицензия ──►  Inc (Delaware, USA)           │
│  ЮKassa, Мир                      Stripe, PayPal               │
│  Рубли                            USD                           │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  УРОВЕНЬ 2: ИНФРАСТРУКТУРА И СЕРВИСЫ                            │
│  Дублируем по контурам, связываем протоколами                   │
│                                                                 │
│  Бот-RU    Ory-RU    CRM-RU       Бот-EU   Ory-EU   CRM-EU    │
│  PG-RU (PII)                       Neon-EU (PII)               │
│       │                                 │                       │
│       └────────── протоколы ─────────────┘                      │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  УРОВЕНЬ 3: СООБЩЕСТВО И КУЛЬТУРА                               │
│  Единое, без границ                                             │
│                                                                 │
│  ┌───────────────── Community Layer ──────────────────┐         │
│  │  Баллы (Points Engine)    — единая система вклада  │         │
│  │  Цифровой двойник (ЦД)    — один человек = один ЦД │         │
│  │  Знания (Knowledge MCP)  — общие курсы и материалы │         │
│  │  Identity (canonical ID) — один аккаунт навсегда   │         │
│  │  Команды (Б5)            — без границ по составу   │         │
│  └────────────────────────────────────────────────────┘         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Как это работает технически

**Псевдонимизация** — ключевой ход. В Community Layer нет персональных данных (ФИО, email, телефон). Только `ory_id` (UUID). Ключ «ory_id → ФИО» хранится в контурном Ory, в юрисдикции пользователя.

```
КОНТУР РФ                    COMMUNITY LAYER               КОНТУР МИР
┌────────────────┐     ┌──────────────────────┐     ┌────────────────┐
│ ory_id → ФИО   │     │ ory_id → баллы       │     │ ory_id → name  │
│ ory_id → email │     │ ory_id → прогресс    │     │ ory_id → email │
│ ory_id → тел.  │     │ ory_id → квалификация│     │ ory_id → addr  │
│ (PII, 152-ФЗ)  │     │ (псевдоним, ОК)      │     │ (PII, GDPR)    │
└────────────────┘     └──────────────────────┘     └────────────────┘
```

Результат: баллы, ЦД, знания — юридически не являются персональными данными. Не требуют локализации ни по 152-ФЗ, ни по GDPR.

### Единый клуб, единый knowledge base

Клуб (Discourse), knowledge base, LMS — **один инстанс каждый**, не два. Два клуба = два сообщества = смерть peer review, менторства, культурной диффузии (подробно: [03-community §4c](03-community-and-culture.md)).

```
Иван (РФ) → Ory-RU → JWT (ory_id=abc, ник=ivan_s) ─┐
                                                      ├→ Клуб (один) → тема, рецензия, дискуссия
Maria (EU) → Ory-EU → JWT (ory_id=xyz, ник=maria_s) ─┘
```

Клуб авторизует через федерацию Ory: принимает JWT от любого контурного Ory. Клуб **знает** только ory_id + публичный ник. **Не знает** ФИО, email, юрисдикцию. Интерфейс клуба — мультиязычный (Discourse i18n).

### Как выполняются законы о персональных данных

**Принцип: PII на границе, не в центре.** ПД (ФИО, email, тел.) хранятся в контурных Ory (каждый в своей юрисдикции). Community Layer содержит только UUID — юридически не является ПД. Законы обеих юрисдикций выполняются.

> Подробно: 152-ФЗ, GDPR, UGC, consent в оферте, сценарии удаления/миграции, чеклист → [05-compliance.md](05-compliance.md)

## 4. Согласование: проблема → решение

Проверка: решение (§3) отвечает на каждое давление из проблемы (§1).

| Давление | Как решается трёхуровневой моделью |
|----------|------------------------------------|
| **Юридическое** (152-ФЗ / GDPR) | PII хранится в контурных Ory (каждый в своей юрисдикции). Community Layer — псевдонимизирован, не содержит ПД. Оба закона выполняются |
| **Финансовое** (изолированные платёжные системы) | Уровень 1: два юрлица, два платёжных провайдера. Пользователь платит в своей валюте, маршрутизация прозрачна |
| **Инфраструктурное** (блокировки, нестабильность) | Уровень 2: дублирование сервисов по контурам + outbox-паттерн. PG-RU как буфер при недоступности Neon EU |
| **Инвариант** (единое сообщество) | Уровень 3: Community Layer без PII, единый клуб, единый knowledge base. Псевдонимизация позволяет соблюсти законы **и** сохранить единство |

## 6. Сценарии

### 6.1. Участник платит из-за рубежа

```
Maria (Берлин) → Бот @aist_me → /subscribe
  → CRM-EU → Stripe → IWE Inc (Delaware) → подписка активна
  → Событие: subscription_created (source_region='world')
  → Activity Hub → Points Engine → баллы начислены
  → Maria видит баллы в /points (общие, не «международные»)
```

### 6.2. Участник переезжает из РФ за рубеж

1. Баллы, ЦД, прогресс — **ничего не меняется** (привязаны к canonical_ory_id)
2. Подписка переоформляется: ЮKassa → Stripe, ООО → Inc
3. Identity: Ory-RU деактивируется, Ory-EU активируется, identity_links обновляется
4. Для участника: один переключатель. Для сообщества: невидимо

### 6.3. Neon EU недоступна из РФ

```
Событие в RU → Outbox (PG-RU) → retry → Neon EU (когда доступна)
```

Баллы начисляются с задержкой (часы). Для нефинансовой метрики — допустимо. Пользователь не замечает.

## Приложения

### A. Почему единое сообщество — архитектурный инвариант

> «Угрозы и технологии растут намного быстрее, чем растёт общечеловеческий интеллект.»
> — Манифест экосистемы

Экосистема создаёт **созидателей, которые работают на улучшение мира в целом**, а не на вендоров и страны. Сообщество — не аудитория продукта, а **среда развития**: человек развивается через взаимодействие с другими (peer review, менторство, команды Б5, культурная диффузия). Разделить сообщество по юрисдикциям = разделить среду = замедлить развитие каждого.

Три аргумента:
1. **Вклад не имеет гражданства.** Системное мышление одинаково ценно независимо от паспорта
2. **Среда развития неделима.** Уменьшить пул = замедлить каждого участника
3. **Культурная диффузия глобальна.** Участники несут системное мышление в общество — на рабочие места, в стартапы, корпорации, государственные организации

> «Каждый человек талантлив. Но не каждому повезло раскрыть свой потенциал — для этого нужен доступ к лучшим знаниям, к среде единомышленников и к ресурсам развития. Мы создаём экосистему, которая даёт это каждому — **независимо от стартовой точки, географии или социального статуса**.»
> — Манифест экосистемы

Подробное обоснование: [03-community-and-culture.md §0](03-community-and-culture.md)

### B. Почему частичное разделение не работает

Нельзя «разделить всё, кроме баллов». Баллы начисляются за взаимодействие (peer review, менторство, обсуждения в клубе). Если клуб, knowledge base и команды разделены — Мария из Берлина не может проверить работу Ивана из Москвы. Событий для начисления баллов **кратно меньше**. Экономика вклада мертва. Подробно: [03-community §4c](03-community-and-culture.md).
