# Анализ: реалистичность и интерес для инвесторов

> **Контекст:** оценка всех документов Regional-Split (00-05) на реалистичность реализации и привлекательность для инвесторов.

---

## 1. Реалистичность

### 1.1. Что сильно и реалистично

**Трёхуровневая модель — архитектурно грамотная.** Разделение «экономика локально / инфраструктура по контурам / сообщество глобально» — не изобретение, а проверенный паттерн (JetBrains, Linux Foundation, GitHub). Псевдонимизация через `ory_id` — стандартный ход, юридически обоснованный по GDPR Art. 4(5).

**Эволюционный подход — правильный.** Не строить всё сразу, а двигаться триггерами (Этап 1 → 2 → 3). Снижает риск overengineering.

**BYOK для LLM — единственно верное решение.** Зафиксировано, не обсуждается.

### 1.2. Проблемы с реалистичностью

| # | Проблема | Суть | Статус |
|---|---------|------|--------|
| 1 | **РФ + инвесторы — неразрешённое противоречие** | Олег: «РФ нету и не планируется — единственный приемлемый ответ для инвестора». Два бренда (совет Олега) — единственный реалистичный путь, но кратно усложняет всё | Открытый вопрос (§8.4 в 05-compliance) |
| 2 | **РФ-инфраструктура дорого для текущего масштаба** | k8s на Selectel/Яндекс + Ory self-hosted + PG-RU = $500-2000/мес (оценка Олега). При десятках пользователей — непропорционально | Решение не принято (WP-73, 13 апр) |
| 3 | **Юридические решения блокируют друг друга** | 9 решений в п.7, из которых 5 юридических. Все зависят от WP-186 (юрподготовка), который не начат. Без юриста ничего не двигается | WP-186 не начат |
| 4 | **Foundation model преждевременна** | При десятках пользователей Foundation = governance overhead без отдачи. Горизонт — годы, не кварталы | Документы это признают |
| 5 | **Смена региона юридически сложнее** | Apple/Google создают новую учётку + новый SLA. Текущая модель (canonical_ory_id, бесшовная миграция) может не пройти юридическую проверку | Требует юроценки |

## 2. Интерес для инвесторов

### 2.1. Что привлечёт

- **Глобальная амбиция** — экосистема развития интеллекта, не SaaS для одного рынка
- **Vendor-agnostic архитектура** — нет зависимости от одного LLM-провайдера
- **Единое сообщество с экономикой вклада** — если баллы/Points Engine заработают, это moat (конкурентное преимущество)
- **Delaware уже есть** — VC-friendly юрисдикция из коробки

### 2.2. Что оттолкнёт

| # | Фактор | Почему проблема для инвестора |
|---|--------|------------------------------|
| 1 | **РФ в уравнении** | Санкционные риски. Не может объяснить LP, зачем портфельная компания обслуживает РФ-рынок. Красный флаг при due diligence |
| 2 | **Нет revenue и traction** | 7 документов архитектуры, но нет данных о пользователях, retention, revenue, unit economics. Инвестор вкладывает в бизнес, не в архитектуру |
| 3 | **Сложность** | 9 решений, 8 РП в зависимостях, 3 этапа эволюции, 2 юрлица, 2 контура — для десятков пользователей выглядит как overengineering |
| 4 | **Foundation model без масштаба** | Foundation = «вы отдадите IP некоммерческой организации». Работает при масштабе Linux, пугает при масштабе стартапа |

## 3. Рекомендации

### 3.1. Для инвесторов нужен другой документ

Не архитектурная концепция разделения, а pitch:
- Проблема → рынок → решение → traction → команда → ask
- РФ вообще не упоминается (или: «open-source код доступен третьим сторонам по лицензии» — совет Олега)
- Фокус на US/EU сообщество + экономика вклада как moat

### 3.2. Текущие документы — внутренняя ценность

Документы Regional-Split ценны как внутренняя архитектурная проработка для команды, но не для инвестора. Инвестор не будет читать 7 файлов про pseudonymization и canonical_ory_id.

### 3.3. Главное нерешённое

**Определиться: РФ-рынок в этом юрлице или нет.** Пока это «открытый вопрос», вся архитектура висит в воздухе.

Совет Олега (два бренда, код по лицензии) — реалистичный путь:
- US/EU компания: разработка, документация, инвесторы. Только US/EU юзеры
- РФ: отдельный бренд, отдельное юрлицо, тот же код по лицензии. Никаких финансовых/юридических связей

Это нужно принять как решение, а не держать как опцию.
