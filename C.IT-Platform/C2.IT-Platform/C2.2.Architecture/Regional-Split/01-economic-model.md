# Экономика и компания: модель двух юрлиц

> **Контекст:** Принято решение о полном разделении финансов по юрлицам (ИТ-встреча 5, 9 апр).
> **Открытый вопрос:** юрисдикция международного юрлица, модель revenue sharing.
> **Связь:** WP-186 (юрподготовка), WP-183 (CRM-сервис).

---

<details open>
<summary><b>1. Три модели корпоративной структуры</b></summary>

### 1.1. Модель A: Зонтичная компания (Umbrella)

```
        ┌──────────────────┐
        │ Holding Co        │  ← международная юрисдикция
        │ (IP, бренд, код) │     (Кипр, Нидерланды, ОАЭ)
        └────────┬─────────┘
                 │
        ┌────────┴─────────┐
        │                  │
┌───────▼──────┐  ┌────────▼─────┐
│ ООО «ИВЭ»    │  │ IWE Ltd / BV │
│ Россия       │  │ мир           │
│ ЮKassa, Мир  │  │ Stripe, PayPal│
│ 152-ФЗ       │  │ GDPR          │
└──────────────┘  └──────────────┘
```

| + | - |
|---|---|
| IP централизован, защищён | Нужна третья юрисдикция для холдинга |
| Revenue sharing через лицензионные платежи | Трансфертное ценообразование = налоговый риск |
| Масштабируется на N юрисдикций | Дороже в обслуживании |

**Кто так делает:** Wise (10+ юрлиц), JetBrains (Чехия → дочки), Miro (US → NL → дочки).

### 1.2. Модель B: Два независимых юрлица + лицензия

```
┌──────────────┐            ┌──────────────┐
│ ООО «ИВЭ»    │◄──────────►│ IWE Ltd      │
│ Россия       │  лицензия   │ мир           │
│              │  на код     │              │
└──────────────┘            └──────────────┘
```

| + | - |
|---|---|
| Проще всего юридически (два договора) | Нет единого владельца IP |
| Максимальная изоляция рисков | Сложнее координировать развитие |
| Нет холдинга = меньше расходов | При конфликте учредителей — риск форка |

**Кто так делает:** Ранний JetBrains (до консолидации), некоторые локальные дистрибьюторы ПО.

### 1.3. Модель C: Фонд + коммерческие компании (Foundation)

```
        ┌──────────────────┐
        │ IWE Foundation    │  ← некоммерческий фонд
        │ (сообщество,     │     (Швейцария, Нидерланды)
        │  баллы, ЦД, код) │
        └────────┬─────────┘
                 │ governance
        ┌────────┴─────────┐
        │                  │
┌───────▼──────┐  ┌────────▼─────┐
│ ООО «ИВЭ»    │  │ IWE Ltd      │
│ Россия       │  │ мир           │
│ коммерция    │  │ коммерция    │
└──────────────┘  └──────────────┘
```

| + | - |
|---|---|
| Сообщество юридически независимо от коммерции | Сложная структура для текущего масштаба |
| Максимальная защита общего слоя | Фонд требует governance, board |
| Привлекательно для инвесторов (бизнес отделён от community) | |
| Масштабируется на N коммерческих партнёров | |

**Кто так делает:** Linux Foundation + Red Hat/Google/etc., TON Foundation + Telegram, Kotlin Foundation + JetBrains/Google.

</details>

<details>
<summary><b>2. Рекомендация: эволюционный путь</b></summary>

| Этап | Модель | Когда |
|------|--------|-------|
| **Сейчас** | РФ-юрлицо (активное) + US-юрлицо в Delaware (неактивное) | -- |
| **Этап 1** | **Модель B** — активировать Delaware + лицензия между юрлицами | Ближайший квартал |
| **Этап 2** | **Модель A** — зонтичная (если нужен EU-рынок отдельно) | При масштабировании |
| **Целевой** | **Модель C** — фонд (если сообщество вырастет) | При 1000+ активных участников |

**Почему не сразу C:** Foundation model требует governance overhead (board, bylaws, reporting). На текущем масштабе (десятки пользователей) это избыточно. Но архитектурно закладываем разделение «сообщество ≠ коммерция» уже сейчас.

**Delaware vs EU:** Delaware = стандарт для US-инвесторов, но не EU (нет GDPR-adequacy). Если фокус на EU-пользователей — может понадобиться EU-дочка (NL/EE). Если фокус на русскоязычных за рубежом — Delaware достаточен.

</details>

<summary><b>3. Финансовая модель: два юрлица</b></summary>

### 3.1. Потоки денег

```
Пользователь (РФ)                    Пользователь (мир)
      │                                     │
      ▼                                     ▼
┌─────────────┐                     ┌─────────────┐
│ ЮKassa / Мир│                     │ Stripe      │
│ ООО «ИВЭ»   │                     │ IWE Ltd     │
│ Рубли       │                     │ USD / EUR   │
└──────┬──────┘                     └──────┬──────┘
       │                                   │
       ▼                                   ▼
 ┌───────────┐                       ┌───────────┐
 │ CRM-RU    │                       │ CRM-EU    │
 │ Оплаты,   │                       │ Оплаты,   │
 │ подписки  │                       │ подписки  │
 └───────────┘                       └───────────┘
       │                                   │
       └───────────────┬───────────────────┘
                       │ событие: «оплата»
                       ▼
              ┌──────────────────┐
              │ Community Layer   │
              │ (баллы, ЦД)      │
              │ Оператор: ?      │
              └──────────────────┘
```

### 3.2. Revenue sharing между юрлицами

| Модель | Как | Налоговые последствия |
|--------|-----|---------------------|
| **Лицензионные платежи** | РФ-юрлицо платит % за использование IP | Роялти, налог у источника |
| **Сервисный контракт** | Одно юрлицо оказывает услуги другому | НДС, трансфертное ценообразование |
| **Единый контент, раздельные продажи** | Каждое юрлицо продаёт доступ к общему контенту | Нужна лицензия на контент |

**Рекомендация:** лицензионная модель. Международное юрлицо владеет IP (код, контент). Российское юрлицо = лицензиат. Простая, понятная регуляторам схема. Олег Царёв рекомендует этот подход.

### 3.3. Выбор юрисдикции для международного юрлица

| Юрисдикция | + | - | Применимость |
|-----------|---|---|-------------|
| **Кипр** | Низкие налоги (12.5% CIT), EU, русскоязычные юристы | Репутация «оффшора», FATF мониторинг | Средняя |
| **Нидерланды** | EU, IP-режим (Innovation Box 9%), солидная репутация | Дорогое обслуживание, substance requirements | Высокая |
| **Эстония** | 0% CIT на нераспределённую прибыль, e-Residency, EU | Маленький рынок, ограниченная инфраструктура | Высокая |
| **ОАЭ (DIFC/ADGM)** | 0% налогов, нейтральная юрисдикция, русскоязычное сообщество | Не EU (нет GDPR-adequacy), далеко от клиентов | Средняя |
| **Великобритания** | Солидная репутация, IP-режим (Patent Box), понятное право | Brexit = не EU, отдельный GDPR-UK | Средняя |
| **Делавэр (США)** | Стандарт для стартапов, VC-friendly | Сложная налоговая система, далеко от EU-данных | Низкая (если фокус на EU) |

**Текущая ситуация IWE:** Уже есть компания в Делавэре (США), пока неактивная. Это даёт VC-friendly юрисдикцию из коробки. Вопрос не «где открывать», а «активировать Delaware или открыть EU-юрлицо дополнительно».

### 3.4. Пользователь переезжает

| Сценарий | Что происходит |
|----------|---------------|
| РФ → мир | Аккаунт мигрирует в международное юрлицо. Подписка переоформляется. Баллы, ЦД, прогресс — остаются (общий слой) |
| Мир → РФ | Аналогично, в обратную сторону |
| СНГ | Решить: к какому юрлицу? Рекомендация: к международному (один Stripe на все не-РФ страны) |
| Беларусь | Отдельный вопрос: санкции, платёжные ограничения. Скорее всего — к международному с ограничениями |
