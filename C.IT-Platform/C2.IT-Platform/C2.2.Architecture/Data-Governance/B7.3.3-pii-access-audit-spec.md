# Спецификация аудита доступа к PII / encrypted columns / secrets

> **Назначение.** Ответ на открытый вопрос Павла из переписки 29 апр: «Аудит вот только понять бы как делать». Зафиксировать единый контракт: **что** логируется, **куда**, **под какие правила алертов**, **на сколько хранится**, **кто читает**.

---

## 1. Что логируется

> Один формат для трёх классов чувствительных данных. Триггер записи — **факт обращения**, не только успех/отказ.

| Поле | Тип | Обязательность | Пример |
|------|-----|----------------|--------|
| `id` | UUID | ✅ | `01H8N4...` |
| `ts` | timestamptz | ✅ | `2026-04-29T15:32:11Z` |
| `actor_kind` | enum (`user`, `service_account`, `admin`, `bot`) | ✅ | `service_account` |
| `actor_id` | text | ✅ | `personal-mcp@cf-worker` (для service); `user_uuid` (для user); `admin@github_login` (для admin) |
| `subject_user_uuid` | UUID | ✅ если применимо | `019018af-...` (чьи данные читаем; NULL если platform-wide read) |
| `data_class` | enum (`pii`, `payment_credentials`, `secret`) | ✅ | `pii` |
| `resource_kind` | enum (`column`, `endpoint`, `file`) | ✅ | `column` |
| `resource_ref` | text | ✅ | `persona.ory_identity.email` или `payment.SAVED_PAYMENT_TOOL.payment_method_id` или `oauth_tokens` |
| `operation` | enum (`read`, `decrypt`, `write`, `delete`, `export`) | ✅ | `decrypt` |
| `purpose_code` | enum (см. §2) | ✅ | `support_ticket_resolution` |
| `request_id` | UUID | ✅ | trace-id, привязка к Langfuse/CF logs |
| `result` | enum (`success`, `denied`, `error`) | ✅ | `success` |
| `bytes_returned` | int | ⚠️ опционально | для bulk-detect аномалий |
| `client_ip` | inet | ⚠️ опционально (PII!) | hash, не raw, кроме admin-actions |
| `decryption_key_version` | int | ⚠️ если operation=decrypt | связка с B7.3.2 ротацией |

**Важно:** само значение PII в логе **не сохраняется**. Лог фиксирует факт + контекст, не содержимое.

## 2. Purpose codes (зачем читали)

> Без purpose-кода аудит не имеет ценности — нельзя отделить легитимный read от вторжения. Каждый сервис при чтении PII обязан декларировать namespace-кодированную причину.

| Код | Когда используется | Допустимые actor_kind |
|-----|---------------------|------------------------|
| `service_routine` | Регулярная обработка (projection-worker, ETL, scheduled job) | service_account |
| `user_self_view` | Пользователь смотрит свой профиль через UI/MCP | user (subject == actor) |
| `user_request` | Пользователь явно запросил действие, требующее PII (export, send-email, push notification) | user (subject == actor) |
| `support_ticket_resolution` | Оператор поддержки решает обращение | admin (с reference на ticket-id) |
| `legal_compliance` | Запрос регулятора, GDPR Art.15 export | admin (с reference на запрос) |
| `gdpr_erasure` | Чтение для целевого удаления по GDPR Art.17 / 152-ФЗ ст.21 | service_account или admin (с reference на erasure-запрос) |
| `compliance_audit` | Внутренний compliance-цикл (DPA-проверка, ежегодное ревью) | admin (с reference на audit-id) |
| `security_incident` | Расследование инцидента | admin (с incident-id) |
| `internal_analytics` | Агрегатные расчёты для аналитики | service_account (только агрегация, без identifying-полей наружу) |
| `migration` | Однократный перенос данных | admin |

**Правило:** обращение без декларации purpose-кода → автоматически `result=denied`. Реализуется через wrapper-слой в БД-клиенте (Python: `db.read_pii(..., purpose=...)`; JS: `db.readPII(..., { purpose })`).

## 3. Куда писать (БД #8 health)

| Аспект | Решение |
|--------|---------|
| Локация | Neon БД `health` (#8 в карте 12 БД), таблица `pii_access_log` |
| Партиционирование | Range по `ts` (per-month) для retention drop через pg_partman |
| Индексы | `(subject_user_uuid, ts DESC)`, `(actor_id, ts DESC)`, `(data_class, ts DESC)`, `(purpose_code, ts DESC)` |
| Запись | Async через очередь (НЕ блокирует основной запрос); fallback в file-log при недоступности БД (chmod 600, ротация logrotate) |
| Защита | RLS с `FORCE ROW LEVEL SECURITY` (Neon): только `audit_reader` role читает. Owner таблицы ≠ `neondb_owner` (отдельная audit-owner role); чтения через `neondb_owner` тоже логируются как `actor_kind=admin` (self-monitoring через детектор A7). Без `FORCE RLS` superuser обходит политики — обязательно включить |
| Доставка | service-сторона: middleware Python-клиента БД; для прямого SQL (Metabase, ad-hoc) — DENIED через RLS |

**Почему БД #8 health, не БД источника:** см. различение в `distinctions.md` «Системная БД ≠ Health-хранилище» — health-факты живут в #8, чтобы при сбое сервиса диагностика оставалась доступной. Аудит — health-факт.

## 4. Retention

| Класс | Срок хранения | Обоснование |
|-------|---------------|-------------|
| `payment_credentials` access | **7 лет** | Налоговое законодательство РФ + 152-ФЗ обязательство на финансовые операции |
| `pii` access (не финансовое) | **18 месяцев** | Балансировка GDPR/152-ФЗ (необходимый минимум для расследования инцидентов: 90-180 дней — типовой инцидент-период; 18 мес покрывает повторные инциденты) |
| `secret` access | **3 года** | Compromise-расследование может вскрываться позже (ротация q90d → 12 раз × 90 дней + буфер) |
| Failed attempts (`result=denied`) | **3 года** | Паттерны попыток несанкционированного доступа; tampering forensics |

**Авто-удаление:** pg_partman drop-old партиций по retention-категории (отдельные партиции по `data_class`).

## 5. Alert-rules (детекторы аномалий)

> Алерты идут в TG канал `@aisystant_status` (admin-only) через alerter (см. WP-244). Каждое правило имеет cooldown — DB-backed, чтобы предотвратить шторм.

| # | Правило | Триггер | Severity | Cooldown | Действие |
|---|---------|---------|----------|----------|----------|
| **A1** | Bulk read одного user | actor читает >50 rows одного `subject_user_uuid` за 1 минуту AND `purpose_code NOT IN (user_self_view, user_request, gdpr_erasure)` (легитимный self-export может превышать 50 rows) | high | 10 мин | TG alert + автоматический block actor (revoke session) при >500 rows |
| **A2** | Bulk read across users | actor читает >100 разных `subject_user_uuid` за 5 минут AND `purpose_code NOT IN (service_routine, support_ticket_resolution, legal_compliance, compliance_audit, internal_analytics)` (поддержка/compliance/projection-worker = legitimate bulk) | high | 30 мин | TG alert + freeze actor до approve |
| **A3** | Cross-user read | `actor_kind=user` AND `subject_user_uuid != actor_id` AND `purpose_code != support_ticket_resolution` | critical | 0 (каждое срабатывание) | TG alert + блокировка немедленно |
| **A4** | Unknown actor | `actor_id` не в whitelist registered services | high | 10 мин | TG alert + reject |
| **A5** | Decryption spike | >100 decrypt-операций одного actor за 1 минуту | medium | 60 мин | TG warning |
| **A6** | Failed access spike | >20 `result=denied` от одного actor за 5 минут | high | 30 мин | TG alert + temporary throttle |
| **A7** | Off-hours admin access | `actor_kind=admin` AND `ts` вне 9:00-22:00 МСК AND нет declared incident | medium | 0 | TG warning (не блокирует) |
| **A8** | Payment credentials read | любое чтение `data_class=payment_credentials` AND `actor_kind != service_account` (для `service_routine` авторизованного pipeline) | critical | 0 | TG alert + аудит-review обязателен |

**Реализация:** SQL queries поверх `pii_access_log` через `pg_cron` (every 1 min) → emit в alerter (см. WP-244 Ф3 alerter-rules). Source-of-truth детекторов — `health.alerter_rules` таблица.

## 6. Кто читает audit log

| Роль | Доступ |
|------|--------|
| `audit_reader` (Olej Tsarev для R8 ревью; security-team) | Полный read, без write |
| `pasha-secgate@github` (B7.3.6 read-only) | Полный read через Read-only DB role + IP whitelist |
| Owner (Tseren) | Полный read через Neon admin (логируется как `actor_kind=admin`, что само по себе попадает в audit log → self-monitoring) |
| Все остальные | DENIED через RLS |

**Self-monitoring:** чтения admin'ом тоже логируются — детекторы A7 алертят при off-hours доступе.

## 7. Open questions

1. **Bulk export GDPR Art.15** — пользователь запрашивает все свои данные → один read = >1000 rows одного `subject_user_uuid`, попадёт в A1. Решение: добавить `purpose_code=user_request` исключение для `subject == actor` AND `bytes_returned < 100MB`.
2. **Метрики для Metabase** — агрегаты типа `COUNT(users) GROUP BY tier` не должны падать в audit log (это не чтение PII, а агрегация). Решение: middleware решает по `result_shape` — если возвращается `count`/`avg`/`sum` без identifying-колонок, не логировать.
3. **Retention на `failed attempts`** — 3 года достаточно? Compliance может требовать 5 лет в некоторых юрисдикциях. Open для Pasha review.
4. **Подключение к WP-244 alerter rules** — добавить 8 правил выше в `health.alerter_rules` через миграцию или отдельный РП?
