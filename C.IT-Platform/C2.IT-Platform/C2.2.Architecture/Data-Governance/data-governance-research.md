# WP-214: Концепция учёта персональных данных в IWE

> **Тип:** исследование → концепция
> **Источник аналогии:** elrussia.ru, Концепция регулирования государственного учёта, вер. 2.0 (июнь 2006)
> **Пост:** #108 «Парламент вместо президента»

## 1. Гипотеза

IWE — аналог государственных ведомств, парламента и ограниченного президента. Персональные данные пользователя:
1. Создаются и хранятся в разных доменах (как ведомства ведут свой учёт)
2. Доступ организован по чёткому протоколу (как законодательство регулирует межведомственный обмен)
3. Не один ИИ-агент, а распределённый «парламент»
4. Каждый агент имеет доступ только к определённой информации по протоколу
5. Всю информацию в одном месте собрать нельзя — архитектурное ограничение, не баг

## 2. Источник: триада учёт–доступ–аудит (elrussia.ru)

ФЦП «Электронная Россия» (2002–2010). Группа при Минэкономразвития предложила разделить государственную информацию на три института:

- **Учёт** — каждое ведомство ведёт учёт в своей зоне ответственности. Учёт это нормативно-регулируемая деятельность, а не база данных
- **Доступ** — отдельный институт регулирует, кто и к каким данным имеет доступ. Разделение учёта и доступа как разных институтов
- **Аудит** — третий институт проверяет, что первые два соблюдают правила. Независимая проверка

Триада образует систему сдержек и противовесов для информации, аналогичную разделению властей.

## 3. Маппинг на IWE

| Государство (elrussia, 2006) | IWE (2026) | Комментарий |
|------------------------------|------------|-------------|
| Ведомство (ведёт учёт в своей зоне) | Pack-репозиторий | Source-of-truth домена. OwnerIntegrity: один факт в одном месте |
| Парламент (создаёт законы) | CLAUDE.md + протоколы ОРЗ + SC + Gates | Правила: какой агент имеет доступ к каким данным, при каких условиях |
| Президент (ограниченный) | Оркестратор (Стратег) | Координирует, но не хранит данные. Архитектурно не может собрать всё |
| Гражданин | Пользователь | Субъект данных |
| Закон об учёте | Протокол ОРЗ | Кто что записывает, в каком порядке |
| Закон о доступе | SC-clauses + MCP permissions | Кто что видит. DP.D.031: публичный vs приватный |
| Аудит | Верификация (Haiku R23, context isolation) | Проверяет результат, не процесс создания |
| Инфраструктура доверия | OAuth + MCP + Ory | Удостоверение, авторизация между агентами и сервисами |
| Закон о личной тайне | RLS (Row-Level Security) + user_id фильтрация | WP-212: B4.13-B4.15 |

## 4. Что уже работает

- **Pack = ведомство:** каждый Pack хранит знания своего домена, OwnerIntegrity
- **CLAUDE.md = парламент:** правила доступа, gates, протоколы
- **MCP Access Model (DP.D.031):** публичный (knowledge-mcp) vs приватный (digital-twin-mcp)
- **RLS:** user_id фильтрация в knowledge-mcp и digital-twin-mcp (WP-212)
- **Верификация:** Haiku R23 с context isolation
- **OAuth:** GitHub, Linear, Ory через бота (WP-212 B4.11)

## 5. Чего не хватает

| Разрыв | Описание | Где решать |
|--------|----------|-----------|
| **Протокол межагентного доступа** | Нет явного «закона», описывающего какой агент к каким данным имеет право. Сейчас регулируется неявно через структуру репозиториев | WP-214 Ф3 |
| **Межсервисная аутентификация** | Сервисы не проверяют друг друга. HW-checker принимает запросы от кого угодно | WP-73 |
| **Consent layer** | Нет явного механизма consent (как Индия DEPA): что, кому, зачем, на какой срок | WP-214 |
| **Аудит вызовов** | Нет систематического логирования кто-кого-когда вызвал | Пока не в плане |
| **Реестр сервисов** | Нет единого реестра: какие сервисы существуют, какие данные хранят, какие API выставляют | MAP.002 |
| **Security Server pattern** | Нет шлюза на границе каждого домена с access control и подписью запросов | WP-214 |

## 6. Исследование: государственная инфраструктура (9 апр)

### 6.1. Эстония: X-Road — эталон

**Архитектура:** peer-to-peer. Каждое ведомство разворачивает Security Server (шлюз). Central Server распространяет конфигурацию и доверие, но НЕ маршрутизирует данные. 450+ организаций, 3000+ сервисов, 25+ стран.

**Безопасность:** mutual TLS + цифровая подпись каждого сообщения + OCSP-проверка сертификатов + batch timestamping. Три типа логов: audit (конфигурация), message (каждый запрос подписан), system (события).

**Ключевой принцип:** Centralize Trust, Decentralize Data. Данные НЕ копируются в центральное хранилище. Мотивация: утечка 1996 года, когда подрядчик создал «суперкадастр» из данных нескольких ведомств.

**Для IWE:** Security Server = MCP-сервер домена (шлюз с access control). Central Server = оркестратор (реестр доверия, не данные). Message log = audit trail каждого обращения агента.

### 6.2. Индия: India Stack — consent as infrastructure

**Архитектура:** 5 слоёв, каждый управляется отдельной организацией. Aadhaar (ID), UPI (платежи), DigiLocker (документы), DEPA (согласие), ABDM (здоровье).

**DEPA (Data Empowerment and Protection Architecture):** гражданин выдаёт Consent Artifact — цифровая запись: какие данные, кому, на какой срок, для какой цели. Account Aggregator = посредник, не хранит данные, только маршрутизирует на основании consent.

**Для IWE:** Consent layer — обязательный. Каждый доступ агента к персональным данным через consent mechanism с ограничением по времени, цели, объёму. Три разные роли: кто хранит ≠ кто передаёт ≠ кто даёт согласие.

### 6.3. Сингапур: infrastructure-first

**Подход:** сначала платформа (ID, API, design system), потом сервисы. MyInfo — «tell-us-once»: данные вводятся один раз, переиспользуются через API. Гражданин видит и контролирует, что передаётся.

**Для IWE:** Platform-first = сначала протоколы и ID-слой, потом доменные агенты. Personal data vault — единое место контроля (не хранения!) данных.

### 6.4. ЕС: федерация суверенных данных

**Три столпа:** eIDAS v2 (Digital Identity Wallet для каждого гражданина), GAIA-X (федеративная инфраструктура), European Data Spaces (15 секторальных доменов).

**Принципы:** data sovereignty (данные не покидают юрисдикцию владельца), self-sovereign identity (гражданин — хозяин), секторальные домены (каждый data space = authority).

**Для IWE:** Data Spaces = Pack-домены. Self-sovereign identity = пользователь владеет идентичностью. Digital Clearing House = compliance gateway перед допуском агента.

### 6.5. Россия: СМЭВ — централизованная шина

**Архитектура:** hub (84 узла на 7 ЦОДах). 200+ млн обменов/год. СМЭВ 4: «витрины данных» — предвычисленные ответы для скорости.

**Для IWE:** Витрины = materialized views для часто запрашиваемых данных. Антипаттерн: центральная шина = бутылочное горлышко при масштабе.

### 6.6. Антипаттерн: NHS NPfIT (10 млрд фунтов потерь)

Top-down централизация, one-size-fits-all, rushed procurement, игнорирование пользователей. **Вывод:** попытка создать единую суперсистему для всех доменов = гарантированный провал.

### 6.7. Сравнительная таблица

| | Эстония | Индия | Сингапур | ЕС | Россия |
|---|---------|-------|----------|-----|--------|
| **Топология** | Peer-to-peer | Слоёная | Platform-first | Федеративная | Hub (шина) |
| **Хранение** | У ведомств | По слоям | Профиль + источники | У владельцев | Распределённое |
| **Consent** | Межведомственный | Consent Artifact (granular) | Singpass | Self-sovereign wallet | ЕСИА |
| **Аутентификация** | Mutual TLS + PKI | Биометрия | OAuth2 + PKI | Certification framework | ViPNet + ЭЦП |
| **Audit trail** | 3 типа логов | По слоям | Централизованный | По data space | Лог СМЭВ |
| **Once-only** | Да | Частично | Да | Да | Частично |
| **Открытость** | Open source (MIT) | Closed | Открытый API | Open standards | Закрытая |

## 7. Исследование: SOTA персональных ИИ-систем (9 апр)

### 7.1. Personal Data Stores

**Solid (Tim Berners-Lee):** спецификация зрелая, adoption бедный. Pod-модель = правильная метафора, но реализация должна быть проще (файлы + API, не полный Solid-сервер).

**DIDs/Verifiable Credentials:** W3C DIDs v1.1 (Candidate Rec, апрель 2026). VC v2.0 = W3C Recommendation. Зрелый стандарт для идентификации агентов и полномочий.

**Big Tech:** ни один не запустил полноценного personal data vault. Фрагментированные хранилища (iCloud, Google Account, Microsoft Graph).

### 7.2. Agent frameworks: подходы к данным

| Фреймворк | State/Memory | Permissions |
|-----------|-------------|-------------|
| **LangGraph** | Explicit reducer-driven state, checkpointing, time travel | На уровне кода |
| **CrewAI** | Short/long/entity memory, task output propagation | Role-based |
| **AutoGen/AG2** | Conversation history | На уровне conversation |
| **OpenAI Agents SDK** | Guardrails (PII, jailbreak) | Developer responsibility |
| **Google ADK** | OAuth user auth, in-tool guardrails | IAM + service accounts |
| **MS Agent Framework 1.0** | Session state, middleware, telemetry | RBAC, audit, lifecycle |

**AWS Bedrock AgentCore Policy (GA март 2026):** Cedar-language policies для agent-to-tool. Natural language → Cedar → formal verification → enforcement. Первая система с формально верифицированными agent permissions.

### 7.3. Memory layer: Mem0

Гибридное хранилище: vector (семантический поиск) + graph (отношения) + key-value (факты). Extraction pipeline автоматически маршрутизирует знание. 26% рост accuracy, 91% снижение latency vs full-context.

### 7.4. Policy-as-code для агентов

| Engine | Модель | AI-ready |
|--------|--------|----------|
| **Cedar (AWS)** | RBAC, ABAC, ReBAC | Production (Bedrock AgentCore) |
| **OPA** | RBAC, ABAC | Широкая экосистема, Rego сложен |
| **Ory Keto** | ReBAC (Zanzibar) | Open-source, self-hosted |

**ReBAC (Relation-Based Access Control)** — наиболее выразительная модель для IWE: отношения пользователь-агент-данные (агент X member-of workspace Y owner-of document Z).

### 7.5. Privacy

| Технология | Зрелость | Для IWE |
|-----------|---------|---------|
| Differential Privacy | Production (Apple, Google) | Для аналитики поведения |
| Federated Learning | Production | Обучение без передачи данных |
| Secure Enclaves (TEE) | Production | Cloud inference через TEE |
| Homomorphic Encryption | Research | Слишком дорого для персонального use case |

### Различение: агент ≠ роль ≠ домен (DP.D.025)

**ИИ-агент** = вычислительный процесс с определённой моделью (Claude Opus, Haiku, GPT-5). Это «тело».
**Роль** = контекст, в котором агент работает (Стратег, Кодировщик, Верификатор). Это «личность» — определяет как агент мыслит, какие принципы применяет, какой опыт и знания использует.
**Домен** = область данных с определённым уровнем конфиденциальности (здоровье, образование, финансы).

Агент **исполняет** роль, а не **является** ею. Один агент может исполнять разные роли. Одну роль могут исполнять разные агенты. Правильно: «ИИ-агент Claude Opus в роли Стратега». Неправильно: «агент Стратег».

### Два измерения: домены × роли

Разделение по **доменам данных** (не по ролям!). Роли ортогональны доменам:

```
               Домены (данные)
               ┌──────────┬──────────┬──────────┬──────────┐
               │Здоровье  │Образов.  │Финансы   │Дети      │
Роли           ├──────────┼──────────┼──────────┼──────────┤
(функции)      │          │          │          │          │
 Стратег       │    ✓     │    ✓     │    ✓     │    ✓     │
 Аналитик      │    ✓     │    ✓     │    ✓     │    ✓     │
 Исполнитель   │    ✓     │    ✓     │    ✓     │    ✓     │
               └──────────┴──────────┴──────────┴──────────┘
               Наддоменный уровень: Стратег видит агрегаты
               из нескольких доменов (витрины, не сырые данные)
```

### Три типа агентов по доступу

| Тип | Доступ | Пример | Может ли видеть другие домены? |
|-----|--------|--------|-------------------------------|
| **Доменный** | Только один домен | ИИ-агент в роли аналитика медицинских данных. Видит только PACK-health | **Нет. Архитектурно невозможно** |
| **Междоменный** | Несколько доменов по протоколу (consent) | ИИ-агент в роли Стратега. Видит агрегаты из нескольких доменов | **Только витрины, не сырые данные** |
| **Инфраструктурный** | Метаданные, не данные | Оркестратор. Знает какие домены существуют и какие агенты допущены | **Видит структуру, не содержимое** |

### P13: Минимальное количество агентов

**Формулировка:** минимальное число независимых ИИ-агентов = количество доменов данных с разной конфиденциальностью + 1 междоменный + 1 инфраструктурный.

**Формула: N доменов + 2 = минимум агентов.**

Если 4 домена (здоровье, образование, финансы, дети) → минимум 6 агентов:
- 4 доменных (каждый изолирован, не может видеть другие домены)
- 1 междоменный (видит агрегаты/витрины для стратегических решений)
- 1 инфраструктурный (маршрутизация, реестр доверия)

**Почему доменный агент не может быть «общим»:** если один агент обслуживает и здоровье, и финансы, он имеет доступ к обоим доменам. Это нарушает P12 (никогда не собирать всё в одном месте). Даже если policy запрещает доступ, в context window агента могут оказаться данные из обоих доменов. Физическая изоляция (разные процессы, разные MCP-серверы) = единственная гарантия.

### Ускорение: где overhead и как его снимать

| Принцип | Замедляет? | Как ускорить на уровне IWE |
|---------|-----------|---------------------------|
| P1 Decentralize Data | Нет (локальные файлы = мс) | Уже быстро |
| P2 Security Server | Да (mTLS = 50-100мс) | Для локальных агентов: Unix socket без TLS. mTLS только для облачных |
| P3 Consent | Да, если интерактивный | **Consent policies, не consent prompts.** Один раз описал в Cedar/YAML, дальше автоматически |
| P4 Declarative Policies | Минимально (Cedar check = микросекунды) | Политики компилируются в lookup table при старте сессии |
| P5 ReBAC (Keto) | Да (сетевой запрос 5-20мс) | **Session-level permission cache** (TTL = 1 час). Embedded policy engine для local |
| P9 Audit trail | Минимально | Async write, batch flush раз в 30 сек |
| P12 Не собирать всё | Может (3 запроса вместо 1) | **Витрины данных** (materialized views для частых запросов) |

**Принцип: security на границах, скорость внутри.**
- Внешний мир (облако): mTLS + Keto + audit (полный стек)
- Gateway/MCP шлюз: consent check, policy enforcement
- Локальные агенты: Unix socket, кэш разрешений, без TLS
- Pack-файлы на диске: прямое чтение, мс

**Реальный overhead в штатном режиме: ~0.** Pack = файлы на диске, агенты = процессы на машине, policies = предкомпилированы. Overhead только при обращении к облачным сервисам, а там и так LLM inference = секунды.

## 9. Межсервисная аутентификация (из WP-73)

Проблема: сервисы не подтверждают друг другу, что они «свои». С учётом исследования:

| Вариант | Плюсы | Минусы | Аналог в госсистемах | Рекомендация |
|---------|-------|--------|---------------------|-------------|
| **API key** | Просто | Утечка = полный доступ | Нет аналога (слишком слабо) | Временно (текущее) |
| **mTLS** | Надёжно, стандарт | Сложная настройка | **X-Road** (mutual TLS + PKI) | Целевое для critical path |
| **Keto check** | Централизованная политика | Зависимость от Keto | **DEPA** consent check | Целевое для authorization |
| **mTLS + Keto** | Аутентификация + авторизация | Максимальная сложность | **X-Road + DEPA** | Финальная архитектура |

Рекомендация из исследования: **mTLS для аутентификации (кто ты?) + Keto для авторизации (что тебе можно?)**. Как X-Road: mutual TLS на транспорте, проверка полномочий через централизованную политику.
