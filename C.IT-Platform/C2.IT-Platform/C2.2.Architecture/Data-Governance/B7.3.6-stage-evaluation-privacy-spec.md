# B7.3.6: Privacy-спецификация stage-evaluation-worker

> **Контекст.** SR.001-SR.004 вычисляют переходы между ступенями мастерства (1-5) через события из `domain_event`. Этот документ отвечает на Security Gate B7.3 чеклист ДО реализации worker.

---

## 1. Новые таблицы и их PII-классификация

### `learning.stage_transitions`

| Поле | Тип | Классификация | Обоснование |
|------|-----|---------------|-------------|
| `id` | UUID | public | технический идентификатор |
| `account_id` | UUID | **косвенно PII** | Ory identity UUID, позволяет идентифицировать пользователя |
| `from_stage` | int | public | 1-5, не PII |
| `to_stage` | int | public | 1-5, не PII |
| `triggered_by` | text | public | 'SR.001', ..., 'SR.004' |
| `occurred_at` | timestamptz | public | временная метка без контента |
| `evidence` | jsonb | **косвенно PII** | может содержать event-IDs, от которых прослеживается пользователь |

**Вывод:** таблица содержит `account_id` → требует RLS.

### `learning.graduation_log`

| Поле | Тип | Классификация | Обоснование |

| `evidence_impact` | text | public | описание факта без персданных |

**Вывод:** `account_id` → требует RLS.

## 2. RLS-политики

### 2.1. Пользователь видит только свои переходы

```sql
-- learning.stage_transitions
ALTER TABLE learning.stage_transitions ENABLE ROW LEVEL SECURITY;

CREATE POLICY stage_transitions_user_isolation ON learning.stage_transitions
  USING (account_id = current_setting('app.current_user_id')::uuid);

-- Сервисные роли (projection_writer, stage_evaluator) обходят RLS через BYPASSRLS
-- Нельзя использовать просто BYPASSRLS без явного SQL-фильтра (уязвимость L2-PRIVACY)
-- Правило: projection_worker всегда добавляет WHERE account_id = $1
```

```sql
-- learning.graduation_log
ALTER TABLE learning.graduation_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY graduation_log_user_isolation ON learning.graduation_log
  USING (account_id = current_setting('app.current_user_id')::uuid);
```

### 2.2. club.members RLS (открытый вопрос WP-214 №3 → ЗАКРЫТ)

**Решение:** RLS по `ory_identity_id` обязателен. Пользователь не должен видеть `discourse_user_id` другого.

```sql
ALTER TABLE club.members ENABLE ROW LEVEL SECURITY;

CREATE POLICY club_members_user_isolation ON club.members
  USING (ory_identity_id = current_setting('app.current_user_id')::text);
```

> ⚠️ **Риск L2-PRIVACY:** сервисные роли с BYPASSRLS получают ВСЕ строки. Обязательно: каждый projection-worker-запрос к club.members добавляет явный `WHERE ory_identity_id = $account_id`.

## 3. Согласие пользователя (opt-in) — WP-214 открытый вопрос №1

**Контекст:** нужно ли явное opt-in для трекинга активности в domain_event?

**Решение (принятое):**

Двухступенчатая схема:
1. **Ступень 1-2 (Случайный/Практикующий):** события `day_plan_closed`, `iwe_session` собираются без отдельного opt-in — это базовая работа с IWE, пользователь принял TOS при регистрации. Аналог: история браузера для отображения прочитанного.
2. **Ступень 3+ и клубная активность:** требуется явное согласие на "расширенное отслеживание для расчёта баллов и ступени". Фраза в TOS клуба: «Ваша активность в systemsworld.club используется для расчёта баллов и ступени мастерства».

**Реализация:**
- Поле `learning.opt_in_tracking` в таблице пользователя (boolean, default: TRUE для новых)
- При opt-out: перестать включать события пользователя в stage_evaluation_worker, сохранять уже начисленную ступень
- **Не блокирует Ф10.3:** stage-worker при расчёте читает opt_in_flag; если false → skip

```sql
-- Таблица с opt-in статусом
CREATE TABLE IF NOT EXISTS learning.tracking_consent (
  account_id    UUID PRIMARY KEY,
  opt_in        BOOLEAN NOT NULL DEFAULT TRUE,
  opted_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  scope         TEXT[] DEFAULT ARRAY['stage_evaluation', 'club_activity']
);
```

## 4. Право на удаление (right-to-forget) — WP-214 открытый вопрос №2

**Контекст:** при выходе из клуба или удалении аккаунта → что происходит с `domain_event` и `stage_transitions`?

**Решение (принятое):**

| Таблица | Действие при deletion | Срок |
|---------|-----------------------|------|
| `domain_event` | Soft-delete: `account_id → NULL` + `payload JSONB удаляется` | В течение 30 дней после запроса |
| `learning.stage_transitions` | Soft-delete: запись сохраняется с `account_id = NULL` для аналитики агрегатов | 30 дней |
| `learning.graduation_log` | Полное удаление строки | Немедленно |
| `club.members` | Полное удаление строки | Немедленно |

**Ограничения architecture:**
- `domain_event` — append-only (Neon WAL, projection-workers); физическое удаление строки нарушает projection idempotency
- Решение: `tombstone_event` с `event_type='account_deleted'` триггерует anonymization-worker, который перезаписывает `account_id → NULL` и удаляет payload

**Реализация anonymization-worker:**
```python
# При получении tombstone 'account_deleted':
UPDATE domain_event
SET account_id = NULL,
    payload = '{"anonymized": true}'::jsonb
WHERE account_id = $deleted_account_id;

DELETE FROM learning.stage_transitions WHERE account_id = $deleted_account_id;
DELETE FROM learning.graduation_log WHERE account_id = $deleted_account_id;
DELETE FROM club.members WHERE ory_identity_id = $deleted_ory_id;
```

> ℹ️ **Бюджет:** anonymization-worker — отдельный РП (~2h), не входит в WP-214 Ф10.3. Блокирует только production-запуск, не pilot.

## 5. Audit trail для stage_transitions

Согласно B7.3.3 audit-spec: каждое обращение к PII-содержащей таблице логируется.

Для `stage_transitions` и `graduation_log`:
- **actor_kind:** `service_account` (stage_evaluation_worker)
- **purpose_code:** `service_routine` (расчёт ступени)
- **Логируется:** факт перехода (from_stage → to_stage), не evidence JSONB

```sql
-- Тригер на stage_transitions INSERT
CREATE OR REPLACE FUNCTION audit_stage_transition() RETURNS trigger AS $$
BEGIN
  INSERT INTO audit.pii_access_log (
    actor_kind, actor_id, subject_user_uuid, data_class,
    resource_kind, resource_ref, operation, purpose_code, result
  ) VALUES (
    'service_account', 'stage-evaluation-worker',
    NEW.account_id, 'pii',
    'column', 'learning.stage_transitions.account_id',
    'write', 'service_routine', 'success'
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_stage_transition_trigger
  AFTER INSERT ON learning.stage_transitions
  FOR EACH ROW EXECUTE FUNCTION audit_stage_transition();
```

## 6. W-индекс и R28 Диагност — privacy boundary

**Проблема:** W-индекс (мировоззрение) получается из диалога с R28 Диагностом. Ответы пользователя — высококонтекстные персданные.

**Решение:**
- W-рефлексии хранятся в `learning.w_reflections(account_id, session_at, quality_score, depth_level)` — **без хранения текста ответов**
- Текст диалога остаётся в контексте Telegram-сессии (не логируется в Neon)
- `w_calibrated_at` — флаг наличия оценки для SR.002/SR.003 W-fallback

```sql
CREATE TABLE IF NOT EXISTS learning.w_reflections (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id    UUID NOT NULL,
  session_at    TIMESTAMPTZ NOT NULL,
  quality_score INT CHECK (quality_score BETWEEN 1 AND 5),
  depth_level   INT CHECK (depth_level BETWEEN 1 AND 3)
  -- НЕТ поля content: текст диалога не хранится
);
```
