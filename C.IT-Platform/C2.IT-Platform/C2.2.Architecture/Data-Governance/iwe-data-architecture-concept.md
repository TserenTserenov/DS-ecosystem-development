# Концепция устройства IWE: данные, домены, агенты, роли

> **Назначение:** общая архитектурная картина IWE как персонального помощника на уровне данных, доменных областей и ИИ-агентов с ролями.
> **Источники:** ФЦП «Электронная Россия» (триада учёт-доступ-аудит), X-Road (Эстония), India Stack (DEPA), SOTA ИИ-агентов (MCP, Cedar, Keto).
> **Исследование:** [data-governance-research.md](data-governance-research.md). **Пост:** [#108 Парламент вместо президента](../../../../DS-Knowledge-Index-Tseren/docs/2026/09-апрель/2026-04-09-108-personal-data-accounting-parliament-not-president/108-1-club-2026-04-09.md).

---

## 1. Главная идея

IWE — это не один ИИ-помощник, который знает о человеке всё. IWE — распределённая система, архитектурно похожая на государство: независимые домены данных (как ведомства), протоколы взаимодействия между ними (как законодательство), и ни одна точка системы не имеет полного доступа ко всему (как разделение властей).

Персональные данные человека распределены по доменам. Каждый домен — единственный authority в своей области. Доступ между доменами — только по протоколу, с явным consent, с audit trail. ИИ-агенты исполняют роли внутри доменов и между ними, но доменный агент физически изолирован от других доменов.

Три института обеспечивают целостность: **учёт** (кто что фиксирует), **доступ** (кто что видит), **аудит** (кто проверяет). Это аналог разделения властей, но для информации.

## 3. Домены данных

**Домен** = область данных с определённым уровнем конфиденциальности и своим source-of-truth.

### Принципы доменов

- Каждый домен = единственный authority в своей области (OwnerIntegrity)
- Данные не копируются между доменами (Once-Only, P8)
- Каждый домен имеет шлюз (MCP-сервер) на границе (Security Server Pattern, P2)
- Внутренняя структура домена скрыта за шлюзом (P11: be consistent, not uniform)

### Текущие домены IWE

| Домен | Source-of-truth | MCP-сервер | Конфиденциальность |
|-------|----------------|------------|-------------------|
| Доменные знания (образование) | PACK-MIM, PACK-personal | knowledge-mcp (публичный) | Низкая (общие знания) |
| Цифровой двойник (персональные данные) | Neon DB | digital-twin-mcp (приватный) | Высокая (user_id фильтрация) |
| Стратегия (планы, приоритеты) | DS-my-strategy | — (локальные файлы) | Высокая (только владелец) |
| Платформа (архитектура, код) | PACK-digital-platform, DS-* | — (локальные файлы) | Средняя |
| Руководства | Aisystant | guides-mcp (публичный) | Низкая |

### Будущие домены (по мере роста)

| Домен | Что хранит | Конфиденциальность |
|-------|-----------|-------------------|
| Здоровье | Медицинские данные, привычки, сон | Критическая |
| Финансы | Доходы, расходы, инвестиции | Критическая |
| Семья (дети) | Данные о детях, их развитие | Критическая |
| Социальные связи | Контакты, взаимодействия | Высокая |
| Карьера | CV, навыки, достижения | Средняя |

## 4. ИИ-агенты

**ИИ-агент** = вычислительный процесс с определённой моделью. Это «тело». Агент не равен роли (DP.D.025).

### Три типа агентов по доступу к данным

| Тип | Доступ | Изоляция | Аналог в государстве |
|-----|--------|----------|---------------------|
| **Доменный** | Только один домен. Физически не может видеть другие | Отдельный процесс, свой MCP-сервер, свои permissions | Ведомство |
| **Междоменный** | Несколько доменов, но только витрины (агрегаты, не сырые данные) | Видит предвычисленные представления по consent | Межведомственная комиссия |
| **Инфраструктурный** | Метаданные: какие домены существуют, какие агенты допущены | Видит структуру, не содержимое | Канцелярия |

### Минимальное количество агентов (P13)

**Формула: N доменов + 2**

- N доменных агентов (по одному на каждый домен с уникальной конфиденциальностью)
- 1 междоменный (видит витрины для стратегических решений)
- 1 инфраструктурный (маршрутизация, реестр доверия)

Текущий IWE (5 доменов): минимум 7 агентов.
Будущий IWE (10 доменов): минимум 12 агентов.

### Почему физическая изоляция, а не policy

Если один агент обслуживает и здоровье, и финансы, в его context window могут оказаться данные из обоих доменов. Policy может запретить, но LLM может «проболтаться» (prompt injection, утечка через reasoning). Физическая изоляция (разные процессы, разные MCP) = единственная гарантия.

## 5. Роли

**Роль** = контекст, личность. Определяет как агент мыслит, какие принципы применяет, какой опыт использует.

### Роли ортогональны доменам

Один и тот же Стратег может работать:
- В домене «здоровье» (стратегия лечения)
- В домене «образование» (стратегия развития)
- На наддоменном уровне (жизненная стратегия, видит витрины из нескольких доменов)

Но **доменный агент** в роли Стратега в домене «здоровье» физически не видит домен «финансы». А **междоменный агент** в роли Стратега видит витрины из обоих.

### Текущие роли IWE

| Роль | Что делает | Где может работать |
|------|-----------|-------------------|
| Стратег (R1) | Планирование, приоритеты, стратегические решения | В любом домене + наддоменный |
| Кодировщик (R6) | Реализация, код, конфигурации | В домене «платформа» |
| Верификатор (R23) | Проверка по чеклистам, context isolation | В любом домене (read-only) |
| Консультант | Ответы на вопросы по доменным знаниям | В домене «знания» |
| Навигатор (R27) | Траектория развития ученика | В домене «образование» + ЦД |
| Диагност (R28) | Определение ступени мастерства | В домене «образование» + ЦД |
| Автор контента | Написание постов | В домене «знания» (read) |

### Доменные роли vs общие роли

| Тип роли | Описание | Пример |
|----------|----------|--------|
| **Общая** | Может исполняться в любом домене и на наддоменном уровне | Стратег, Верификатор |
| **Доменная** | Имеет смысл только в конкретном домене | Навигатор (только образование), Кодировщик (только платформа) |

Общая роль на доменном агенте = ограничена одним доменом (агент изолирован).
Общая роль на междоменном агенте = видит витрины нескольких доменов.

## 6. Триада учёт–доступ–аудит

Три института, которые обеспечивают целостность системы:

### 5.1. Учёт (кто что фиксирует)

| Компонент IWE | Что учитывает | Принцип |
|--------------|--------------|---------|
| Pack-репозиторий | Доменные знания (source-of-truth) | OwnerIntegrity: один факт в одном месте |
| DS-репозиторий | Реализационные решения | HD #29: Pack-знание ≠ реализационное |
| Цифровой двойник | Персональные данные, прогресс | user_id фильтрация |
| CLAUDE.md + протоколы | Правила работы системы | Version-controlled, auditable |

### 5.2. Доступ (кто что видит)

| Механизм | Что регулирует | Статус |
|----------|---------------|--------|
| MCP permissions | Какой сервер какие данные выставляет | Работает (DP.D.031) |
| RLS (Row-Level Security) | Фильтрация по user_id | Работает (WP-212) |
| CLAUDE.md gates | Блокировка действий без проверки | Работает |
| SC-clauses | Обещания пользователям | Работает |
| Consent policies (Cedar/YAML) | Декларативные permissions агентов | **Не реализовано** |
| Keto (ReBAC) | Relation-based access control | **Планируется** (WP-73) |

### 5.3. Аудит (кто проверяет)

| Механизм | Что проверяет | Статус |
|----------|-------------|--------|
| Верификатор (Haiku R23) | Чеклисты, context isolation | Работает |
| Git history | Все изменения прослеживаемы | Работает |
| Audit log вызовов | Кто-кого-когда вызвал | **Не реализовано** |
| Batch timestamping | Юридическая доказательность | **Не реализовано** |

## 7. 13 архитектурных принципов

> Полные формулировки и источники: [data-governance-research.md](data-governance-research.md) §8

| # | Принцип | Короткая формулировка |
|---|---------|----------------------|
| P1 | Централизуй доверие, децентрализуй данные | Оркестратор = реестр доверия, данные в доменах |
| P2 | Каждый домен = шлюз | MCP-сервер: access control + audit + подпись |
| P3 | Consent обязателен | Без consent artifact = отказ в доступе |
| P4 | Декларативные политики | Permissions в Cedar/YAML, не в коде |
| P5 | ReBAC как модель доступа | Отношения: агент → роль → домен → данные |
| P6 | Три разделённые роли | Хранит ≠ транспортирует ≠ аудирует |
| P7 | Local-first | Чувствительное на устройстве |
| P8 | Once-only (OwnerIntegrity) | Один факт — одно место |
| P9 | Audit trail обязателен | Каждое обращение = лог |
| P10 | Infrastructure-first | Сначала протоколы, потом агенты |
| P11 | Be consistent, not uniform | Единые границы, свобода внутри |
| P12 | Никогда не собирать всё в одном месте | Ни один агент не видит всё |
| P13 | Минимум агентов = домены + 2 | N доменных + 1 междоменный + 1 инфраструктурный |

## 8. Ускорение: security на границах, скорость внутри

```
Внешний мир (облако)     ←— mTLS + Keto + audit (полный стек)
        │
   [Gateway / MCP шлюз]  ←— consent check, policy enforcement
        │
Локальные агенты          ←— Unix socket, кэш разрешений, без TLS
        │
   [Pack-файлы на диске]  ←— прямое чтение, мс
```

- Session-level permission cache (TTL = 1 час)
- Consent = policy, не prompt (один раз описал, дальше автоматически)
- Витрины данных (materialized views для частых кросс-доменных запросов)
- Async audit (buffer, flush раз в 30 сек)
- Реальный overhead в штатном режиме: ~0 (файлы на диске, policies предкомпилированы)
