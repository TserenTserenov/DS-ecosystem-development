# Карта Персона / Память / Контекст

> **Назначение.** Однозначно ответить на вопрос Павла из переписки 29 апр: «промпты без показывания клиентских кредов — но трудность к привязыванию к двойнику. Накопительная история должна быть и привязка к человеку».
>
> **Решение.** Расщепить «двойника» на три слоя с разными writer/owner. Накопительная история живёт в **Памяти** (под RLS+Fernet), в **Контекст** LLM уходят только агрегаты-производные **без PII**, привязка — через `user_uuid`, не через email/login/telegram_id.
>
> **Терминология.** В этом документе «Контекст» используется как читаемый синоним канонической сущности **Проекция** (DP.ARCH.007). В архитектурном дискурсе и кодовых ссылках использовать «Проекция»; «Контекст» — обиходный термин для LLM context window / промпт-сборки. См. DP.D.052 §2 примечание.

---

## 1. Три слоя (writer + owner)

| Слой | Writer (кто пишет) | Owner (где source-of-truth) | Что хранит | TTL |
|------|--------------------|------------------------------|------------|-----|
| **Персона** | Пользователь (или агент по acceptance) | Git пользователя (приватный/публичный) | PACK-personal, captures, fleeting-notes, цели, preferences, Ory identity (декларация: email, telegram_id) | Постоянно, владелец удаляет |
| **Память.Observed** | Платформа (в момент события) | Neon (БД `learning`, `payment`, `engagement`) | Append-only events: test_answered, payment_succeeded, session_start | Регулируется retention (B3.5) |
| **Память.Derived** | Платформа (при пересчёте) | Neon (БД `indicators`, `persona`) | Агрегаты без сырых событий: BKT skill_mastery, HLR memory_decay, baseline/potential, RCS | Перевычисляется из Observed |
| **Контекст** (= Проекция) | Агент в runtime | Не хранится долго (in-flight) | LLM-промпт, OLM view, nudge message | Жизнь одного запроса |

**Тест границы** — «что пропадёт, если удалю X?»: Git пользователя → Персона. Neon → Память. Прервал LLM-вызов → Контекст.

---

## 2. Что уходит наружу (LLM-провайдер)

> **Главный инвариант (отвечает на вопрос Павла):** в LLM-промпт уходит **Контекст**, собранный из **Памяти.Derived** (агрегатов без PII) + (опционально) кусков **Персоны** под GDPR-согласие. **Сырые Observed-события и идентификаторы реального человека не уходят никогда.**

| Что | Можно ли в LLM-промпт | Обоснование |
|-----|----------------------|-------------|
| `user_uuid` (Ory UUID) | ✅ Да (как opaque ID для атрибуции трейсинга) | Не привязан к реальному человеку без БД-связки |
| `skill_mastery` (BKT-агрегат) | ✅ Да | Derived, без PII |
| `baseline`/`potential` метрики | ✅ Да | Derived, без PII |
| Текст captures из PACK-personal | ✅ Только под consent (GDPR Art.6) | Персона, владелец сам публикует в LLM |
| `email`, `telegram_id`, ФИО, login | ❌ Никогда | PII, не нужно для рассуждения LLM |
| Сырое сообщение пользователя боту | ⚠️ Только текущая реплика, через PII-redact/sanitize слой | Не накопительная история; identifying фрагменты (email/телефон в тексте) удаляются на входе в LLM. Накопление — в Память.Observed под RLS, в LLM не возвращается |
| OAuth `access_token`/`refresh_token` | ❌ Никогда | Secrets, никогда не в промптах |

**Атрибуция в Langfuse-трейсах** — через `user_uuid`, без `email`/`telegram_id`. Если расследование требует выйти на конкретного человека — JOIN в БД `persona` через RLS+аудит чтения (B7.3.3).

## 3. Где живут PII (защита)

| Слой | Защита | Подробно |
|------|--------|----------|
| Персона (Git пользователя) | Приватный репозиторий, владелец сам управляет | Платформа read-only через `personal_*` MCP-инструменты |
| Память.Observed (Neon `learning`/`engagement`) | RLS per `user_uuid` + Fernet column-level на чувствительных полях | RLS — WP-228 Ф6; Fernet — WP-234 / B2.5 |
| Память.Derived (Neon `indicators`/`persona`) | RLS per `user_uuid` | Агрегаты не содержат PII raw, но `user_uuid` остаётся → RLS обязателен |
| Контекст (LLM in-flight) | Минимизация: только Derived; Langfuse PII-фильтр на input/output | DP.SC.112 freemium subscription gate; Langfuse trace `user_id=user_uuid` |

**Классы данных** — см. `B7.3.1-l2-data-classification-map.md`. Специально для этой карты:
- email/telegram_id/ФИО → **PII** (защита: RLS + Fernet в Памяти; запрет в LLM-промпт)
- ory_id (UUID) → **PII**, но **opaque** (можно в трейс/промпт как ID)
- OAuth tokens → **secrets ∩ PII** (Fernet + RLS, никогда в промпт)
- payment_method_id → **payment_credentials** (строже PII, никогда в промпт)

## 4. «Накопительная история» — как реализовано

Ответ на ключевой пункт Павла «накопительная история должна быть и привязка к человеку»:

1. **История событий** — Память.Observed (append-only event log в Neon, привязка через `user_uuid`).
2. **Состояние ученика** — Память.Derived (агрегаты, привязка через `user_uuid`).
3. **Промпт LLM** — Контекст, собранный агентом: только Derived-агрегаты + текущая реплика. История в промпт **не уходит сырьём**; LLM получает «состояние к моменту», не «список действий».
4. **Привязка к реальному человеку** — JOIN `user_uuid → email/telegram_id` живёт в БД `persona` под RLS, доступен только service-роли с аудитом чтения (B7.3.3).

**Принцип:** LLM рассуждает о состоянии абстрактного `user_uuid`, не о «Дмитрии с email`@domain`». Расследование/поддержка — отдельный путь через БД с аудитом.

## 5. Открытые вопросы для Павла

1. **Embeddings personal-документов** (knowledge-MCP). Вектор содержит семантику личного содержания → формально PII производное. Сейчас живёт в БД `knowledge` под RLS. Нужно ли Fernet column-level для `embedding`? (B7.3.5 решит на уровне выбора слоёв.)
2. **Captures-flow** — пользователь явно говорит боту «запомни X». Сейчас это идёт в Память.Observed как event, далее предлагается в Персону через `personal_propose_capture`. Согласие на хранение — через UI бота, явно или implicit? (Зависит от B8.0 ToS/Privacy.)
3. **Pre-Grant** (proto-Persona) — лид до Ory identity, лежит как `subscription_grants.claim_token`. Не имеет `user_uuid` ещё. Атрибуция в LLM (если он что-то пишет лиду до claim) — через `claim_token` (opaque) или вообще не в LLM? (Текущий ответ — анонимный пресет, без обращения к LLM с привязкой.)
