# B7.3.1 L2 Data Classification Map

> **Назначение.** Single-source матрица «класс × слой × режим» для архитектурных решений по логированию, шифрованию, доступу, cross-user агрегации. Дополняет распределённое описание классификации в `DP.ARCH.004 §2 П6.1` и `distinctions.md` («PII ≠ payment_credentials»). Без этой карты ArchGate §Б чек-лист не имеет точки опоры — каждое РП с PII решает «что считать PII» заново.
>
> **Источник классов:** `distinctions.md` («PII ≠ payment_credentials», DP.ARCH.004 §2 П6.1).
> **Источник слоёв:** WP-250 Ф-F онтология IWE (L1 платформенный stack / L2 Aisystant продукт / L3 publishable template / L4 personal).
> **Связь с MVP 1 мая (WP-250):** B7.3.1 — блокер MVP доверия холодной когорты. Без формализованной классификации команда не может ответить на «как защищены мои данные?» однозначно.

## 1. Четыре класса данных

> Класс = ответ на вопрос «что произойдёт при утечке этого поля?».

| Класс | Что попадает | Тест принадлежности | Утечка → последствие |
|-------|-------------|---------------------|----------------------|
| **public** | Контент платформы (L1 знания, L2 руководства), агрегаты без индивидуальных полей (`COUNT(users) GROUP BY tier`), публичные ID (название курса) | «Этот факт можно опубликовать в README, ничьё доверие не нарушится?» Да → public. | Нет последствий |
| **PII** | email, telegram_id, ory_id (как идентификатор реального человека), first/last_name, location, raw содержимое личных документов, `digital_twins.data` (профиль), `user_events.payload` с user_uuid | «Этот факт привязан к конкретному человеку и помогает его идентифицировать или раскрывает его поведение?» Да и не payment → PII. | Регуляторный риск (GDPR/152-ФЗ), потеря доверия пользователя, доксинг |
| **payment_credentials** | YooKassa `payment_method_id`, `provider_token_ref` для автосписаний, `masked_id` (BIN+last4), webhook signature secrets | «Можно ли с этим инициировать списание у пользователя или раскрыть его платёжные привычки?» Да → payment_credentials. | **Финансовый ущерб пользователю**, не только приватности. PCI DSS требования. Регуляторный риск выше PII. |
| **secrets** | API keys (Anthropic/OpenAI/Linear/Railway/CF), DB passwords, Ory client_secret, GitHub PAT, Telegram bot token, RSA/Ed25519 private keys, HMAC keys | «Можно ли с этим выдать себя за нашу систему или получить наш privileged-доступ?» Да → secrets. | Полный compromise сервиса/всех пользователей, не отдельной записи |

**Граничные случаи:**

- **`telegram_id`** = PII (привязан к реальному пользователю Telegram). Тест: можно ли по нему написать конкретному человеку через @username? Да (через `tg://user?id=...`). Не путать с `chat_id` для публичного канала — там public.
- **`ory_id` (UUID)** = PII (внутренний идентификатор реального человека) — даже без email/telegram_id, утечка списка `ory_id` + поведенческих событий = поведенческий fingerprint. **Не secret** (не даёт привилегий).
- **`finance_payments.amount`** = PII (раскрывает поведение конкретного пользователя), но **не payment_credentials** (с amount нельзя инициировать списание).
- **OAuth `access_token`/`refresh_token` в БД (`ory_tokens`, `github_connections`, `dt_tokens`)** = **secrets** (с токеном можно выдать себя за пользователя в чужой системе), хранятся в БД пользователя — формально приписаны к нему, но утечка одного = compromise integration, не одного личного факта. Защищаются как secrets (Fernet, encrypted at-rest, не логируются), но **владение** — пользователь, не платформа. Двойная природа: secrets + per-user RLS.
- **OpenAI embeddings вектор личного документа** = PII производное (вектор содержит семантику личного содержания, при reverse-engineering можно частично восстановить). Класс: PII (наследует от исходного документа).

## 3. Режим обращения с данными

> Что значит «защита» по классу. Шесть осей режима.

### 3.1. Логирование

| Класс | Application logs (stdout, Railway, CF tail) | Audit log (security/compliance) | LLM трейсы (Langfuse) |
|-------|---------------------------------------------|--------------------------------|----------------------|
| public | ✅ свободно | ✅ свободно | ✅ свободно |
| PII | ⚠️ **только маскированно** (`u-{first8}`, `tg-***`) или **тип события без значения** (`event=user_login`, без user_id raw) | ✅ полно (audit-log = legitimate purpose, см. WP-237 7-year retention для платежей) | ⚠️ **disable если PII в промпте**: пропускать LANGFUSE_SECRET_KEY если в context user data, или sanitize до отправки |
| payment_credentials | 🚫 **запрещено в любом виде**, даже маскированно (last4 в логах = частичная утечка) | ✅ только в `audit.log` JSONB (с RLS audit_reader access) | 🚫 запрещено |
| secrets | 🚫 **запрещено в любом виде** | ⚠️ только metadata факта использования (`secret_used: api_key_anthropic`), не значение | 🚫 запрещено (детектор pre-output: B7.7) |

**Правило `FORBIDDEN_FIELDS` (event-driven):** при сериализации event payload, поля класса PII/payment_credentials/secrets фильтруются на etl-уровне. Реализовано в WP-268 Ф2 (`telegram_id` убран из 7 event payloads, см. commit `9a71434`). Источник конфигурации — отдельный yaml-файл с listing per-event-type.

### 3.2. Шифрование

| Класс | At-rest (Neon managed AES-256) | Column-level (application-side, Fernet/pgcrypto) | In-transit |
|-------|-------------------------------|-------------------------------------------------|------------|
| public | ✅ inherited | ❌ не нужно | ✅ TLS обязательно |
| PII | ✅ inherited (B3.4) | ⚠️ **рекомендовано для high-risk полей**: содержимое `digital_twins.data`, raw payload `user_events`, личные документы (если хранятся отдельно от Knowledge embeddings) | ✅ TLS обязательно |
| payment_credentials | ✅ inherited | ✅ **обязательно**: `provider_token_ref`, `masked_id` — Fernet через app-level (не Neon-side, чтобы Neon admin не видел) | ✅ TLS обязательно |
| secrets | ✅ inherited (если в БД) | ✅ **обязательно**: OAuth-токены (WP-234 Fernet), HMAC, API keys в БД | ✅ TLS обязательно |

**Конкретная цель WP-234** = column-level Fernet encryption для OAuth-токенов (5 таблиц: `ory_tokens`, `dt_tokens`, `github_connections`, `google_calendar_connections`, `user_integrations`). Класс этих данных — secrets с per-user RLS. Без B7.3.1 непонятно, почему их шифровать — теперь явно: secrets ∩ persisted-in-DB.

### 3.3. Доступ (Authorization)

| Класс | Per-user (RLS) | Role-based (Metabase, Directus) | Admin (DBA, neondb_owner) |
|-------|----------------|---------------------------------|---------------------------|
| public | свободно | свободно | свободно |
| PII | ✅ строгий (RLS на all PII tables, см. B9.x) | ⚠️ **только views без PII** (`metabase_reader` SELECT (`telegram_id`, `region`, `created_at`), без `email`/`first_name`) | ⚠️ только при инциденте, аудит чтений (DP.SC.032 `dt_view_requested`) |
| payment_credentials | ✅ строгий + дополнительный audit при каждом доступе | ❌ **запрещено даже в маскированном виде** (Metabase не должен видеть payment_methods) | ⚠️ только service-account для платёжного pipeline + аудит (WP-237 trigger) |
| secrets | N/A (secrets не в RLS-таблицах per-user, кроме OAuth-токенов которые имеют двойную природу) | ❌ запрещено | ⚠️ только при rotation (B2.6 cascade runbook) |

**Правило обязательного RLS:** любая новая таблица содержащая PII или payment_credentials с user-level ownership ДОЛЖНА иметь RLS-политику до первого insert на prod. Lint-проверка в CI (B4.26, расширение в B9.5). Без user-level (агрегаты/каталоги) — pomet `platform_only` в `sync-manifest.yaml`.

### 3.4. Retention (срок хранения)

| Класс | Минимум | Максимум | Триггер удаления |
|-------|---------|----------|------------------|
| public | как нужно | indefinite | по решению владельца контента |
| PII | как требует функционал | indefinite (для активных аккаунтов), 30-90 дней rolling (sessions, request_traces, FSM state) | (а) GDPR Art.17 запрос пользователя; (б) account closure; (в) TTL по B3.5 |
| payment_credentials | как требует функционал | **7 лет** (compliance, налоговое законодательство — WP-237 audit log) | account closure → автосписания отозваны, аудит-лог сохраняется 7 лет с обезличиванием |
| secrets | как требует функционал | **до ротации**, не дольше расписания B2.3 | revoke по B2.6 cascade rotation runbook |

### 3.5. Cross-user агрегация (новое правило, B7.3.1)

> **Принципиальное правило:** L2-агрегация поверх PII разрешена только в одном из режимов: (а) явный consent на participation в исследовании, (б) агрегаты без индивидуальных полей (`COUNT/AVG/SUM GROUP BY` без `WHERE user_id` или с большой когортой).

| Сценарий | Разрешено? |
|----------|-----------|
| `SELECT COUNT(*) FROM users` (общее число) | ✅ public-агрегат |
| `SELECT COUNT(*) FROM users GROUP BY region` (агрегат с N>=10 в каждой группе) | ✅ public-агрегат с k-anonymity threshold k=10 |
| `SELECT email, last_login FROM users` (индивидуальные строки) | ❌ только с consent или admin-аудитом |
| `SELECT user_id, AVG(session_duration) FROM events GROUP BY user_id` (per-user статистика) | ⚠️ **только для self-service** (пользователь видит свои метрики) или с явным consent (исследование) |
| ML-обучение на агрегированных моделях поведения (без attribution) | ⚠️ требует privacy review + consent в ToS |
| Sharing данных партнёрам (даже агрегаты) | 🚫 **запрещено без отдельного DPA** |

**k-anonymity threshold:** для агрегатов содержащих чувствительные группировки (region, qualification_level) применять минимум k=10 (если в группе <10 человек — fold в "other") при экспорте за пределы платформы.

### 3.6. Внешние интеграции (data leaving the system)

> См. полную карту в B3.2. Здесь — правила решения «можно ли отправлять X в Y».

| Direction | public | PII | payment_credentials | secrets |
|-----------|--------|-----|---------------------|---------|
| → Anthropic Claude API | ✅ | ⚠️ **минимизировать** в system prompt; role codes вместо имён; никаких email/telegram_id | 🚫 | 🚫 |
| → OpenAI Embeddings | ✅ | ⚠️ **только если пользователь явно загрузил личный документ** (consent через GitHub repo); raw текст уходит, embedding **нельзя забыть** post-факт | 🚫 | 🚫 |
| → OpenAI Reranking (gpt-4o-mini) | ✅ | ⚠️ sanitize перед отправкой (убрать имена/контакты) | 🚫 | 🚫 |
| → Langfuse (traces) | ✅ | ⚠️ **disable если PII в промпте** (опт-аут) | 🚫 | 🚫 |
| → Aisystant LMS (legacy) | ✅ | ⚠️ только `telegram_id`/`aisystant_suser_id` через Bridge-2 (mutual mapping) | 🚫 | 🚫 |
| → YooKassa | ✅ | `email` (для receipt) + `metadata.telegram_id` (для reconciliation) | ✅ (это и есть назначение) | 🚫 (наш HMAC только мы знаем) |
| → GitHub API (Personal MCP) | ✅ | ✅ (per-user OAuth scope) | ✅ (приватный репо пользователя) | 🚫 (наши GitHub App secrets никогда не уходят) |

**Правило «cross-user агрегация во внешнюю систему»:** даже агрегаты не отправляются в третьесторонний LLM/analytics без явного consent. Внешние системы могут стать источником утечки.

## 5. Применение карты в ArchGate (B7.3 Security Gate)

> §Б чек-лист в `archgate` SKILL.md обновляется так:

```
§Б. Security Gate (BLOCKING для РП с PII / payment_credentials / secrets):

1. Какой класс данных затронут? (public / PII / payment_credentials / secrets)
   → если public → Б-блок пропустить.

2. На каком слое размещаются? (L1 / L2 / L3 / L4)
   → проверить таблицу из B7.3.1 §2 «где какие классы могут жить»: разрешено?

3. Логирование: соответствует §3.1 B7.3.1?
   → для PII: только маскированно или тип-без-значения.
   → для payment_credentials/secrets: запрещено в любом виде.

4. Шифрование at-rest + column-level: соответствует §3.2?
   → для secrets/payment_credentials: column-level Fernet обязателен.

5. RLS: есть политика per-user (если user-level ownership)?
   → если нет — лента до первого insert на prod БЛОКЕР.

6. Cross-user агрегация: соблюдено правило §3.5?
   → если экспорт PII во внешнюю систему — DPA подписан?

≥2 ❌ → Безопасность = ❌ → ArchGate FAIL.
```

## 6. Открытые вопросы (для ревью Паши)

1. **Двойная природа OAuth-токенов** (secrets ∩ PII): согласен ли Паша с тем что владение — пользователь (RLS), но защита — как secrets (Fernet)? Альтернатива: считать pure-secrets, без RLS — тогда любой бэкенд service-account видит все токены при компрометации.

2. **`finance_payments` НЕ payment_credentials**: достаточна ли защита через RLS + аудит без column-level encryption? Аргумент за — `amount + telegram_id` информация о поведении, не credential для списания. Аргумент против — в комбинации с другими данными даёт fingerprint.

3. **k-anonymity threshold k=10** для агрегатов с группировкой — взято as-is из общей практики (HIPAA Safe Harbor использует k=10 для small-cell suppression). Нужен ли мини-эксперимент: текущие наши агрегаты (region, qualification_level) — везде ли проходят? Возможно для МСК/СПб проходят, для редких регионов — нет.

4. **OpenAI embeddings retention**: открытый вопрос с 14 апр (B3.1 «Open question»). Если пользователь удаляет личный документ, embedding в OpenAI остаётся **навсегда** (нельзя forget). Решение должно быть в B7.3.1 или в WP-214 Data Governance? Кандидат: явный consent перед первым личным embedding + warning в UI.

5. **`audit.log` ретеншн**: 7 лет применимо ко всем PII-логам или только финансовым? Возможно для не-финансовых событий 90-180 дней достаточно (GDPR-совместимо).

6. **Проксирование external LLM требует ли local-redaction layer**: Прокси-сервис между нашими бэкендами и OpenAI/Anthropic, который применяет правила §3.6 на лету — переоценка vs делать sanitize в каждом call-site? Если делать — это новый сервис (WP-NN), бюджет ~10-15h.
