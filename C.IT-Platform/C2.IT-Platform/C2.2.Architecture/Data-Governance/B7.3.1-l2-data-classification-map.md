# L2 Data Classification Map

> **Назначение.** Single-source матрица «класс данных × слой × режим обращения» для архитектурных решений по логированию, шифрованию, доступу, retention, cross-user агрегации и внешним интеграциям. Без формализованной классификации каждое архитектурное решение с PII решает «что считать чувствительным» заново — это и есть основной риск, который этот документ устраняет.
>
> **Срок:** до 1 мая (блокер MVP-доверия для холодной когорты).

## 1. Четыре класса данных

> Класс = ответ на вопрос «что произойдёт при утечке этого поля?».

| Класс | Что попадает | Тест принадлежности | Утечка → последствие |
|-------|-------------|---------------------|----------------------|
| **public** | Контент платформы (открытые знания, руководства), агрегаты без индивидуальных полей (`COUNT(users) GROUP BY tier`), публичные ID (название курса) | «Этот факт можно опубликовать в README, ничьё доверие не нарушится?» Да → public. | Нет последствий |
| **PII** | email, telegram_id, ory_id (как идентификатор реального человека), first/last_name, location, raw содержимое личных документов, профиль пользователя, события активности с user_id | «Этот факт привязан к конкретному человеку и помогает его идентифицировать или раскрывает его поведение?» Да и не payment → PII. | Регуляторный риск (GDPR/152-ФЗ), потеря доверия пользователя, доксинг |
| **payment_credentials** | YooKassa `payment_method_id`, `provider_token_ref` для автосписаний, `masked_id` (BIN+last4), webhook signature secrets | «Можно ли с этим инициировать списание у пользователя или раскрыть его платёжные привычки?» Да → payment_credentials. | **Финансовый ущерб пользователю**, не только приватности. PCI DSS требования. Регуляторный риск выше PII. |
| **secrets** | API keys (Anthropic/OpenAI/Linear/Railway/CF), DB passwords, Ory client_secret, GitHub PAT, Telegram bot token, RSA/Ed25519 private keys, HMAC keys | «Можно ли с этим выдать себя за нашу систему или получить наш privileged-доступ?» Да → secrets. | Полный compromise сервиса/всех пользователей, не отдельной записи |

**Граничные случаи:**

- **`telegram_id`** = PII (привязан к реальному пользователю Telegram). Тест: можно ли по нему написать конкретному человеку через @username? Да (через `tg://user?id=...`). Не путать с `chat_id` для публичного канала — там public.
- **`ory_id` (UUID)** = PII (внутренний идентификатор реального человека) — даже без email/telegram_id, утечка списка `ory_id` + поведенческих событий = поведенческий fingerprint. **Не secret** (не даёт привилегий).
- **`finance_payments.amount`** = PII (раскрывает поведение конкретного пользователя), но **не payment_credentials** (с amount нельзя инициировать списание).
- **OAuth `access_token`/`refresh_token` в БД** = **secrets** (с токеном можно выдать себя за пользователя в чужой системе), хранятся в БД пользователя — формально приписаны к нему, но утечка одного = compromise integration. Защищаются как secrets (Fernet, encrypted at-rest, не логируются), но **владение** — пользователь, не платформа. Двойная природа: secrets + per-user RLS.
- **OpenAI embeddings вектор личного документа** = PII производное (вектор содержит семантику личного содержания, при reverse-engineering можно частично восстановить). Класс: PII (наследует от исходного документа).

## 2. Слои IWE

> Где «живут» данные.

| Слой | Что | Кто видит | Канал распространения |
|------|-----|-----------|----------------------|
| **L1 Stack** (платформенное знание: онтология, методы, агенты) | Знание — формализованная общедоступная база | Все (open source) | git public repos |
| **L2 Aisystant** (массовый продукт: бот, MCP-сервисы, БД health/learning/persona/payment/subscription/indicators/knowledge/community/lead/rewards) | Все пользователи (multi-tenant) | Backend services, Cloudflare Workers, Railway, Neon | Web/Bot UI, MCP API, не git |
| **L3 Template** (открытый шаблон рабочей среды) | Структура для нового пилота (плоский шаблон) | Любой публичный пользователь | git public repo |
| **L4 Personal** (`~/IWE/` пользователя) | Per-user content: личные знания, заметки, captures, fleeting-notes | Только владелец | git private repo (или приватный hosting), Personal MCP |

**Где какие классы могут жить:**

| Класс | L1 Stack | L2 Aisystant | L3 Template | L4 Personal |
|-------|---------|--------------|-------------|-------------|
| public | ✅ всё | ✅ всё | ✅ всё | ✅ всё |
| PII | ❌ запрещено | ✅ обязательно (с защитой по §3) | ❌ запрещено (template — public) | ✅ владелец сам распоряжается |
| payment_credentials | ❌ запрещено | ✅ только в БД payment-registry с защитой по §3 | ❌ запрещено | ❌ запрещено (платёжные данные не должны попадать в Personal) |
| secrets | ❌ запрещено (только placeholder в `params.yaml.template`) | ✅ обязательно (env / wrangler secrets / Railway / `.secrets/`) | ❌ запрещено | ❌ запрещено в git, ✅ можно в `~/IWE/.secrets/` |

## 3. Режим обращения с данными

> Что значит «защита» по классу. Шесть осей режима.

### 3.1. Логирование

| Класс | Application logs (stdout, Railway, CF tail) | Audit log (security/compliance) | LLM трейсы (Langfuse) |
|-------|---------------------------------------------|--------------------------------|----------------------|
| public | ✅ свободно | ✅ свободно | ✅ свободно |
| PII | ⚠️ **только маскированно** (`u-{first8}`, `tg-***`) или **тип события без значения** (`event=user_login`, без user_id raw) | ✅ полно (audit-log = legitimate purpose, 7-year retention для платежей) | ⚠️ **disable если PII в промпте**: пропускать LANGFUSE_SECRET_KEY если в context user data, или sanitize до отправки |
| payment_credentials | 🚫 **запрещено в любом виде**, даже маскированно (last4 в логах = частичная утечка) | ✅ только в `audit.log` JSONB (с RLS audit_reader access) | 🚫 запрещено |
| secrets | 🚫 **запрещено в любом виде** | ⚠️ только metadata факта использования (`secret_used: api_key_anthropic`), не значение | 🚫 запрещено (детектор pre-output) |

**Правило `FORBIDDEN_FIELDS` (event-driven):** при сериализации event payload, поля класса PII/payment_credentials/secrets фильтруются на ETL-уровне. Реализовано (например, `telegram_id` убран из 7 event payloads). Источник конфигурации — отдельный yaml-файл с listing per-event-type.

### 3.2. Шифрование

| Класс | At-rest (Neon managed AES-256) | Column-level (application-side, Fernet/pgcrypto) | In-transit |
|-------|-------------------------------|-------------------------------------------------|------------|
| public | ✅ inherited | ❌ не нужно | ✅ TLS обязательно |
| PII | ✅ inherited | ⚠️ **рекомендовано для high-risk полей**: содержимое профиля пользователя, raw payload событий, личные документы (если хранятся отдельно от Knowledge embeddings) | ✅ TLS обязательно |
| payment_credentials | ✅ inherited | ✅ **обязательно**: `provider_token_ref`, `masked_id` — Fernet через app-level (не Neon-side, чтобы Neon admin не видел) | ✅ TLS обязательно |
| secrets | ✅ inherited (если в БД) | ✅ **обязательно**: OAuth-токены (Fernet), HMAC, API keys в БД | ✅ TLS обязательно |

**Главный приоритет column-level шифрования сейчас** = OAuth-токены в 5 таблицах (`ory_tokens`, `dt_tokens`, `github_connections`, `google_calendar_connections`, `user_integrations`). Класс этих данных — secrets с per-user RLS. Без явной классификации непонятно, почему их шифровать — теперь явно: secrets ∩ persisted-in-DB.

### 3.3. Доступ (Authorization)

| Класс | Per-user (RLS) | Role-based (Metabase, Directus) | Admin (DBA, neondb_owner) |
|-------|----------------|---------------------------------|---------------------------|
| public | свободно | свободно | свободно |
| PII | ✅ строгий (RLS на all PII tables) | ⚠️ **только views без PII** (`metabase_reader` SELECT (`telegram_id`, `region`, `created_at`), без `email`/`first_name`) | ⚠️ только при инциденте, аудит чтений |
| payment_credentials | ✅ строгий + дополнительный audit при каждом доступе | ❌ **запрещено даже в маскированном виде** (Metabase не должен видеть payment_methods) | ⚠️ только service-account для платёжного pipeline + аудит-trigger |
| secrets | N/A (secrets не в RLS-таблицах per-user, кроме OAuth-токенов которые имеют двойную природу) | ❌ запрещено | ⚠️ только при rotation (cascade runbook) |

**Правило обязательного RLS:** любая новая таблица содержащая PII или payment_credentials с user-level ownership ДОЛЖНА иметь RLS-политику до первого insert на prod. Без user-level (агрегаты/каталоги) — пометка `platform_only` в манифесте.

### 3.4. Retention (срок хранения)

| Класс | Минимум | Максимум | Триггер удаления |
|-------|---------|----------|------------------|
| public | как нужно | indefinite | по решению владельца контента |
| PII | как требует функционал | indefinite (для активных аккаунтов), 30-90 дней rolling (sessions, request_traces, FSM state) | (а) GDPR Art.17 запрос пользователя; (б) account closure; (в) TTL по retention-политике |
| payment_credentials | как требует функционал | **7 лет** (compliance, налоговое законодательство) | account closure → автосписания отозваны, аудит-лог сохраняется 7 лет с обезличиванием |
| secrets | как требует функционал | **до ротации**, не дольше расписания rotation policy | revoke по cascade rotation runbook |

### 3.5. Cross-user агрегация

> **Принципиальное правило:** L2-агрегация поверх PII разрешена только в одном из режимов: (а) явный consent на participation в исследовании, (б) агрегаты без индивидуальных полей (`COUNT/AVG/SUM GROUP BY` без `WHERE user_id` или с большой когортой).

| Сценарий | Разрешено? |
|----------|-----------|
| `SELECT COUNT(*) FROM users` (общее число) | ✅ public-агрегат |
| `SELECT COUNT(*) FROM users GROUP BY region` (агрегат с N>=10 в каждой группе) | ✅ public-агрегат с k-anonymity threshold k=10 |
| `SELECT email, last_login FROM users` (индивидуальные строки) | ❌ только с consent или admin-аудитом |
| `SELECT user_id, AVG(session_duration) FROM events GROUP BY user_id` (per-user статистика) | ⚠️ **только для self-service** (пользователь видит свои метрики) или с явным consent (исследование) |
| ML-обучение на агрегированных моделях поведения (без attribution) | ⚠️ требует privacy review + consent в ToS |
| Sharing данных партнёрам (даже агрегаты) | 🚫 **запрещено без отдельного DPA** |

**k-anonymity threshold:** для агрегатов содержащих чувствительные группировки (region, qualification_level) применять минимум k=10 (если в группе <10 человек — fold в "other") при экспорте за пределы платформы.

### 3.6. Внешние интеграции (data leaving the system)

> Правила решения «можно ли отправлять X в Y».

| Direction | public | PII | payment_credentials | secrets |
|-----------|--------|-----|---------------------|---------|
| → Anthropic Claude API | ✅ | ⚠️ **минимизировать** в system prompt; role codes вместо имён; никаких email/telegram_id | 🚫 | 🚫 |
| → OpenAI Embeddings | ✅ | ⚠️ **только если пользователь явно загрузил личный документ** (consent через GitHub repo); raw текст уходит, embedding **нельзя забыть** post-факт | 🚫 | 🚫 |
| → OpenAI Reranking (gpt-4o-mini) | ✅ | ⚠️ sanitize перед отправкой (убрать имена/контакты) | 🚫 | 🚫 |
| → Langfuse (traces) | ✅ | ⚠️ **disable если PII в промпте** (опт-аут) | 🚫 | 🚫 |
| → Aisystant LMS (legacy) | ✅ | ⚠️ только `telegram_id`/`aisystant_suser_id` через bridge (mutual mapping) | 🚫 | 🚫 |
| → YooKassa | ✅ | `email` (для receipt) + `metadata.telegram_id` (для reconciliation) | ✅ (это и есть назначение) | 🚫 (наш HMAC только мы знаем) |
| → GitHub API (Personal MCP) | ✅ | ✅ (per-user OAuth scope) | ✅ (приватный репо пользователя) | 🚫 (наши GitHub App secrets никогда не уходят) |

**Правило «cross-user агрегация во внешнюю систему»:** даже агрегаты не отправляются в третьесторонний LLM/analytics без явного consent. Внешние системы могут стать источником утечки.

## 5. Применение карты в Security Gate

> Чек-лист безопасности для архитектурных решений (БЛОКИРУЮЩЕЕ для решений с PII / payment_credentials / secrets):

```
1. Какой класс данных затронут? (public / PII / payment_credentials / secrets)
   → если public → блок безопасности пропустить.

2. На каком слое размещаются? (L1 / L2 / L3 / L4)
   → проверить таблицу §2 «где какие классы могут жить»: разрешено?

3. Логирование: соответствует §3.1?
   → для PII: только маскированно или тип-без-значения.
   → для payment_credentials/secrets: запрещено в любом виде.

4. Шифрование at-rest + column-level: соответствует §3.2?
   → для secrets/payment_credentials: column-level Fernet обязателен.

5. RLS: есть политика per-user (если user-level ownership)?
   → если нет — БЛОКЕР до первого insert на prod.

6. Cross-user агрегация: соблюдено правило §3.5?
   → если экспорт PII во внешнюю систему — DPA подписан?

≥2 ❌ → Безопасность = ❌ → решение FAIL.
```

## 6. Открытые вопросы для ревью

1. **Двойная природа OAuth-токенов** (secrets ∩ PII): согласен ли ты с тем что владение — пользователь (RLS), но защита — как secrets (Fernet)? Альтернатива: считать pure-secrets, без RLS — тогда любой бэкенд service-account видит все токены при компрометации.

2. **`finance_payments` НЕ payment_credentials**: достаточна ли защита через RLS + аудит без column-level encryption? Аргумент за — `amount + telegram_id` информация о поведении, не credential для списания. Аргумент против — в комбинации с другими данными даёт fingerprint.

3. **k-anonymity threshold k=10** для агрегатов с группировкой — взято as-is из общей практики (HIPAA Safe Harbor использует k=10 для small-cell suppression). Нужен ли мини-эксперимент: текущие наши агрегаты (region, qualification_level) — везде ли проходят? Возможно для МСК/СПб проходят, для редких регионов — нет.

4. **OpenAI embeddings retention**: открытый вопрос. Если пользователь удаляет личный документ, embedding в OpenAI остаётся **навсегда** (нельзя forget). Кандидат: явный consent перед первым личным embedding + warning в UI.

5. **`audit.log` ретеншн**: 7 лет применимо ко всем PII-логам или только финансовым? Возможно для не-финансовых событий 90-180 дней достаточно (GDPR-совместимо).

6. **Проксирование external LLM требует ли local-redaction layer**: Прокси-сервис между нашими бэкендами и OpenAI/Anthropic, который применяет правила §3.6 на лету — переоценка vs делать sanitize в каждом call-site? Если делать — это новый сервис, бюджет ~10-15h.
