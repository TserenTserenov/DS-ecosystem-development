# B3.4 Neon Encryption at-rest

## Статус Neon managed encryption

Neon PostgreSQL предоставляет encryption at-rest на уровне хранилища:

| Уровень | Статус | Метод |
|---------|--------|-------|
| **Disk encryption** | ✅ Включено по умолчанию | AES-256 (AWS EBS/S3 encryption) |
| **WAL archive encryption** | ✅ Included | Шифруется перед записью в S3 |
| **In-transit (TLS)** | ✅ Обязательно | TLS 1.2+ для всех connections |
| **Column-level encryption** | ❌ Не реализовано | B2.5 pending — pgcrypto или app-level |

**Вывод:** Neon покрывает disk-level encryption автоматически. Дополнительных действий для B3.4 не требуется.

**Рекомендация для B2.5:** Column-level encryption для `*_token` полей в `user_integrations`, `github_connections`, `google_calendar_connections`, `dt_tokens`, `ory_tokens` — через pgcrypto или application-level AES-256-GCM. Задача: WP-212 B2.5 (pending, ~4-6h).

# B3.5 Data Retention Policy

## Политика хранения по типу данных

| Тип данных | Таблица | Срок хранения | Автоматическое удаление | Правило |
|-----------|---------|--------------|------------------------|---------|
| **Сессии FSM** | `development.user_state`, `development.fsm_states` | 30 дней неактивности | ✅ midnight job (B4.7, DONE) | `WHERE updated_at < NOW() - INTERVAL '30 days'` |
| **Event stream** | `development.user_events` | 90 дней rolling | ❌ Pending (требует cron job) | Партиционирование по `fetched_at` — см. migration 004 |
| **Raw events** | `development.raw_events` | 90 дней | ❌ Pending | Партиция `LIST(source)` + `RANGE(fetched_at)` — pg_partman auto-drop |
| **Request traces** | Langfuse | 30 дней | ✅ Langfuse managed | Настраивается в Langfuse project settings |
| **Личные знания (L4)** | `documents` (user_id IS NOT NULL) | До удаления репо / GDPR request | ❌ Manual only | Пользователь удаляет репо → нужен webhook→cleanup |
| **Платформенные знания (L2)** | `documents` (user_id IS NULL) | Indefinite (платформенный контент) | N/A | Управляется платформой |
| **Digital Twin** | `digital_twins` | До удаления аккаунта | ❌ Pending B3.6 | GDPR erasure workflow |
| **Users / Identity** | `users`, `ory_tokens`, `dt_tokens` | До удаления аккаунта | ❌ Pending B3.6 | GDPR erasure workflow |
| **OAuth connections** | `github_connections`, `google_calendar_connections` | До отзыва интеграции / удаления аккаунта | ❌ Partial (bot /revoke command) | Полный cascade при удалении аккаунта — B3.6 |
| **Финансовые записи** | `finance_payments`, `workshop_payments` | 7 лет (налоговое законодательство РФ) | ❌ Manual | Нельзя удалять по GDPR-запросу — override налоговым правом |
| **Embeddings** | `documents.embedding` | Синхронно с документом | ⚠️ Orphan risk | При удалении `content` — embedding остаётся в Neon, но данные в OpenAI несвязаны |

## Pending cron jobs для автоматической очистки

```sql
-- user_events: очистка > 90 дней (activity-hub Neon, partitioned table)
-- pg_partman настроен через migration 004, auto-drop при правильных параметрах
-- Проверить: SELECT partman.check_retention('development.raw_events');

-- qa_history: очистка > 90 дней (если таблица существует в aist_bot)
DELETE FROM qa_history WHERE created_at < NOW() - INTERVAL '90 days';

-- user_state: очистка неактивных > 30 дней (DONE B4.7)
DELETE FROM development.user_state WHERE updated_at < NOW() - INTERVAL '30 days';
```

## OpenAI Embeddings — Open Question

**Проблема:** OpenAI генерирует embedding по тексту и кеширует его. После удаления документа из Neon:
- Запись в `documents` удалена ✅
- Neon больше не хранит текст ✅
- Embedding в поле `documents.embedding` тоже удалён ✅ (CASCADE)
- **НО:** OpenAI может кешировать embedding на своей стороне

**Решение (pending WP-214):** Решить при разработке GDPR erasure flow — учесть что "правое на забвение" в отношении OpenAI embeddings технически ограничено.
