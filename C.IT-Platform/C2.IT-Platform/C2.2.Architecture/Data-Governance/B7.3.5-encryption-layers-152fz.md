# Слои шифрования данных под 152-ФЗ

> **Назначение.** Ответ на радикал Павла из переписки 29 апр: «практика такая была, где 152-ФЗ маячит — шифрование всей БД на уровне БД или прямо диска». Зафиксировать выбранный набор слоёв L1/L2/L3 и обосновать достаточность для текущей фазы (MVP-доверие, дедлайн 1 мая) с явным upgrade-path при росте регуляторных требований.

---

## 1. Три возможных слоя

| Слой | Что шифрует | Защищает от чего | Стоимость | Стандарт 152-ФЗ |
|------|-------------|------------------|-----------|------------------|
| **L1: Disk/at-rest (Neon managed)** | Весь физический volume в S3/EBS | Кража физического диска / снапшота AWS | $0 — включено по умолчанию | Базовый минимум (orgrn 1119 §13) |
| **L2: Column-level (Fernet AES-128-CBC + HMAC)** | Конкретные поля per-column в приложении до записи | Утечка БД (admin Neon, дамп через `pg_dump`, leaked credentials) | Низкая (~10-50ms на ключевые поля, лента ключей через KMS/secret store) | Эквивалент СКЗИ КС1-КС2 при правильной ротации (B7.3.2) |
| **L3: Whole-table (pgcrypto/Transparent)** | Все строки целевых таблиц через pgcrypto или TDE | Утечка дампа БД, защита аналитика-инсайдера | Высокая (10-30% latency, ломает индексы по зашифрованным полям, ломает Metabase-аналитику) | Эквивалент СКЗИ КС3 |

---

## 2. Решение для текущей фазы (MVP-доверие, апрель-май 2026)

| Слой | Статус | Покрытие |
|------|--------|----------|
| **L1 (Neon at-rest)** | ✅ Active | Все 9 БД (по умолчанию, AES-256 EBS/S3) |
| **L2 (Fernet column-level)** | 🔄 In progress (WP-234, B2.5) | OAuth-токены (`github_connections`, `google_calendar_connections`, `ory_tokens`, `dt_tokens`, `user_integrations`); payment_credentials (будущая `payment.SAVED_PAYMENT_TOOL.payment_method_id` / `provider_token_ref` / `masked_id` — `finance_payments` остаётся под RLS, не Fernet, см. B7.3.1 §4) |
| **L3 (whole-table pgcrypto)** | ❌ Defer | — |

**Решение:** L1 + L2, без L3.

**Обоснование:**
1. **L1 покрывает базовый сценарий** — кража физического носителя (Neon = AWS managed, AES-256 EBS).
2. **L2 покрывает реалистичные сценарии утечки** — leaked admin credentials Neon, утечка `pg_dump`, инсайдер с read-only DB-доступом. Защита целевая: только PII и payment_credentials (см. B7.3.1 классы).
3. **L3 избыточен на текущем масштабе:**
   - Ломает индексы по зашифрованным полям → перестаёт работать поиск по `email`/`telegram_id` без расшифровки.
   - Ломает Metabase-аналитику (Metabase читает БД напрямую).
   - Замедляет прод-запросы на 10-30%.
   - Pgcrypto whole-table управление ключами через `pgcrypto.key_id` = вторая система ключей (помимо Fernet) → дублирование операционной поверхности.
   - **152-ФЗ не требует L3 явно для категории К3-К4** (см. §3); L3 = «параноидальная страховка» при К1-К2 (медданные, биометрия).

## 3. Соответствие 152-ФЗ

> **Две независимые оси классификации.** (1) **Категория ПДн** (что обрабатывается) — 4 типа. (2) **Уровень защищённости (УЗ)** ИСПДн (какие меры обязательны) — 4 уровня по ПП РФ №1119, определяется комбинацией категории ПДн × числа субъектов × актуальных угроз. Путать оси — частая ошибка. Документ ниже использует обе корректно.

### 3.1. Категория ПДн в IWE

| Категория ПДн (152-ФЗ ст.10) | Применимость к IWE | Что попадает |
|-------------------------------|---------------------|--------------|
| **Специальные** (раса, политика, здоровье, интим) | ❌ Не собираем | — |
| **Биометрические** (фото лица для идентификации, отпечатки) | ❌ Не собираем | — |
| **Общедоступные** (по ст.8: общедоступные источники с согласия) | ❌ Не наша модель | — |
| **Иные ПДн** (любые ПДн, не попадающие в три выше) | ✅ Это про нас | email, telegram_id, ФИО, события активности, captures |

### 3.2. Уровень защищённости ИСПДн (ПП РФ №1119)

УЗ определяется тройкой: **категория ПДн × число субъектов × тип актуальных угроз** (1-3-й тип).

| УЗ | Когда применяется | Применимо к IWE | Достаточен ли L1+TLS+L2+RLS? |
|----|-------------------|------------------|--------------------------------|
| **УЗ-1** | Специальные ПДн любого объёма + угрозы 1-2-го типа; биометрия + угрозы 1-го типа | ❌ Нет (специальных/биометрии не обрабатываем) | — |
| **УЗ-2** | Иные ПДн >100k субъектов + угрозы 1-го типа; специальные ПДн + угрозы 3-го типа | ❌ Нет (текущая модель угроз — 3-й тип, без угроз НДВ ОС/ПО) | — |
| **УЗ-3** | **Иные ПДн >100k субъектов + угрозы 3-го типа** | ✅ Это наш сценарий при росте до >100k пилотов | ✅ Да (см. ПП №1119 §13: СЗИ от НСД, антивирус, межсетевой экран, регламент криптозащиты при передаче — TLS) |
| **УЗ-4** | Иные ПДн ≤100k субъектов + угрозы 3-го типа; обезличенные ПДн | ✅ Это наш сценарий **сейчас** (десятки пилотов) | ✅ Да (требования мягче УЗ-3) |

### 3.3. Что обязательно для УЗ-3/УЗ-4

ПП РФ №1119 §13 не требует:
- Шифрования всей БД на уровне СУБД (L3 pgcrypto whole-table) — **не требуется**.
- Использования сертифицированных СКЗИ КС3 для самих ПДн в БД — **не требуется** (нужно для УЗ-1/УЗ-2 со специальными категориями).

Требуется:
- СЗИ от несанкционированного доступа (✅ покрывается RLS + аудит чтения).
- Регламент криптозащиты канала передачи (✅ TLS 1.2+).
- Антивирус, межсетевой экран на инфраструктуре (✅ Cloudflare + Neon managed).

**Вывод:** L1 (Neon at-rest) + TLS + L2 (Fernet column-level на OAuth/payment_credentials) + RLS + аудит = **достаточный набор контролей для УЗ-3 и УЗ-4**.

### 3.4. Когда нужен upgrade до L3

- Расширение в **специальные категории** (медицинский трекер health-метрик с диагнозами, биометрическая идентификация) → переход в УЗ-1/УЗ-2 → требуется СКЗИ КС2-КС3 → L3 уровень шифрования.
- Появление **актуальных угроз 1-2-го типа** (модель угроз пересматривается ФСТЭК-методиками при изменении ландшафта) → возможно переход на УЗ выше при тех же ПДн.
- B2B-сегмент с регуляторными контрактами (банковский compliance, госзаказчики, ПДн ст.10 ч.1) → отдельные ТЗ выше базовых УЗ-3.

## 4. Управление ключами Fernet (L2)

| Аспект | Решение | Подробно |
|--------|---------|----------|
| **Алгоритм** | Fernet (AES-128-CBC + HMAC-SHA256), стандарт `cryptography` Python | Симметричное authenticated encryption, secure-by-default |
| **Master key storage** | Cloudflare Worker secrets (для gateway-mcp / personal-mcp) + Railway env (для bot, content-pipeline) | Не в БД, не в git, не в логах |
| **Key derivation** | Один master key per environment (prod / pilot), не per-user | Простота rotate / restore |
| **Rotation policy** | См. **B7.3.2** — q90d плановая, immediate при compromise. Dual-key window 30 дней для re-encryption всех existing rows | Подробности — отдельный артефакт |
| **Audit** | Каждая расшифровка — запись в `health.pii_access_log` (см. **B7.3.3**) | `user_uuid`, `column`, `actor`, `ts` |
| **Disaster recovery** | Master key backup в офлайн-хранилище (печатный лист в сейфе), процедура — B6.1 backup-dr-policy | Потеря master без backup = потеря всех зашифрованных данных |

## 5. Что **не** делаем (и почему)

| Подход | Решение | Обоснование |
|--------|---------|-------------|
| Whole-table pgcrypto | ❌ Defer | См. §2: ломает индексы и Metabase, не требуется К3 |
| Application-level encryption всех PII (email, telegram_id) | ❌ Не делаем | email/telegram_id используются в индексах, поиске, JOINах — шифрование сделает массовые операции невозможными. Защищаются через RLS + L1 + аудит чтения (B7.3.3) |
| Hashing (one-way) email вместо encryption | ❌ Не подходит | Hashing ломает обратную операцию (поддержка пользователя «найти меня по email»). Шифрование с возможностью расшифровки под аудит — лучшее решение |
| Encryption keys в Vault (HashiCorp / OpenBao) | ⚠️ Defer (Q3 2026) | Согласовано с Павлом 28-29 апр: «начал бы со стабилизации в целом стека» → Vault после стабилизации. CF Worker secrets / Railway env — достаточно для MVP |
| pgcrypto для FK-полей (`user_uuid`, `payment_id`) | ❌ Не делаем | Эти поля — opaque UUID, не PII raw. Шифрование UUID = поломка JOINов без выигрыша приватности |
