# Процесс онбординга: Персональное руководство пилота

> ⚠️ Этот файл — рабочий черновик. **Source-of-truth:** `DP.M.038` в `PACK-digital-platform/pack/digital-platform/03-methods/`.
> Актуальная версия с исправленным порядком этапов и ВДВ-проверкой — там.

---

## Этап 0. Получение всего необходимого

| | |
|---|---|
| **Вход** | Новый пилот без аккаунта на платформе |
| **Действие** | Пилот регистрируется на платформе → создаёт GitHub-аккаунт (если нет) → оформляет подписку → даёт согласие на обработку персональных данных → платформа связывает идентификаторы в `identity_map` |
| **Выход** | 5 записей в Neon, подтверждающих готовность: |

| Артефакт | Физическое подтверждение | Где |
|---|---|---|
| Аккаунт платформы | Строка в `users` | Neon (`learning` DB) |
| GitHub-аккаунт | `github_login` заполнен в `identity_map` | Neon |
| Подписка | Строка в `subscriptions` / `contracts` | Neon (`learning` DB) |
| Согласие на персданные | `opt_in=true, scope=stage_evaluation` в `tracking_consent` | Neon (`learning` DB) |
| Связка identity | Строка `(ory_uuid, github_id, telegram_id)` в `identity_map` | Neon |

**Gate:** все 5 записей существуют → переход к Этапу 1. Отсутствует любая → процесс не начинается.

## Этап 1. Создание репо

| | |
|---|---|
| **Вход** | 5 записей в Neon (выход Этапа 0) + репо `personal-guide` отсутствует на GitHub-аккаунте пилота |
| **Действие** | Агент в роли **Оркестратора** вызывает `create_repository(name: "personal-guide")` → GitHub API |
| **Выход** | Пустой репо `github.com/<pilot>/personal-guide` (0 файлов) + `personal_guide_repo_url` записан в `identity_map` (Neon) |

**Подтверждение в двух местах:**
- GitHub — репо существует и доступен
- Neon — `identity_map.personal_guide_repo_url` заполнен

**Идемпотентность:** если репо уже существует (HTTP 409) — записать URL в `identity_map` и перейти к Этапу 2.

---

## Этап 2. Доставка контента

| | |
|---|---|
| **Вход** | Пустой репо (выход Этапа 1) + RCS-слоты `(W, M1, M2, M4)` в `Память.Derived` (Neon) |
| **Действие** | Агент в роли **Навигатора (R27)** вызывает `/personal-guide-render`: читает RCS → вычисляет ступень `stage_raw = min(W, M1, M2, M4)` и bottleneck → выбирает заготовку `stage-N × domain-X` из `PACK-personal/personal-guide-seeds/` → подставляет плейсхолдеры → записывает файлы через `personal_write` → commit + push |
| **Выход** | 8 артефактов в репо + commit в GitHub: |

| Файл | Содержимое |
|---|---|
| `README.md` | Навигация, дата сборки, ступень, домен |
| `profile.md` | RCS-профиль, bottleneck, неудовлетворённости, целевой ритм |
| `worldview.md` | Фаза мировоззренческой дуги (FORM.087), мемы |
| `methods.md` | Методы под bottleneck из доменной вставки |
| `weekly/YYYY-Www.md` | Гипотеза недели, слоты, ДЗ |
| `daily/YYYY-MM-DD.md` | Тактика дня, критерий закрытия, вопрос для чата |
| `history/reflection-template.md` | Шаблон рефлексии (5 вопросов, 5 мин) |
| `.claude/skills/` | 5 скиллов для работы в любом канале Claude Code |

**Подтверждение:** GitHub — commit существует, 8 файлов в репо.

**Если RCS отсутствует** (пилот не проходил оценку): использовать fallback заготовку `stage-1-random.md` → только `profile.md` с одним вопросом.

## Этап 3. Регистрация GitHub App

| | |
|---|---|
| **Вход** | Репо с 8 файлами (выход Этапа 2) + GitHub App «Aisystant Personal Guide» не установлена на репо пилота |
| **Действие** | Агент в роли **Оркестратора** вызывает `/connect-guide` → строит URL `https://…/auth/github_app/setup?telegram_user_id={N}` → пилот открывает браузер, логинится в GitHub, выбирает репо `personal-guide`, нажимает «Install» |
| **Выход** | Webhook-запись на сервере платформы: `(repo=personal-guide, pilot_id, events=[push])` + `webhook_registered=true` в `identity_map` (Neon) |

**Подтверждение в двух местах:**
- Сервер платформы — webhook активен
- Neon — `identity_map.webhook_registered=true`

**Примечание для реализации:** установка App требует ручного действия пилота (OAuth-согласие GitHub). Автоматизировать через GitHub Apps API нельзя — пилот должен явно выбрать репо. Минимизация трения: предзаполнить URL репо, показать скриншот-инструкцию.

## Этап 4. Работа с руководством

| | |
|---|---|
| **Вход** | Репо с 8 файлами (выход Этапа 2) + активный webhook (выход Этапа 3) |
| **Действие** | Пилот работает через один из трёх интерфейсов. Агент в роли **Навигатора (R27)** читает `profile.md`, `methods.md` → ведёт урок. Агент в роли **Фиксатора** пишет `daily/` и `history/` → commit + push → webhook → reindex в `Память.Derived` (≤30 сек) |
| **Выход** | Заполненные файлы в репо: `daily/YYYY-MM-DD.md`, `weekly/YYYY-Www.md`, `history/*.md` + обновлённые агрегаты в `Память.Derived` (Neon) |

**Интерфейсы:**

| Интерфейс | Читает | Пишет |
|---|---|---|
| Бот (Telegram) | `profile.md`, `methods.md`, `weekly/` | `daily/`, `history/` |
| Браузер (claude.ai/code) | Все файлы | Все файлы |
| VS Code | Все файлы | Все файлы (прямое редактирование + скиллы) |

**Цикл обновления руководства:** при изменении RCS (пересчёт ступени) → Навигатор вызывает `/personal-guide-render` без `first-run` → архивирует прежние `weekly/daily` в `history/` → перезаписывает 6 основных файлов.

## Сводная схема

```
Этап 0                    Этап 1              Этап 2                 Этап 3              Этап 4
Предусловия       →       Репо         →      Контент         →      Webhook      →      Работа
5 записей Neon            Пустой репо          8 файлов в репо        Webhook активен     daily/weekly/history
                          + URL в Neon         + commit                + флаг в Neon       + агрегаты в Neon
```
