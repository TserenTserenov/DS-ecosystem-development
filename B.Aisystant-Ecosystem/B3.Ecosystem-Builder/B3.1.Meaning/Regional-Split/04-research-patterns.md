# Исследование паттернов: мультиюрисдикционные платформы

> **Цель:** собрать опыт компаний, решавших аналогичные задачи трёхуровневого разделения (экономика, инфраструктура, сообщество), чтобы не изобретать велосипед.
>
> **Методология:** анализ публичных источников (SEC filings, пресс-релизы, техническая документация, корпоративные реестры).

<summary><b>1. Telegram: оффшорная платформа с единым глобальным сервисом</b></summary>

### Корпоративная структура

| Юрлицо | Юрисдикция | Роль |
|--------|-----------|------|
| Telegram FZ-LLC | ОАЭ (DIFC, Дубай) | Основная операционная компания |
| Telegram Group Inc | Великобритания (ранее) | Холдинг (исторический) |
| TON Foundation | Швейцария (Цуг) | Блокчейн TON, юридически отделён от Telegram (после SEC settlement 2020) |

**Нет российского юрлица.** Дуров вывел компанию из РФ в 2014 году.

### Три уровня

| Уровень | Как решено |
|---------|-----------|
| **Экономика** | Юрлицо в ОАЭ. Premium и Stars -- через App Store/Google Play (Apple/Google как платёжные агенты). Реклама -- через TON/крипто. Нет привязки к конкретной юрисдикции |
| **Инфраструктура** | Единый глобальный сервис. 5 DC-кластеров (США, Европа, Сингапур, Ближний Восток). Шардирование по регионам, но один продукт |
| **Сообщество** | Полностью единое. Нет «российской версии». Каналы, группы -- глобальные |

### Блокировка 2018-2020

- Роскомнадзор заблокировал IP-адреса (апрель 2018)
- Telegram использовал domain fronting и облачные провайдеры для обхода
- Роскомнадзор заблокировал диапазоны AWS/GCP → collateral damage для других сервисов
- Июнь 2020: Роскомнадзор снял блокировку (фактически признал провал)
- **Урок:** блокировка глобального сервиса -- дорого и неэффективно для регулятора

### Данные и 152-ФЗ

Не локализуют данные в РФ. Штраф -- 4 млн руб (2021). Стратегия: «штраф дешевле, чем compliance».

### Применимость к экосистеме

| Паттерн | Подходит? | Почему |
|---------|----------|--------|
| Оффшорная юрисдикция (ОАЭ) | Частично | Нейтральная, но нет EU-adequacy для GDPR |
| Единый глобальный сервис | Да | Цель экосистемы -- единое сообщество |
| Игнорирование 152-ФЗ | Нет | Экосистема имеет российских клиентов с оплатой. Штрафы = репутационный риск |
| App Store как платёжный агент | Нет | Экосистема не мобильное приложение |

</details>

<summary><b>2. Miro: миграция юрисдикции</b></summary>

### Корпоративная структура

| Юрлицо | Юрисдикция | Роль |
|--------|-----------|------|
| Miro Inc | США (Сан-Франциско, Делавэр) | Головная компания |
| Miro BV | Нидерланды | Европейские операции |
| ООО «РеалТаймБорд» | Россия (Пермь) | R&D-центр (свёрнут после 2022) |

### История разделения

- 2011: основан в Перми как RealtimeBoard
- 2019: ребрендинг в Miro, HQ перенесён в Сан-Франциско
- 2020-2021: привлечение $400M+ инвестиций (US VCs)
- 2022: закрытие доступа для пользователей из РФ и Беларуси (compliance с санкциями). Релокация сотрудников из Перми в Амстердам, Берлин, Остин

### Три уровня

| Уровень | Как решено |
|---------|-----------|
| **Экономика** | US-головная + NL-дочка. Единая подписка в USD. После 2022 -- РФ-рынок отрезан |
| **Инфраструктура** | Единый SaaS (AWS). Data residency для enterprise: EU (Frankfurt) или US |
| **Сообщество** | Было единым. После 2022 -- российские пользователи потеряли доступ |

### Применимость к экосистеме

| Паттерн | Подходит? | Почему |
|---------|----------|--------|
| Перенос HQ для инвесторов | Да | WP-186 (инвесторы). Международное юрлицо = must |
| Полное отключение РФ | **Нет** | Экосистема принципиально за объединение, не разделение |
| R&D в РФ, продажи из-за рубежа | Частично | Если оставить разработку в РФ |
| NL-юрлицо для EU | Да | Нидерланды = надёжная юрисдикция |

**Главный урок:** Miro пожертвовал российским рынком ради инвесторов. Экосистема Aisystant хочет сохранить оба рынка. Это сложнее, но не невозможно.

### Три уровня

|---------|-----------|
| **Экономика** | Чешская головная + дочки по регионам. Лицензионная модель. Community editions -- бесплатно. После 2022: приостановка продаж в РФ, но fallback-лицензии работают |
| **Инфраструктура** | Десктопные IDE не требуют серверов. JetBrains Space/TeamCity Cloud -- SaaS в AWS/GCP. License Server -- свой |

### Применимость к экосистеме

| Паттерн | Подходит? | Почему |
|---------|----------|--------|
| Чехия как EU-юрисдикция | Частично | Альтернатива NL/EE |
| Open-source ядро + коммерция | Да | Community layer может быть open-source |
| Kotlin Foundation model | **Да** | Общий код под Foundation, коммерция отдельно |
| Fallback-лицензии | Нет | У экосистемы подписочная SaaS-модель |

| Юрлицо | Юрисдикция | Лицензия |
|--------|-----------|---------|
| Wise PLC | Великобритания (LSE) | FCA (E-Money Institution) |
| Wise Europe SA | Бельгия | ЕЦБ |

### Три уровня

| Уровень | Как решено |
|---------|-----------|

| Паттерн | Подходит? | Почему |
|---------|----------|--------|
| Юрлицо на юрисдикцию | Избыточно | Экосистема не финтех, не нужна лицензия в каждой стране |
| Единый UX при разных backend'ах | **Да** | Один бот, разные платёжные интеграции |
| Прозрачная маршрутизация | **Да** | Пользователь не должен думать о контуре |

<summary><b>5. Foundation model: Linux Foundation, Apache, CNCF</b></summary>

### Структура

```
┌─────────────────────────────┐
│  Foundation (некоммерческий)  │
│  • Governance                │
│  • Trademark                 │
│  • Community infrastructure  │
│  • Нейтральная территория    │
└──────────────┬──────────────┘
               │
     ┌─────────┼─────────┐
     │         │         │
┌────▼───┐ ┌──▼───┐ ┌───▼────┐
│Red Hat │ │Google│ │Huawei  │
│(USA)   │ │(USA) │ │(China) │
│RHEL    │ │GKE   │ │Cloud   │
└────────┘ └──────┘ └────────┘
```

### Ключевые принципы

1. **Фонд = стюард, не владелец.** Код под открытой лицензией. Фонд владеет trademark, governance-процессом
2. **Нейтральная территория.** Ни одна компания не контролирует проект. Contributors из любой страны равны
3. **Коммерция отделена.** Red Hat продаёт support, Google продаёт managed service. Фонд не конкурирует
4. **Governance = merit-based.** Право голоса пропорционально вкладу, не размеру взноса

### Формы фондов

| Форма | Юрисдикция | Примеры | Для экосистемы |
|-------|-----------|---------|---------|
| 501(c)(3) | США | Apache | Нет (US-centric) |
| 501(c)(6) | США | Linux Foundation | Нет |
| Stichting | Нидерланды | Blender Foundation, Mozilla | **Да** (EU, простая форма) |
| Verein | Швейцария | TON Foundation | Частично |
| AISBL | Бельгия | Eclipse Foundation | Частично |

### Применимость к экосистеме

| Паттерн | Подходит? | Когда |
|---------|----------|-------|
| Community = Foundation | **Да** (целевой) | При масштабе 1000+ участников |
| Код под открытой лицензией | Частично | Возможно для Community Layer |
| Merit-based governance | **Да** | Баллы экосистемы = аналог contribution metrics |
| Trademark в Foundation | **Да** | Защищает бренд от юрисдикционных рисков |

</details>

<summary><b>7. Юридические и технические паттерны</b></summary>

### 7.1. Standard Contractual Clauses (SCC)

Стандартные договорные условия ЕС для передачи ПД в третьи страны. Обязательны после решения Schrems II (2020). Используют все SaaS, работающие с EU-данными из-за пределов ЕС.

**Для экосистемы:** SCC нужны, если:
- Данные EU-пользователей обрабатываются в РФ (или наоборот)
- Community Layer в EU обрабатывает данные РФ-пользователей

**Митигация через псевдонимизацию:** если в Community Layer нет PII (только ory_id), SCC может не требоваться для community data.

### 7.2. Псевдонимизация

Замена идентификаторов на токены. GDPR (Recital 26): псевдонимизированные данные = менее рискованные. 152-ФЗ: спорнее, но Роскомнадзор штрафует за хранение ПД (ФИО, email), а ory_id без контекста -- не ПД.

**Для экосистемы:** ключевой технический ход. Community Layer хранит ory_id → баллы/прогресс. Ory хранит ory_id → ФИО/email. Разделение = юридическая защита.

### 7.3. Федерация (Matrix, ActivityPub)

| Протокол | Как работает | Проблема |
|---------|-------------|---------|
| **Matrix** | Каждый homeserver в своей юрисдикции. Федерация через протокол | Данные реплицируются на все серверы в «комнате» — расплывание по юрисдикциям |
| **ActivityPub** | Инстанс = юрисдикция. Админ инстанса = compliance officer | Нет гарантии deletion (Art. 17 GDPR) на федерированных инстансах |

**Для экосистемы:** федерация -- **не подходит** для community data. Мы хотим один source-of-truth (Community Layer), а не распределённые копии. Федерация подходит для identity (два Ory), но не для баллов.

### 7.4. Data Mesh

| Принцип | Как в экосистеме |
|---------|-----------|
| Домены владеют данными | Контурные домены (CRM-RU, Ory-RU) владеют PII. Community Layer владеет баллами |
| Data-as-a-product | API Community Layer = продукт для контурных сервисов |
| Federated governance | Правила доступа едины (CLAUDE.md, consent policies) |
| Self-serve infrastructure | Контуры могут добавляться без изменения Community Layer |

</details>
