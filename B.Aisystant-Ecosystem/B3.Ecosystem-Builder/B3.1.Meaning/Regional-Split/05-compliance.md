# Выполнение законов о персональных данных

> **Концепция:** [00-concept.md](00-concept.md) · **Персональные данные (концепция учёта):** WP-214

---

## 1. Принцип: PII на границе, не в центре

Вся персональная информация хранится в **контурных Ory** (каждый в своей юрисдикции). Community Layer (баллы, ЦД, прогресс, квалификации, клуб, команды) содержит только `ory_id` (UUID) + публичный ник. Community Layer **физически не содержит** персональных данных.

```
КОНТУР РФ                    COMMUNITY LAYER               КОНТУР МИР
┌────────────────┐     ┌──────────────────────┐     ┌────────────────┐
│ ory_id → ФИО   │     │ ory_id → баллы       │     │ ory_id → name  │
│ ory_id → email │     │ ory_id → прогресс    │     │ ory_id → email │
│ ory_id → тел.  │     │ ory_id → квалификация│     │ ory_id → addr  │
│ (PII, 152-ФЗ)  │     │ (псевдоним, ОК)      │     │ (PII, GDPR)    │
└────────────────┘     └──────────────────────┘     └────────────────┘
```

## 2. 152-ФЗ (Россия)

### Что требует

Федеральный закон 152-ФЗ «О персональных данных» (ст. 18 ч. 5): при сборе персональных данных граждан РФ оператор обязан обеспечить запись, систематизацию, накопление, хранение ПД с использованием баз данных, находящихся на территории РФ.

### Что является персональными данными

ФИО, email, номер телефона, адрес, паспортные данные, IP-адрес (в контексте идентификации).

### Что НЕ является персональными данными

UUID (ory_id), публичный ник (выбранный пользователем), баллы, прогресс обучения, квалификации — это **псевдонимизированные данные**. Без доступа к Ory невозможно связать их с конкретным человеком.

### Как выполняем

1. **ПД граждан РФ** (ФИО, email, телефон) хранятся в **Ory-RU** на сервере в РФ (Selectel / VPS в РФ)
2. **Community Layer** (Neon EU) хранит UUID + публичный ник — это не ПД по 152-ФЗ
3. **CRM-RU** (платёжные данные для ЮKassa) — на сервере в РФ
4. **Клуб** (Discourse) получает от Ory-RU только JWT с ory_id + ник. ФИО, email, телефон в клуб **не передаются**

### Штрафы

- Первичное нарушение: до 6 млн руб (ст. 13.11 КоАП)
- Повторное: до 18 млн руб
- Роскомнадзор может заблокировать сайт

## 3. GDPR (ЕС)

### Что требует

General Data Protection Regulation (EU 2016/679): legal basis для обработки, право на доступ/удаление/переносимость, псевдонимизация как рекомендуемая мера (Art. 25, 32).

### Псевдонимизация по GDPR

Art. 4(5): «обработка персональных данных таким образом, что они не могут быть более отнесены к конкретному субъекту данных без использования дополнительной информации».

Community Layer содержит только `ory_id`. Дополнительная информация (ory_id → ФИО) хранится в Ory-EU **отдельно**. Это соответствует определению псевдонимизации по GDPR.

### Как выполняем

1. **Legal basis:** consent в оферте платформы — пункт: «участие в программе сообщества, включая псевдонимизированный учёт вклада, peer review, отображение публичного профиля»
2. **Право на доступ (Art. 15):** пользователь запрашивает экспорт через бот → данные из Ory-EU + Community Layer по ory_id
3. **Право на удаление (Art. 17):**
   - Ory-EU: удаление ПД (ФИО, email)
   - Community Layer: анонимизация ory_id → `deleted_<hash>`. Вклад (рецензии, посты) остаётся анонимизированным (аналог GitHub/SO)
4. **Право на переносимость (Art. 20):** экспорт данных в машиночитаемом формате
5. **Контроллер vs Процессор:** юрлицо (Inc Delaware или EU-дочка) = контроллер. Neon, Railway = процессоры → DPA (Data Processing Agreement)

### Штрафы

- До 20 млн EUR или 4% мирового оборота (что больше)

## 4. UGC (пользовательский контент)

Когда пользователь сам публикует текст в клубе, рецензию, комментарий — это его выбор, а не наша обработка ПД. Legal basis: consent (пользователь нажал «Опубликовать»).

**Наши обязанности:**
- Модерация: удалять чужие ПД из публикаций (если кто-то опубликовал телефон другого)
- Notice: в правилах клуба — «не публикуйте чужие персональные данные»
- Реакция: удаление по обращению субъекта данных

---

## 5. Consent в оферте

В оферте (публичный договор-оферта на сайте) должен быть пункт:

> «Принимая оферту, Пользователь соглашается на участие в программе сообщества Aisystant, включая:
> - псевдонимизированный учёт вклада (баллы, квалификации)
> - peer review и менторство с другими участниками
> - отображение публичного профиля (ник, квалификация, баллы) в клубе и на платформе
>
> Персональные данные (ФИО, email, телефон) хранятся в юрисдикции Пользователя и не передаются в Community Layer.»

Это обеспечивает legal basis и по 152-ФЗ, и по GDPR.

## 6. Сценарии обработки запросов

### 6.1. Пользователь из РФ просит удалить данные

1. Ory-RU: удаление ПД (ФИО, email, тел.)
2. CRM-RU: анонимизация платёжных записей (хранить для бухгалтерии, но без привязки к ФИО)
3. Community Layer: ory_id → `deleted_<hash>`. Баллы и вклад — анонимизированы, но сохранены (аналог Wikipedia)
4. Клуб: посты остаются от имени `[удалённый участник]`

### 6.2. Пользователь переезжает РФ → EU

1. Создаётся аккаунт в Ory-EU, привязывается к тому же canonical_ory_id
2. ПД переносятся из Ory-RU в Ory-EU
3. Ory-RU деактивируется (ПД удаляются из РФ-контура)
4. Community Layer: ничего не меняется (работает по ory_id)

### 6.3. Роскомнадзор запрашивает проверку

Демонстрируем: ПД граждан РФ хранятся в Ory-RU на сервере в РФ. Community Layer (EU) содержит только UUID — не является ПД.

## 8. Открытые вопросы (по итогам обсуждения 3 апр)

> Источник: обсуждение с Олегом Царёвым, ИТ-встреча 5.

### 8.1. Railway и РФ

Railway не имеет серверов в РФ. РФ-контур (Ory-RU, CRM-RU, PG-RU) **не может** работать на Railway — нужен российский хостинг (Selectel, Яндекс Облако, VPS в РФ). Вариант: generic k8s stack (Talos Linux) на российском ДЦ.

**Статус:** решение не принято. Зависит от WP-73 (обсуждение с Андреем 13 апр).

### 8.2. Email серверы в РФ

152-ФЗ может потребовать, чтобы email-рассылки для РФ-пользователей отправлялись с серверов в РФ. Текущие провайдеры (Mailgun, SendGrid) не имеют серверов в РФ.

**Статус:** требует юридической оценки (WP-186).

### 8.3. LLM gateway и санкционные ограничения

Олег указал на риск: если платформа предоставляет РФ-пользователям доступ к Claude/GPT через gateway — это нарушает ToS провайдеров (Anthropic, OpenAI запрещают использование из РФ).

**Принятое решение:** LLM gateway для РФ-пользователей = **только BYOK** (Bring Your Own Key). Пользователь сам регистрирует аккаунт у доступного ему LLM-провайдера и подключает свой ключ. Экосистема = маршрутизатор (прокси), не провайдер LLM-доступа. Биллинг за LLM для РФ-пользователей **запрещён**.

**Риск для инвесторов:** при due diligence аудиторы могут не разбираться в vendor-agnostic архитектуре. Наличие РФ-пользователей в платформе = красный флаг. Важно: документация и презентации должны чётко фиксировать BYOK-модель.

### 8.4. Due diligence и два бренда

Совет Олега: для инвестиционной привлекательности — полное разделение:
- US/EU компания: разработка, документация, инвесторы. Только US/EU юзеры
- РФ: отдельный бренд, отдельное юрлицо, тот же код по лицензии (open-source ядро или лицензионный договор). Никаких финансовых/юридических связей

**Статус:** обсуждается. Может повлиять на модель юрлиц (WP-186) и Этап 1 эволюции.

### 8.5. Смена региона пользователем

Олег: Apple/Google при смене региона создают **новую учётку**, новый SLA, новый billing. Данные экспортируются из старой и импортируются в новую.

**Текущая модель (00-concept §6.2):** canonical_ory_id сохраняется, Ory-RU → Ory-EU. Это технически проще, но может не соответствовать юридическим требованиям (новый SLA = новый договор = новая учётка).

**Статус:** требует юридической оценки. Если нужна новая учётка — canonical_ory_id всё равно связывает старый и новый ory_id для сохранения баллов и прогресса в Community Layer.
