# Описание системы приёма оплаты

Платёжная подсистема экосистемы для приёма фиатных платежей за образовательные услуги, подписки и токены. Обеспечивает безопасную обработку платежей через множество провайдеров, поддерживает разные валюты и методы оплаты, автоматически синхронизируется с биллингом и токеномикой. Реализует принципы FPF: эволюционность через оптимизацию конверсии платежей, сквозную целостность от намерения оплатить до зачисления средств, дидактическую прозрачность каждой транзакции.

## Назначение

Система приема оплаты фиатом — это **платёжный шлюз экосистемы**, который превращает намерение оплатить в реальные финансовые поступления с минимальными трениями для плательщика.

### Ключевые функции

1. **Прием платежей через множество каналов**
   - Банковские карты (Visa, Mastercard, МИР)
   - Электронные кошельки (PayPal, Apple Pay, Google Pay)
   - Банковские переводы (SWIFT, SEPA, СБП)
   - Криптовалюты как фиат (через конвертацию)
   - Мобильные платежи (carrier billing)

2. **Поддержка множества валют**
   - Мультивалютность (USD, EUR, RUB и др.)
   - Автоматическая конвертация по актуальным курсам
   - Расчёты в валюте плательщика
   - Выплаты в валюте получателя

3. **Управление подписками**
   - Recurring payments (автоматические списания)
   - Trial periods (пробные периоды)
   - Upgrade/downgrade планов
   - Отмена и возвраты

4. **Безопасность и compliance**
   - PCI DSS compliance (хранение карточных данных)
   - Fraud detection (обнаружение мошенничества)
   - 3D Secure для карточных платежей
   - KYC/AML для крупных транзакций
   - GDPR compliance для персональных данных

5. **Финансовая отчётность**
   - Учёт всех транзакций
   - Формирование финансовых документов (чеки, инвойсы)
   - Reconciliation (сверка с провайдерами)
   - Налоговая отчётность

### Роль в архитектуре

Платёжная система — точка входа денег в экосистему:
- Получает запросы на оплату от **Billing** (подписки, курсы)
- Интегрируется с множеством **Payment Providers** (Stripe, PayPal, ЮKassa)
- Уведомляет **CRM** о платежах для triggering onboarding
- Передаёт данные в **Tokenomics** для конвертации в токены
- Логирует транзакции в **Activity Hub**

## Архитектура

### Компоненты

#### 1. Payment Gateway (Платёжный шлюз)

**Назначение:** Центральная точка обработки всех платежей.

**Функции:**
- **Payment Request Handling** — приём запросов на оплату
- **Provider Selection** — выбор подходящего провайдера (на основе метода, валюты, геолокации)
- **Payment Processing** — инициация платежа через провайдера
- **Callback Handling** — обработка callbacks от провайдеров (success, failure, pending)
- **Retry Logic** — повторные попытки при временных ошибках

**Поддерживаемые провайдеры:**
- **Stripe** — международные платежи (карты, wallets)
- **PayPal** — международные платежи (PayPal accounts)
- **ЮKassa** — российские платежи (карты, СБП, ЮMoney)
- **CloudPayments** — российские и международные карты
- **Robokassa** — российские платежи (множество методов)
- **Cryptocurrency gateways** — криптовалюты с конвертацией в фиат (CoinGate, BitPay)

**Payment flow:**
```
1. User clicks "Pay" → Create Payment Intent
2. Gateway selects Provider (based on rules)
3. Redirect user to Provider (hosted payment page or embedded form)
4. User completes payment
5. Provider sends Callback → Gateway validates
6. Gateway confirms Payment → Update Billing
7. Send receipt to User
```

**Интеграции:**
- Billing (4.9) — инициация платежей
- ORY (4.14) — identity для привязки платежей
- Activity Hub (4.6) — логирование транзакций

#### 2. Subscription Manager (Менеджер подписок)

**Назначение:** Управление recurring payments (автоплатежи).

**Функции:**
- **Subscription Creation** — создание подписки при первой оплате
- **Recurring Billing** — автоматические списания по расписанию
- **Failed Payment Handling** — обработка неудачных списаний (retry logic)
- **Subscription Lifecycle** — active, paused, cancelled, expired
- **Upgrade/Downgrade** — изменение планов с prorated billing

**Lifecycle подписки:**
```
Created → Trial (опционально) → Active → (Paused) → Cancelled/Expired
```

**Retry logic для failed payments:**
```
Day 0: Payment fails → Retry immediately
Day 1: Retry again
Day 3: Retry + send email notification
Day 7: Last retry + send warning email
Day 10: Suspend subscription
Day 30: Cancel subscription
```

**Proration (при upgrade/downgrade):**
- При upgrade: immediate charge за разницу (prorated)
- При downgrade: credit на следующий период

**Интеграции:**
- Billing (4.9) — синхронизация статусов подписок
- CRM (4.7) — уведомления о проблемах с оплатой
- Digital Twin (4.11) — обновление access прав

#### 3. Fraud Detection (Обнаружение мошенничества)

**Назначение:** Предотвращение мошеннических транзакций.

**Функции:**
- **Risk Scoring** — оценка риска транзакции (0-100)
- **Rule-based checks** — проверка по правилам
- **ML-based detection** — машинное обучение для anomaly detection
- **3D Secure enforcement** — требование 3DS для рискованных транзакций
- **Blacklist/Whitelist** — списки заблокированных/доверенных

**Risk factors:**
- Необычная геолокация (не соответствует профилю)
- Множество неудачных попыток оплаты
- Несоответствие IP и billing address
- Использование VPN/proxy
- Транзакция из blacklisted страны
- Карта из другой страны (не соответствует IP)

**Actions на основе risk score:**
- **Low risk (0-30):** Allow automatically
- **Medium risk (31-70):** Require 3D Secure
- **High risk (71-100):** Block or require manual review

**Интеграции:**
- Payment Providers — получение fraud signals
- External fraud services (Sift, Forter)
- Activity Hub (4.6) — логирование fraud events

#### 4. Currency Converter (Конвертер валют)

**Назначение:** Конвертация между валютами по актуальным курсам.

**Функции:**
- **Real-time rates** — получение актуальных курсов (от провайдеров или FX APIs)
- **Conversion calculation** — расчёт сумм с учётом комиссий
- **Multi-currency display** — показ цен в валюте пользователя
- **Settlement currency** — конвертация в валюту получателя

**Источники курсов:**
- Payment providers (Stripe, PayPal rates)
- External FX APIs (exchangerate-api.com, fixer.io)
- Oracle services (Chainlink для crypto rates)

**Формула конвертации:**
```
Amount_target = Amount_source * Exchange_Rate * (1 - Fee_percentage)
```

**Интеграции:**
- Billing (4.9) — для расчёта цен
- Tokenomics (4.10) — конвертация фиат → токены

#### 5. Refund Manager (Менеджер возвратов)

**Назначение:** Обработка возвратов и chargeback.

**Функции:**
- **Refund Processing** — инициация возврата через провайдера
- **Partial Refunds** — частичные возвраты
- **Chargeback Handling** — обработка disputes от банков
- **Refund Accounting** — учёт возвратов для финансовой отчётности

**Refund policies:**
- **Full refund:** в течение 14 дней (для курсов с <10% прохождения)
- **Partial refund:** prorated на основе использования
- **No refund:** для завершённых курсов (>90% прохождения)

**Chargeback flow:**
```
1. Bank initiates chargeback
2. Provider notifies Gateway
3. Gateway notifies Admin + freeze user account (if fraud suspected)
4. Admin reviews and responds (provide evidence)
5. Bank decision (resolved or lost)
6. If lost: deduct from balance
```

**Интеграции:**
- Billing (4.9) — обновление балансов
- CRM (4.7) — уведомления support team
- Activity Hub (4.6) — логирование refunds

#### 6. Receipt Generator (Генератор чеков)

**Назначение:** Формирование финансовых документов для плательщиков.

**Функции:**
- **Receipt Generation** — генерация чеков после успешной оплаты
- **Invoice Generation** — генерация инвойсов для b2b клиентов
- **Fiscal compliance** — соответствие фискальным требованиям (54-ФЗ для РФ)
- **Email delivery** — отправка документов на email

**Форматы документов:**
- **Receipt (чек):** простой receipt для b2c (PDF)
- **Invoice:** детальный инвойс для b2b (PDF)
- **Fiscal receipt:** чек с QR-кодом (для РФ) через ОФД

**Контент чека:**
- Merchant info (название, ИНН, адрес)
- Payment details (сумма, валюта, дата, метод)
- Order details (название продукта/услуги)
- Tax breakdown (если применимо)
- Transaction ID, receipt number

**Интеграции:**
- Fiscal services (ОФД провайдеры: АТОЛ, OrangeData для РФ)
- Email service для отправки
- Storage (S3) для архивирования

## Процессы

### П1. Одноразовая оплата курса (One-time purchase)

**Назначение:** Оплата образовательного продукта (курс, программа).

**Этапы:**
1. **Initiate Payment** — пользователь нажимает "Купить курс"
2. **Create Payment Intent** — Gateway создаёт intent с amount, currency, metadata
3. **Select Provider** — выбор провайдера на основе правил
4. **Redirect to Provider** — redirect на hosted payment page провайдера
5. **User Pays** — пользователь вводит данные карты и подтверждает
6. **Provider Callback** — провайдер отправляет webhook о статусе
7. **Validate Callback** — Gateway проверяет signature и статус
8. **Confirm Payment** — обновление статуса в Billing → выдача доступа к курсу
9. **Send Receipt** — отправка чека на email
10. **Log Event** — публикация события в Activity Hub

**Retry при ошибках:**
- Если payment fails (insufficient funds, card declined) → показать ошибку пользователю
- Если webhook не пришёл (timeout) → polling status у провайдера

**Интеграции:**
- Billing (4.9) — создание order, подтверждение оплаты
- LMS (4.4) — выдача доступа к курсу
- CRM (4.7) — trigger onboarding sequence
- Activity Hub (4.6) — `payment.completed` event

### П2. Создание подписки (Subscription creation)

**Назначение:** Оформление recurring подписки на платформу.

**Этапы:**
1. **Select Plan** — пользователь выбирает тариф (monthly, annual)
2. **Trial Setup** — если есть trial период, установка trial_end_date
3. **First Payment** — одноразовый платёж для активации (или $0 для trial)
4. **Create Subscription** — создание subscription entity в Gateway + Provider
5. **Schedule Recurring** — провайдер настраивает автоплатежи
6. **Activate Access** — выдача доступа к функциям подписки
7. **Send Confirmation** — email с подтверждением и деталями подписки

**Trial handling:**
- Trial period: 7-30 дней (no charge)
- Перед окончанием trial: email reminder (за 3 дня)
- После trial: автоматическое списание или требование добавить карту

**Интеграции:**
- Billing (4.9) — создание подписки
- CRM (4.7) — onboarding для подписчиков
- Digital Twin (4.11) — обновление subscription tier

### П3. Recurring billing (автоплатёж)

**Назначение:** Автоматическое списание за подписку.

**Этапы:**
1. **Trigger Billing** — наступление billing date (monthly/annual anniversary)
2. **Charge Attempt** — провайдер пытается списать с сохранённой карты
3. **Success Path:**
   - Payment succeeds → update subscription (next_billing_date)
   - Send receipt
   - Log event
4. **Failure Path:**
   - Payment fails → trigger retry logic (см. выше)
   - Send email notification (проблема с оплатой)
   - Если не удалось после всех retry → suspend subscription

**Dunning management:**
- Автоматические retry (smart retry logic по дням недели, времени)
- Персонализированные emails (не агрессивные, helpful tone)
- Предложение альтернативных методов оплаты
- Pause subscription вместо immediate cancel

**Интеграции:**
- Billing (4.9) — обновление статуса подписки
- CRM (4.7) — retention campaigns при failed payments
- Activity Hub (4.6) — события billing cycle

### П4. Возврат средств (Refund)

**Назначение:** Обработка запроса на возврат.

**Этапы:**
1. **Refund Request** — пользователь или support инициирует возврат
2. **Eligibility Check** — проверка refund policy (сроки, использование)
3. **Approval** — автоматическое (если в policy) или manual approval
4. **Process Refund** — инициация refund через провайдера
5. **Provider Confirmation** — провайдер подтверждает (может занять несколько дней)
6. **Update Records** — обновление баланса и доступов
7. **Notify User** — email с подтверждением возврата

**Proration при refund подписки:**
- Если отмена в середине периода: prorated refund (неиспользованное время)
- Если использовано >80% периода: no refund (по policy)

**Интеграции:**
- Billing (4.9) — обновление балансов
- LMS (4.4) / Digital Twin (4.11) — отзыв доступов
- CRM (4.7) — сбор feedback (почему возврат?)

### П5. Fraud review process

**Назначение:** Ручная проверка подозрительных транзакций.

**Этапы:**
1. **Fraud Alert** — система детектирует high-risk transaction
2. **Hold Payment** — транзакция помещена на hold (не завершена)
3. **Manual Review** — fraud analyst проверяет (user profile, payment history, geolocation)
4. **Decision:**
   - **Approve:** complete payment, whitelist user (если ложное срабатывание)
   - **Decline:** refund, blacklist card/IP, notify user
5. **Log Decision** — документирование решения для audit trail

**Review criteria:**
- User history (новый vs returning)
- Order value (крупные транзакции = выше risk)
- Geolocation consistency
- Device fingerprinting

## API и контракты

### Основные эндпоинты

**Payment Intents:**
```
POST /payments/intents                  # Создать payment intent
GET  /payments/intents/:id              # Получить статус intent
POST /payments/intents/:id/confirm      # Подтвердить payment
POST /payments/intents/:id/cancel       # Отменить payment
```

**Subscriptions:**
```
POST /payments/subscriptions            # Создать подписку
GET  /payments/subscriptions/:id        # Получить подписку
PUT  /payments/subscriptions/:id        # Обновить подписку (upgrade/downgrade)
POST /payments/subscriptions/:id/cancel # Отменить подписку
GET  /payments/subscriptions/:id/invoices # История инвойсов
```

**Payment Methods:**
```
POST /payments/methods                  # Добавить payment method (card)
GET  /payments/methods                  # Список payment methods
DELETE /payments/methods/:id            # Удалить payment method
PUT  /payments/methods/:id/default      # Установить default
```

**Refunds:**
```
POST /payments/refunds                  # Создать возврат
GET  /payments/refunds/:id              # Статус возврата
GET  /payments/refunds                  # Список возвратов
```

**Webhooks:**
```
POST /payments/webhooks/:provider       # Webhook endpoint для провайдеров (Stripe, PayPal, etc)
```

**Reports:**
```
GET  /payments/transactions             # Список транзакций (с фильтрами)
GET  /payments/reports/reconciliation   # Reconciliation report
GET  /payments/reports/revenue          # Revenue report
```

### События

**Payment Events:**
- `payment.initiated` — платёж инициирован
- `payment.processing` — платёж обрабатывается
- `payment.succeeded` — платёж успешен
- `payment.failed` — платёж не прошёл
- `payment.cancelled` — платёж отменён

**Subscription Events:**
- `subscription.created` — подписка создана
- `subscription.activated` — подписка активирована (после first payment)
- `subscription.renewed` — подписка продлена (recurring payment)
- `subscription.payment_failed` — неудачная попытка оплаты
- `subscription.cancelled` — подписка отменена
- `subscription.expired` — подписка истекла

**Refund Events:**
- `refund.requested` — запрос на возврат
- `refund.approved` — возврат одобрен
- `refund.completed` — возврат завершён
- `chargeback.received` — получен chargeback от банка

**Fraud Events:**
- `fraud.detected` — обнаружена подозрительная транзакция
- `fraud.review_required` — требуется manual review
- `fraud.blocked` — транзакция заблокирована

## Хранилище данных

### Схема данных

**Таблицы (PostgreSQL):**

```sql
-- Платежи
CREATE TABLE Payment (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  amount DECIMAL NOT NULL,
  currency VARCHAR(3) NOT NULL,
  status VARCHAR,  -- initiated, processing, succeeded, failed, cancelled, refunded
  payment_method VARCHAR,  -- card, paypal, bank_transfer, etc
  provider VARCHAR,  -- stripe, paypal, yookassa, etc
  provider_payment_id VARCHAR,  -- ID в системе провайдера
  metadata JSONB,  -- order info, user details
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  completed_at TIMESTAMP
);

-- Подписки
CREATE TABLE Subscription (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  plan_id UUID NOT NULL,
  status VARCHAR,  -- trialing, active, paused, cancelled, expired
  payment_method_id UUID,
  current_period_start TIMESTAMP,
  current_period_end TIMESTAMP,
  trial_end TIMESTAMP,
  cancel_at_period_end BOOLEAN DEFAULT false,
  provider VARCHAR,
  provider_subscription_id VARCHAR,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  cancelled_at TIMESTAMP
);

-- Payment Methods (токенизированные карты)
CREATE TABLE PaymentMethod (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  type VARCHAR,  -- card, paypal_account, bank_account
  provider VARCHAR,
  provider_method_id VARCHAR,  -- токен карты у провайдера
  last4 VARCHAR,  -- последние 4 цифры карты
  brand VARCHAR,  -- visa, mastercard, etc
  exp_month INT,
  exp_year INT,
  is_default BOOLEAN DEFAULT false,
  created_at TIMESTAMP
);

-- Возвраты
CREATE TABLE Refund (
  id UUID PRIMARY KEY,
  payment_id UUID REFERENCES Payment(id),
  amount DECIMAL NOT NULL,
  reason VARCHAR,
  status VARCHAR,  -- requested, approved, processing, completed, failed
  provider_refund_id VARCHAR,
  created_at TIMESTAMP,
  completed_at TIMESTAMP
);

-- Invoices (для подписок)
CREATE TABLE Invoice (
  id UUID PRIMARY KEY,
  subscription_id UUID REFERENCES Subscription(id),
  user_id UUID NOT NULL,
  amount DECIMAL NOT NULL,
  currency VARCHAR(3),
  status VARCHAR,  -- draft, open, paid, void, uncollectible
  period_start TIMESTAMP,
  period_end TIMESTAMP,
  due_date TIMESTAMP,
  paid_at TIMESTAMP,
  invoice_pdf_url VARCHAR,
  created_at TIMESTAMP
);

-- Чеки/Receipts
CREATE TABLE Receipt (
  id UUID PRIMARY KEY,
  payment_id UUID REFERENCES Payment(id),
  user_id UUID NOT NULL,
  receipt_number VARCHAR UNIQUE,
  receipt_url VARCHAR,  -- PDF в S3
  fiscal_receipt_url VARCHAR,  -- фискальный чек (для РФ)
  created_at TIMESTAMP
);

-- Fraud Reviews
CREATE TABLE FraudReview (
  id UUID PRIMARY KEY,
  payment_id UUID REFERENCES Payment(id),
  risk_score INT,  -- 0-100
  risk_factors JSONB,
  status VARCHAR,  -- pending, approved, declined
  reviewed_by UUID,
  reviewed_at TIMESTAMP,
  decision_reason TEXT,
  created_at TIMESTAMP
);
```

### Retention policy

- **Payments:** 7 лет (налоговые требования)
- **Subscriptions:** Бессрочно (пока активны), 7 лет после cancellation
- **Refunds:** 7 лет
- **Invoices:** 7 лет
- **Receipts:** 7 лет
- **Fraud Reviews:** 3 года

### Алерты

**Критические:**
- Payment gateway unavailable
- Provider API down (Stripe, PayPal)
- Database connection lost
- Webhook processing stopped
- Failed payment spike (>10% in 1 hour)

**Предупреждения:**
- High payment failure rate (>5%)
- Unusual refund rate (>5%)
- Chargeback rate increasing (>1%)
- Fraud detection score spike
- Subscription renewal failures (>20%)
- Provider latency high (>10 sec)
